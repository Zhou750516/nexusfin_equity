## 告警解读速查表

### P0 级告警（紧急，需立即处理）

| 告警信息 | 含义 | 可能原因 | 立即检查 | 估计恢复时间 |
|---------|------|---------|---------|------------|
| **订单创建成功率 < 95%** | 核心业务不可用 | 1. ECS 资源耗尽 2. 数据库连接池满 3. 外部服务故障 | 1. CPU/内存 2. RDS 连接数 3. QW/Allinpay 可达性 | 5-30分钟 |
| **首扣代扣成功率 < 95%** | 支付系统故障 | 1. Allinpay 服务异常 2. 网络连接问题 3. 代码异常 | 1. Allinpay 状态页 2. 网络延迟 3. 应用日志 | 10-60分钟 |
| **QW 调用成功率 < 95%** | 云卡系统故障 | 1. QW API 不可用 2. 网络延迟 3. 请求异常 | 1. QW 状态页 2. 网络诊断 3. 日志查询 | 5-30分钟 |
| **RDS 连接数 > 80%** | 数据库连接即将耗尽 | 1. 连接泄漏 2. 长连接占用 3. 高并发 | 1. RDS 活跃连接 2. 慢查询 3. 应用日志 | 立即处理 |
| **ECS CPU > 85%** | 服务器过热，可能雪崩 | 1. 高流量 2. 应用内存泄漏 3. 数据库查询慢 | 1. CPU 使用率分布 2. 进程 CPU 占用 3. RDS CPU | 5-15分钟 |

### P1 级告警（重要，需要关注）

| 告警信息 | 含义 | 可能原因 | 初步检查 | 优先级 |
|---------|------|---------|---------|--------|
| **请求延迟 P99 > 2s** | 用户体验变差 | 1. 数据库慢查询 2. 外部服务延迟 3. GC 暂停 | 1. 慢查询日志 2. 外部服务延迟 3. JVM GC 日志 | 立即 |
| **错误率 > 0.5%** | 系统异常增加 | 1. 业务异常 2. 依赖故障 3. 代码bug | 1. 应用日志 2. 外部服务状态 3. 错误日志聚合 | 立即 |
| **云卡同步延迟 > 3s** | 下游系统响应变慢 | 1. QW API 性能问题 2. 网络延迟 3. 本地处理慢 | 1. QW 响应时间 2. 网络诊断 3. 应用处理耗时 | 30分钟内 |
| **对账失败** | 财务数据不一致 | 1. 订单未同步 2. 支付状态不一致 3. 数据错误 | 1. 对账日志 2. 差异订单详情 3. 支付状态日志 | 1小时内 |

### P2 级告警（监控，可备忘）

| 告警信息 | 含义 | 可能原因 | 建议处理 |
|---------|------|---------|---------|
| **RDS 慢查询 > 20/小时** | 数据库 SQL 性能问题 | 1. 索引缺失 2. 查询不优化 3. 数据量大 | 优化 SQL / 添加索引 |
| **JVM 堆内存 > 85%** | 内存紧张 | 1. 内存泄漏 2. 缓存过大 3. 高并发 | 优化代码 / 调整 JVM 参数 |
| **数据库连接 > 30** | 连接使用过多 | 1. 业务高峰 2. 连接泄漏 | 监控 / 代码审查 |

## 故障诊断决策树

### 一级：按告警来源分类

```
告警事件来源
├─ 基础设施告警（ECS/RDS）
│   └─ [决策树 1]
├─ 应用性能告警（CPU/内存/延迟）
│   └─ [决策树 2]
├─ 业务流程告警（订单/支付/云卡）
│   └─ [决策树 3]
└─ 外部服务告警
    └─ [决策树 4]
```

### 决策树 1：基础设施故障诊断

```
ECS/RDS 告警
    │
    ├─ ECS CPU > 85%?
    │   ├─ YES → 检查应用进程 CPU 占用
    │   │   ├─ 单个进程 CPU > 50%?
    │   │   │   ├─ YES → 可能代码死循环或高算力操作
    │   │   │   │   ├─ 步骤 A1.1：抓取 Java 线程栈
    │   │   │   │   ├─ 步骤 A1.2：分析耗时线程
    │   │   │   │   └─ 步骤 A1.3：优化代码或限流
    │   │   │   └─ NO → 多个进程共同消耗
    │   │   │       └─ 步骤 A1.4：检查是否有异常进程
    │   │   └─ 检查数据库 CPU > 75%?
    │   │       ├─ YES → 可能慢查询或全表扫描
    │   │       │   ├─ 步骤 A2.1：查看 RDS 慢查询日志
    │   │       │   ├─ 步骤 A2.2：分析慢 SQL
    │   │       │   └─ 步骤 A2.3：优化 SQL 或添加索引
    │   │       └─ NO → 网络异常或其他组件
    │   │           └─ 步骤 A2.4：检查网络流量
    │   │
    │   └─ NO → 正常范围，监控数据可能滞后
    │
    ├─ RDS 连接数 > 80%?
    │   ├─ YES → 检查是否有连接泄漏
    │   │   ├─ 步骤 B1.1：查看活跃连接明细
    │   │   ├─ 步骤 B1.2：识别长连接和空闲连接
    │   │   ├─ 是否有大量 SLEEP 状态连接?
    │   │   │   ├─ YES → 应用连接泄漏，需要重启应用
    │   │   │   │   ├─ 步骤 B1.3：灰度重启应用
    │   │   │   │   ├─ 步骤 B1.4：监控连接数恢复
    │   │   │   │   └─ 步骤 B1.5：检查代码中的连接管理
    │   │   │   └─ NO → 高并发或流量突增
    │   │   │       ├─ 步骤 B1.6：检查请求数是否高
    │   │   │       ├─ 步骤 B1.7：评估是否需要扩容
    │   │   │       └─ 步骤 B1.8：临时限流保护系统
    │   │   └─ 步骤 B1.9：长期优化连接池配置
    │   │
    │   └─ NO → 正常范围
    │
    ├─ RDS 慢查询 > 20/小时?
    │   ├─ YES → 检查 SQL 性能
    │   │   ├─ 步骤 C1.1：从 RDS 控制台查看慢查询日志
    │   │   ├─ 步骤 C1.2：分析是否为同一 SQL 重复执行
    │   │   ├─ 是否有 FULL TABLE SCAN?
    │   │   │   ├─ YES → 缺少索引或索引失效
    │   │   │   │   ├─ 步骤 C1.3：确认表和查询条件
    │   │   │   │   ├─ 步骤 C1.4：创建合适的索引
    │   │   │   │   └─ 步骤 C1.5：验证查询性能改进
    │   │   │   └─ NO → 查询复杂度高或数据量大
    │   │   │       ├─ 步骤 C1.6：分析 EXPLAIN 执行计划
    │   │   │       ├─ 步骤 C1.7：优化 SQL 逻辑（JOIN、子查询等）
    │   │   │       └─ 步骤 C1.8：考虑添加查询缓存或分页
    │   │   └─ 步骤 C1.9：建立 SQL 优化规范
    │   │
    │   └─ NO → 数据库性能良好
    │
    ├─ 磁盘空间 > 90%?
    │   ├─ YES → 紧急处理，可能导致服务异常
    │   │   ├─ ECS 磁盘满?
    │   │   │   ├─ YES
    │   │   │   │   ├─ 步骤 D1.1：检查日志文件大小
    │   │   │   │   ├─ 步骤 D1.2：清理历史日志
    │   │   │   │   ├─ 步骤 D1.3：查看其他大文件
    │   │   │   │   └─ 步骤 D1.4：规划磁盘扩容
    │   │   │   └─ NO → RDS 磁盘满
    │   │   │       ├─ 步骤 D1.5：联系 RDS 技术支持
    │   │   │       ├─ 步骤 D1.6：审视数据清理策略
    │   │   │       └─ 步骤 D1.7：规划表空间优化
    │   │   └─ 步骤 D1.8：定期监控磁盘趋势
    │   │
    │   └─ NO → 磁盘空间充足
    │
    └─ 网络丢包 > 0.1%?
        ├─ YES → 网络不稳定
        │   ├─ 步骤 E1.1：检查网络连接质量
        │   ├─ 步骤 E1.2：运行 ping 和 traceroute 诊断
        │   ├─ 步骤 E1.3：联系阿里云网络支持
        │   └─ 步骤 E1.4：检查是否有 DDoS 攻击迹象
        │
        └─ NO → 网络正常
```

### 决策树 2：应用性能故障诊断

```
应用性能告警（延迟/错误率/吞吐量）
    │
    ├─ 请求延迟 P99 > 2s?
    │   ├─ YES → 分析延迟来源
    │   │   ├─ 步骤 F1.1：查看请求延迟分布
    │   │   │   ├─ 是否所有请求都慢?
    │   │   │   │   ├─ YES → 系统整体问题
    │   │   │   │   │   ├─ 步骤 F1.2：检查 ECS/RDS CPU
    │   │   │   │   │   └─ 步骤 F1.3：检查网络延迟
    │   │   │   │   └─ NO → 部分请求慢
    │   │   │   │       ├─ 步骤 F1.4：识别慢端点
    │   │   │   │       └─ 步骤 F1.5：分析该端点的依赖
    │   │   │   │
    │   │   ├─ 步骤 F1.6：查看外部服务调用延迟
    │   │   │   ├─ 是否有外部服务延迟高（>1s）?
    │   │   │   │   ├─ YES → 外部服务故障，见决策树 4
    │   │   │   │   └─ NO → 继续诊断
    │   │   │   │
    │   │   ├─ 步骤 F1.7：查看数据库查询延迟
    │   │   │   ├─ 是否有慢查询（>100ms）?
    │   │   │   │   ├─ YES → 见决策树 1 的慢查询处理
    │   │   │   │   └─ NO → 继续诊断
    │   │   │   │
    │   │   ├─ 步骤 F1.8：查看 GC 日志
    │   │   │   ├─ 是否有频繁的 FULL GC（>500ms）?
    │   │   │   │   ├─ YES → 内存问题
    │   │   │   │   │   ├─ 步骤 F1.9a：检查堆内存使用率
    │   │   │   │   │   ├─ 步骤 F1.9b：分析内存泄漏
    │   │   │   │   │   └─ 步骤 F1.9c：调整 JVM 参数
    │   │   │   │   └─ NO → GC 正常
    │   │   │   │
    │   │   └─ 步骤 F1.10：查看应用代码耗时
    │   │       └─ 可能是业务逻辑复杂或加密操作
    │   │
    │   └─ NO → 延迟正常
    │
    ├─ 错误率 > 0.5%?
    │   ├─ YES → 识别错误原因
    │   │   ├─ 步骤 G1.1：查看错误日志聚合
    │   │   │   ├─ 是否全是 5xx 错误?
    │   │   │   │   ├─ YES → 应用异常
    │   │   │   │   │   ├─ 步骤 G1.2：查看异常堆栈
    │   │   │   │   │   ├─ 步骤 G1.3：分析错误类型
    │   │   │   │   │   │   ├─ NullPointerException? → 代码 bug
    │   │   │   │   │   │   ├─ TimeoutException? → 依赖超时
    │   │   │   │   │   │   ├─ ConnectionException? → 数据库/网络故障
    │   │   │   │   │   │   └─ 其他? → 业务异常
    │   │   │   │   │   └─ 步骤 G1.4：根据错误类型处理
    │   │   │   │   └─ NO → 混合 4xx 和 5xx
    │   │   │   │       ├─ 步骤 G1.5：分别分析 4xx 和 5xx 错误
    │   │   │   │       └─ 4xx 高 → 可能被攻击
    │   │   │   │
    │   │   ├─ 步骤 G1.6：检查是否与特定操作关联
    │   │   │   ├─ 是否某个端点错误率特别高?
    │   │   │   │   ├─ YES → 该端点的依赖故障
    │   │   │   │   │   └─ 步骤 G1.7：分析该端点的外部调用
    │   │   │   │   └─ NO → 系统整体问题
    │   │   │   │
    │   │   └─ 步骤 G1.8：如果是依赖故障，见决策树 4
    │   │
    │   └─ NO → 错误率正常
    │
    ├─ 吞吐量 < 50 req/s?
    │   ├─ YES → 请求被拒或处理变慢
    │   │   ├─ 步骤 H1.1：检查请求被拒数量
    │   │   │   ├─ 是否有大量 429/503 错误?
    │   │   │   │   ├─ YES → 限流或超载
    │   │   │   │   │   ├─ 步骤 H1.2：检查请求队列长度
    │   │   │   │   │   ├─ 步骤 H1.3：检查线程池状态
    │   │   │   │   │   ├─ 步骤 H1.4：考虑临时限流保护
    │   │   │   │   │   └─ 步骤 H1.5：规划扩容
    │   │   │   │   └─ NO → 真实吞吐量低
    │   │   │   │
    │   │   └─ 步骤 H1.6：诊断原因
    │   │       ├─ 每个请求处理时间是否很长? → 见延迟诊断
    │   │       ├─ 并发数是否低? → 可能是流量正常减少
    │   │       └─ 数据库连接是否满? → 见决策树 1
    │   │
    │   └─ NO → 吞吐量正常
    │
    └─ JVM 堆内存 > 85%?
        ├─ YES → 内存紧张，可能导致 OOM
        │   ├─ 步骤 I1.1：查看堆内存使用趋势
        │   │   ├─ 是否持续上升?
        │   │   │   ├─ YES → 可能内存泄漏
        │   │   │   │   ├─ 步骤 I1.2：dump 堆内存快照
        │   │   │   │   ├─ 步骤 I1.3：使用 MAT 分析泄漏对象
        │   │   │   │   ├─ 步骤 I1.4：定位泄漏代码
        │   │   │   │   └─ 步骤 I1.5：修复代码并发布
        │   │   │   └─ NO → 短期波动，可能是正常业务高峰
        │   │   │
        │   ├─ 步骤 I1.6：检查 GC 频率和暂停时间
        │   │   ├─ 是否频繁 FULL GC?
        │   │   │   ├─ YES
        │   │   │   │   ├─ 步骤 I1.7：优化 JVM 堆大小配置
        │   │   │   │   ├─ 步骤 I1.8：检查缓存大小
        │   │   │   │   └─ 步骤 I1.9：优化代码，减少临时对象
        │   │   │   └─ NO → GC 正常
        │   │   │
        │   └─ 步骤 I1.10：考虑临时重启应用，长期需要优化
        │
        └─ NO → 内存充足
```

### 决策树 3：业务流程故障诊断

```
业务指标告警（订单/支付/云卡/对账）
    │
    ├─ 订单创建成功率 < 95%?
    │   ├─ YES → 业务核心流程故障
    │   │   ├─ 步骤 J1.1：检查失败订单日志
    │   │   │   ├─ 失败原因分布如何?
    │   │   │   │   ├─ 多是数据库错误? → 见决策树 1
    │   │   │   │   ├─ 多是验证失败? → 业务规则问题
    │   │   │   │   │   ├─ 步骤 J1.2a：检查请求参数有效性
    │   │   │   │   │   └─ 步骤 J1.2b：可能客户端更新规则
    │   │   │   │   ├─ 多是外部服务失败? → 见决策树 4
    │   │   │   │   └─ 多是其他异常? → 步骤 J1.2c：分析具体错误
    │   │   │   │
    │   │   ├─ 步骤 J1.3：检查云卡同步是否成功
    │   │   │   ├─ 是否卡在 sync_qw_member 阶段?
    │   │   │   │   ├─ YES → QW 服务问题，见决策树 4
    │   │   │   │   └─ NO → 继续
    │   │   │   │
    │   │   ├─ 步骤 J1.4：检查数据库约束是否触发
    │   │   │   ├─ 例如：唯一约束冲突、外键约束等
    │   │   │   ├─ 是否看到 Duplicate Key 错误?
    │   │   │   │   ├─ YES → 可能数据重复或幂等性问题
    │   │   │   │   │   ├─ 步骤 J1.5a：检查幂等性记录
    │   │   │   │   │   └─ 步骤 J1.5b：可能是客户端重试
    │   │   │   │   └─ NO → 继续
    │   │   │   │
    │   │   └─ 步骤 J1.6：监控订单创建是否正常恢复
    │   │       └─ 如果已恢复，则为暂时故障
    │   │
    │   └─ NO → 订单创建正常
    │
    ├─ 首扣代扣成功率 < 95%?
    │   ├─ YES → 支付核心功能故障（最严重）
    │   │   ├─ 步骤 K1.1：立即检查 Allinpay 服务状态
    │   │   │   ├─ Allinpay 是否可达?
    │   │   │   │   ├─ 否 → Allinpay 服务故障

```

```
   ├─ 记录根本原因

2. 进行事后复盘（Post Mortem）
   ├─ 分析为什么没有提前预警
```

```
1. 查看这些订单的同步状态
   ├─ 是否 sync_status = PENDING（等待同步）?
   ├─ 是否 sync_status = FAILED（同步失败）?
   └─ 是否 sync_status = SUCCESS（声称已同步，但实际没有）?

2. 查看同步日志
   ├─ 是否有重试队列中的消息?
   ├─ 是否有失败日志记录?
   └─ 失败原因是什么? (网络错误、QW 拒绝、超时等)

3. 确定数据丢失的时间点
   ├─ 这些订单是何时创建的?
   ├─ 同步失败是何时发生的?
   └─ 是否对应某个已知的故障窗口? (如 QW 宕机)
```

**第 30-60 分钟（数据修复）**:
```
方案 1：手工重新同步
├─ 步骤 L1.6b：触发重试机制
├─ 监控这些订单的同步状态
└─ 验证 QW 端是否接收到

方案 2：手工补录 QW
├─ 导出差异订单的关键信息
├─ 通过 QW 后台手工创建或更新
└─ 更新本系统的 sync_status

方案 3：数据补偿
├─ 记录这些差异订单
├─ 启动人工审核流程
└─ 根据审核结果进行补偿或调整
```

**第 60+ 分钟（事后处理）**:
```
1. 完整的问题溯源
   ├─ 查看当时的系统日志
   ├─ 查看网络和服务可用性记录
   └─ 确定根本原因（是否是系统bug、网络故障、服务故障）

2. 防止再次发生
   ├─ 如果是代码 bug → 修复并发布
   ├─ 如果是网络问题 → 增加重试机制、改进超时设置
   ├─ 如果是外部服务 → 增加回调机制、改进容错
   └─ 增强同步监控告警

3. 合规审查
   ├─ 检查是否需要向监管部门报告
   ├─ 准备审计日志
   └─ 更新合规文档
```

## 应急处理SOP

### SOP 1：应用无响应（ECS 卡死）

**症状**:
- 请求超时，应用无响应
- ECS 仍在运行，但 CPU/内存可能很高
- 无法建立新连接

**处理步骤**:
```
第 1 步：确认问题
└─ 尝试多次调用，确保不是个别请求问题

第 2 步：快速诊断（<5 分钟）
├─ 检查 ECS CPU 是否 > 95%
├─ 检查 ECS 磁盘是否满（100%）
├─ 检查网络连接是否建立
└─ 检查应用日志是否有异常

第 3 步：临时恢复（<10 分钟）
├─ 方案 A：重启应用（最激进）
│   ├─ 在灰度环境重启 10% 流量
│   ├─ 如果恢复，逐步提升到 100%
│   └─ 如果仍不恢复，检查其他原因
│
├─ 方案 B：限流保护（有损服务）
│   ├─ 启用客户端限流
│   ├─ 放弃一部分流量以保护系统
│   └─ 避免级联故障
│
└─ 方案 C：扩容（如果是高并发）
    ├─ 临时启动新的应用实例
    ├─ 将流量分散
    └─ 评估是否需要永久扩容

第 4 步：事后诊断（故障消除后）
├─ 收集详细的系统日志和指标
├─ dump JVM 堆内存快照（如果怀疑内存泄漏）
├─ 分析根本原因
└─ 提交改进方案
```

---

### SOP 2：数据库无法连接

**症状**:
- 应用日志出现 `Connection refused` 或 `Connection timeout`
- 业务流程中断
- 部分操作失败

**处理步骤**:
```
第 1 步：确认 RDS 状态
├─ 登录阿里云 RDS 控制台
├─ 检查实例是否在运行
├─ 检查是否有告警通知
└─ 查看最近的维护日志

第 2 步：检查网络连接
├─ SSH 登录 ECS
├─ 运行 telnet ip:3306（或 RDS 端口）
├─ 尝试 nc -zv ip:3306
└─ 检查连接是否通

第 3 步：检查 RDS 配置
├─ 检查白名单中是否包含 ECS IP
├─ 检查 VPC 和安全组规则
├─ 检查 RDS 参数中的 max_connections
└─ 是否需要调整?

第 4 步：恢复措施
├─ 如果是白名单问题 → 添加 ECS IP
├─ 如果是 max_connections 满 → 分析连接泄漏
├─ 如果是 RDS 宕机 → 等待 RDS 重启或故障转移
├─ 如果是 RDS 故障 → 联系阿里云支持
└─ 监控恢复进度

第 5 步：应用恢复
├─ 应用会自动重连（如果配置了重试）
├─ 如果没有，手工重启应用
├─ 验证数据库连接是否恢复
└─ 检查是否有遗漏的更新（从宕机时起）
```

---

### SOP 3：外部服务（Allinpay/QW）不可达

**症状**:
- 调用外部服务超时或连接拒绝
- 特定的业务流程失败（如支付、权益行权）
- 外部服务可能宣布故障

**处理步骤**:
```
第 1 步：确认问题
├─ 检查外部服务的官方状态页面
├─ 查看是否有公告故障
└─ 查看最近 24 小时的故障历史

第 2 步：网络诊断
├─ SSH 登录 ECS
├─ 运行 ping 外部服务的域名
├─ 运行 traceroute 诊断路由
├─ 查看 DNS 解析是否正常
└─ 检查是否能连接（telnet）

第 3 步：应用侧检查
├─ 检查 API 端点是否正确
├─ 检查认证信息（密钥、签名）是否正确
├─ 检查是否被限流或黑名单
├─ 查看最近的 API 调用日志
└─ 是否有异常错误信息?

第 4 步：应急降级
├─ 如果是 Allinpay 故障，启用兜底支付
├─ 如果是 QW 故障，启用本地缓存或排队
├─ 如果是认证故障，可能需要刷新或重新获取凭证
└─ 通知用户可能的延迟

第 5 步：监控和恢复
├─ 定期尝试重新连接
├─ 监控恢复进度
├─ 外部服务恢复后，进行数据同步和对账
└─ 启动 fallback 恢复流程
```

---

### SOP 4：告警风暴（告警信息爆炸）

**症状**:
- 告警消息频繁发送（每秒数十条）
- 通知渠道被刷屏
- 无法快速定位真实问题

**处理步骤**:
```
第 1 步：立即止损
├─ 临时关闭或延迟告警通知（降低噪音）
├─ 但继续在后台收集告警数据
└─ 专注于最关键的 P0 告警

第 2 步：快速诊断
├─ 检查是否是连锁故障
│   ├─ 如 ECS 故障导致所有应用告警
│   ├─ 如数据库故障导致所有业务告警
│   └─ 如果是，解决根本原因
│
└─ 检查是否是告警配置不合理
    ├─ 是否阈值设置太低?
    ├─ 是否检测间隔太短?
    └─ 是否需要调整?

第 3 步：恢复
├─ 解决根本问题
├─ 验证告警已消除
├─ 恢复正常的告警通知
└─ 重新审视告警规则的准确性

第 4 步：改进告警规则
├─ 分析这次风暴的原因
├─ 调整告警阈值或条件
├─ 增加告警聚合或抑制规则
└─ 避免下次风暴
```

## 故障排查工具

### 工具 1：日志查询

**在阿里云日志服务 SLS 中查询**:
```sql
-- 查询订单创建失败的日志
source: "BenefitOrderServiceImpl" AND "createOrder" AND "failed"
| stats count() as failure_count by error_reason

-- 查询支付处理超时
source: "PaymentServiceImpl" AND "handleCallback" AND "timeout"
| stats count() as timeout_count, avg(duration) as avg_duration_ms

-- 查询 QW 调用错误
source: "QwBenefitClientImpl" AND "error"
| stats count() as error_count by error_code
```

### 工具 2：CloudMonitor 查询

```bash
# 查询 ECS CPU 使用率（最近 1 小时）
aliyun cms DescribeMetricList --MetricName CPUUtilization --Dimensions '[{"name":"instanceId","value":"i-xxxxx"}]'

# 查询 RDS 连接数
aliyun cms DescribeMetricList --MetricName Connections --Dimensions '[{"name":"instanceId","value":"rds-xxxxx"}]'
```

### 工具 3：RDS 慢查询分析

```bash
# 登录 RDS MySQL
mysql -h rds-host -u user -p

# 查看慢查询日志
SELECT * FROM mysql.slow_log;

# 分析执行计划
EXPLAIN SELECT ...;

# 查看表索引
SHOW INDEX FROM table_name;
```

### 工具 4：JVM 诊断

```bash
# 列出 Java 进程
jps -l

# 查看线程信息
jstack pid > threads.dump

# 查看堆内存使用
jstat -gc pid 1000

# dump 堆内存快照
jmap -dump:live,format=b,file=heap.bin pid

# 使用 MAT 分析（在本地）
mat heap.bin
```

### 工具 5：网络诊断

```bash
# 检查网络连接
ping api.allinpay.com

# 追踪路由
traceroute api.allinpay.com

# 检查 DNS 解析
nslookup api.allinpay.com

# 检查端口连接
telnet api.allinpay.com 443

# 查看网络连接统计
ss -s
netstat -an | grep ESTABLISHED | wc -l
```

## 总结与最佳实践

### 快速诊断的关键原则

1. **分层诊断**（从下到上）
   - 基础设施 → 应用 → 业务 → 外部服务
   - 先排除已知的常见原因

2. **数据驱动**
   - 不要猜测，看数据
   - 查看日志、指标、追踪
   - 对比时间线

3. **快速假设验证**
   - 提出假设（根本原因）
   - 快速验证（通过日志或诊断命令）
   - 得出结论

4. **及时升级**
   - 如果 5 分钟内无法定位，升级告警
   - 寻求专家或技术负责人帮助
   - 不要试图一个人解决复杂问题

### 故障处理的黄金时间

```
P0 告警处理时间目标：
├─ 0-2 分钟：确认问题的真实性
├─ 2-10 分钟：定位根本原因
├─ 10-30 分钟：采取恢复措施
└─ 30+ 分钟：事后复盘和改进
```
