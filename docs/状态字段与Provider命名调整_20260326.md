# 状态字段与 Provider 命名调整说明

## 1. 调整背景

当前项目最初以齐为作为单一权益执行方推进，因此代码中出现了如下命名：

- `qw_first_deduct_status`
- `qw_fallback_deduct_status`
- `qw_exercise_status`
- `payment_record.channel_name = "QW"`

这类命名把单一合作方编码写进了核心领域模型，不利于后续接入多家权益方。

## 2. 本次调整原则

### 2.1 状态字段保持业务中性

面向代码语义和 API 输出，统一改为：

- `firstDeductStatus`
- `fallbackDeductStatus`
- `exerciseStatus`

对应含义分别是：

- 首扣状态
- 兜底代扣状态
- 行权状态

这些字段只表达“业务阶段结果”，不再夹带具体合作方编码。

### 2.2 来源渠道与执行方分离

订单模型中原先的 `channelCode` 容易让人误解为“权益执行方”。

本次在 Java 模型里改为：

- `sourceChannelCode`

它表示：

- 会员来源渠道
- 渠道绑定渠道
- 科技平台侧识别渠道

它不表示：

- 首扣执行方
- 兜底代扣执行方
- 行权执行方
- 权益合作方

### 2.3 支付流水改用 `providerCode`

`payment_record` 代码语义从 `channelName` 调整为：

- `providerCode`

它表达的是当前支付或执行动作由哪一家合作方/权益方承担。

当前默认值仍为 `QW`，但这个值现在只表示“当前默认执行方配置”，不再写入字段名和状态名。

## 3. 本次数据库迁移策略

本次已经继续推进到物理列层面，但仍采用相对安全的迁移方式。

原因：

1. 当前项目默认复用本地已有数据库，不会自动重建表。
2. 现有本地 MySQL 可能仍保留旧列名，直接切换 ORM 到新列会导致插入和查询失败。

因此本次采用的策略是：

- 新建库直接使用新物理列名
- 启动时自动检查旧列是否存在
- 如果存在旧列且新列不存在，则自动执行列重命名
- 代码、API 输出和数据库物理列统一收敛到新命名

这意味着当前实现已经不是“仅代码语义重命名”，而是完成了：

- 代码命名收敛
- API 输出收敛
- 库表初始化脚本收敛
- 本地存量库自动迁移

## 4. 当前落地结果

### 4.1 代码层

- `BenefitOrder.qwFirstDeductStatus` -> `BenefitOrder.firstDeductStatus`
- `BenefitOrder.qwFallbackDeductStatus` -> `BenefitOrder.fallbackDeductStatus`
- `BenefitOrder.qwExerciseStatus` -> `BenefitOrder.exerciseStatus`
- `BenefitOrder.channelCode` -> `BenefitOrder.sourceChannelCode`
- `PaymentRecord.channelName` -> `PaymentRecord.providerCode`
- `BenefitOrderStatusResponse` 的响应字段同步切换为中性命名

### 4.2 数据库层

`benefit_order` 新物理列名：

- `channel_code` -> `source_channel_code`
- `qw_first_deduct_status` -> `first_deduct_status`
- `qw_fallback_deduct_status` -> `fallback_deduct_status`
- `qw_exercise_status` -> `exercise_status`

`payment_record` 新物理列名：

- `channel_name` -> `provider_code`

启动期由 `SchemaMigrationRunner` 自动执行旧列到新列的迁移。

### 4.3 配置层

新增：

- `nexusfin.business.default-provider-code`

当前默认值是：

- `QW`

含义是当前默认执行方配置，而不是领域字段名的一部分。

## 5. 后续建议

虽然本次已经完成物理列迁移，但仍有两项后续工作值得单独推进：

1. 更新所有历史设计文档和数据库标准文档中的旧列名，避免文档与代码脱节。
2. 如果后面还会继续拆分“来源渠道”和“执行方”，可以在 `benefit_order` 继续补充更细粒度的 provider 字段，例如：
   - `first_deduct_provider_code`
   - `fallback_deduct_provider_code`
   - `exercise_provider_code`
