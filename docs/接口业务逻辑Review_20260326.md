#这个是reivew 接口业务逻辑的修改建议（是在从“静默注册 + 下单”切换为“SSO 登录 + 下单”：之前）：

##1 `POST /api/users/register`
1.入参增加电话号码
2.先解密用户四项姓名、身份证、银行卡、手机号。
由于是第三方传递，需要根据channelCode 提供解密方法。解密方法参考配置文件。
3.然后判断四项是否符合格式要求。
4.然后四项信息统一采用：AES-256-GCM加密入库。加密方法参考配置文件。
5.加密、解密配置文件：
{
  "crypto": {
    "kek": {
      "keyId": "KEK_USER_MASTER",
      "keyVersion": 1,
      "algorithm": "AES-256",
      "purpose": "wrap_dek",
      "status": "ACTIVE"
    },
    "dek": {
      "keyId": "DEK_USER_PROFILE",
      "keyVersion": 1,
      "algorithm": "AES-256-GCM",
      "purpose": "encrypt_user_sensitive_fields",
      "status": "ACTIVE",
      "wrappedDek": "BASE64_ENCRYPTED_DEK_PLACEHOLDER"
    }
  }
}
##2 `GET /api/equity/products/{productCode}`
1.需要考虑登录问题，参考单点登录实现方案。
2.齐为的权益信息，可以单独先配置到数据库表中。

##3 `POST /api/equity/orders`
1.benefit_order 数据表中的状态字段需要合并成一个order_status（first_deduct_status、fallback_deduct_status、exercise_status、refund_status、grant_status）
  用order_status 表示状态机的字段
2.同步更新数据库表benefit_order、同步更新代码逻辑，并跑通单测
3.同步更新docs/数据库ER关系说明；  

注：当前项目已将 `qw_*` 状态字段重命名为中性命名，但未采纳“压平成单一 order_status”的方案。
