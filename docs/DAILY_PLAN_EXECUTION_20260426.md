## 3. 根因分析（按贡献度排序）

### 3.1 主因：**导航 / 汇总文档腐烂**（影响最大、最致命）

`README.md` 和 `阶段性 TODO 清单` 不会随新文件自动更新——必须手工回填。一旦回填中断（你在 4-21 之后没回填），所有 “索引依赖入口" 的工作流都退化成 “盲目搜索"：

- `grep -r 异步补偿 docs/plan/` → 出来 9 份文档，不知道哪份是最新口径
- 想问 “小花联合登录现在卡在哪” → 要读 4-15 / 4-16 / 4-17 / 4-21 / 4-23 / 4-24 / 4-26 → 7 份文档接力推断

**这一项一旦发生，其它问题都被放大**。

### 3.2 次要因：**主题信息跨日跨文件碎片化**

同一主题（齐为签约、小花联合登录、异步补偿、云卡状态机）的演进散落在 5–10 份按日期命名的文件里。每个新决策是 “补丁式” 落在新文件，旧文件不更新。

例 · 异步补偿主题文档时间线：

```
20260417_异步消息补偿机制方案与实施计划.md   (688 行)
20260417_异步消息补偿机制开发任务拆解.md    (809 行)
20260417_异步消息补偿机制_checklist.md
20260418_checklist.md（含异步补偿小节）
20260419_异步补偿开发任务拆解_调度接入版.md
20260419_异步补偿人工处理SQL与操作模板.md
20260419_异步补偿运维与人工处理说明.md
20260419_异步补偿自动调度接入建议方案.md
20260420_自动调度运行口径与启用说明.md
20260420_自动调度启用后观察与人工介入SOP.md
```

要回答 “异步补偿现在是什么状态”，**必须读 10 份**，且没有一份是 “canonical 当前口径”。

### 3.3 三因：**没有归档机制 → 已完成项继续占注意力**

`specs/001-equity-service-baseline` 已完成、3 月初的整体计划文档（`20260329_项目开发到上线整体计划.md` 555 行）已不再是当前主线，但全部和今日待办平铺在同目录。每次 `ls docs/plan/` 都看到一坨历史。

### 3.4 四因：**plan 文件本身体量增长**

10 份单文件 >500 行（最大 809 行）。这些往往是 “开发任务拆解” 类，本来该一次性消费完归档，结果一直散在主目录里。Claude Code 做主题搜索时这些大文件会被读取多次。

### 3.5 五因：**plan 起步无模板，结构不统一**

每天的 `<date>.md` / `<date>_checklist.md` 结构其实高度相似（目标 / 优先级 / 动作 / 完成标准 / 风险），但每次都从 0 写。同时不同主题的 “实施计划 / 任务拆解 / checklist” 三段式也没固化模板，导致：

- 同类文档结构略不同 → grep 不到
- 起步成本累积 → 每天写 plan 的时间在涨

### 3.6 六因：**两套规划体系并存**（specs/ vs docs/plan/）

`specs/` 是早期 spec-kit 风格的正式规格目录（plan.md、data-model.md、quickstart.md、tasks.md），`docs/plan/` 是日常迭代日志。两者：

- 都包含 "tasks" 描述
- 现状是 docs/plan/ 是主战场、specs/ 渐渐废弃
- 但 specs/004/tasks.md 还会被工具检索（CI、grep）

→ 用户必须显性标注 “specs/004 已不再是主线”（在 4-26 plan 里写过），但工具不知道。

### 3.7 七因：**测试 / 构建慢**（已部分修复）

`mvn test` 32s → 17.6s 已通过 surefire/failsafe 拆分修好。但 `mvn verify` 仍 62s——这本身不算大问题，但每天跑多次时累积感知 “慢”。

### 3.8 八因（次要）：**Claude Code session 上下文成本**

每次新会话：
- 扫描代码库（含 27k 行 plan）
- 加载 CLAUDE.md / memory
- 多次 grep 大文件

这一项不是主因，但和上面 1–4 项叠加放大。

## 4. 解决方案（按 ROI 排序）

### 4.1 Tier 1 · 立竿见影（半天投入，反复受益）

#### T1.1 — 归档 + 物理隔离（最高 ROI）

```
docs/plan/
├── README.md                       ← 重写成自动生成的导航
├── CURRENT_STATE.md                ← 唯一 "当前口径" 入口（每周末更新）
├── active/                         ← 进行中：最近 7 天 + 阻塞项
│   ├── 20260426.md
│   ├── 20260426_checklist.md
│   └── 20260426_*.md
├── archive/                        ← 已完成 / 已 superseded
│   ├── 202603/
│   └── 202604/                     ← 4-21 之前都进这里
└── topics/                         ← 主题工作目录
    ├── 齐为/
    │   ├── README.md               ← 当前状态 + 时间线
    │   └── 20260421_*.md           ← 历史快照
    ├── 小花科技/
    ├── 异步补偿/
    ├── 云卡/
    └── 电子签章/
```

- **执行**：写个 `scripts/archive_plan.sh` 把 7 天前的 `<date>.md` 移到 `archive/<YYYYMM>/`，主题专题文档移到对应 `topics/` 下
- **收益**：`ls docs/plan/active/` 永远只看到当前在战的几份文件
- **投入**：脚本 1h（Claude Code 15min），首次手工分类 1–2h

#### T1.2 — 自动生成 README

```bash
# scripts/gen_plan_readme.sh
# 扫描最近 7 天的 plan 文件
# 按 topic（从文件名前缀提取）分组
# 输出 README.md 的 "当前活跃文档" 段
```

- **执行**：每天首次运行 / git pre-commit hook 自动跑
- **收益**：README 永不腐烂
- **投入**：30min 脚本（Claude Code 10min）

#### T1.3 — `CURRENT_STATE.md` 单点真相

每周末（或每个里程碑）由你手动写一份：

```markdown
# 当前项目状态 · 2026-04-26

## 主线状态
- 用信新链路改造方案：v0.3，待云卡接口定版
- ...

## 5 方依赖确认状态
| 方 | 待确认项 | 状态 | 文档 |
|----|---------|------|------|
| 齐为 | 签约接口字段 | ✅ 已确认（4-21） | topics/齐为/README.md |
| 小花科技 | 4.22 细节 | ⏳ 等回复 | topics/小花科技/README.md |
| 云卡 | 状态机定义 | ⏳ 等接口文档 | topics/云卡/README.md |
| 科技平台 | 接口改造 | ✅ 已收口（4-08） | archive/202604/... |
| 电子签章 | 法大大资料 | ⏳ 等商务 | topics/电子签章/README.md |

## 阻塞项（按解除条件分类）
...

## 这周的核心决定
...
```

- **收益**：每天开工只读 1 份 → CURRENT_STATE.md
- **投入**：每周末 30min（不是天天写）

### 4.2 Tier 2 · 工具 / 模板（1 天投入，长期受益）

#### T2.1 — 模板化（消除起步成本）

```
docs/plan/templates/
├── daily_plan.md         ← <date>.md 模板
├── daily_checklist.md    ← <date>_checklist.md 模板
├── topic_design.md       ← XX 方案设计 模板
├── topic_tasks.md        ← XX 任务拆解 模板
└── topic_checklist.md    ← XX checklist 模板
```

- 起步从 “复制模板 + 改 frontmatter” 开始
- 一致结构 = 一致 grep 模式 = AI 协作更快

#### T2.2 — frontmatter 元数据

```markdown
---
date: 2026-04-26
type: daily | topic-design | topic-tasks | checklist
topic: 齐为 | 小花科技 | 异步补偿 | 云卡 | 电子签章 | shared
status: active | superseded | archived
supersedes: [20260417_异步消息补偿机制方案与实施计划.md]
---
```

- 配脚本可以查询 “topic=齐为 AND status=active” 的所有文档
- 取代手工维护 README

#### T2.3 — 每周 retro / 5-minute 收口

每周五 10–15min：

```bash
# 自动出报告
scripts/weekly_plan_summary.sh
# 列出本周新增 plan 文件、touched topics、未完成 checklist 项
```

- 防止又长一周还没回填导航
- 强制每周一次 “归档 + 更新 CURRENT_STATE”

### 4.3 Tier 3 · 流程改造（中长期）

| 方向 | 投入 | 收益 |
|------|------|------|
| 引入轻量项目管理工具（Linear / GitHub Projects） | 1 天 setup + 习惯切换 | 多方协调 / 状态追踪从 markdown 转到 issue tracker，markdown 只留 “方案沉淀" |

| `docs/plan/` 目录纳入轻量索引（如 ripgrep 配置 + AI 索引） | 1h | grep 命中率提升，找决策更快 |

## 7. 风险与边界

| 风险 | 缓解 |
|------|------|
| 归档过程中误移正在用的文件 | 先 `git mv` + commit，错了能 revert |
| 自动 README 把不该展示的 internal note 暴露 | 模板里加 `visibility: internal` 字段，脚本过滤 |
| 模板 / frontmatter 让写文档变重 | 模板要 “最小可用"——3 个 section + 5 行 frontmatter，不强求填满 |
| 你不习惯每周五收口 | 先观察 2 周，看是不是必要；不是的话改成 “产出新 plan 时回填 README" |
| `specs/` 目录被 CI / 脚本依赖 | 归档前 grep 一遍 `specs/004` 在哪些地方被引用 |

## 8. 不推荐做的事（避免常见过度优化）

- ❌ **整体迁移到 Notion / 飞书 / 等等**：迁移成本远大于收益，且失去 git 版本控制。markdown + git 已经是最好的工程师友好选择
- ❌ **试图把 27k 行 plan 自动总结成一份大文档**：AI 总结会丢失关键边界条件，反而引入错误
- ❌ **强制所有历史文件加 frontmatter**：只对 active/ 和最近 7 天的加，archive 不动
- ❌ **引入复杂的 “知识图谱" / 向量检索**：当前规模（132 文件 / 27k 行）远没到这个必要
