# 当前项目 API 业务逻辑梳理

## 1. 文档目的

本文档基于当前代码实现，梳理 `nexusfin-equity` 项目全部对外 API 的业务逻辑、关键处理机制和当前实现边界，重点覆盖以下内容：

- 接口的业务用途和处理链路
- 关键编排步骤
- 并发、幂等、事务、异常处理方式
- 资损、安全、状态一致性相关关注点

本文档描述的是“当前代码事实”，不是理想设计稿。如果代码存在明显边界或风险，文档会直接指出，便于后续逐条修订。

## 2. 接口总览

当前项目共有 13 个 HTTP API：

| 分类 | 方法 | 路径 | 说明 |
| --- | --- | --- | --- |
| 健康检查 | `GET` | `/api/equity/health` | 服务健康检查 |
| SSO 认证 | `GET` | `/api/auth/sso-callback` | 科技平台 token 回调登录 |
| 当前用户 | `GET` | `/api/users/me` | 获取当前登录用户 |
| 产品页 | `GET` | `/api/equity/products/{productCode}` | 查询产品展示信息 |
| 下单 | `POST` | `/api/equity/orders` | 创建权益订单 |
| 订单查询 | `GET` | `/api/equity/orders/{benefitOrderNo}` | 查询订单状态 |
| 行权链接 | `GET` | `/api/equity/exercise-url/{benefitOrderNo}` | 获取行权链接 |
| 支付回调 | `POST` | `/api/callbacks/first-deduction` | 首扣结果回调 |
| 支付回调 | `POST` | `/api/callbacks/fallback-deduction` | 兜底代扣结果回调 |
| 通知回调 | `POST` | `/api/callbacks/grant/forward` | 放款结果通知 |
| 通知回调 | `POST` | `/api/callbacks/repayment/forward` | 还款状态通知 |
| 通知回调 | `POST` | `/api/callbacks/exercise-equity` | 行权结果通知 |
| 通知回调 | `POST` | `/api/callbacks/refund` | 退款结果通知 |

## 3. 平台级统一机制

### 3.1 统一返回结构

所有 Controller 都返回 `Result<T>` 结构。

- 成功时通过 `Result.success(...)` 返回
- 业务异常通过 `BizException` 抛出后，被 `GlobalExceptionHandler` 统一转换
- 参数校验异常返回 `400`
- JWT 未登录由过滤器直接返回 `401`

### 3.2 TraceId 与日志

每个请求都会经过 `TraceIdFilter`：

- 读取请求头 `X-Trace-Id`
- 如果没有则自动生成
- 回写到响应头
- 在日志 MDC 中绑定 `traceId`

当前关键业务日志一般会打印 `traceId + bizOrderNo`，便于串联链路。

### 3.3 认证与访问控制

系统有两套入口保护机制：

1. JWT Cookie 认证

- 由 `JwtAuthenticationFilter` 负责
- 保护路径主要包括：
  - `/api/equity/products`
  - `/api/equity/orders`
  - `/api/equity/exercise-url`
  - `/api/users/me`
- SSO 登录成功后，服务端写入本地 JWT Cookie
- 业务接口从 `AuthContextUtil` 读取当前 `memberId`

2. 回调签名校验

- 由 `SignatureInterceptor` 负责
- 保护路径是 `/api/callbacks/**`
- 校验请求头：
  - `X-App-Id`
  - `X-Timestamp`
  - `X-Nonce`
  - `X-Signature`
- 当前签名算法只覆盖 `appId + timestamp + nonce`，不包含 body

### 3.4 参数校验

所有写接口基本都使用了 `@Valid`。

典型校验包括：

- `requestId`、`benefitOrderNo` 等必填
- 金额必须为正数或非负数
- 回调状态字段必须非空

需要注意：

- 路径参数如 `productCode`、`benefitOrderNo` 当前没有加格式注解，只做“是否查得到记录”的业务校验

### 3.5 事务边界

当前主要事务边界落在 Service 层：

- `AuthServiceImpl.loginWithTechToken`
- `BenefitOrderServiceImpl.createOrder`
- `PaymentServiceImpl.handleFirstDeductCallback`
- `PaymentServiceImpl.handleFallbackDeductCallback`
- `NotificationServiceImpl.handleGrant`
- `NotificationServiceImpl.handleRepayment`
- `NotificationServiceImpl.handleExercise`
- `NotificationServiceImpl.handleRefund`
- `FallbackDeductServiceImpl.triggerFallback`
- `AgreementServiceImpl.ensureAgreementArtifacts`

事务设计意图是：

- 一个业务动作内的“写主表 + 写流水/日志 + 写幂等记录”尽量保持同成同败
- 避免只更新了订单状态但没写日志，或只写了流水没更新主单

### 3.6 幂等实现

系统的幂等主要依赖 `idempotency_record` 表和部分业务表重复查询。

当前使用方式分为两类：

1. 主动业务幂等

- 创建订单通过 `requestId` 查 `idempotency_record`
- 已处理则直接返回历史结果

2. 回调幂等

- 支付回调先查 `payment_record.request_id`
- 再查 `idempotency_record`
- 通知回调主要查 `idempotency_record`

当前实现特点：

- `IdempotencyServiceImpl` 是“先查再插”
- 没有分布式锁
- 没有乐观锁版本号
- 主要依赖数据库主键约束兜底最终一致性

### 3.7 状态模型

`benefit_order` 不是单一状态字段模型，而是“主状态 + 子状态并存”：

- `order_status`：订单主状态
- `first_deduct_status`：首扣状态
- `fallback_deduct_status`：兜底代扣状态
- `exercise_status`：行权状态
- `refund_status`：退款状态
- `grant_status`：放款状态
- `sync_status`：下游同步状态

当前代码明确依赖这种拆分式状态表达。后续如要合并状态字段，改动面会非常大。

补充说明：

- Java 代码和 API 输出已经切换为中性命名，不再暴露 `qw_*`
- 新库初始化直接使用中性物理列名
- 对已有本地库，启动阶段会自动把旧列名迁移为新列名
- `benefit_order.sourceChannelCode` 表示来源渠道，不表示权益执行方
- 支付流水中的 `providerCode` 表示当前执行方/权益方

## 4. 当前业务主链

从当前代码看，核心业务主链是：

1. 前端携带科技平台 token 进入 `/api/auth/sso-callback`
2. 服务端向科技平台换取用户资料
3. 本地即时建档或更新会员
4. 下发本地 JWT Cookie
5. 前端访问产品页 `/api/equity/products/{productCode}`
6. 用户提交下单 `/api/equity/orders`
7. 系统写订单、协议任务、合同归档、幂等记录
8. 渠道回调首扣结果 `/api/callbacks/first-deduction`
9. 首扣成功时走直连同步；首扣失败时等待放款结果决定是否走兜底代扣
10. 放款通知 `/api/callbacks/grant/forward`
11. 如果“首扣失败 + 放款成功”，系统自动触发兜底代扣
12. 后续接收兜底代扣、行权、退款、还款等通知

## 5. 逐接口业务逻辑

### 5.1 `GET /api/equity/health`

#### 业务用途

服务健康检查接口，返回应用名、状态和当前时间。

#### 处理流程

1. Controller 直接构造 `HealthStatusResponse`
2. 返回 `service=nexusfin-equity`
3. 返回 `status=UP`
4. 返回当前 UTC 时间字符串

#### 并发、幂等、事务

- 无状态接口
- 天然幂等
- 无事务

#### 异常与风险

- 几乎无业务风险
- 不做数据库或下游连通性探测，因此只能代表“进程活着”，不能代表“依赖都可用”

#### 资损关注

- 无直接资损风险

### 5.2 `GET /api/auth/sso-callback`

#### 业务用途

承接科技平台浏览器跳转，完成本地登录态建立。

#### 处理流程

1. 接收 `token` 和可选 `redirect_url`
2. `AuthServiceImpl.loginWithTechToken` 校验 token 不可为空
3. `TechPlatformUserClientImpl` 调科技平台用户接口换取用户资料
4. 取回的资料包含：
   - `userId`
   - `phone`
   - `realName`
   - `idCard`
   - 可选 `channelCode`
5. 系统根据 `channelCode` 做入站解码
6. 对手机号、身份证、姓名做格式校验
7. 对敏感字段做 AES-GCM 加密，并写 hash 索引
8. 按 `techPlatformUserId` 查本地会员
9. 已存在则更新会员资料；不存在则新建会员
10. 按 `channelCode + externalUserId` 保证会员渠道绑定存在
11. 校验 `redirect_url` 是否命中白名单
12. 生成本地 JWT
13. 通过 `Set-Cookie` 写回浏览器
14. 返回 `302` 跳转

#### 关键处理

1. 敏感信息处理

- 手机号、身份证、姓名入库前会加密
- 手机号和身份证还会写 SHA-256 hash
- 当前代码没有解密展示接口，主要服务于安全存储

2. 即时建档

- 通过 `tech_platform_user_id` 识别会员
- 这是当前替代“静默注册”的实现

3. 登录态建立

- JWT subject 是 `memberId`
- Cookie 默认 `HttpOnly`
- Cookie 可配置 `Secure`、`SameSite`

#### 并发、幂等、事务

- 方法带事务
- 没有显式幂等表
- 幂等依赖会员唯一标识 `tech_platform_user_id`
- 如果同一用户并发首次登录，可能出现并发插入竞争

当前代码特点：

- 先查再插，没有显式“查不到后重试查询”的补偿
- 一旦数据库唯一键冲突，事务会回滚
- 当前代码没有专门捕获并转义“重复键”异常

#### 异常与风险

- 上游 token 校验失败会抛业务异常
- `redirect_url` 不在白名单内会拒绝
- 新会员如果缺少手机号、身份证、姓名，会直接失败
- 旧会员同步资料时允许部分字段缺失，不会强制失败

#### 资损与安全关注

- 该接口本身不处理资金，但会影响订单归属
- 如果用户资料绑定错会员，会导致后续订单归属错误
- 当前 `redirect_url` 使用 `startsWith` 白名单校验，能拦截明显外域，但仍依赖配置准确性

### 5.3 `GET /api/users/me`

#### 业务用途

返回当前登录用户信息。

#### 处理流程

1. `JwtAuthenticationFilter` 校验 Cookie 中 JWT
2. 解析出 `memberId` 和 `techPlatformUserId`
3. `AuthServiceImpl.getCurrentUser` 查询 `member_info`
4. 返回：
   - `memberId`
   - `techPlatformUserId`
   - `externalUserId`
   - `memberStatus`

#### 并发、幂等、事务

- 只读接口
- 天然幂等
- 无事务

#### 异常与风险

- 缺少 Cookie 返回 `401`
- JWT 失效或非法返回 `401`
- JWT 解析成功但本地会员不存在，也会返回 `401`

#### 资损关注

- 无直接资损
- 但它是前端判断登录态的基础接口，若鉴权错误会影响后续下单主体

### 5.4 `GET /api/equity/products/{productCode}`

#### 业务用途

查询权益产品展示页数据，并带出当前登录会员的基本归属信息。

#### 处理流程

1. JWT 过滤器校验登录态
2. Controller 从认证上下文取 `memberId`
3. `BenefitOrderServiceImpl.getProductPage` 查询产品主表
4. 产品不存在或不是 `ACTIVE` 则报错
5. 查询当前会员
6. 返回：
   - 产品编码
   - 产品名称
   - 费率
   - 固定协议类型列表
   - 当前会员 `memberId`
   - 当前会员 `externalUserId`

#### 关键处理

- 接口不接受外部传入 `memberId`
- 会员身份完全由登录态决定
- Controller 只负责取认证上下文，不拼业务数据

#### 并发、幂等、事务

- 只读接口
- 天然幂等
- 无事务

#### 异常与风险

- 产品不存在返回 `PRODUCT_NOT_FOUND`
- 当前实现没有校验“产品是否属于特定渠道/特定会员”

#### 资损关注

- 风险较低，但它决定下单前展示信息
- 如果展示产品与实际可下单范围不一致，可能引发错误下单

### 5.5 `POST /api/equity/orders`

#### 业务用途

创建权益订单，是当前业务核心入口。

#### 入参约束

- `requestId` 必填
- `productCode` 必填
- `loanAmount` 必须大于 0
- `agreementSigned` 必填

#### 处理流程

1. JWT 过滤器校验登录态
2. Controller 从认证上下文取 `memberId`
3. 先按 `requestId` 查幂等表
4. 如果已处理：
   - 取历史 `benefitOrderNo`
   - 查订单主表
   - 直接回放历史结果
5. 校验 `agreementSigned=true`
6. 查询产品，要求产品为 `ACTIVE`
7. 查询当前会员
8. 查询会员最近一条渠道绑定
9. 构造订单主表 `benefit_order`
10. 初始化状态：
    - `order_status = FIRST_DEDUCT_PENDING`
    - `first_deduct_status = PENDING`
    - `fallback_deduct_status = NONE`
    - `exercise_status = NONE`
    - `refund_status = NONE`
    - `grant_status = PENDING`
    - `sync_status = SYNC_PENDING`
11. 写入订单
12. 生成协议任务和合同归档
13. 写入幂等记录
14. 返回 `benefitOrderNo`

#### 关键处理

1. 订单号设计

- 订单主键是 `benefitOrderNo`
- 后续支付、回调、通知、签约、归档都围绕该单号串联

2. 协议前置

- 代码要求下单前协议已经签署
- 通过 `agreementSigned` 做前置校验

3. 协议资料沉淀

- 下单后立即创建两类协议任务
- 同时生成合同归档记录
- 这是后续审计与争议处理的基础数据

#### 并发、幂等、事务

- 方法带事务
- 幂等键是 `requestId`
- 幂等记录在订单和协议写入后落表

当前并发语义：

- 正常串行场景可确保同一个 `requestId` 只生成一单
- 并发重复请求时，两个线程都可能在早期看到“未处理”
- 但后续幂等表主键冲突会让后到事务回滚

需要注意：

- 这种实现属于“数据库主键兜底型幂等”
- 没有显式锁
- 没有 `benefit_order.request_id` 唯一约束

#### 异常与风险

- 产品不存在、会员不存在、渠道绑定不存在都会失败
- 如果协议生成失败，整个订单事务会回滚

#### 资损关注

- 这是最关键的资损入口之一
- 当前代码没有校验 `loanAmount` 是否与产品规则匹配
- 当前代码没有校验会员是否对该产品有准入资格
- 当前代码也没有防止同一会员短时间内多笔业务重复下单

### 5.6 `GET /api/equity/orders/{benefitOrderNo}`

#### 业务用途

查询订单当前状态视图。

#### 处理流程

1. JWT 过滤器校验登录态
2. 根据 `benefitOrderNo` 直接查订单主表
3. 返回：
   - `benefitOrderNo`
   - `orderStatus`
   - `firstDeductStatus`
   - `fallbackDeductStatus`
   - `exerciseStatus`
   - `grantStatus`

#### 并发、幂等、事务

- 只读接口
- 天然幂等
- 无事务

#### 异常与风险

- 查不到订单返回 `ORDER_NOT_FOUND`

当前明显边界：

- 代码没有校验该订单是否属于当前登录会员
- 也没有校验当前用户是否有权查看该订单

#### 资损与安全关注

- 虽然它本身不改资金状态，但会暴露订单进度
- 如果订单号可枚举，存在越权查询风险

### 5.7 `GET /api/equity/exercise-url/{benefitOrderNo}`

#### 业务用途

返回一个行权链接和过期时间。

#### 处理流程

1. JWT 过滤器校验登录态
2. 按 `benefitOrderNo` 查询订单
3. 拼接固定格式的假定 URL
4. 返回“当前时间 + 1 天”的过期时间

#### 并发、幂等、事务

- 只读接口
- 天然幂等，但由于过期时间取当前时间，返回值不是严格恒定
- 无事务

#### 异常与风险

- 查不到订单返回 `ORDER_NOT_FOUND`

当前明显边界：

- 没有校验订单归属
- 没有校验订单是否处于允许行权状态
- 返回的是拼接 URL，不是下游真实签发链接

#### 资损与安全关注

- 如果后续接真实行权链路，这里会成为高风险入口
- 当前代码只是演示实现，不能直接视为生产级链接签发逻辑

### 5.8 `POST /api/callbacks/first-deduction`

#### 业务用途

接收齐为首扣结果回调。

#### 入参约束

- `requestId` 必填
- `benefitOrderNo` 必填
- `qwTradeNo` 必填
- `deductStatus` 必填
- `deductAmount` 必须大于 0

#### 处理流程

1. `SignatureInterceptor` 校验签名
2. `PaymentServiceImpl.handleFirstDeductCallback` 执行事务
3. 先按 `payment_record.request_id` 查重复
4. 再查 `idempotency_record` 是否已处理
5. 如果已处理，直接返回已有支付记录
6. 查询订单
7. 写一条 `payment_record`
8. 根据 `deductStatus` 更新订单状态机
9. 首扣成功或失败后都调用 `downstreamSyncService.syncOrder`
10. 更新订单主表
11. 写幂等记录
12. 返回支付处理结果

#### 关键处理

1. 支付流水留痕

- 每次回调都会沉淀 `payment_record`
- 便于对账、查单、争议排查

2. 状态机驱动

- 成功：
  - `order_status = FIRST_DEDUCT_SUCCESS`
  - `first_deduct_status = SUCCESS`
- 失败：
  - `order_status = FIRST_DEDUCT_FAIL`
  - `first_deduct_status = FAIL`

3. 首扣后同步下游

- 成功走直连继续
- 失败进入“可兜底”路径

#### 并发、幂等、事务

- 方法带事务
- 幂等依赖 `requestId`
- 同时使用 `payment_record` 和 `idempotency_record` 做双保险

需要注意：

- 当前仍然是“先查再写”模式
- 没有数据库唯一键保证 `payment_record.request_id` 唯一
- 高并发下主要依赖末尾写幂等记录时的主键约束回滚事务

#### 异常与风险

- 订单不存在会直接抛异常，不会吞掉
- 下游同步失败会让整个事务失败

#### 资损关注

- 这是直接影响资金状态的核心接口
- 当前代码没有校验 `deductAmount` 是否等于预期扣款金额
- 当前代码没有校验回调状态的时序是否合理
- 当前代码没有校验渠道交易号是否与历史记录冲突

### 5.9 `POST /api/callbacks/fallback-deduction`

#### 业务用途

接收兜底代扣结果回调。

#### 处理流程

整体与首扣回调相同，差异点在于：

- 写入的 `paymentType = FALLBACK_DEDUCT`
- 调用 `OrderStateMachine.applyFallbackResult`
- 不再触发下游同步

成功时：

- `order_status = FALLBACK_DEDUCT_SUCCESS`
- `fallback_deduct_status = SUCCESS`

失败时：

- `order_status = FALLBACK_DEDUCT_FAIL`
- `fallback_deduct_status = FAIL`

#### 并发、幂等、事务

- 与首扣回调一致
- 事务型处理
- 幂等键仍为 `requestId`

#### 异常与风险

- 订单不存在会直接失败
- 没有额外校验该回调是否真的对应“已发起兜底代扣”的订单

#### 资损关注

- 直接影响代扣成功/失败结论
- 如果乱序通知进入，当前代码缺少更严格的状态时序校验

### 5.10 `POST /api/callbacks/grant/forward`

#### 业务用途

接收放款结果通知。

#### 处理流程

1. 签名拦截器先校验签名
2. 先查幂等表，若已处理直接返回成功
3. 先写 `notification_receive_log`，状态为 `RECEIVED`
4. 查询订单
5. 如果订单不存在：
   - 将通知日志标记为 `FAILED`
   - 方法直接返回，不抛异常
6. 如果订单存在：
   - 更新 `grant_status`
   - 补齐 `loanOrderNo`
   - 若“放款成功且当前订单主状态为首扣成功”，则进入 `EXERCISE_PENDING`
7. 若“放款成功且当前主状态为首扣失败”，触发自动兜底代扣
8. 更新订单
9. 将通知日志标记为 `PROCESSED`
10. 写幂等记录

#### 关键处理

1. 先记通知，再做业务

- 这是通知类接口统一设计
- 即便业务失败，也能保留通知原始痕迹

2. 自动兜底代扣触发

- 条件是“首扣失败 + 放款成功”
- 不满足条件不会触发

#### 并发、幂等、事务

- 方法带事务
- 幂等依赖 `requestId`
- 自动触发兜底代扣也在同一事务语义内

#### 异常与风险

- 订单不存在时不会抛异常，而是吞掉业务失败，仅记录日志
- 这意味着调用方看到的是成功响应，但本地业务其实未处理成功

当前边界：

- 放款金额 `actualAmount` 当前只传给兜底代扣记录，不参与订单金额核对
- 如果放款成功通知早于首扣失败通知到达，自动兜底不会触发

#### 资损关注

- 放款成功是高资损节点
- 当前代码没有校验 `actualAmount` 与订单金额一致
- 通知乱序可能影响兜底流程是否及时发起

### 5.11 `POST /api/callbacks/repayment/forward`

#### 业务用途

接收还款状态通知。

#### 处理流程

1. 签名校验
2. 幂等检查
3. 记录通知日志
4. 查询订单
5. 订单不存在则将日志标记 `FAILED` 后返回
6. 订单存在则直接将日志标记 `PROCESSED`
7. 写幂等记录

#### 关键处理

当前代码中，这个接口只做“通知留痕 + 幂等记录”，不更新订单状态，也不写单独的还款记录表。

#### 并发、幂等、事务

- 方法带事务
- 幂等键为 `requestId`
- 逻辑简单，事务主要保证“日志状态 + 幂等记录”一致

#### 异常与风险

- 当前没有根据 `repaymentStatus`、`termNo`、`paidAmount` 更新任何业务模型
- `loanOrderNo` 也不做一致性校验

#### 资损关注

- 该接口当前更多是“审计占位”
- 如果进入生产化对账，它的信息粒度明显不够
- 目前无法支撑精细化的还款核销、逾期识别、还款对账

### 5.12 `POST /api/callbacks/exercise-equity`

#### 业务用途

接收行权结果通知。

#### 处理流程

1. 签名校验
2. 幂等检查
3. 记录通知日志
4. 查询订单
5. 订单不存在则日志标记失败并返回
6. 订单存在则根据 `exerciseStatus` 更新：
   - `exercise_status`
   - `order_status`
7. 更新订单
8. 日志标记成功
9. 写幂等记录

#### 并发、幂等、事务

- 方法带事务
- 幂等键为 `requestId`

#### 异常与风险

- 缺少对当前订单是否“允许行权”的前置状态校验
- 如果通知乱序或重复状态回退，当前状态机会直接覆盖结果

#### 资损关注

- 行权是实质业务结果节点
- 当前缺少更细的前置状态约束，生产化时容易引入状态污染

### 5.13 `POST /api/callbacks/refund`

#### 业务用途

接收退款结果通知。

#### 处理流程

2. 幂等检查
3. 记录通知日志
4. 查询订单
5. 订单不存在则日志标记失败并返回
6. 订单存在则根据 `refundStatus` 更新：
   - `refund_status`
   - `order_status`
7. 更新订单
8. 日志标记成功
9. 写幂等记录

#### 并发、幂等、事务

- 方法带事务
- 幂等键为 `requestId`

#### 异常与风险

- 当前不校验 `refundAmount` 是否与应退金额一致
- 也不校验是否已发生过成功退款

#### 资损关注

- 这是资金流出节点
- 当前代码缺少金额一致性校验和重复退款防控的精细规则

## 6. 支撑服务逻辑

### 6.1 协议任务与合同归档

`AgreementServiceImpl.ensureAgreementArtifacts` 会在下单后补齐两类协议：

- `EQUITY_AGREEMENT`
- `DEFERRED_AGREEMENT`

处理方式：

1. 按合同类型查是否已有签约任务
2. 没有则创建 `sign_task`
3. 再查是否已有合同归档
4. 没有则创建 `contract_archive`

关键点：

- 该服务也是事务方法
- 目标是让订单一创建就有完整协议追溯材料

### 6.2 下游同步

`DownstreamSyncServiceImpl.syncOrder` 由首扣回调触发。

处理方式：

1. 以 `benefitOrderNo + orderStatus` 生成同步请求号
2. 如果该请求号已处理，则跳过
3. 根据订单状态判断同步路线：
   - `FIRST_DEDUCT_SUCCESS` -> `DIRECT_CONTINUE`
   - `FIRST_DEDUCT_FAIL` 或 `FALLBACK_DEDUCT_PENDING` -> `FALLBACK_ELIGIBLE`
4. 必填字段缺失则更新 `sync_status=SYNC_FAIL` 并报错
5. 合法则更新 `sync_status=SYNC_SUCCESS`
6. 写幂等记录

关键边界：

- 这是“本地下游同步占位实现”
- 当前没有真正调用外部下游系统
- 只是更新状态并写一个 payload 字符串

### 6.3 自动兜底代扣

`FallbackDeductServiceImpl.triggerFallback` 由放款成功通知推动。

处理方式：

1. 先查是否已有 `FALLBACK_DEDUCT` 支付记录
2. 有则直接返回现有记录
3. 校验当前订单必须是 `FIRST_DEDUCT_FAIL`
4. 创建一条 `PENDING` 的兜底代扣支付记录
5. 更新订单为：
   - `order_status = FALLBACK_DEDUCT_PENDING`
   - `fallback_deduct_status = PENDING`

关键边界：

- 当前并不会真的调用代扣渠道
- 只是把“兜底代扣已触发”的事实沉淀到库里

## 7. 当前代码中的关键风险与待补点

### 7.1 并发与幂等

1. 幂等大多是“先查后插”，没有分布式锁，也没有乐观锁。
2. 并发重复请求主要靠数据库主键冲突回滚事务兜底。
3. `payment_record` 没有看到针对 `request_id` 的数据库唯一约束，支付回调在极端并发下存在重复写入窗口。

### 7.2 鉴权与越权

1. `GET /api/equity/orders/{benefitOrderNo}` 没有校验订单归属当前登录会员。
2. `GET /api/equity/exercise-url/{benefitOrderNo}` 也没有做订单归属校验。
3. 回调签名当前不覆盖 body，也没有 nonce 去重存储，抗重放能力有限。

### 7.3 状态时序

1. 放款通知、首扣通知、兜底通知如果乱序到达，当前状态机约束偏弱。
2. 行权、退款接口没有严格限制前置状态。
3. 还款通知没有进入正式状态模型，只做日志留痕。

### 7.4 金额与资损控制

1. 下单时没有校验 `loanAmount` 是否符合产品规则。
2. 首扣/兜底代扣回调没有校验回调金额与订单应扣金额一致。
3. 放款通知没有校验 `actualAmount` 与订单金额一致。
4. 退款通知没有校验退款金额和可退余额。

### 7.5 外部调用占位实现

1. 下游同步当前不是真实外部调用。
2. 兜底代扣触发当前不是真实渠道请求。
3. 行权链接当前是拼接 URL，不是真实签发。

## 8. 总结

当前代码已经形成一个可跑通的基线闭环：

- SSO 登录建档
- 本地 JWT 鉴权
- 产品查询
- 创建订单
- 协议归档
- 首扣结果处理
- 放款驱动兜底代扣
- 行权、退款、还款通知留痕

但从生产级角度看，当前实现仍然偏“基线版”：

- 资损控制规则不够细
- 并发与幂等主要依赖数据库兜底
- 部分接口缺少归属校验
- 若干外部交互仍是占位逻辑

后续如果你要逐条改这份文档，我建议优先讨论以下四块：

1. 订单查询/行权链接的归属校验
2. 支付、放款、退款金额一致性校验
3. 回调乱序与状态机补强
4. 幂等与重放防护的数据库约束完善
