# 2026-05-08 阿里云联调环境准备方案

## 1. 目标

今天开始和小花科技做真实联调，当前先落一个**可用、可观测、可快速改配置**的阿里云联调环境。

这次目标不是一步到位做生产级平台化，而是先把下面 4 件事做稳：

1. 后端能从本地 `MOCK / stub` 切到阿里云上的真实联调配置
2. H5 能在公网域名下正常访问，并能和后端联通
3. Yunka / 小花科技真实请求能打出去，日志能留证
4. 出问题时能快速判断是代码问题、配置问题，还是外部环境问题

---

## 2. 当前代码真值

先把现在仓库里的事实说清楚，避免联调时误判：

1. 后端是 `Spring Boot 3.2.12 + Java 17 + MySQL`
2. H5 不是纯静态站点，生产启动方式是：
   - `pnpm build`
   - `pnpm start`
3. 数据库表初始化已经在项目里：
   - `src/main/resources/db/schema.sql`
   - `application.yml` 已配置 `spring.sql.init.schema-locations`
4. Yunka、QW、auth 的关键配置都已经支持环境变量覆盖
5. 当前仓库**没有**现成部署资产：
   - 没有 `Dockerfile`
   - 没有 `docker-compose`
   - 没有现成 `nginx` 配置
   - 没有现成 `systemd` 服务文件

所以，今天最合适的路线不是先做容器化，而是先走：

1. 阿里云 `ECS`
2. ECS 本机 `MySQL`
3. 后端 `jar`
4. H5 `Node production server`
5. `Nginx` 做反向代理和 HTTPS

## 3. 推荐部署方案

### 3.1 先用最快可落地方案

建议先上这一版：

1. 1 台 ECS
2. ECS 本机 MySQL
3. ECS 上同时跑：
   - 后端 `8080`
   - H5 `3000`
   - `nginx` 对外暴露 `80/443`

这样做的原因很直接：

1. 成本低
2. 改配置快
3. 联调定位简单
4. 先把“真实请求能跑通”这件事做成，再考虑拆分

### 3.2 机器建议

联调环境建议最小配置：

1. ECS：`2C4G`
2. 磁盘：`40G` 以上
3. 系统：`Alibaba Cloud Linux 3` 或 `Ubuntu 22.04 LTS`
4. 本机数据库：`MySQL 8.0`

### 3.3 域名建议

至少准备两个域名，或者两个二级域名：

1. H5：
   - 例如 `equity-test.xxx.com`
2. 后端：
   - 例如 `equity-api-test.xxx.com`

如果你想省事，也可以先只用一个域名，通过 `nginx` 转发：

1. `/` -> H5 `3000`
2. `/api/` -> backend `8080`

这版更适合今天先跑通。

## 5. 联调环境配置原则

### 5.1 必须明确改掉的默认值

现在代码默认很多值还是本地或 mock 口径。只要上阿里云，就不能继续混着用。

今天现场要按下面真值收口：

1. 阿里云出口白名单 IP：`101.201.236.106`
2. Yunka 域名：`https://www.yongcar.com.cn`
3. 数据库：`127.0.0.1:3306 / nexusfin_equity / work`
4. `YUNKA_MODE=REST`
5. `QW_MODE=MOCK`
6. `TECH_PLATFORM_API_ENABLED=false`
7. `START_TECH_USER_STUB=0`

说明：

1. `YUNKA_BASE_URL` 今天必须按域名用，不要改成 IP。
2. 白名单是否已在 Yunka 侧生效，属于外部前置；ABS 本地代码无法替代这一步。
3. `MYSQL_PASSWORD` 当前现场值已确认，但不直接写进仓库文档；部署时按现场机密值填入。

至少要明确覆盖这些：

1. `SERVER_PORT=8080`
2. `MYSQL_HOST`
3. `MYSQL_PORT`
4. `MYSQL_DATABASE`
5. `MYSQL_USERNAME`
6. `MYSQL_PASSWORD`
7. `AUTH_COOKIE_SECURE=true`
8. `AUTH_JWT_SECRET`
9. `NEXUSFIN_APP_SECRET`
10. `YUNKA_MODE=REST`
11. `YUNKA_BASE_URL=https://www.yongcar.com.cn`
12. `YUNKA_GATEWAY_PATH=/api/gateway/proxy`
13. `YUNKA_CHANNEL_CODE=ABS`
14. `YUNKA_SIGN_SECRET=<现场配置，不要写回仓库>`

### 5.2 QW 配置怎么处理

这里要分两种情况：

1. 如果今天真实联调主线还是 `小花科技 / Yunka`，齐为先不切真：
   - `QW_MODE=MOCK`
2. 如果今天连 `benefiturl` 依赖的 QW 也要切真：
   - 再补真实 `QW_BASE_URL / QW_SIGN_KEY / QW_AES_KEY_BASE64 / 证书类配置`

当前建议是：

1. **先不要在第一天同时切 Yunka 真环境和 QW 真环境**
2. 优先把 `Yunka` 真联调先打通
3. `QW` 如果外部物料没齐，先继续保留 `MOCK`

原因：

1. `benefiturl` 现在虽然已经接线，但其运行时来源仍是 `QW exercise redirect URL`
2. 如果 Yunka 和 QW 同时切真，出问题时定位面会变大

### 5.3 科技平台配置原则

继续沿用当前已经定下的口径：

1. 艾博生不直接感知科技平台服务
2. 科技平台相关服务细节由云卡统一收口
3. 除非存在直接 `token / 回调 / 跳转` 边界，否则不向科技平台索要接口级细节
4. 今天允许单独保留的只有 ABS 自身 auth 直连边界，例如 `sso-callback`、`/api/users/me`

所以这次上阿里云时：

1. **不要默认去接科技平台真实服务**
2. `TECH_PLATFORM_BASE_URL` 只有在确认存在直连边界时再切真
3. 否则仍按当前边界处理，不把科技平台拉进来扩大问题面

## 7. 部署顺序

建议严格按下面顺序做，不要跳。

### 7.1 先建云资源

1. 创建 ECS
2. 准备 ECS 本机 MySQL
3. 配安全组
4. 配域名解析
5. 申请 HTTPS 证书

### 7.2 再装运行时

ECS 上安装：

1. `JDK 17`
2. `Maven`
3. `Node`
4. `pnpm`
5. `nginx`

### 7.3 再发代码

部署方式先用最直接的：

1. 拉代码
2. 后端执行：
   - `mvn -q -DskipTests package`
3. H5 执行：
   - `cd H5 && pnpm install`
   - `cd H5 && pnpm build`

### 7.4 再起数据库

1. 确认 ECS 本机 MySQL 已启动
2. 创建数据库 `nexusfin_equity`
3. 给应用账号 `work` 授最小必要权限
4. 首次启动让应用自动执行 `schema.sql`

### 7.5 再起后端

先用最朴素的命令启动：

```bash
java -jar target/nexusfin-equity-0.0.1-SNAPSHOT.jar
```

启动时通过环境变量读取真实配置。

### 7.6 再起 H5

```bash
cd H5
pnpm start
```

### 7.7 最后用 nginx 暴露

建议先做成：

1. `https://equity-test.xxx.com/` -> H5 `3000`
2. `https://equity-test.xxx.com/api/` -> backend `8080`

这样前端跨域、cookie、联调入口都更省事。

## 8. 联调当天必须看的启动日志

环境起完后，不要急着点页面，先看日志。

至少确认这几条：

1. `qw benefit client routing initialized mode=...`
2. `yunka gateway client initialized enabled=true mode=REST baseUrl=... gatewayPath=...`
3. `Started NexusfinEquityApplication`

判定规则：

1. 如果 `Yunka mode` 还是 `MOCK`，说明环境根本没切对
2. 如果 `baseUrl` 还是 `127.0.0.1:18081`，说明还在打本地 stub
3. 如果 `baseUrl` 还是 `https://yunka.local`，说明真实地址没配进去
4. 如果 `QW mode` 不是你预期值，要在联调记录里明确写出来

## 9. 联调 smoke 顺序

阿里云环境起来后，按这个顺序做最小验证：

### 9.1 基础健康检查

1. 打开 H5 首页
2. 请求后端：
   - `GET /api/equity/health`

### 9.2 认证和联登

1. 验证 `joint-login`
2. 验证 cookie 能正常写入
3. 验证跳转页能打开
4. token 是否有效，以实际调用返回值判断，不按本地解析 JWT `exp` 预判

### 9.3 核心主链

1. `POST /api/loan/calculate`
2. `GET /api/benefits/card-detail`
3. `POST /api/benefits/activate`
4. `GET /api/repayment/info/{loanId}`

### 9.4 关键日志

每条链路至少能回查到：

1. `traceId`
2. `path`
3. `requestId`
4. `bizOrderNo`
5. `requestBodyJson`
6. `responseBodyJson`
7. `elapsedMs`
8. `errorNo / errorMsg`

如果阿里云环境里看不到完整出站日志，这个环境还不能算“可联调”。

## 10. 今天最容易踩的坑

### 10.1 cookie 问题

公网 HTTPS 环境下：

1. `AUTH_COOKIE_SECURE` 必须是 `true`

本地 HTTP 环境下：

1. `AUTH_COOKIE_SECURE` 才需要 `false`

不要把本地值原样带上云。

### 10.2 Yunka 路径真值

当前本地已经证明 `/loan/trial` 可用，但外部文档历史上有 `/loan/trail` 歧义。

所以阿里云联调前，必须让云卡给最终口径，不要自己猜。

### 10.3 benefiturl 语义

要继续按现在的真值对外说明：

1. `benefiturl` 已经接入主链并真实出站
2. 当前运行时来源是 `QW exercise redirect URL`
3. 不能把它说成“所有权益场景通用联登 URL”

### 10.4 不要默认接科技平台真实服务

除非云卡明确说存在必须直连边界，否则：

1. 不去接科技平台 token 校验接口
2. 不去接科技平台用户信息接口
3. 不去扩展科技平台内部 path / 域名 / 签名细节

### 10.5 token 判断口径

今天先按运行时证据说话：

1. 不要因为看到 JWT `exp` 字段就直接下结论说 token 已过期。
2. 以 ABS 调 Yunka 后的真实返回为准。
3. 联调时结合 `joint-login` 响应、页面落点、backend 日志一起判断。

## 12. 今天主决策结论

今天不建议一上来就做复杂平台化。

当前最稳的决策是：

1. 先用 **1 台 ECS + 本机 MySQL + nginx + backend jar + H5 Node** 起联调环境
2. 先把 Yunka 真实联调打通
3. `QW` 没必要和 `Yunka` 在第一天同时切真
4. 科技平台继续按“默认不直连”处理
5. 先补部署资产和 smoke 清单，再上云

如果今天就要开始动，我建议执行顺序就是：

1. 先开阿里云 ECS，并确认本机 MySQL 与 Yunka 白名单前置
2. 让开发会话补部署资产
3. 让测试会话补阿里云 smoke 清单
4. 资源和文档齐了以后，再正式上云部署
