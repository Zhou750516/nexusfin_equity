# 2026-03-31 出站接口联调执行说明

## 1. 目标

本文档用于为 `M3.1` 出站接口联调提供固定执行顺序、通过标准和观察点，确保外部条件一旦到位，可以直接按清单执行，而不是临场再决定怎么联调。

## 2. 联调顺序

建议严格按以下顺序推进：

1. `abs.push.order`
2. `abs.token.get`
3. `abs.lending.notify`

原因：

1. `abs.push.order` 是下单主链起点，不打通后续链路无意义。
2. `abs.token.get` 依赖前置订单和会员上下文，更适合在建单链路确认后再验证。
3. `abs.lending.notify` 依赖放款结果推进，适合放在第三步。

## 3. 接口级执行说明

### 3.1 `abs.push.order`

**本地入口**

- `BenefitOrderServiceImpl.createOrder()`

**联调输入**

1. 已认证会员
2. 可用 `productCode`
3. 合法 `loanAmount`
4. 可复用的 `requestId`
5. 对应齐为所需会员、手机号、产品和支付协议号字段

**成功判定**

1. 我方创建订单成功
2. 齐为返回有效订单号
3. 本地订单记录到齐为侧返回的订单 / 卡信息
4. 日志中可串起 `traceId + bizOrderNo`

**失败判定**

1. 齐为返回业务失败码
2. HTTP 调用超时或网络失败
3. 返回报文格式不符或解密失败
4. 本地误把失败结果当成功落库

**需要保留的证据**

1. 请求报文或关键字段截图
2. 响应报文或关键字段截图
3. 本地订单落库结果
4. 对应应用日志

### 3.2 `abs.token.get`

**本地入口**

- `BenefitOrderServiceImpl.getExerciseUrl()`

**联调输入**

1. 已存在的有效 `benefitOrderNo`
2. 订单关联的齐为用户上下文

**成功判定**

1. 齐为返回真实 `redirectUrl`
2. 返回 URL 可被前端或测试页消费
3. 本地接口返回值完整，含过期时间或等价字段

**失败判定**

1. 订单不存在或关联上下文缺失
2. 齐为返回业务失败
3. URL 为空或结构不符合预期

**需要保留的证据**

1. 请求参数
2. 返回的 `redirectUrl`
3. 应用日志
4. 对应订单号

### 3.3 `abs.lending.notify`

**本地入口**

- `NotificationServiceImpl.handleGrant()`

**联调输入**

1. 已存在的有效订单
2. 放款成功或失败通知样例
3. 齐为要求的订单号和状态字段

**成功判定**

1. 我方向齐为发送放款通知成功
2. 齐为返回有效响应
3. 本地日志能关联到订单号和通知结果

**失败判定**

1. 齐为返回业务拒绝
2. 请求超时或网络失败
3. 我方在失败时没有明确日志和错误分类

**需要保留的证据**

1. 发送时的关键请求字段
2. 齐为返回结果
3. 本地通知处理日志
4. 订单状态变化结果

## 4. 联调观察点

联调当天建议重点看以下信息：

1. `traceId`
2. `bizOrderNo`
3. 齐为 method 名称
4. 请求是否命中 `HTTP` 模式
5. 上游返回码 / 错误信息
6. 本地订单状态是否发生预期变化

## 5. 最小通过标准

`M3.1` 出站接口联调可认为通过，需要同时满足：

1. 三个接口都至少完成一轮真实环境调用。
2. 每个接口都有成功或失败证据留存。
3. 日志中可通过 `traceId` 或 `bizOrderNo` 回溯完整调用链。
4. 失败场景能区分是参数问题、网络问题还是协议问题。

## 6. 备注

当前文档只覆盖出站接口联调，不包含：

1. 齐为入站回调联调
2. 科技平台 SSO 真联调闭环
3. 异常回放与重放攻击验证

这些内容应在 `M3.2` 和 `M3.3` 阶段单独执行。
