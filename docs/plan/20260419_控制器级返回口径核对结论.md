# 2026-04-19 控制器级返回口径核对结论

## 1. 目的

本文档用于核对当前“异步补偿已接入后”的用户接口返回口径，重点确认：

1. 同步失败时，前端到底看到什么
2. 哪些场景会进入异步补偿
3. 当前代码实现是否与已确认业务口径一致

本轮重点只核对两条主链路：

1. 权益购买同步失败 / 超时
2. 用信提交同步失败 / 超时

## 2. 核对范围

### 2.1 相关接口

1. `POST /api/loan/apply`
2. `POST /api/equity/orders`

### 2.2 相关补偿任务

1. `QW_BENEFIT_PURCHASE_RETRY`
2. `YUNKA_LOAN_APPLY_RETRY`

## 3. 当前已确认业务口径

### 3.1 用信提交同步超时

已确认口径：

1. 用户点击前端页面
2. 艾博生前端调用后端接口
3. 后端内部调用云卡（用信提交）超时
4. 后端进入异步补偿流程
5. 前端页面应展示“已提交”或“审核中”

### 3.2 权益购买同步超时 / 报错

已确认口径：

1. 若用户购买权益时，齐为接口超时或报错
2. 需要写入异步补偿任务

本轮**未收到“前端必须展示审核中”的额外业务确认**，因此当前仍按“同步失败，但补偿任务保留”的口径承接。

## 4. 当前代码与测试核对结果

## 4.1 场景一：权益购买同步超时（QW 超时）

### 当前行为

当前路径为：

1. `LoanServiceImpl.apply()`
2. 调用 `BenefitOrderServiceImpl.createOrder()`
3. `QwBenefitClient.syncMemberOrder()` 超时
4. 本地将权益订单 `syncStatus` 更新为 `SYNC_FAIL`
5. 写入 `QW_BENEFIT_PURCHASE_RETRY` 补偿任务
6. 接口继续向上抛出 `QW_SYNC_TIMEOUT`
7. 控制器最终返回失败 `Result`

### 当前前端看到的结果

接口返回：

1. `code = -1`
2. `message` 包含 `QW_SYNC_TIMEOUT`
3. `data = null`

### 本轮新增修复结论

本轮新增集成测试发现一个关键问题：

1. 原实现中，`loan/apply` 外层事务会把“权益订单 + 补偿任务”一起回滚
2. 导致虽然接口报了 `QW_SYNC_TIMEOUT`，但本地没有保留可补偿的数据

现已完成修复：

1. 为“权益购买同步超时但需要保留补偿”的场景引入专用异常
2. 在 `BenefitOrderServiceImpl.createOrder()` 与 `LoanServiceImpl.apply()` 上对该异常关闭事务回滚
3. 现在该场景下：
   - 接口仍同步返回失败
   - 但权益订单和补偿任务会保留

### 测试证据

已通过：

1. `Phase9TaskGroupECompensationIntegrationTest`
2. `Phase9TaskGroupEIntegrationTest`
3. `BenefitOrderServiceTest`

### 结论

当前实现与“要入异步补偿任务”的业务要求**一致**，但与“前端展示审核中”无关；当前前端仍会看到同步失败，这是当前阶段可接受的口径。

## 4.2 场景二：权益购买成功，但云卡用信提交失败（非超时）

### 当前行为

当前路径为：

1. 权益购买成功
2. 调用云卡 `loan/apply`
3. 云卡明确返回失败或空响应
4. 后端返回业务态 `loan_failed`

### 当前前端看到的结果

接口返回：

1. `status = loan_failed`
2. `benefitsActivated = true`
3. `message = 权益购买成功，借款申请失败：...`
4. 不再返回可继续查询结果页的 `applicationId`

### 结论

该场景与已确认业务口径**一致**：

1. 明确告诉用户借款失败
2. 同时明确权益已购买成功

## 4.3 场景三：权益购买成功，但云卡用信提交超时

### 当前行为

当前路径为：

1. 权益购买成功
2. 调用云卡 `loan/apply`
3. 云卡调用超时
4. 本地保存 `applicationId -> loanId` 映射，状态为 `PENDING_REVIEW`
5. 写入 `YUNKA_LOAN_APPLY_RETRY` 补偿任务
6. 同步接口直接返回“已提交 / 审核中”

### 当前前端看到的结果

接口返回：

1. `status = pending`
2. `benefitsActivated = true`
3. `message = 借款申请已提交，正在审核中`

### 结论

该场景与已确认业务口径**完全一致**。

## 5. 本轮核对后的总体结论

### 5.1 已一致的口径

以下口径已经与业务确认一致：

1. **权益成功 + 云卡失败（非超时）**
   - 前端显示借款失败
   - 同时明确权益已成功
2. **权益成功 + 云卡超时**
   - 前端显示“已提交 / 审核中”
   - 后端进入异步补偿

### 5.2 已修复的一处关键实现偏差

以下问题已完成修复：

1. **权益购买超时时，补偿任务和权益订单被事务一起回滚**
2. 现已调整为：
   - 接口仍同步失败
   - 但本地权益订单和补偿任务会保留

### 5.3 当前仍需明确但不阻塞本轮推进的点

当前仍有一个**可继续观察、但不阻塞今天收口**的点：

1. **权益购买同步超时场景，前端是否未来也要调整为“处理中 / 稍后查看”**

目前没有明确业务确认要求该场景改成 `pending`，因此本轮保持：

1. 同步失败
2. 后端保留补偿任务

## 6. 推荐对外统一表述

当前建议统一对外表述如下：

1. **权益购买同步超时**
   - 当前仍按同步失败承接
   - 但后端已保留补偿任务，可继续补偿处理
2. **权益购买成功后，用信提交超时**
   - 当前按“已提交 / 审核中”承接
   - 后端进入异步补偿
3. **权益购买成功后，用信提交明确失败**
   - 当前按“借款失败，但权益已成功”承接

## 7. 一句话结论

当前控制器级返回口径已基本与业务确认一致；本轮额外修复了一处关键事务问题，确保“权益购买同步超时”场景下虽然接口仍返回失败，但本地补偿数据不会再丢失。
