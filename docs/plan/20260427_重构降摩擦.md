## 二、根因归一

把以上压缩成 4 条结构性原因，每一条都直接对应"Codex 越改越慢"：

1. **认知半径过大**：plan 文档体量、双轨规划、AGENTS.md 1.7k 行单文件 → 每次 Codex 入场要"先理解现状"才能动手。
2. **横切关注点没抽出**：日志 / 助手 / Yunka 调用模板 / UUID / 金额转换在多处复刻 → 一个口径改动要改 N 个地方，Codex 经常漏点。
3. **垂直切片不够纯**：业务 service 把 DTO 校验、上游协议、补偿入队、I18n、日志、状态机映射、Repo 全揽 → 单文件 500–700 行，diff 半径大。
4. **第三方 / UI 库依赖未裁剪**：`thirdparty/qw/` 过细拆 + skeleton 平行实现；shadcn 全套与 i18n 单文件 → 项目搜索 / 类型推导 / Codex 检索都被拖慢。

## 三、分阶段重构 plan

每一阶段都给出**目的**、**动作**、**完成判据**（Codex 迭代速度可以观测的指标）。建议按顺序做，优先级从高到低；前一阶段不收口不进下一阶段。

> 总体目标度量（建议在动手前打基线，之后每阶段对一次）：
> - **P50 文件长度** 后端 < 250 行 / 前端 < 300 行
> - **重复 helper 数** 0
> - **plan 目录入场文件** ≤ 5
> - **后端 IT context 风味数** ≤ 3
> - 每个迭代回合 Codex 平均消耗 token 下降 30%+

---

### Phase 0 · 认知半径瘦身（1–2 天，高 ROI）

**目的**：让 Codex 进任何任务都只读 ≤ 5 份顶层文档。

**动作**
1. `docs/plan/` 仅保留：`README.md`（重写为活索引）+ 当周 plan + 当周 checklist + 1 份"决策汇编"（per topic 一节）。其余迁到 `docs/plan/_archive/<yyyy-mm>/`。
2. `specs/004-benefit-loan-flow/` 整体迁到 `docs/_archive/specs-004/`，README 内显式标注非主线（与 4-26 plan 一致）。
3. AGENTS.md 拆成 3 份：
   - `AGENTS.md`（30 行入口，列禁止事项 + 入口链接）
   - `docs/AGENTS_BACKEND.md`、`docs/AGENTS_H5.md`（细则各自归位）
4. 增加 `.codexignore` / `.aiignore`（如果用的工具支持），把 `docs/_archive/`、`docs/plan/_archive/`、`thirdparty/qw/demo/` 排除。
5. `pom.xml` 的 `<excludes>` 文件直接物理删 / 移到 `docs/third-part/qw-demo/`，不再放在 `src/main/java/`。

**完成判据**
- `find docs/plan -maxdepth 1 -type f | wc -l ≤ 5`
- AGENTS.md ≤ 60 行
- `src/main/java/com/nexusfin/equity/thirdparty/qw/demo/` 不存在

### Phase 1 · 后端横切抽取（2–3 天，机械性强、可让 Codex 执行）

**目的**：消灭重复 helper 与日志 boilerplate，让单业务变更落到 ≤ 50 行 diff。

**动作**
1. 新增 `util/MoneyUnits.java`：集中 `yuanToCent / centsToYuan / CENTS_PER_YUAN`，删掉 `LoanServiceImpl`、`RepaymentServiceImpl` 等的私有副本。
2. 新增 `util/BizIds.java`：集中 `newCompactUuid()`、统一前缀策略（`LC- / LA- / RT- / LQ-` 走 enum）。
3. 新增 `service/support/YunkaCallTemplate.java`：把"构 `YunkaGatewayRequest` → `proxy()` → 三段日志 → `requireSuccessfulYunkaData`"包成 `<R> R execute(YunkaCall<R> call)`。所有 service 改用模板。
4. 引入 `@GatewayCall` AOP 或在模板内打日志，去掉手写的 `traceId/bizOrderNo/requestId/path/elapsedMs` 重复块。
5. `RepaymentServiceImpl`、`LoanServiceImpl`、`XiaohuaGatewayServiceImpl`、`YunkaLoanApplyCompensationExecutor` 全量改造，跑 IT 验证。

**完成判据**
- `grep "newCompactUuid\|yuanToCent\|elapsedMs(" src/main/java -rn | wc -l` 显著下降；service/impl 仅引用而不定义
- `LoanServiceImpl` ≤ 350 行、`RepaymentServiceImpl` ≤ 300 行
- 所有 yunka 相关 IT 通过

---

### Phase 2 · 后端纵向拆分（3–4 天）

**目的**：把"god service"按动词拆，单一职责，下次 Codex 只需打开一个文件。

**动作**
1. `LoanServiceImpl` 拆为：
   - `LoanCalculatorService`（calculator-config + calculate）
   - `LoanApplicationService`（apply + 入队补偿）
   - `LoanApprovalQueryService`（approval-status + approval-result + repay-plan 查询）
   - 共享部分（`LoanApplicationMapping` CRUD、status 映射）下沉为 `LoanApplicationGateway`。
2. `RepaymentServiceImpl` 同样拆 `RepaymentInfoService` / `RepaymentSmsService` / `RepaymentSubmitService`。
3. 把内嵌的 `LoanApplyForwardData` / `LoanQueryForwardData` / `RepayTrialForwardData` 等 `record` 集中到 `thirdparty/yunka/payload/`，确保 Yunka 协议字段只在一处定义。
4. `AsyncCompensationEnqueueService.enqueue(...)` 当前要拼一个非常长的 `"""{...}"""` JSON。改成 strongly typed `EnqueuePayload.YunkaLoanApply(...)`，由 enqueue service 内部序列化。
5. 把 `XiaohuaGatewayServiceImpl` 与 `YunkaGatewayClient` 的职责边界写进类注释（30 行内）：谁负责协议、谁负责业务编排。

**完成判据**
- 任一 `service/impl/*.java` ≤ 350 行
- `grep -n '""".*{' src/main/java/com/nexusfin/equity/service/impl | wc -l == 0`（不再有内嵌 JSON 模板）
- IT 全绿

## 五、风险与建议

- **风险 1：拆 god service 引入回归。** → Phase 1 先把 IT 跑稳并锁版，再做 Phase 2，每个子 service 一个 PR。
- **风险 2：删 shadcn 组件可能误删。** → 用 `grep "from \"@/components/ui/<name>\"" -r` 全量验证再删。
- **风险 3：i18n 拆分破坏键路径。** → key 不变，只拆文件，单测覆盖 `useI18n.t(key)` 命中即可。
- **建议**：动手前给 `target/` 之外建一份"基线指标"文件（最大文件长度 / 重复 helper 数 / `mvn test` 耗时 / Codex 平均 token），完成每阶段后对一次，避免"感觉变快了"无证据。
