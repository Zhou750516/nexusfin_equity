# 2026-04-03 支付协议号状态二阶段改造点清单

## 1. 文档目的

本文档用于承接 `2026-04-03` 当天已经沉淀完成的“支付协议号状态定义与主流程取值规则”，进一步拆出后续第二阶段需要补的代码与文档改造点。

补充说明：

1. 当前第二阶段改造点不绑定“通联必须是最终支付渠道”这一前提
2. 该清单主要围绕“用户级协议号状态流转能力”展开
3. 即使后续支付渠道从通联调整为其他渠道，本文档的大部分改造点仍可继续复用

当前阶段已经完成的是：

1. 真实协议号主链最小闭环
2. `ACTIVE` 协议号回写与读取
3. 测试 override 的环境边界保护

当前阶段尚未覆盖的是：

1. 非 `ACTIVE` 状态回写
2. 有效协议号失效化处理
3. 历史协议与当前协议切换策略

## 2. 当前已实现边界

截至今天，当前代码边界明确如下：

1. `PaymentProtocolCallbackServiceImpl` 当前仅接受 `ACTIVE`
2. `PaymentProtocolServiceImpl` 当前仅查询 `ACTIVE`
3. 主流程无有效协议号时：
   - 测试环境允许按受控条件使用 override
   - 正式环境直接失败

因此第二阶段改造的目标，不是重做第一轮结构，而是在现有结构上补齐“状态流转和失效处理”。

## 3. 建议的二阶段目标

建议第二阶段聚焦以下三件事：

### 3.1 目标一：支持非 `ACTIVE` 状态回写

后续建议支持至少以下状态的回写：

1. `INACTIVE`
2. `EXPIRED`
3. `REVOKED`

目标价值：

1. 让支付侧能同步用户协议状态变化
2. 避免本地长期把已失效协议继续当作有效协议

### 3.2 目标二：补齐当前有效协议号失效化

当收到：

1. `EXPIRED`
2. `REVOKED`

时，建议补以下逻辑：

1. 更新当前协议状态
2. 设置 `expiredTs` 或对应失效时间
3. 让主流程后续不再将其视为可用协议

### 3.3 目标三：明确历史协议与当前协议切换规则

后续建议明确：

1. 一个用户若再次签约，是否覆盖旧协议
2. 是否允许保留多条历史记录
3. 当前“生效中的唯一协议”如何识别

## 4. 推荐改造点

## 4.1 回写接口层

建议改造：

- `src/main/java/com/nexusfin/equity/service/impl/PaymentProtocolCallbackServiceImpl.java`

建议内容：

1. 将当前“只允许 `ACTIVE`”的硬编码判断调整为可识别多状态
2. 根据不同状态分发不同处理逻辑

建议分支如下：

1. `ACTIVE`：
   - 正常写入 / 覆盖当前有效协议
2. `INACTIVE`：
   - 记录状态，但不允许主流程使用
3. `EXPIRED`：
   - 标记失效
   - 清理当前可用口径
4. `REVOKED`：
   - 标记撤销
   - 清理当前可用口径

## 4.2 协议号服务层

建议改造：

- `src/main/java/com/nexusfin/equity/service/impl/PaymentProtocolServiceImpl.java`

建议内容：

1. 新增“保存非激活状态”的处理逻辑
2. 新增“失效当前协议”的处理逻辑
3. 明确再次签约时如何更新当前记录

建议最小新增能力：

1. `saveActiveProtocol`
2. `saveInactiveProtocol`
3. `expireProtocol`
4. `revokeProtocol`

## 4.3 数据层

建议补充确认：

1. `member_payment_protocol` 是否只维护一条当前记录
2. 是否需要增加历史流水表
3. 若暂不加历史表，是否至少保留状态更新时间与失效时间

当前建议：

1. 第二阶段先不扩大到独立历史表
2. 仍在现有表结构内完成最小闭环
3. 若后续审计要求提升，再独立拆历史表

## 4.4 文档层

建议同步更新：

1. `docs/third-part/20260403_通联服务对接流程与上线切换说明.md`
2. `docs/third-part/齐为/20260401_齐为线上部署说明.md`
3. 若进入实现，再补一份状态流转专项说明

## 5. 推荐执行顺序

建议后续第二阶段按以下顺序推进：

1. 先补状态语义与回写规则确认
2. 再改回写接口和服务层
3. 再补测试
4. 最后更新文档与上线检查口径

## 6. 当前建议

今天不建议继续直接落第二阶段代码，原因如下：

1. 当前第一轮闭环刚完成，需要先把正式文档和边界沉淀稳定
2. 第二阶段改造会扩大到状态流转语义，需要先确认支付侧是否确实会回写非 `ACTIVE` 状态
3. 当前最优先的生产阻塞仍是“真实协议号来源接通”，不是状态流转扩边界

## 7. 一句话结论

> 第二阶段改造的核心，不是继续补更多 override，而是让“协议号状态变化”可以被正式接收、落库、失效化，并与主流程解耦。
