## 4. 现有实现与 H5 目标接口的详细差异

### 4.1 已有 Controller 与目标接口差异

当前已有 Controller 主要为：

1. `AuthController`
2. `BenefitOrderController`
3. `NotificationCallbackController`
4. `PaymentCallbackController`
5. `PaymentProtocolCallbackController`
6. `HealthController`

当前尚不存在以下 H5 目标 Controller：

1. `LoanController`
2. `BenefitsController`
3. `RepaymentController`

结论：

- `Phase 9` 的 10 个 H5 接口中，除认证相关能力外，目标 Controller 基本都需要新增。

### 4.2 权益相关能力的复用判断

#### 接口 `GET /api/benefits/card-detail`

可复用点：

1. 当前已有权益产品实体 `BenefitProduct`
2. 当前已有产品查询能力 `BenefitOrderService.getProductPage()`
3. 当前已有权益产品页接口 `/api/equity/products/{productCode}`

主要差异：

1. 现有 `ProductPageResponse` 只包含 `productCode`、`productName`、`feeRate`、`agreements` 等字段
2. H5 需要的返回结构包含：
   - `cardName`
   - `price`
   - `totalSaving`
   - `features`
   - `categories`
   - `tips`
   - `protocols`
3. 现有数据库实体 `BenefitProduct` 也不足以完整覆盖 H5 所需展示字段

结论：

- 该接口不能直接复用现有 `/api/equity/products/{productCode}` 对外暴露。
- 可复用现有产品数据与服务入口，但仍需新增 H5 专用 DTO 和适配逻辑。
- 若现有数据库没有足够字段，建议部分展示内容先由本地配置承载，而不是阻塞在数据库改造上。

#### 接口 `POST /api/benefits/activate`

可复用点：

1. 当前已有 `BenefitOrderService.createOrder()`
2. 当前已有齐为同步下单 `QwBenefitClient.syncMemberOrder()`
3. 当前已有幂等处理、协议归档、支付协议解析、订单落库能力

主要差异：

1. 现有对外接口是 `/api/equity/orders`
2. 现有请求体 `CreateBenefitOrderRequest` 为：
   - `requestId`
   - `productCode`
   - `loanAmount`
   - `agreementSigned`
3. H5 目标请求体为：
   - `applicationId`
   - `cardType`
4. H5 目标响应体为：
   - `activationId`
   - `status`
   - `message`

结论：

- 主业务能力可复用，但不能直接把现有接口暴露给 H5。
- 需要新增 `BenefitsService.activate()` 作为 H5 适配层，把 H5 参数转换为现有下单主链所需参数。

### 4.3 借款与还款相关能力的差异

当前代码中尚未看到以下能力：

1. H5 借款试算接口
2. H5 借款申请接口
3. H5 审批进度查询接口
4. H5 审批结果查询接口
5. H5 还款详情接口
6. H5 还款提交接口
7. H5 还款结果接口
8. 云卡 gateway 客户端
9. `applicationId` 映射实体 / 仓储 / 服务

结论：

- 贷款与还款 7 个接口当前均属于新增开发范围。
- 其中核心新增不只是 Controller，而是“云卡统一 gateway 封装 + `applicationId` 映射持久化 + H5 DTO 映射”。

### 4.4 云卡转发基础设施差异

当前第三方调用模式主要有两类：

1. `TechPlatformUserClientImpl`
   - 适合参考其 `RestClient + 超时配置 + 重试 + 异常转换` 模式
2. `QwBenefitClientImpl`
   - 适合参考其“客户端接口 + 实现类 + 路由代理”的分层方式

当前尚未具备的云卡能力：

1. 云卡基础域名配置
2. 云卡统一 gateway URI 配置
3. `path + data + requestId + bizOrderNo` 的统一请求封装
4. 云卡统一响应解包与错误映射
5. `/loan/trail`、`/loan/query`、`/loan/apply`、`/repay/trial`、`/repay/apply`、`/repay/query` 的路径配置

结论：

- 任务组 B 基本为全新增，但可沿用现有第三方客户端实现范式。

### 4.5 鉴权、统一返回、迁移能力的对齐判断

#### `9.5 请求校验`

- 当前项目已普遍使用 `@Valid` 和 `jakarta.validation`
- 新增接口只需延续现有风格

结论：基础能力已具备，但新接口 DTO 仍需逐个补齐注解。

#### `9.6 JWT 认证过滤器`

- 当前 `JwtAuthenticationFilter` 已存在
- 需要做的是把新 H5 接口路径加入受保护路径配置

结论：该项不属于从零实现，而是配置纳管。

#### `9.7 统一响应格式`

- 当前 `Result<T>` 已满足 Phase 9 统一返回要求

结论：已实现，无需新建响应包装规范。

#### `9.8 数据库迁移`

- 当前项目已有 `SchemaMigrationRunner`
- 已存在通过程序启动时自动补齐字段与索引的做法

结论：

- Phase 9 如需新增 `applicationId` 映射表或字段，可以延续现有迁移模式。
- 不建议仅为本轮任务改造成 Flyway/Liquibase，除非你希望统一全项目迁移机制。

## 6. 可直接执行的开发清单

建议按以下顺序执行：

### 第一阶段：先完成最小可交付骨架

1. 新增 `LoanController`
2. 新增 `BenefitsController`
3. 新增 `RepaymentController`
4. 新增 H5 专用服务接口：
   - `LoanService`
   - `BenefitsService`
   - `RepaymentService`
5. 新增 H5 目标 DTO 目录与基础类

交付目标：

- 先把 `Phase 9` 的目标接口骨架立起来，与当前 `/api/equity/*` 老接口边界分离。

### 第二阶段：完成任务组 A

1. 实现 `GET /api/loan/calculator-config`
   - 使用本地配置
2. 实现 `GET /api/benefits/card-detail`
   - 优先复用 `BenefitProduct` 基础数据
   - 不足字段用本地配置补齐
3. 实现 `POST /api/benefits/activate`
   - H5 入参适配到现有 `BenefitOrderService.createOrder()`
4. 补充单元测试 / Controller 集成测试

交付目标：

- H5 首页与权益页先可联调，不依赖云卡服务就绪。

### 第三阶段：完成任务组 B

1. 新增云卡配置类
2. 新增云卡 gateway 客户端接口与实现
3. 统一请求结构：
   - `requestId`
   - `path`
   - `bizOrderNo`
   - `data`
4. 把 `/loan/trail` 等路径抽到配置项
5. 编写客户端封装测试

交付目标：

- 云卡服务即使尚未完工，艾博生侧也能先完成文档契约下的请求封装。

### 第四阶段：完成任务组 C + D

1. 新增 `applicationId` 映射实体 / 仓储 / 服务
2. 实现 `POST /api/loan/calculate`
3. 实现 `GET /api/loan/approval-status/{applicationId}`
4. 实现 `GET /api/loan/approval-result/{applicationId}`
5. 实现 `GET /api/repayment/info/{loanId}`
6. 实现 `POST /api/repayment/submit`
7. 实现 `GET /api/repayment/result/{repaymentId}`

执行重点：

1. `approval-status` 与 `approval-result` 共用云卡 `/loan/query`
2. `repayment/result/{repaymentId}` 不本地生成 `repaymentId`
3. 所有返回都映射成 H5 DTO，而不是透传云卡原始报文

### 第五阶段：最后完成 `loan/apply` 编排

1. 本地生成 `applicationId`
2. 落库保存 `applicationId` 映射
3. 先调用权益购买
4. 权益成功后调用云卡 `/loan/apply`
5. 明确处理“齐为成功、云卡失败”的部分成功语义
6. 补充编排顺序测试、部分成功测试

交付目标：

- 完成 Phase 9 最复杂的复合接口闭环。

## 7. 一句话结论

> 当前项目已具备统一返回、鉴权、异常处理、齐为权益主链和第三方调用范式，但 `Phase 9` 的 10 个 H5 接口整体上仍以新增开发为主；可复用的重点在于“权益下单主链”和“第三方客户端实现模式”，建议按任务组 A → B → C/D → E 顺序推进。
