## 2. 已确认事实

当前已确认的业务事实如下：

1. 放款成功后的可行权权益提醒 push 由科技平台触发
2. push 可从科技平台 App 跳转到艾博生分发页面
3. 艾博生分发页面再跳转到齐为供应商页面
4. 权益订单页行权入口跳转到艾博生页面
5. 权益包退款入口跳转到艾博生页面
6. 权益状态变更需要推送给科技平台
7. 联合登录采用“科技平台携带 token 跳转艾博生，艾博生后端再调小花后端”的方案
8. 联合登录核心入口本地骨架已落地：
   - `/api/auth/joint-login`
   - `/joint-entry`
   - `/joint-dispatch`
   - `/joint-refund-entry`

基于以上事实，可先明确：

1. 分发页、退款页和状态回推都应复用联合登录的用户身份口径
2. 当前已落地的 `/joint-dispatch` 与 `/joint-refund-entry` 可视为正式方案的占位入口
3. 在外部契约未到位前，不应直接进入真实供应商跳转、真实退款申请或真实状态回推实现

## 3. 总体设计原则

### 3.1 统一入口原则

科技平台跳艾博生的外部入口建议统一先落到联合登录承接页，再根据 `scene` 分流到具体业务页。

推荐统一入口链路：

```text
科技平台 -> /joint-entry -> /api/auth/joint-login -> 目标业务页
```

### 3.2 身份复用原则

分发页、退款页、状态回推均统一复用以下三类上下文：

1. 科技平台 / 小花用户唯一标识
2. 艾博生本地用户标识
3. 当前权益订单 / 场景上下文

### 3.3 页面与接口解耦原则

当前阶段先把页面职责、入参来源、后端接口边界定义清楚，不把：

1. 齐为真实跳转 URL
2. 退款正式接口
3. 科技平台正式状态回推协议

直接写死在当前骨架中。

### 3.4 可渐进扩展原则

当前方案应支持以下演进路径：

1. 先是占位页可联调
2. 再替换真实供应商跳转 / 真实退款流程
3. 最后补状态回推与完整监控日志

## 4. 分发页方案骨架

### 4.1 页面职责

分发页建议定义为“权益行权统一分发页”，负责承接：

1. 可行权提醒 push 入口
2. 权益订单页行权入口

当前不建议让分发页直接承担退款流程。

### 4.2 推荐路由与场景

当前建议沿用已有占位页路径：

- `GET /joint-dispatch`

由联合登录入口根据 `scene` 进行分流，推荐场景值：

| `scene` | 含义 | 进入页 |
|---|---|---|
| `push` | 放款成功后的权益提醒 | `/joint-dispatch` |
| `exercise` | 权益订单页行权入口 | `/joint-dispatch` |

### 4.3 推荐页面行为

分发页进入后建议按以下顺序执行：

1. 校验联合登录是否完成
2. 加载当前权益订单上下文
3. 判断当前订单是否允许继续行权 / 跳转
4. 决定：
   - 直接跳齐为供应商页面
   - 展示中间说明页
   - 展示异常页

### 4.4 后端接口边界

分发页正式化后，建议后端至少提供以下能力：

1. `GET /api/benefit-dispatch/context/{benefitOrderNo}`
   - 返回当前订单状态
   - 返回当前场景
   - 返回是否允许跳转供应商
2. `POST /api/benefit-dispatch/resolve`
   - 计算目标跳转地址
   - 记录跳转流水
   - 返回前端是否应直接重定向

当前阶段这两个接口只作为方案骨架，先不实现。

### 4.5 当前占位页与后续关系

当前已落地的：

- `H5/client/src/pages/JointDispatchPage.tsx`

可继续演进为正式分发页，后续只需逐步替换：

1. 占位文案
2. 静态说明
3. 真实上下文接口
4. 真实供应商跳转行为

## 5. 退款页方案骨架

### 5.1 页面职责

退款页当前建议定义为“退款入口承接页”，职责是：

1. 承接科技平台跳来的退款场景
2. 展示当前订单上下文与退款说明
3. 为后续退款申请 / 查询 / 状态页留出扩展入口

### 5.2 推荐路由与场景

当前建议沿用已有占位页路径：

- `GET /joint-refund-entry`

推荐由联合登录入口根据以下场景值进入：

| `scene` | 含义 | 进入页 |
|---|---|---|
| `refund` | 权益包退款入口 | `/joint-refund-entry` |

### 5.3 推荐页面行为

退款入口页进入后建议依次处理：

1. 校验联合登录是否完成
2. 拉取退款前置信息
3. 判断当前订单是否允许退款
4. 根据结果进入：
   - 退款说明态
   - 退款申请态
   - 退款状态态
   - 当前不可退款异常态

### 5.4 后端接口边界

退款页正式化后，建议后端补以下能力：

1. `GET /api/refund/info/{benefitOrderNo}`
   - 返回可退状态
   - 返回可退金额 / 规则说明
   - 返回当前退款状态
2. `POST /api/refund/apply`
   - 提交退款申请
   - 返回退款受理编号
3. `GET /api/refund/result/{refundId}`
   - 返回退款处理中 / 成功 / 失败

当前阶段上述接口只作为推荐方案骨架，先不实现。

### 5.5 当前占位页与后续关系

当前已落地的：

- `H5/client/src/pages/JointRefundEntryPage.tsx`

可作为后续退款入口页的第一页骨架，后续在此基础上扩展：

1. 正式退款规则展示
2. 申请按钮与表单
3. 退款状态查询
4. 失败重试与客服指引

## 6. 权益状态回推方案骨架

### 6.1 目标

状态回推的目标是：当权益状态发生关键变化时，让科技平台可以感知并同步展示。

### 6.2 推荐触发事件

当前建议把状态回推事件分为三类：

#### A. 权益生命周期事件

1. 权益订单创建
2. 权益开通成功
3. 权益开通失败

#### B. 行权相关事件

1. 行权成功
2. 行权失败
3. 行权失效 / 已过期

#### C. 退款相关事件

1. 退款申请
2. 退款成功
3. 退款失败

当前是否全部回推，仍待科技平台确认。

### 6.3 推荐回推接口模式

建议优先采用“艾博生主动 HTTP 回推科技平台”的模式。

推荐预留后端能力：

1. 事件组装器
2. 回推客户端
3. 重试机制
4. 幂等记录
5. 回推日志

### 6.4 推荐回推字段

当前建议预留以下字段：

1. `eventId`
2. `eventType`
3. `externalUserId`
4. `benefitOrderNo`
5. `productCode`
6. `statusBefore`
7. `statusAfter`
8. `eventTime`
9. `reason`
10. `sourceSystem`

### 6.5 推荐后端能力边界

状态回推正式化后，建议后端具备：

1. `BenefitStatusPushService`
2. `BenefitStatusPushLogRepository`
3. `TechPlatformStatusPushClient`

当前阶段仅作为方案命名骨架，不进入实现。
