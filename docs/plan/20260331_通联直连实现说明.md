# 2026-03-31 通联直连实现说明

## 1. 目标

本次实现的目标是：

1. 保留现有业务入口不变
2. 为齐为适配层增加通联证书直连模式的配置与路由能力
3. 落地通联直连 client 的基础骨架代码
4. 明确当前已实现范围与未实现边界

本次**不包含**通联 `ProcessServlet` 的具体报文组装和真实接口调用逻辑，因为该部分仍依赖齐为 / 通联补齐直连协议资料。

配套文档：

1. 实现现状与代码边界：本文档
2. 外部待确认项：`docs/plan/20260331_齐为通联直连外部确认清单.md`
3. 联调字段逐项回填模板：`docs/plan/20260331_通联直连联调字段待填模板.md`
4. 联调执行清单：`docs/plan/20260331_通联直连联调执行清单.md`

### 2.2 协议路由层

已新增 `RoutingQwBenefitClient`，并作为 `@Primary` Bean 生效。

当前路由策略：

| 模式 | 实际委派实现 |
| --- | --- |
| `MOCK` | `QwBenefitClientImpl` |

这意味着上层业务服务仍然只依赖 `QwBenefitClient`，不需要感知底层协议差异。

### 2.3 通联证书装载基础设施

已新增 `AllinpayCertificateLoader`，支持：

1. 装载 `PKCS12` 商户证书包
2. 装载 `X509` 验签证书
3. 对文件不存在、格式错误等情况抛出明确业务异常

### 2.4 通联直连 client 骨架

已新增 `AllinpayDirectQwBenefitClient`，当前具备：

1. 校验通联直连关键配置是否齐全
2. 在首次调用时装载 `PKCS12` 与验签证书
3. 初始化以下基础设施：
   - `AllinpayRequestSigner`
   - `AllinpayResponseVerifier`
   - `AllinpaySslContextFactory`
   - `AllinpayRestClientFactory`
4. 对缺失配置抛出：
   - `ALLINPAY_DIRECT_CONFIG_MISSING`
5. 对协议尚未落地部分抛出：
   - `ALLINPAY_DIRECT_PROTOCOL_UNIMPLEMENTED`

### 2.5 第二轮基础设施补充

为后续真实联调继续补充了以下基础设施：

| 组件 | 作用 |
| --- | --- |
| `AllinpayRequestSigner` | 基于 `PKCS12` 私钥对请求载荷进行 `SHA256withRSA` 签名 |
| `AllinpayResponseVerifier` | 基于 `X509` 证书公钥对响应签名进行验签 |
| `AllinpaySslContextFactory` | 基于商户私钥证书和验签证书构建 `SSLContext` |
| `AllinpayRestClientFactory` | 基于 `SSLContext` 和超时配置创建通联直连 `RestClient` |

说明：

这四个组件当前已经能完成“证书加载 -> 私钥签名 -> 公钥验签 -> SSL 上下文初始化 -> `RestClient` 构建”的基础闭环，但尚未接入真实通联报文调用。

### 2.6 请求工厂层补充

为避免后续直接在 client 中硬编码业务动作和 `serviceCode`，本次继续新增了请求工厂层：

| 组件 | 作用 |
| --- | --- |
| `AllinpayDirectOperation` | 统一定义三类直连业务动作：`MEMBER_SYNC`、`EXERCISE_URL`、`LENDING_NOTIFY` |
| `AllinpayDirectInvocation` | 承载一次直连调用的内部上下文，包括 `operation`、`serviceCode`、目标地址、商户凭证和原始业务请求 |
| `AllinpayDirectRequestFactory` | 根据当前 `QwProperties.direct.*` 配置，为三类业务动作构造统一调用上下文 |

当前 `AllinpayDirectQwBenefitClient` 的执行顺序已经变为：

1. 校验直连证书与基础配置
2. 初始化签名器、验签器、SSL、`RestClient`
3. 通过 `AllinpayDirectRequestFactory` 构造内部调用上下文
4. 在具体协议映射尚未实现时，抛出带 `operation + serviceCode` 的未实现异常

这意味着后续接真实报文时，只需要继续补“调用上下文 -> 通联请求报文”的映射层，而不必再改现有骨架的初始化和路由逻辑。

### 2.7 映射层与内部信封对象

为了继续往“可接真实报文”推进，又补了一层与外部协议弱耦合的内部抽象：

| 组件 | 作用 |
| --- | --- |
| `AllinpayDirectPayloadMapper<T>` | 定义“业务请求 -> 规范化内部 payload”的映射接口 |
| `AllinpayMemberSyncPayloadMapper` | 将 `QwMemberSyncRequest` 映射为规范化 payload |
| `AllinpayExerciseUrlPayloadMapper` | 将 `QwExerciseUrlRequest` 映射为规范化 payload |
| `AllinpayLendingNotifyPayloadMapper` | 将 `QwLendingNotifyRequest` 映射为规范化 payload |
| `AllinpayDirectPayloadMapperRegistry` | 按 `operation + requestType` 路由到对应 mapper |
| `AllinpayDirectEnvelopeHead` | 统一承载 `serviceCode`、商户号、用户名、密码、时间戳 |
| `AllinpayDirectEnvelope` | 统一承载目标地址、报文头、规范化业务 payload |
| `AllinpayDirectEnvelopeFactory` | 基于调用上下文和 payload 构造内部信封对象 |

当前 `AllinpayDirectQwBenefitClient` 已经改为：

1. 先构造 `AllinpayDirectInvocation`
2. 再通过 `PayloadMapperRegistry` 生成规范化业务 payload
3. 再通过 `EnvelopeFactory` 生成统一内部信封对象
4. 最后在报文协议尚未实现时，抛出带 `operation + serviceCode` 的未实现异常

这层的价值在于：

后续即使拿到通联 XML / 表单协议文档，也只需要新增“内部信封对象 -> 真实外部报文”的转换，而不必重改上层业务请求对象。

当前三个业务方法：

| 方法 | 当前状态 |
| --- | --- |
| `syncMemberOrder()` | 已接入通联直连 skeleton |
| `getExerciseUrl()` | 已接入通联直连 skeleton |
| `notifyLending()` | 已接入通联直连 skeleton |

### 2.8 协议序列化与签名准备边界

在内部信封对象之上，本次继续补齐了“待发送请求”的边界层，目标是把通联直连 client 再向前推进一步，但仍然不伪造真实外部协议细节。

新增组件：

| 组件 | 作用 |
| --- | --- |
| `AllinpayDirectProtocolSerializer` | 定义“内部信封对象 -> 待发送请求”的协议序列化接口 |
| `AllinpayDirectSkeletonProtocolSerializer` | 当前占位实现，先将内部信封序列化为稳定 JSON 结构，作为骨架请求表示 |
| `AllinpayDirectSerializedRequest` | 承载序列化后的请求对象，包括目标地址、内容类型、请求体、签名原文 |
| `AllinpayDirectRequestPreparer` | 串联协议序列化与 `SHA256withRSA` 签名，生成已完成签名准备的请求对象 |
| `AllinpayDirectPreparedRequest` | 承载已准备好的请求对象，包括目标地址、内容类型、请求体、签名结果 |
| `AllinpayDirectTransportMapper` | 定义“已签名请求 -> 可外发 transport request”的映射接口 |
| `AllinpayDirectSkeletonTransportMapper` | 默认占位映射器，将已签名请求转为稳定的 POST transport 结构，并保留签名属性 |
| `AllinpayDirectTransportRequest` | 承载 transport 层请求对象，包括方法、内容类型、body、headers、附加属性 |
| `AllinpayDirectHttpExecutor` | 定义“已签名请求 -> 原始响应”的执行器接口 |
| `AllinpayDirectSkeletonHttpExecutor` | 默认占位执行器，明确阻断真实外发请求，避免在协议资料未确认前误联调 |
| `AllinpayDirectRestHttpExecutor` | 通用 `RestClient` 真发送器，可按 transport request 发起真实 HTTP 调用并回收状态码、body、headers |
| `AllinpayDirectResponseParser` | 定义“原始响应 -> 业务响应”的解析接口 |
| `AllinpayDirectSkeletonResponseParser` | 当前占位实现，明确在未拿到真实协议文档前禁止伪造响应解析 |
| `AllinpayDirectRawResponse` | 统一承载后续真实 HTTP 返回的状态码、响应体、签名字段 |
| `AllinpayDirectResponseSignatureResolver` | 定义“原始响应 -> 签名值”的提取接口 |
| `AllinpayDirectHeaderResponseSignatureResolver` | 从指定响应头提取签名值 |
| `AllinpayDirectResponseVerificationStage` | 定义“原始响应 -> 已验签响应”的统一阶段接口 |
| `AllinpayDirectSkeletonResponseVerificationStage` | 默认占位验签阶段，明确阻断未确认协议下的响应验签 |
| `AllinpayDirectResponseVerificationAdapter` | 串联“签名提取 + 证书验签”，生成已验签响应对象 |
| `AllinpayDirectVerifiedResponse` | 承载已通过签名校验的响应内容 |

`AllinpayDirectQwBenefitClient` 当前执行顺序已经进一步推进为：

1. 校验直连基础配置与证书路径
2. 加载 `PKCS12` 和验签证书
3. 初始化签名器、验签器、SSL、`RestClient`
4. 构造 `AllinpayDirectInvocation`
5. 映射为规范化业务 payload
6. 组装 `AllinpayDirectEnvelope`
7. 通过 `AllinpayDirectProtocolSerializer` 生成待发送请求
8. 通过 `AllinpayDirectRequestPreparer` 完成签名准备
9. 通过 `AllinpayDirectTransportMapper` 生成 transport 层请求对象
10. 通过 `AllinpayDirectHttpExecutor` 进入执行层
11. 通过 `AllinpayDirectResponseVerificationStage` 进入响应验签阶段
12. 通过 `AllinpayDirectResponseParser` 解析业务响应
13. 默认由 `AllinpayDirectSkeletonHttpExecutor / AllinpayDirectSkeletonResponseVerificationStage / AllinpayDirectSkeletonResponseParser` 明确阻断未确认协议的真实联调

这样当前剩余未落地部分已经进一步收敛为两段：

1. 真实通联 `ProcessServlet` 协议映射
2. 真实响应报文解析与验签字段对接

补充说明：

`AllinpayDirectQwBenefitClient` 现在已经支持通过构造参数注入自定义 `ProtocolSerializer / TransportMapper / HttpExecutor / ResponseVerificationStage / ResponseParser`。这意味着后续拿到齐为 / 通联的正式协议资料时，只需要替换底层五个适配组件，就可以复用现有的配置校验、证书装载、签名准备和业务调用主流程。

本次继续补充说明：

1. 已新增 `AllinpayDirectRestHttpExecutor`，用于承接未来真实联调时的通用 HTTP 外发
2. 该发送器会基于 `transport request` 回收：
   - HTTP 状态码
   - 响应 body
   - 响应 headers
3. 当前默认 Bean 仍然不会启用该发送器，而是继续使用 `AllinpayDirectSkeletonHttpExecutor`
4. 这样做的目的是：把“真实发送能力”提前准备好，但在协议字段、签名位置、响应格式未确认前，仍然强制阻断误联调

本次再继续补充响应侧预留层：

1. 已新增 `ResponseSignatureResolver`，用于屏蔽“签名值从响应头、响应体还是其他位置提取”的差异
2. 已新增 `ResponseVerificationStage`，并已正式接入 `AllinpayDirectQwBenefitClient` 主流程
3. `ResponseVerificationAdapter` 负责把“提取签名 -> 公钥验签 -> 生成已验签响应”串成独立闭环
4. 当前 parser 仍然是 skeleton，但未来真实 parser 可以直接复用该验签闭环
5. 这意味着后续真实联调在响应侧剩余的核心工作已经进一步收敛为：
   - 明确签名字段来源
   - 明确响应体结构
   - 将已验签响应映射为业务响应对象

### 2.9 Spring 装配边界补充

为避免后续拿到齐为 / 通联协议资料后还要改 `AllinpayDirectQwBenefitClient` 本体，本次继续把协议相关默认装配从 client 内部抽出到 Spring 配置层。

新增组件：

| 组件 | 作用 |
| --- | --- |
| `AllinpayDirectConfiguration` | 统一提供通联直连协议组件的默认 Bean 装配 |

当前装配策略：

| 接口 | 默认 Bean |
| --- | --- |
| `AllinpayDirectProtocolSerializer` | `AllinpayDirectSkeletonProtocolSerializer` |
| `AllinpayDirectTransportMapper` | `AllinpayDirectSkeletonTransportMapper` |
| `AllinpayDirectHttpExecutor` | `AllinpayDirectSkeletonHttpExecutor` |
| `AllinpayDirectResponseVerificationStage` | `AllinpayDirectSkeletonResponseVerificationStage` |
| `AllinpayDirectResponseParser` | `AllinpayDirectSkeletonResponseParser` |

补充说明：

1. 上述默认 Bean 通过 `@ConditionalOnMissingBean` 提供。
2. `AllinpayDirectQwBenefitClient` 已改为优先通过 Spring 注入上述五个协议组件。
3. 当前默认行为仍然是安全 skeleton，不会误发真实请求。
4. 后续若齐为 / 通联协议资料补齐，只需要新增或替换对应 Bean，即可覆盖默认 skeleton，而不需要再修改 client 主流程。
5. 同时，`AllinpayDirectRequestFactory`、`AllinpayDirectPayloadMapperRegistry`、`AllinpayDirectEnvelopeFactory` 也已进入统一配置装配，后续真实联调可以继续通过替换 Bean 收敛改动范围。
