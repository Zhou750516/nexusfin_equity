# 2026-04-26 外部待确认项与本侧可执行项拆分

## 1. 文档目标

本文档用于把当前项目中“必须等待外部确认”的事项，与“艾博生本侧可以立即推进”的事项拆开，避免后续推进再次混用。

拆分原则：

1. 外部接口语义未定，视为待确认；
2. 本地状态、映射、文档、DTO、facade、配置等，优先视为本侧可执行；
3. 能先做替换点，就先不抢跑真实业务实现。

---

## 2. 外部待确认项

1. 建放款订单接口路径与字段；

4. 云卡状态查询接口；

6. Yunka 转发回调的 envelope 结构；

### 影响

2. 本地状态与云卡状态映射；

## 4. 建议执行顺序

建议当前按以下顺序推进：

1. 先做本侧可执行的方案收口；
2. 再做 Shared Foundation 第一批基础层；
3. 再做权益页动态化与不依赖状态机的局部对齐；
4. 最后等待云卡 / 小花 / 齐为 / 法大大关键资料进一步定版后，再进入高返工主链改造。

---

1. 把“外部待确认项”误当成本地立刻可实现；
2. 把“本地可先定的状态与映射”继续拖到外部确认之后；

4. 在云卡接口未定时直接大改 `/api/loan/apply` 主链。

## 6. 一句话结论

> 当前最稳妥的推进方式，是把“外部待确认项”和“本侧可执行项”彻底拆开：前者持续跟进、后者立即落地，避免团队在等待和抢跑之间来回摇摆。
