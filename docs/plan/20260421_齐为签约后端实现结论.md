# 2026-04-21 齐为签约后端实现结论

## 1. 本轮已完成

基于 `plan/20260421_齐为签约后端接入设计方案.md`，当前已经完成以下后端骨架落地：

### 1.1 艾博生后端接口

已新增 3 个接口：

| 接口 | 方法 | 说明 |
|---|---|---|
| `/api/bank-card/sign-status` | `GET` | 查询当前银行卡是否已签约 |
| `/api/bank-card/sign-apply` | `POST` | 发起签约申请，发送短信 |
| `/api/bank-card/sign-confirm` | `POST` | 提交短信验证码，确认签约 |

当前接口风格：

1. 统一走登录态；
2. 统一返回 `Result<T>`；
3. 由后端从本地会员信息中解密获取手机号、姓名、身份证号；
4. 前端当前只需要传：
   - 查询：`accountNo`
   - 申请：`accountNo`
   - 确认：`accountNo`、`verificationCode`

### 1.2 齐为 client 扩展

已在现有 `QwBenefitClient` / `QwBenefitClientImpl` 上扩展：

1. `querySignStatus(...)`
2. `applySign(...)`
3. `confirmSign(...)`

并补齐了对应 DTO：

1. `QwSignStatusRequest / Response`
2. `QwSignApplyRequest / Response`
3. `QwSignConfirmRequest / Response`

### 1.3 本地 mock 骨架

已补齐签约 3 个接口的 mock 分支，用于本地联调与后端自测：

1. `accountNo` 以 `8` 结尾时，签约查询返回已签约；
2. 签约申请返回 `mock-sign-apply-xxxx`；
3. 签约确认返回 `mock-sign-confirm-xxxx`，并视为 `ACTIVE`。

### 1.4 本地协议记录落库

签约确认成功后，当前已回写 `member_payment_protocol`：

| 字段 | 当前实现 |
|---|---|
| `provider_code` | `QW_SIGN` |
| `protocol_no` | `QW-SIGN-{requestNo}` |
| `protocol_status` | `ACTIVE` |
| `sign_request_no` | 签约确认返回或本地生成的 `requestNo` |
| `channel_code` | 取会员最近一条渠道绑定记录 |
| `signed_ts` | 当前后端确认时间 |

说明：

1. 这里没有复用 `ALLINPAY`，避免影响当前已存在的首扣 / 支付协议解析逻辑；
2. 当前 `protocol_no` 为本地稳定占位规则；
3. 当前实现不依赖齐为返回正式签约号。

### 1.5 认证保护

已将 `/api/bank-card` 加入认证保护路径：

1. `src/main/resources/application.yml`
2. `src/test/resources/application.yml`

因此未登录访问会返回 `401`。

## 3. 当前待补项

以下事项当前仍建议保留为下一步任务：

### 3.1 会员同步字段升级未完全落地

`2026-04-21` 文档中会员同步新增了：

1. `mobile`
2. `username`
3. `accountNo`
4. `cardNo`

当前待补原因：

1. 现有后端主链中，`accountNo` / `cardNo` 还没有稳定本地来源；
2. 若直接替换现有 `payProtocolNo` 口径，容易影响现有权益购买同步链路；
3. 建议等绑卡页、银行卡信息模型或齐为回传口径进一步明确后，再整体升级。

收口方案已单独整理：

1. `docs/plan/20260421_齐为会员同步字段升级收口方案.md`

### 3.2 H5 绑卡页面未接入

当前只完成后端能力，尚未包含：

1. 绑卡页 UI；
2. 短信验证码交互；
3. 倒计时 / 重发；
4. 页面态与错误提示翻译。

后续 H5 接入说明已单独整理：

1. `docs/plan/20260421_H5绑卡页接入后端接口说明.md`

### 3.3 齐为正式联调待接入

当前按接口文档与 mock 骨架落地，待后续齐为联调时继续确认：

1. 签约申请真实业务 `data` 结构；
2. 签约确认真实业务 `data` 结构；
3. 是否有正式签约号 / 协议号可回填；
4. 失败码和重试口径。

## 4. 建议下一步

建议按下面顺序继续：

1. 先保持当前后端签约骨架不动，等待 H5 绑卡页面提供；
2. 等银行卡字段来源明确后，再推进会员同步 `accountNo / cardNo` 升级；
3. 与齐为做一次真实联调，确认申请 / 确认返回报文；
4. 联调后再补错误码映射、幂等和重试策略。
