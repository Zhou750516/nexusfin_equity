# 2026-04-21 齐为签约后端接入设计方案

## 1. 文档目标

本文档用于基于 `third-part/齐为/20260421_会员同步与签约接口整理.md`，为艾博生侧输出一版**只覆盖后端、齐为接口接入和本地记录落库**的技术方案。

本次范围明确不包含：

1. H5 绑卡页面；
2. H5 交互细节；
3. 状态流转、重试、幂等的完整设计；
4. 签约号 / 协议号 / 申请流水号的页面依赖与编排。

本次目标是先把“齐为签约相关后端能力”设计成可独立开发、可后续挂接 H5 的稳定骨架。

---

## 2. 范围定义

## 2.1 本次包含

1. 齐为会员同步接口字段升级：
   - `mobile`

5. 艾博生后端对外提供 3 个签约能力接口
6. 签约成功后的本地记录落库
7. 基础测试与文档补齐

## 2.2 本次不包含

1. H5 页面与页面流程实现
2. 验证码倒计时、重发、按钮态等前端逻辑
3. 复杂错误码翻译体系
4. 重试、幂等、防重放等增强机制
5. 新增独立签约表结构

## 3. 设计结论

## 3.1 总体方案

本次采用以下设计策略：

### 方案结论

1. **新增独立的“齐为签约后端链路”**
2. **复用现有 `member_payment_protocol` 作为签约成功本地记录模型**
3. **扩展现有会员同步请求模型以承接齐为新字段**
4. **对外新增 3 个艾博生后端接口，但不依赖 H5 页面先行落地**

### 为什么这样做

1. 齐为签约接口与现有借款 / 还款主链属于不同业务子域，独立建后端链路更清晰
2. 你已明确“签约成功需要本地记录”，而当前最合适的承接位置就是 `member_payment_protocol`
3. 你已明确“本次不需要把签约号作为本侧接口依赖”，因此当前没有必要先新建独立签约表
4. H5 页面尚未提供，因此后端接口应先稳定下来，等待后续页面直接接入

## 4. 后端接口设计

## 4.1 艾博生后端对外接口

建议新增以下 3 个后端接口：

| 接口 | 方法 | 说明 |
|---|---|---|
| `/api/bank-card/sign-status` | `GET` | 查询当前银行卡是否已签约 |
| `/api/bank-card/sign-apply` | `POST` | 发起签约申请，下发短信 |
| `/api/bank-card/sign-confirm` | `POST` | 提交短信验证码，确认签约 |

### 设计说明

1. 仍沿用当前项目接口风格：不带版本号
2. 统一返回 `Result<T>`
3. Controller 只做参数接收、鉴权上下文获取和 service 调用
4. 真正业务逻辑放在 service 层

## 4.2 与齐为接口的映射关系

| 艾博生后端接口 | 齐为 method | 说明 |
|---|---|---|
| `/api/bank-card/sign-status` | `abs.sign.query` | 查询是否签约 |
| `/api/bank-card/sign-apply` | `abs.sign.apply` | 申请签约短信 |
| `/api/bank-card/sign-confirm` | `abs.sign.confirm` | 短信验证确认签约 |

## 5. 代码结构设计

## 5.1 Controller 层

建议新增：

- `src/main/java/com/nexusfin/equity/controller/BankCardSignController.java`

职责：

1. 暴露 3 个签约相关接口
2. 获取当前登录用户上下文
3. 调用 `BankCardSignService`
4. 统一返回 `Result<T>`

## 5.2 Service 层

建议新增：

- `src/main/java/com/nexusfin/equity/service/BankCardSignService.java`
- `src/main/java/com/nexusfin/equity/service/impl/BankCardSignServiceImpl.java`

职责：

1. 校验请求参数
2. 组织调用齐为 client
3. 解析齐为公共返回结构和业务返回结构
4. 在签约确认成功后回写本地协议记录
5. 统一业务异常口径

## 5.3 Third-party 层

建议扩展现有齐为 client，而不是新起完全平行的 client 体系。

建议方式：

1. 扩展 `QwBenefitClient`
2. 在 `QwBenefitClientImpl` 中新增 3 个方法：
   - `querySignStatus(...)`
   - `applySign(...)`
   - `confirmSign(...)`

这样做的原因：

1. 当前 `QwBenefitClientImpl` 已经封装了齐为公共协议：
   - AES 加密
   - MD5 签名
   - 公共 envelope
   - 通用响应解包
2. 新增 3 个 method 复用同一套出站协议实现最稳妥
3. 不需要重复创建第二套齐为 envelope / 签名 / 解密逻辑

## 5.4 DTO / Request / Response

建议新增以下本侧 DTO：

### 控制器入参 / 出参

- `dto/request/BankCardSignStatusRequest`（若使用 GET Query 可不单独建 record）
- `dto/request/BankCardSignApplyRequest`
- `dto/request/BankCardSignConfirmRequest`
- `dto/response/BankCardSignStatusResponse`
- `dto/response/BankCardSignApplyResponse`
- `dto/response/BankCardSignConfirmResponse`

### 齐为第三方请求 / 返回

建议新增：

- `thirdparty/qw/QwSignStatusRequest`
- `thirdparty/qw/QwSignStatusResponse`
- `thirdparty/qw/QwSignApplyRequest`
- `thirdparty/qw/QwSignApplyResponse`
- `thirdparty/qw/QwSignConfirmRequest`
- `thirdparty/qw/QwSignConfirmResponse`

说明：

1. 即使齐为文档中签约申请 / 确认的成功返回结构尚未展开，也建议先定义最小响应模型
2. 第一版响应模型只保留本侧明确要用到的字段
3. 其他不确定字段先不扩，避免伪造协议

## 6. 本地记录设计

## 6.1 记录模型选择

本次结论：

> 签约成功后，优先复用 `member_payment_protocol`，不新增独立签约表。

## 6.2 原因

1. 当前已有 `member_payment_protocol` 承接协议号 / 状态 / 签约时间
2. 你已明确“本次不需要依赖签约号做接口编排”
3. 当前优先目标是落“签约成功事实”，而不是沉淀完整签约生命周期
4. 这样可以最快与现有权益购买 / 首扣协议链路衔接

## 6.3 回写时机

建议在以下条件满足时回写：

1. `sign-confirm` 调齐为成功
2. 业务返回判定为签约成功

## 6.4 回写内容

建议复用 `PaymentProtocolService.saveActiveProtocol(...)` 能力，但需要适配齐为签约写入场景。

建议回写内容至少包括：

| 字段 | 建议值 |
|---|---|
| `member_id` | 当前登录会员 |
| `external_user_id` | 当前外部用户标识 |
| `provider_code` | 建议使用齐为签约对应 provider 常量 |
| `protocol_no` | 若齐为未明确返回签约号，则先使用稳定本地生成规则或待补规则 |
| `protocol_status` | `ACTIVE` |
| `sign_request_no` | 可先记录本次请求号 / 本地流水 |
| `channel_code` | 当前业务渠道，如 `KJ` |
| `signed_ts` | 当前签约成功时间 |

## 6.5 当前风险点

这里存在一个仍需进一步收口的点：

1. 你已确认“本次不需要用签约号”
2. 但本地表 `member_payment_protocol` 仍要求有 `protocol_no`

因此第一版设计建议：

1. 先在方案里明确：需要一个**本地可稳定生成**的 `protocol_no` 占位规则，专用于齐为签约成功记录
2. 但该值只作为本地记录主键材料，不作为当前页面 / 后端编排依赖
3. 后续若齐为补充正式签约号返回，再评估替换或回填

## 7. 会员同步接口升级设计

## 7.1 当前问题

现有 `QwMemberSyncRequest` 结构为：

1. 已有 `mobile`
2. 已有 `username`
3. 当前仍使用 `payProtocolNo`
4. 未体现文档中的 `accountNo`

这与 `2026-04-21` 齐为文档新口径不完全一致。

## 7.2 本次调整建议

建议将 `QwMemberSyncRequest` 升级为与最新齐为文档一致的字段口径：

1. 保留 `mobile`
2. 保留 `username`
3. `payProtocolNo` 不再继续作为会员同步字段
4. 新增 `accountNo`
5. 保留 `cardNo`

## 7.3 本地取值建议

| 字段 | 建议来源 |
|---|---|
| `mobile` | `MemberInfo` 解密手机号 |
| `username` | `MemberInfo` 解密真实姓名 |
| `accountNo` | 当前有效支付协议 / 绑卡输入 / 后续签约流程沉淀字段 |
| `cardNo` | 首次为空；续期场景按已有会员卡号回传 |

## 7.4 当前待确认点

由于当前库表里没有明确银行卡号字段，本次需要在设计和任务中明确：

1. 第一版会员同步是否先允许 `accountNo` 来源于签约确认入参并回写缓存 / 模型；
2. 若无稳定来源，会员同步字段升级与签约成功本地记录应联动推进；
3. 在未确认前，不建议直接硬编码银行卡号。

## 10. 当前建议结论

本次设计建议按以下顺序实施：

1. 先扩展齐为 client 能力
2. 再新增签约 service 和 controller
3. 再补签约成功本地记录
4. 再升级会员同步字段映射
5. 最后补测试和文档

一句话结论：

> 当前最合适的实现方式是：新增独立的齐为签约后端链路、复用现有 `member_payment_protocol` 做签约成功记录、扩展现有齐为 client 与会员同步字段模型，并先把后端能力闭环起来，等待后续 H5 页面接入。
