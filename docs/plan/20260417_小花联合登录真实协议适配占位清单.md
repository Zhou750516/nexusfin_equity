# 2026-04-17 小花联合登录真实协议适配占位清单

## 1. 文档目标

本文档用于承接当前已完成的“小花联合登录本地骨架”，进一步明确：

1. 真实小花联合登录文档到位后，艾博生侧优先替换哪些位置
2. 哪些现有类和接口可以保持稳定
3. 哪些配置项、错误码、测试和日志能力需要补齐
4. 如何在不推翻当前骨架的前提下，把本地占位实现升级为真实协议适配

本文档不进入真实协议实现，只做“可接入、可替换、可联调”的占位清单。

---

## 2. 当前已固定的本地骨架边界

截至 `2026-04-17`，当前联合登录本地骨架已经固定以下边界：

### 2.1 前端入口边界

当前 H5 已固定以下入口与场景承接方式：

1. `/joint-entry` 作为统一联合登录承接页
2. `/joint-dispatch` 作为分发页占位入口
3. `/joint-refund-entry` 作为退款入口占位页

前端当前固定职责：

1. 读取 URL 参数
2. 调用艾博生后端 `POST /api/auth/joint-login`
3. 根据后端结果跳转到目标页
4. 不直接调用小花后端

### 2.2 后端接口边界

当前后端已固定：

1. 对外入口：`POST /api/auth/joint-login`
2. 内部编排服务：`JointLoginService`
3. 外部适配抽象：`XiaohuaAuthClient`
4. 当前占位实现：`SkeletonXiaohuaAuthClient`
5. 目标页分流器：`JointLoginTargetPageResolver`

### 2.3 当前内部身份模型

当前 `XiaohuaAuthClient` 已约定向上返回：

1. `externalUserId`
2. `phone`
3. `realName`
4. `idCard`
5. `channelCode`

因此，真实小花协议接入时，优先目标不是推翻上层服务，而是把真实返回字段稳定映射到这组内部字段。

---

## 3. 推荐适配原则

### 3.1 保持 Controller 与 H5 请求契约尽量稳定

当前建议优先保持以下输入输出不变：

1. H5 -> 艾博生后端：`token + scene + benefitOrderNo/orderNo + productCode`
2. 艾博生后端 -> H5：`loginSuccess + scene + targetPage + benefitOrderNo + localUserReady`

原因：

1. 当前 H5 页面和联调脚本已经基于这套契约落骨架
2. 真实协议变化应优先封装在后端适配层
3. 这样可以避免外部协议波动再次拖动 H5 页面逻辑

### 3.2 把真实协议变化集中到 `XiaohuaAuthClient`

当前推荐把下列变化尽量收敛在 `XiaohuaAuthClient` 的真实实现中：

1. 小花请求地址
2. 请求方法
3. 请求头
4. 时间戳 / nonce / sign
5. 请求体字段名
6. 响应体字段名
7. 错误码映射

### 3.3 优先映射，不优先扩散

若小花正式字段与当前内部字段不一致，优先策略是：

1. 在适配层做字段映射
2. 在适配层做错误码归一化
3. 仅当内部抽象明显不足时，再谨慎调整 `JointIdentity`

不建议一拿到小花文档，就直接把外部字段名扩散到 Controller、Service 和 H5。

## 4. 真实协议接入时的优先替换点

### 4.1 必换：`SkeletonXiaohuaAuthClient`

当前占位类：

1. `src/main/java/com/nexusfin/equity/service/impl/SkeletonXiaohuaAuthClient.java`

当前行为：

1. 固定抛出 `XIAOHUA_JOINT_LOGIN_NOT_READY`

真实协议接入后，该类需要被以下其中一种方式替换：

1. 直接改造成真实实现
2. 保留 Skeleton 类，再新增 `RestXiaohuaAuthClient`
3. 通过配置开关在 Skeleton / Real 实现之间切换

当前更推荐第 3 种，便于：

1. 本地继续使用 Skeleton
2. 联调环境切换真实实现
3. 未拿到正式文档前仍保持可启动

### 4.2 基本稳定：`JointLoginServiceImpl`

当前服务：

1. `src/main/java/com/nexusfin/equity/service/impl/JointLoginServiceImpl.java`

建议尽量保持以下逻辑稳定：

1. 入参校验
2. 本地用户恢复 / 创建
3. `member_channel` 绑定
4. JWT 建立
5. 目标页分流

真实协议接入后，理论上只需要改变：

1. `xiaohuaAuthClient.exchange(request.token())` 的底层实现
2. 错误码 / 异常口径的映射细节

### 4.3 基本稳定：`JointLoginRequest` / `JointLoginResponse`

当前 DTO 建议继续维持稳定，除非小花正式文档明确要求科技平台跳艾博生时还必须携带额外关键字段。

如果要新增字段，建议优先考虑：

1. `timestamp`
2. `nonce`
3. `sign`
4. `source`
5. `redirectType`

即便新增，也建议：

1. 只在艾博生入口 DTO 扩展
2. 不直接改动现有页面跳转主逻辑

## 6. 推荐字段映射占位

### 6.1 入站：科技平台 -> 艾博生

当前建议继续按以下“最小稳定字段”接入：

| 当前字段 | 说明 | 当前状态 |
|---|---|---|
| `token` | 联合登录核心凭证 | 已固定 |
| `scene` | 场景分流值 | 已固定 |
| `benefitOrderNo` / `orderNo` | 订单上下文 | 已固定 |
| `productCode` | 产品补充信息 | 已固定 |
| `timestamp` | 防重放候选字段 | 占位待确认 |
| `nonce` | 防重放候选字段 | 占位待确认 |
| `sign` | 签名候选字段 | 占位待确认 |

### 6.2 出站：艾博生 -> 小花

当前需要等待小花正式文档确认：

1. 请求方式是 `GET` 还是 `POST`
2. `token` 是否放 Header、Query 还是 Body
3. 是否需要 `appId`
4. 是否需要签名
5. 是否需要时间戳 / nonce
6. 是否需要来源渠道字段

在文档未到位前，建议只保留“请求模型占位”，不猜测最终字段。

### 6.3 回站：小花 -> 艾博生

当前推荐无论小花返回什么字段，都优先收敛映射到：

| 内部字段 | 说明 | 当前用途 |
|---|---|---|
| `externalUserId` | 外部用户唯一标识 | 本地建档、渠道绑定 |
| `phone` | 手机号 | 本地用户信息同步 |
| `realName` | 姓名 | 本地用户信息同步 |
| `idCard` | 身份证 | 本地用户信息同步 |
| `channelCode` | 渠道编码 | `member_channel` 绑定 |

如果小花返回字段更多，例如：

1. 用户状态
2. 会员等级
3. 设备信息
4. 登录来源

当前都不建议直接纳入第一轮联合登录主流程，除非后续业务明确依赖。

## 7. 错误码与异常页占位

真实协议接入时，建议把小花错误统一归并为以下几类本地口径：

### A. token 类错误

1. 无效
2. 过期
3. 已使用
4. 签名错误

建议统一映射为：

1. `JOINT_LOGIN_TOKEN_INVALID`
2. `JOINT_LOGIN_TOKEN_EXPIRED`

### B. 用户态错误

1. 用户不存在
2. 用户被冻结
3. 用户状态异常

建议统一映射为：

1. `JOINT_LOGIN_USER_INVALID`

### C. 系统类错误

1. 小花服务超时
2. 小花服务异常
3. 返回报文无法解析

建议统一映射为：

1. `JOINT_LOGIN_REMOTE_ERROR`

### D. 本地会话类错误

1. 本地建档失败
2. 本地登录态建立失败

建议统一映射为：

1. `JOINT_LOGIN_LOCAL_SESSION_ERROR`

## 8. 需要补的测试占位

真实协议接入后，建议至少补以下几类测试：

### 8.1 后端单测

1. 小花返回正常身份信息时，能正确映射为 `JointIdentity`
2. 小花返回 `token` 失效时，能正确映射本地错误码
3. 小花超时时，能正确抛出本地系统异常
4. 小花返回缺失字段时，能正确做兜底或失败处理

### 8.2 后端集成测试

1. `/api/auth/joint-login` 在真实实现启用时仍能正常返回 Cookie
2. `scene=push` / `exercise` / `refund` 仍能正确分流
3. 异常错误码仍能被前端正确识别

### 8.3 H5 侧验证

1. `/joint-entry` 能处理真实错误码跳转异常态
2. 目标页分流逻辑在真实错误码映射后不受影响
3. 多语言文案能覆盖新增异常场景

## 10. 推荐接入顺序

等小花正式文档到位后，建议按以下顺序推进：

1. 先补真实协议字段映射表
2. 再补配置项
3. 再新增真实 `XiaohuaAuthClient` 实现
4. 跑通联合登录后端测试
5. 跑通 H5 承接页联调
6. 最后再继续推进分发页 / 退款页 / 状态回推的真实业务接入

---

## 11. 当前一句话结论

> 当前最稳妥的推进方式是：保持 H5 入口和联合登录上层编排不动，把未来真实小花协议变化尽量集中封装在 `XiaohuaAuthClient` 的真实实现、配置项和错误码映射里，这样既保住当前骨架，又能在外部文档到位后快速接入。
