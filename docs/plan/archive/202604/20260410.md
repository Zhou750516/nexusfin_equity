# 2026-04-10 重点工作计划

> 2026-04-10 技术方案已调整：**所有 H5 页面由艾博生提供；云卡仅作为 gateway 记录并转发艾博生调用科技平台的接口。** 以下计划已按该边界重排。

## 1. 计划目标

截至 `2026-04-10` 技术方案调整后，当前项目状态进一步收敛为：

1. 所有 H5 页面职责已统一归艾博生，不再按“艾博生页面 + 云卡页面”拆分前端承载。
2. 云卡边界已收敛为 gateway，不再承担 H5 页面，只负责接口记录、转发和网关层编排。
3. 艾博生与科技平台之间的直接接口调用口径取消，改为“艾博生统一调用云卡，再由云卡转科技平台”。
4. 科技平台真实执行条件仍未完全齐备，测试 / 生产域名、登录态、加密加签参数和部分联调口径仍需通过云卡链路补充确认。
5. 齐为主链与支付协议号结构仍保持有效，但正式支付渠道与 `M3.2` 公网条件仍待外部确认。
6. 根据 `20260410_科技平台对接会议纪要`，借款 / 用信 / 还款主链已有一批明确业务结论，可据此收敛页面和接口边界，不再沿旧假设反复发散。

因此 `2026-04-10` 之后的重点，不是立即恢复科技平台直连接口开发，而是围绕“接口边界重定版、云卡网关口径明确、条件分流”推进三件事：

1. 产出并定版一份“艾博生调用云卡 gateway”的接口文档，作为当前对接基线。
2. 把原科技平台改造计划改写为“艾博生 H5 + 云卡 gateway + 科技平台能力提供方”的新责任矩阵。
3. 继续并行跟进 `M3.2` 公网回调条件与支付渠道外部状态，维持主线判断稳定。

当前开发与联调顺序进一步明确为：

1. **先完成艾博生 H5 页面与艾博生后端接口的联通性开发与联调**，先把页面展示、页面提交流程、艾博生内部接口连通性打通。
2. **再完成艾博生调用云卡、由云卡请求科技平台接口的开发与联调**，在内部链路稳定后再推进外部网关链路。

当前已明确的业务口径补充如下：

1. 试算页本期固定只支持 `3` 期，暂不支持灵活期数返回。
2. 试算页本期不展示“放款资方”信息。
3. 借款详情与协议点击统一进入同一个页面，但仍需补一份借款协议。
4. 协议展示需支持半弹窗并强制阅读。
5. 用信审批流程顺序明确为：**先购买权益，再流转云卡，再由云卡对接科技平台**。
6. 绑卡流程涉及短信验证；审批结果能力默认按“查询 + 回调”双机制规划。
7. 还款场景默认按“小花先内部试算，再跳艾博生还款页”理解推进。
8. 云卡需通过 `uid` 查询小花已绑卡列表；未签约卡需补短信验证与签约流程。

说明：

1. 下一步主线是 **云卡 gateway 接口文档定版 + 计划边界重排**。
2. 当前仍以文档推进为主，不恢复高返工风险的代码实现主线。

## 3. 建议执行顺序

建议按以下顺序推进：

1. 先把会议纪要中的已明确结论回填为执行基线
2. 再定版艾博生 -> 云卡 gateway 接口文档
3. 再更新技术方案计划和责任矩阵
4. 先完成艾博生 H5 页面与艾博生后端接口的联通性开发、联调
5. 再完成艾博生调用云卡、请求科技平台接口的开发、联调
6. 再并行确认会议纪要中的外部待确认项、`M3.2` 与支付渠道状态

原因：

1. 会议纪要已经提供了一批可执行口径，应先把这些结论沉淀进计划，而不是继续靠口头理解推进。
2. 当前最需要先统一的是页面、流程和接口边界，而不是代码实现细节。
3. 艾博生 H5 与后端内部联通性先稳定，能更早暴露页面编排与内部接口问题。
4. 云卡 gateway 与科技平台联调应建立在内部链路已稳定的前提上，能减少跨方排障噪音。
5. `M3.2`、支付渠道和会议纪要中的外部确认项仍依赖外部反馈，适合作为并行跟进项。

## 5. 风险提示

`2026-04-10` 的计划主要受以下因素影响：

1. 若联合登录接口、协议接口、卡列表接口和联调排期未及时确认，页面、接口和联调计划仍会保持“内部明确、外部待确认”的状态。
2. 若 `M3.2` 三项公网条件继续无回复，则入站回调联调仍无法启动。
3. 若支付渠道最终方案继续待定，则齐为主链正式切换节奏仍受影响。
4. 若借款金额是否允许修改、活体数据是否必需等问题长时间未定，部分页面交互与接口字段仍需保留弹性。
