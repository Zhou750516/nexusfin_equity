# 2026-04-21 后端日志联调与排障检查清单

## 1. 目标

用于后续 H5、本地后端、云卡、齐为联调时快速确认：

1. 请求是否进入艾博生后端；
2. 业务链路是否进入对应 Service；
3. 第三方调用是否发起；
4. 同一请求是否能通过 `traceId` 串联；
5. 失败时是否能通过 `errorNo / errorMsg` 判断问题归属。

## 2. 通用检查字段

每条关键日志优先检查以下字段：

| 字段 | 含义 | 检查方式 |
|---|---|---|
| `traceId` | 单次前端请求链路 ID | 同一次 H5 请求中 Controller、Service、第三方客户端日志应一致 |
| `bizOrderNo` | 业务检索号 | 借款用 `applicationId` / `loanId`；还款用 `loanId` / `repaymentId`；绑卡用 `bank-card-后四位` |
| `requestId` | 调用第三方请求号 | 云卡 / 齐为调用前后应一致 |
| `path` | 云卡转发路径 | 云卡链路必须能看到，如 `/loan/apply`、`/repay/trial` |
| `elapsedMs` | 耗时 | success / warn / error 日志应带耗时 |
| `errorNo` | 稳定错误码 | 失败日志必须存在，用于分类检索 |
| `errorMsg` | 清晰错误说明 | 应语义准确，避免只输出 `null` 或笼统异常 |

## 3. 日志等级口径

| 等级 | 使用场景 | 示例 |
|---|---|---|
| `INFO` | 请求进入、第三方调用 begin、调用成功 success | `loan apply yunka request begin` |
| `DEBUG` | 非生产细节、mock 跳过、低频调试信息 | `yunka gateway request skipped` |
| `WARN` | 业务失败、参数错误、上游拒绝、可恢复异常、异步补偿入队 | `YUNKA_UPSTREAM_TIMEOUT` |
| `ERROR` | 未知异常、系统异常、不可预期失败、需要堆栈定位的问题 | `INTERNAL_SERVER_ERROR` |

## 4. 借款链路检查

### 4.1 借款试算 `/api/loan/calculate`

| 检查点 | 期望日志 |
|---|---|
| Controller 入口 | `loan calculate requested`，包含 `traceId`、`bizOrderNo=loan-calculate` |
| Service 调云卡前 | `loan calculate yunka request begin`，包含 `bizOrderNo=LC-*`、`requestId=LC-*`、`path=/loan/trail` |
| Service 调云卡成功 | `loan calculate yunka request success`，包含 `elapsedMs` |
| 云卡客户端出站 | `yunka gateway request begin/success`，同一 `traceId`，同一 `requestId` |

### 4.2 用信提交 `/api/loan/apply`

| 检查点 | 期望日志 |
|---|---|
| Controller 入口 | `loan apply requested`，包含 `traceId`、`memberId` |
| 权益购买成功 | `order created and synced`，包含权益订单号 |
| Service 调云卡前 | `loan apply yunka request begin`，包含 `bizOrderNo=APP-*`、`requestId=LA-*`、`benefitOrderNo`、`path=/loan/apply` |
| Service 调云卡成功 | `loan apply yunka request success`，包含 `elapsedMs` |
| 云卡超时 | `WARN`，`errorNo=YUNKA_UPSTREAM_TIMEOUT`，前端返回 `pending` / 审核中，并进入异步补偿 |
| 云卡拒绝 / 空响应 | `WARN`，`errorNo=YUNKA_UPSTREAM_REJECTED` 或 `YUNKA_RESPONSE_EMPTY`，前端返回 `loan_failed` 且表明权益购买成功 |

### 4.3 审核状态 / 审核结果

| 接口 | 期望日志 |
|---|---|
| `/api/loan/approval-status/{applicationId}` | `loan query yunka request begin/success`，`path=/loan/query` |
| `/api/loan/approval-result/{applicationId}` | `loan query yunka request begin/success`，`path=/loan/query` |

## 5. 还款链路检查

| 接口 | 云卡路径 | 期望日志 |
|---|---|---|
| `/api/repayment/info/{loanId}` | `/repay/trial` | `repayment info yunka request begin/success`，`bizOrderNo=loanId` |
| `/api/repayment/submit` | `/repay/apply` | `repayment submit yunka request begin/success`，`bizOrderNo=loanId` |
| `/api/repayment/result/{repaymentId}` | `/repay/query` | `repayment result yunka request begin/success`，`bizOrderNo=repaymentId` |

失败时统一检查：

1. 是否存在 `errorNo`；
2. 是否存在 `errorMsg`；
3. 是否有同一 `traceId` 下的云卡客户端日志；
4. `path` 是否与接口预期一致。

## 6. 绑卡链路检查

绑卡链路当前对接齐为，日志中禁止输出完整银行卡号、短信验证码、身份证号、手机号明文。

| 接口 | 期望日志 | 备注 |
|---|---|---|
| `/api/bank-card/sign-status` | `bank-card sign status qw request begin/success` | `bizOrderNo=bank-card-后四位` |
| `/api/bank-card/sign-apply` | `bank-card sign apply qw request begin/success` | 成功日志包含 `requestNo` |
| `/api/bank-card/sign-confirm` | `bank-card sign confirm qw request begin/success` | 成功后本地记录协议 |

失败时重点检查：

| 场景 | 期望错误 |
|---|---|
| 会员不存在 | `errorNo=MEMBER_NOT_FOUND` |
| 会员手机号缺失 | `errorNo=MEMBER_MOBILE_MISSING` |
| 会员姓名缺失 | `errorNo=MEMBER_NAME_MISSING` |
| 会员身份证缺失 | `errorNo=MEMBER_ID_CARD_MISSING` |
| 签约确认失败 | `errorNo=QW_SIGN_CONFIRM_FAILED` |
| 齐为调用失败 | `errorNo=QW_UPSTREAM_FAILED` 或对应齐为错误 |

## 7. 常用检索方式

### 7.1 按 traceId 串联单次请求

```bash
grep 'traceId=<traceId>' app.log
```

### 7.2 按业务号定位

```bash
grep 'bizOrderNo=<bizOrderNo>' app.log
```

### 7.3 按云卡路径定位

```bash
grep 'path=/loan/apply' app.log
grep 'path=/repay/trial' app.log
```

### 7.4 按错误码定位

```bash
grep 'errorNo=YUNKA_UPSTREAM_TIMEOUT' app.log
grep 'errorNo=QW_SIGN_CONFIRM_FAILED' app.log
```

## 8. 联调验收结论

联调时每条主链路至少满足：

1. Controller 入口日志可见；
2. Service begin / success 或 fail 日志可见；
3. 第三方调用日志可见；
4. 同一前端请求的 `traceId` 能串联；
5. 失败日志包含 `errorNo / errorMsg`；
6. 日志不出现完整银行卡号、身份证号、手机号、token、短信验证码。

## 9. 异步补偿补充检查（2026-04-23）

当 `loan apply` 因云卡超时进入异步补偿时，额外检查以下日志：

| 场景 | 期望日志 |
|---|---|
| 入队成功 | `async compensation task enqueued`，包含 `traceId`、`bizOrderNo`、`taskId`、`taskType`、`bizKey`、`partitionNo`、`targetCode`、`requestPath` |
| 重复入队 | `async compensation task duplicated, ignored`，包含 `errorNo=ASYNC_COMPENSATION_DUPLICATED` |
| worker 认领 | `async compensation task claimed`，包含 `taskId`、`partitionNo`、`workerId`、`requestPath` |
| worker 成功 | `async compensation task succeeded`，包含 `taskId`、`taskType`、`workerId` |
| worker 失败待重试 | `async compensation task failed`，日志级别 `WARN`，包含 `errorNo`、`errorMsg`、`dead=false` |
| worker 达到死任务 | `async compensation task failed`，日志级别 `ERROR`，包含 `errorNo`、`errorMsg`、`dead=true` |

补偿链路排障时，建议先按 `bizOrderNo=<applicationId>` 检索，再按 `taskId` 和 `workerId` 下钻。
