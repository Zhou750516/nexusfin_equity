# 2026-04-16 H5 还款成功与本地账务落库边界说明

## 1. 背景

`2026-04-15` H5 页面维度 MySQL 联调中，页面主流程已从首页跑通到还款成功页，但复核 MySQL 时发现：

- `member_info` 已写入
- `member_payment_protocol` 已写入
- `benefit_order` 已写入
- `loan_application_mapping` 已写入
- `payment_record` 未新增本次 H5 主动还款记录

因此 `2026-04-16` 需要明确：`payment_record` 是否属于当前 H5 主动还款链路的必需落库范围。

---

## 2. 当前代码事实

### 2.1 H5 主动还款接口当前只做云卡转发

当前 H5 还款相关接口由 `RepaymentServiceImpl` 承接：

- `GET /api/repayment/info/{loanId}`
- `POST /api/repayment/submit`
- `GET /api/repayment/result/{repaymentId}`

当前实现逻辑为：

1. `repayment/info` 转发云卡 `/repay/trial`
2. `repayment/submit` 转发云卡 `/repay/apply`
3. `repayment/result` 转发云卡 `/repay/query`
4. 响应结果映射为 H5 页面展示模型

当前 `RepaymentServiceImpl` 未注入 `PaymentRecordRepository`，也未写入 `payment_record`。

### 2.2 `payment_record` 当前职责是支付 / 代扣流水

根据当前代码和数据库文档，`payment_record` 的职责是：

- 记录首扣回调结果
- 记录兜底代扣回调结果
- 记录兜底代扣待处理流水

典型字段口径：

| 字段 | 当前典型值 |
|---|---|
| `payment_type` | `FIRST_DEDUCT` / `FALLBACK_DEDUCT` |
| `provider_code` | `QW` |
| `payment_status` | `PENDING` / `SUCCESS` / `FAIL` |

对应实现入口主要是：

- `PaymentServiceImpl.handleFirstDeductCallback`
- `PaymentServiceImpl.handleFallbackDeductCallback`
- `FallbackDeductServiceImpl.triggerFallback`

### 2.3 还款通知回调当前也不写 `payment_record`

当前 `POST /api/callbacks/repayment/forward` 由 `NotificationServiceImpl.handleRepayment` 处理。

其当前职责是：

1. 写入 `notification_receive_log`
2. 标记处理状态
3. 写入 `idempotency_record`

当前不更新订单状态，也不写 `payment_record`，也没有单独的 `repayment_record`。

## 3. 边界结论

### 3.1 当前 H5 主动还款成功不等于本地 `payment_record` 已落库

当前页面显示“还款成功”的判断依据是：

1. `POST /api/repayment/submit` 调云卡 `/repay/apply` 成功，拿到 `repaymentId`
2. `GET /api/repayment/result/{repaymentId}` 调云卡 `/repay/query` 返回成功状态
3. H5 页面根据云卡返回结果展示“还款成功”

因此，当前链路中：

> H5 还款成功 = 云卡主动还款链路返回成功；不代表本地 `payment_record` 已新增支付流水。

### 3.2 `payment_record` 不是当前 H5 主动还款链路的必需落库表

基于现有实现与历史文档，`payment_record` 当前定位仍是：

- 权益服务费首扣流水
- 兜底代扣流水
- 支付渠道回调与对账留痕

它不是当前 H5 主动还款 `/api/repayment/*` 的设计内落库表。

### 3.3 如后续需要本地账务强一致，不建议直接复用现有 `payment_record`

如果后续业务要求：

- H5 主动还款申请要本地落库
- 还款结果要本地核销
- 需要本地支持还款记录、还款对账、争议处理

建议新增独立模型，例如：

- `repayment_record`
- 或 `loan_repayment_record`

而不是直接把主动还款塞入现有 `payment_record`。

原因：

1. `payment_record` 当前语义偏权益服务费支付 / 代扣
2. 主动还款属于借款还款域，业务语义不同
3. 现有 `payment_type` 枚举不覆盖主动还款
4. 复用 `payment_record` 容易混淆“权益费支付流水”和“贷款还款流水”

---

## 4. 当前推荐口径

### 短期口径

当前 Phase 9 / H5 页面联调阶段：

- 不要求 `POST /api/repayment/submit` 写入 `payment_record`
- 不以 `payment_record` 是否新增作为 H5 主流程成功标准
- 仍以云卡 `/repay/apply` 和 `/repay/query` 返回作为页面成功依据

### 中期口径

进入生产化或对账阶段前，需要单独确认：

1. 是否需要本地还款记录表
2. 是否需要保存云卡 `swiftNumber` / `repaymentId`
3. 是否需要落还款申请、处理中、成功、失败状态
4. 是否需要承接科技平台跳转携带的 `repaymentId`
5. 是否需要将还款状态变更推送给科技平台
