# 2026-04-21 艾博生后端日志完备性 Review 计划

## 1. 背景

当前艾博生后端已具备较多入口访问日志，控制器层大多已经统一输出：

- `traceId`
- `bizOrderNo`

但从“页面请求进入艾博生后端”到“艾博生后端调用云卡，再由云卡转发科技平台”的完整链路看，日志是否**真的可以串起来**，仍需要专项 review。

本次 review 的重点，不是简单统计“有没有日志”，而是确认：

> 是否能通过同一个 `traceId`，把一笔真实请求从 H5 / 前端入口，一直串到艾博生请求云卡的出站调用、返回结果、异常分支和补偿分支。

尤其关注：

- 请求云卡时，是否能从日志中明确看到当前请求的 `traceId`
- 是否能看到该次请求对应的 `path`
- 是否能看到该次请求的 `requestId`
- 是否能看到该次请求的 `bizOrderNo`
- 成功、失败、超时时，日志是否形成闭环

## 2. Review 目标

本次日志完备性 review 目标如下：

1. 明确艾博生后端当前日志覆盖现状；
2. 检查关键链路是否具备统一日志口径；
3. 检查请求云卡时 `traceId` 是否可串联；
4. 检查失败、超时、空响应、异常回退场景是否留痕充分；
5. 输出缺口清单、优先级和后续整改建议。

## 3. Review 范围

本次 review 仅覆盖与当前 H5 / 云卡 / 科技平台主链直接相关的后端日志，不扩展到全部低优先级模块。

### 3.2 本次重点关注对象

本次重点关注以下层次的日志完整性：

1. **Controller 入口日志**
2. **Service 业务日志**

4. **异常处理日志**

## 4. 当前初步判断

基于现有代码快速盘点，可先形成如下初步判断：

### 4.1 已具备的部分

- 大多数 Controller 已有访问日志；
- 控制器日志普遍已带 `traceId + bizOrderNo`；
- 异步补偿模块已有较明确的运行摘要、成功 / 失败日志；
- 科技平台用户校验、权益状态回推、支付协议回调等部分链路已有日志基础。

### 4.2 明显需要重点复核的部分

#### 云卡客户端日志偏弱

当前 `RestYunkaGatewayClient` 更偏“调用封装”，日志证据较弱，需重点核查：

- 是否打印云卡请求开始日志；
- 是否打印云卡返回成功日志；
- 是否打印云卡超时日志；
- 是否打印云卡非超时异常日志；
- 是否打印请求参数中的关键字段：
  - `traceId`
  - `path`
  - `requestId`
  - `bizOrderNo`

#### Service 层链路日志不够闭环

以 `LoanServiceImpl`、`RepaymentServiceImpl` 为代表，虽然业务逻辑完整，但需重点检查：

- 调云卡前是否有统一 begin 日志；
- 返回成功后是否有 success 日志；
- 失败时是否有分类日志；
- 超时转异步补偿时是否有明确留痕；
- “返回前端成功 / 失败 / 审核中”口径是否在日志中有明确说明。

#### 全局异常日志不足

当前 `GlobalExceptionHandler` 负责统一返回，但需要重点 review：

- `BizException` 是否记录日志；
- 参数校验异常是否记录日志；
- 未知异常是否记录带 `traceId` 的错误日志；
- 是否能从统一异常日志回溯到具体接口与业务单号。

## 5. 本次 Review 核心口径

### 5.1 核心判断标准

如果一笔请求发生问题，review 后应能回答以下问题：

1. 这笔请求的 `traceId` 是什么？
2. 对应的 `bizOrderNo` 是什么？
3. 请求进入的是哪个接口？
4. 业务层是否决定要调用云卡？
5. 调用云卡时，请求的 `path` 是什么？
6. 调用云卡时，请求的 `requestId` 是什么？
7. 云卡调用成功、失败还是超时？
8. 如果失败，是同步失败、上游失败、空响应还是解析失败？
9. 如果进入异步补偿，任务是否入队成功？后续是否重试？
10. 最终前端看到的页面口径是什么？

如果以上任一问题无法从日志中快速回答，就说明日志完备性仍不够。

### 5.2 云卡 traceId 串联要求

本次 review 最重要的一条要求：

> 对于每一次请求云卡的调用，必须能够通过日志使用同一个 `traceId` 串起“入口 → 业务组装 → 云卡调用 → 返回 / 异常”。

建议至少满足：

- Controller 入口日志带 `traceId`
- Service 调用云卡前日志带 `traceId`
- `RestYunkaGatewayClient` 出站日志带 `traceId`
- 云卡调用完成日志带 `traceId`
- 超时 / 异常日志带 `traceId`
- 异步补偿入队日志带 `traceId`

### 5.3 错误日志分级要求

本次 review 不仅看“有没有错误日志”，还要看**错误日志分级是否合理**。

建议采用以下口径，并要求配置层支持按环境切换日志级别。

#### 日志等级配置要求

后端需要具备清晰的日志等级设置能力，本次统一按以下四级定义：

| 模式 | 建议级别 | 使用场景 | 说明 |
| --- | --- | --- | --- |
| `DEBUG` 模式 | `DEBUG` | 联调 / 测试环境排查 | 用于查看关键请求参数摘要、下游调用摘要、业务分支判断，不打印敏感明文 |
| `INFO` 模式 | `INFO` | 默认运行模式 | 用于记录正常业务流转、关键状态变化、出站调用结果 |
| `WARN` 模式 | `WARN` | 异常信号收敛 | 用于记录业务可继续但已出现异常信号的场景 |
| `ERROR` 模式 | `ERROR` | 故障收敛 / 降噪模式 | 仅保留失败、异常、不可恢复问题，适合临时降低日志噪音 |

补充说明：

- 配置项至少要能覆盖 `INFO`、`DEBUG`、`WARN`、`ERROR` 四种模式；
- review 时需要检查当前 `application.yml` 或环境变量是否支持快速切换；
- 建议保留包级别配置，例如：

```yaml
logging:
  level:
    root: ${LOG_LEVEL_ROOT:INFO}
    com.nexusfin.equity: ${LOG_LEVEL_APP:INFO}
    com.nexusfin.equity.thirdparty.yunka: ${LOG_LEVEL_YUNKA:INFO}
    com.nexusfin.equity.thirdparty.qw: ${LOG_LEVEL_QW:INFO}
    com.nexusfin.equity.thirdparty.techplatform: ${LOG_LEVEL_TECH_PLATFORM:INFO}
```

联调期建议：

- 默认 `LOG_LEVEL_APP=INFO`
- 排查云卡链路时临时设置 `LOG_LEVEL_YUNKA=DEBUG`
- 排查 traceId 串联时临时设置 `LOG_LEVEL_APP=DEBUG`
- 出现大量噪音时可临时设置局部包为 `ERROR`

注意：

- `DEBUG` / `TRACE` 模式下仍不能打印身份证、手机号、银行卡号、完整 token、完整密文、完整签名密钥；
- 如需打印请求参数，只允许打印脱敏摘要、字段名、长度、hash、业务单号、路径、请求编号等安全信息。

#### `DEBUG`

用于记录调试级业务上下文，例如：

- 云卡请求摘要；
- 科技平台请求摘要；
- 齐为请求摘要；
- 关键 DTO 字段摘要；
- 状态映射前后的值。

使用要求：

- 可在联调环境开启；
- 生产环境默认关闭；
- 不允许输出敏感明文。

#### `INFO`

用于记录正常业务流转结果，例如：

- 请求进入；
- 关键业务动作开始；
- 调用云卡成功；
- 异步补偿执行成功；
- 状态回推成功。

#### `WARN`

用于记录“业务可继续，但已出现异常信号”的场景，例如：

- 上游返回业务失败；
- 上游返回空响应；
- 参数不符合预期但已被业务兜底；
- 进入降级逻辑；
- 进入异步补偿；
- 非致命重试；
- 幂等重复请求。

#### `ERROR`

用于记录“本次处理已失败，或需要立即排查”的场景，例如：

- 云卡调用超时；
- 云卡调用异常中断；
- 科技平台调用异常；
- 未捕获运行时异常；
- 异步补偿达到死任务；
- 回调处理失败；
- 统一异常兜底返回 500；
- 关键数据缺失且无法继续处理。

review 时需要重点检查：

- 是否把真正失败的场景仍然打成 `info`；
- 是否所有失败都只打 `error`，缺少中间层级；
- 是否存在“日志级别过低导致线上检索困难”的问题；
- 是否存在“日志级别过高导致大量噪音”的问题。
- 是否支持通过配置快速切换 `INFO` / `DEBUG` / `WARN` / `ERROR` 模式；
- 是否对云卡、齐为、科技平台等第三方调用包提供独立日志级别配置。

### 5.4 `errorNo / errorMsg` 统一口径要求

本次 review 需要补充检查：

> 出错时是否能输出清晰、稳定、可检索的 `errorNo` 与语义准确的 `errorMsg`。

建议统一原则如下：

#### `errorNo`

`errorNo` 用于：

- 稳定标识错误类别；
- 支撑检索、聚合、监控、告警；
- 避免只依赖自然语言 message 检索失败。

要求：

- 必须稳定，不随文案轻易变化；
- 一类错误一个固定编号 / 编码；
- 同一错误在不同层不要随意改名；
- 推荐优先复用现有 `BizException` code；
- 若是日志侧补充字段，也应保证和返回码语义一致。

示例：

- `YUNKA_UPSTREAM_TIMEOUT`
- `YUNKA_UPSTREAM_FAILED`
- `YUNKA_RESPONSE_EMPTY`
- `YUNKA_RESPONSE_INVALID`
- `TECH_PLATFORM_UPSTREAM_FAILED`
- `PARAM_INVALID`
- `APPLICATION_MAPPING_NOT_FOUND`

#### `errorMsg`

`errorMsg` 用于：

- 面向排障人员提供清晰上下文；
- 清楚表达“哪里错了、怎么错的”；
- 避免模糊、泛化、不可行动的表述。

要求：

- 语义准确；
- 不使用“系统异常”“请求失败”这类过于泛的描述作为唯一信息；
- 能区分“超时”“空响应”“上游业务拒绝”“本地参数错误”“本地状态缺失”等类型；
- 如需拼接上游 message，应保留本地语义前缀；
- 不直接输出敏感明文。

推荐格式：

- `errorNo=<稳定错误码> errorMsg=<清晰语义描述>`

示例：

- `errorNo=YUNKA_UPSTREAM_TIMEOUT errorMsg=Call Yunka gateway timeout when applying loan`
- `errorNo=YUNKA_RESPONSE_EMPTY errorMsg=Yunka gateway returned empty response for repayment submit`
- `errorNo=APPLICATION_MAPPING_NOT_FOUND errorMsg=Loan application mapping not found`

本次 review 需要重点检查：

- 日志里是否只有 message，没有稳定错误编号；
- message 是否过短、过泛；
- 同类错误是否出现多个不同写法；
- 返回给前端的错误口径与日志内部口径是否完全脱节。

## 6. Review 检查项

### 6.1 Controller 入口日志检查

检查项：

- 是否每个关键接口都有访问日志；
- 是否统一输出 `traceId`；
- 是否统一输出 `bizOrderNo`；
- 是否带必要身份上下文，例如：
  - `memberId`
  - `applicationId`
  - `loanId`
  - `repaymentId`

预期结果：

- 每个关键入口至少有一条 `info` 级访问日志；
- 能通过入口日志快速定位请求起点。

### 6.2 Service 业务日志检查

检查项：

- 关键业务分支是否有 begin / result / fail 留痕；
- 业务决策点是否可见，例如：
  - 是否购买权益
  - 是否调用云卡
  - 是否转异步补偿
  - 是否返回“审核中”
- 失败时是否区分：
  - 业务失败
  - 参数不合法
  - 上游失败
  - 超时
  - 空响应

预期结果：

- 能从 Service 日志判断业务走到了哪一步；
- 不是只有入口日志，没有业务中间态。

### 6.3 云卡调用日志检查

这是本次 review 的核心项。

检查项：

- 调用云卡前是否打印 begin 日志；
- begin 日志是否带：
  - `traceId`
  - `bizOrderNo`
  - `requestId`
  - `path`
- 云卡成功返回后是否打印 success 日志；
- success 日志是否带：
  - `traceId`
  - `bizOrderNo`
  - `requestId`
  - `path`
  - 返回码
  - 耗时
- 云卡失败 / 超时后是否打印 fail 日志；
- fail 日志是否区分：
  - timeout
  - upstream_failed
  - response_empty
  - response_invalid

预期结果：

- 任意一笔云卡调用都能完整查到 begin / success / fail；
- 同一条链路中的 `traceId` 保持一致；
- 能清楚区分“调用没发出去”还是“对方返回失败”。

### 6.4 全局异常日志检查

检查项：

- `BizException` 是否统一打印日志；
- `MethodArgumentNotValidException` 是否打印请求参数校验失败日志；
- `Exception` 是否打印完整错误堆栈；
- 是否统一输出：
  - `errorNo`
  - `errorMsg`
- 是否按异常严重程度使用合理日志级别；
- 日志中是否保留：
  - `traceId`
  - `bizOrderNo`
  - 接口信息

预期结果：

- 即使业务层漏记日志，最终异常仍可被统一收口；
- 不出现“前端看到 500，但日志没有任何错误上下文”的情况。

### 6.5 异步补偿日志检查

检查项：

- 入队时是否打印任务类型、业务单号、目标地址；
- worker 执行时是否带 `traceId + bizOrderNo`；
- 失败重试是否带 attempt 信息；
- dead task、回收租约、分片运行摘要是否足够支撑运维排查。

预期结果：

- 超时转异步后，链路仍不断；
- 能从日志追到“同步失败 → 入队 → 重试 → 成功 / 死任务”。

## 7. 优先级划分

### P0：必须优先补齐

1. 云卡客户端出站 begin / success / fail 日志；
2. 云卡请求关键参数日志：
   - `traceId`
   - `bizOrderNo`
   - `requestId`
   - `path`
3. `GlobalExceptionHandler` 的统一异常日志；
4. 明确错误日志分级规则，避免失败场景日志级别失真；
5. 统一 `errorNo / errorMsg` 输出口径；
6. `LoanServiceImpl`、`RepaymentServiceImpl` 调云卡前后的业务日志闭环；
7. 超时转异步补偿时的明确日志留痕。

### P1：建议紧随补齐

1. 绑卡三接口对齐统一日志口径；
2. 科技平台出站接口日志增加耗时与返回码；
3. 下游调用的错误分类日志更细化；
4. 对关键链路补充最小化日志测试；
5. 增加按包维度的日志等级配置，至少支持 `INFO`、`DEBUG`、`WARN`、`ERROR` 模式。

### P2：后续优化项

1. 统一结构化日志字段；
2. 接入日志检索与告警规则；
3. 输出联调期专用日志查询模板；
4. 把核心日志点沉淀到运维 SOP 与测试文档。

## 8. Review 执行步骤

### Step 1：建立日志字段基线

先确认每类日志最少必须包含哪些字段：

- `traceId`
- `bizOrderNo`
- `requestId`
- `path`
- `errorNo`
- `errorMsg`
- `memberId` / `applicationId` / `loanId` / `repaymentId`（按场景）

同时确认日志等级配置基线：

- 是否存在应用默认级别；
- 是否存在云卡包独立级别；
- 是否存在齐为包独立级别；
- 是否存在科技平台包独立级别；
- 是否可通过环境变量切换到 `INFO` / `DEBUG` / `WARN` / `ERROR`。

### Step 2：按链路逐一走读代码

优先按以下顺序 review：

1. 借款试算
2. 借款申请
3. 审批查询
4. 还款试算
5. 还款提交
6. 还款结果查询
7. 绑卡三接口

### Step 3：输出缺口表

按接口输出：

- 当前已有日志点
- 缺失的日志点
- 缺失影响
- 建议补法
- 优先级

### Step 4：形成整改清单

最终形成开发可执行的任务清单：

- 哪些文件需要补日志
- 哪些日志字段需要统一
- 哪些异常日志需要补齐
- 哪些链路需要加验证用例

5. **错误日志分级与 errorNo / errorMsg 规范**

## 10. 本次 Review 的明确结论口径

本次 review 的结论不应只停留在“有日志 / 没日志”，而要明确回答：

1. 艾博生后端现有日志是否足以支撑联调排障？
2. 艾博生调用云卡时，`traceId` 是否可以完整串起来？
3. 云卡相关请求参数是否留痕充分？
4. 成功、失败、超时、补偿链路是否都有日志闭环？
5. 哪些缺口必须马上补，哪些可以后置？

---

如果后续进入开发整改阶段，建议优先以“云卡链路 traceId 串联”作为第一验收标准。
