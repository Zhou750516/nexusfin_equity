# 2026-04-17 本地还款记录模型待确认项

## 1. 文档目标

本文档用于沉淀“是否需要单独建设本地还款记录模型”的后续确认项，避免后续再次把：

1. H5 页面还款成功
2. 云卡代理返回成功
3. 本地账务已完整落库

三者误认为同一件事。

本文档与 `plan/20260416_H5还款成功与本地账务落库边界说明.md` 配套使用。

---

## 2. 当前已确认口径

截至 `2026-04-17`，当前阶段继续保持以下结论不变：

1. H5 主动还款主流程以云卡 `/repay/apply` 与 `/repay/query` 返回结果为主
2. 当前不要求 `POST /api/repayment/submit` 必须落本地还款记录表
3. 当前不以 `payment_record` 是否新增作为 H5 主流程成功判断标准
4. 当前不建议把主动还款直接塞入现有 `payment_record`

因此，短期内继续维持：

> H5 还款成功 = 当前代理链路成功，不等于本地还款账务模型已建设完成。

---

## 3. 为什么要单独管理这个待确认项

如果不单独把该事项挂出来，后续很容易在以下场景再次产生歧义：

1. 验收时追问“为什么页面成功但数据库里没新增对应记录”
2. 对账时追问“本地是否可独立还原还款申请和结果”
3. 查询页或运营页要展示历史还款流水
4. 失败补偿、重试、人工核对需要本地事件轨迹

所以当前更合理的做法是：

1. 先明确本轮不做
2. 再把后续是否要做、做到什么程度单独列清楚

---

## 4. 后续需要确认的核心问题

### 4.1 是否需要新增独立模型

需要业务 / 产品 / 技术共同确认：

1. 是否需要新增 `repayment_record` 或 `loan_repayment_record`
2. 是否只需要申请流水，还是申请 + 结果 + 回调全链路
3. 是否需要支持运营查询、人工补单、对账追踪

### 4.2 模型边界是否覆盖全链路

若未来要建设本地还款记录模型，需要进一步确认至少包括：

1. 还款申请落库
2. 云卡受理结果回写
3. 还款结果查询回写
4. 科技平台回调结果回写
5. 最终状态归档

### 4.3 主键与外部流水如何映射

后续至少要确认以下标识关系：

1. 本地还款单号
2. `loanId`
3. 云卡请求流水
4. 科技平台 `swiftNumber`
5. H5 页面 `repaymentId`

### 4.4 是否需要查询与对账能力

需要确认后续是否要补：

1. 本地还款记录查询接口
2. 运营侧查询页
3. 失败重试或人工处理机制
4. 定时对账 / 差错处理能力

---

## 5. 当前推荐推进口径

### P0：短期口径继续保持不变

1. 当前不要求新增本地还款记录表
2. 当前不改造 `payment_record`
3. 当前不把“页面成功”和“本地账务强一致”绑定验收

### P1：如果业务开始明确要求强一致

届时建议单独立项，而不是在现有链路上临时补丁式修改。建议单独输出：

1. 还款记录模型设计文档
2. 表结构与状态机说明
3. 回调与查询回写方案
4. 对账与补偿方案

### P2：如果只是维持页面主流程

则继续沿用当前口径，不新增模型，不扩展落库范围。

---

## 6. 当前一句话结论

> 当前 H5 主动还款链路可以继续按“云卡代理成功即可完成页面主流程”推进；如后续确实要求本地账务强一致，应单独建设独立还款记录模型，而不是直接复用 `payment_record`。
