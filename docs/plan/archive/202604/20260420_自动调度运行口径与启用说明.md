# 2026-04-20 自动调度运行口径与启用说明

## 1. 目标

在 `2026-04-19` 已完成“自动调度第一轮骨架 + 默认关闭无副作用验证”的基础上，补一版面向研发 / 联调 / 运维同学的**运行口径与启用说明**，重点回答：

1. `workerId` 现在怎么生成、什么时候建议显式配置
2. `heartbeat` 现在代表什么含义
3. `currentTaskId` 在当前阶段的含义是什么
4. 如果要在某环境显式开启自动调度，应该怎么配

## 2. 当前运行口径

### 2.1 `workerId` 口径

当前代码口径：

1. 若显式配置了 `nexusfin.async-compensation.worker-id`
   - 直接使用该值
   - 会自动 `trim`
2. 若未显式配置
   - 退化为 `spring.application.name + runtimeIdentity`
   - 当前 `runtimeIdentity` 优先取 JVM runtime name
   - 若 runtime name 不可用，再退化为主机名

### 当前判断

这个口径对以下场景已经足够：

1. 本地开发
2. 单机测试
3. 明确知道实例部署位置的联调环境

但对以下场景，**更建议显式配置**：

1. 多实例部署
2. 需要人工按实例排查日志
3. 需要明确实例与分片归属关系

### 建议

后续若进入准生产或稳定联调环境，建议：

1. `worker-id` 显式配置
2. 命名中带上环境 / 实例职责 / 分片提示

例如：

1. `async-worker-a`
2. `async-worker-partition-0-3`
3. `staging-async-supervisor-1`

## 2.2 `heartbeat` 口径

当前自动调度已调整为以下语义：

### 场景 A：本次 tick 没有认领到任务

1. 不会上报 `RUNNING`
2. tick 结束时上报：
   - `workerStatus = IDLE`
   - `currentTaskId = null`

### 场景 B：本次 tick 认领到任务

1. `worker service` 在任务被认领后，通过回调把 `taskId` 交回给 `coordinator`
2. `coordinator` 立即上报：
   - `workerStatus = RUNNING`
   - `currentTaskId = taskId`
3. 本次 tick 结束后统一回到：
   - `workerStatus = IDLE`
   - `currentTaskId = null`

### 当前判断

这意味着：

1. `RUNNING` 现在表示“当前 tick 的确已认领到一个具体任务”
2. 不再出现“任务已经处理完了才补打 `RUNNING`”的语义偏差
3. `IDLE` 表示当前 tick 结束后，该分片没有处于持续处理态

## 2.3 `currentTaskId` 口径

当前阶段的 `currentTaskId` 含义是：

> **本次 worker tick 已认领并进入处理的任务 ID**

其生命周期为：

1. 任务认领成功时写入
2. tick 完成时清空

### 当前阶段的边界

当前阶段仍然是：

1. 单次 tick 内同步处理一个任务
2. 不做长任务中途持续心跳
3. 不做并行多任务执行

因此：

1. `currentTaskId` 更适合作为“当前正在处理哪个任务”的瞬时观察值
2. 不适合作为长时运行监控的唯一依据
3. 历史执行留痕仍应以：
   - `async_compensation_task`
   - `async_compensation_attempt`
   - `async_compensation_partition_runtime`
   结合来看

## 3. 当前推荐启用策略

当前仍维持以下总策略不变：

1. **默认关闭**
2. **显式开启**
3. **显式分片**

也就是：

1. 不建议通过默认配置让所有环境自动跑起来
2. 必须显式决定谁跑 worker、谁跑 supervisor
3. 必须显式决定当前实例负责哪些 partition

## 4. 配置项说明

当前关键配置项如下：

1. `nexusfin.async-compensation.enabled`
   - 总开关
2. `nexusfin.async-compensation.worker-enabled`
   - 是否启用 worker 调度
3. `nexusfin.async-compensation.supervisor-enabled`
   - 是否启用 supervisor 调度
4. `nexusfin.async-compensation.worker-id`
   - 当前实例 worker 标识
5. `nexusfin.async-compensation.owned-partitions`
   - 当前实例负责的 partition 列表
6. `nexusfin.async-compensation.worker-poll-interval-ms`
   - worker 轮询周期
7. `nexusfin.async-compensation.supervisor-interval-ms`
   - supervisor 巡检周期

## 5. 单实例启用示例

适用于：

1. 本地手工验证
2. 单实例联调环境
3. 任务量较小的临时环境

示例：

```yaml
nexusfin:
  async-compensation:
    enabled: true
    worker-enabled: true
    supervisor-enabled: true
    worker-id: local-async-node-1
    owned-partitions: 0,1,2,3,4,5,6,7
    worker-poll-interval-ms: 3000
    supervisor-interval-ms: 15000
```

## 6. 双实例显式分片示例

适用于：

1. 小规模多实例联调
2. 需要人工分片归属的环境

### 实例 A

```yaml
nexusfin:
  async-compensation:
    enabled: true
    worker-enabled: true
    supervisor-enabled: false
    worker-id: async-worker-a
    owned-partitions: 0,1,2,3
```

### 实例 B

```yaml
nexusfin:
  async-compensation:
    enabled: true
    worker-enabled: true
    supervisor-enabled: true
    worker-id: async-worker-b
    owned-partitions: 4,5,6,7
```

说明：

1. 第一阶段建议只有一个实例承担 `supervisor`
2. 分片仍采用人工显式归属
3. 当前不做自动抢占与 rebalance

## 7. 启用后重点观察项

若某环境显式开启自动调度，建议重点观察：

### 7.1 运行时表

重点看 `async_compensation_partition_runtime`：

1. `worker_id` 是否符合预期
2. `worker_status` 是否在 `RUNNING / IDLE` 间合理变化
3. `current_task_id` 是否在任务处理期间短暂出现
4. `last_heartbeat_ts` 是否持续更新

### 7.2 任务表

重点看 `async_compensation_task`：

1. `INIT / RETRY_WAIT` 是否逐步下降
2. `SUCCESS` 是否增长
3. `DEAD` 是否异常增长

### 7.3 尝试日志表

重点看 `async_compensation_attempt`：

1. 是否有新的执行记录产生
2. 失败是否具备明确 `errorCode / errorMessage`
3. `worker_id` 是否与实例归属一致

## 8. 当前不承诺事项

本说明只覆盖第一阶段自动调度接入口径，当前仍不承诺：

1. 独立 runner 进程
2. 多实例自动分片编排
3. 长任务持续心跳
4. 动态扩缩容
5. 外部任务平台 / MQ 接入

## 9. 一句话结论

> 当前自动调度已经从“可跑骨架”推进到“有明确 `workerId`、`heartbeat`、`currentTaskId` 口径，且具备显式启用说明”的状态；但仍坚持第一阶段的默认关闭与显式分片策略。
