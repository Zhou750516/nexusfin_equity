# 2026-03-31 通联直连联调执行清单

## 1. 文档目的

本文档用于把当前通联直连主线的联调准备状态压缩成一页，便于：

1. 我方内部快速判断“现在已经做到哪一步”
2. 和齐为 / 通联快速对齐还缺哪些外部资料
3. 拿到协议资料后，按固定路径继续落代码和联调

适用范围：

1. 仅覆盖 `ALLINPAY_DIRECT` 模式
2. 仅覆盖通联 `https://tlt-test.allinpay.com/aipg/ProcessServlet`
3. 不覆盖科技平台 SSO、导流页面、还款页面等其他链路

## 3. 业务主链范围

本轮通联直连只覆盖以下三个既有业务动作：

| 业务动作 | 业务入口 | 下游调用 |
| --- | --- | --- |
| 会员 / 订单同步 | `BenefitOrderServiceImpl.createOrder()` | `QwBenefitClient.syncMemberOrder()` |
| 联登链接获取 | `BenefitOrderServiceImpl.getExerciseUrl()` | `QwBenefitClient.getExerciseUrl()` |
| 放款结果通知 | `NotificationServiceImpl.handleGrant()` | `QwBenefitClient.notifyLending()` |

这三个入口当前均已接入 `ALLINPAY_DIRECT` 主链，但默认仍会在 skeleton 边界阻断真实外发或真实解析。

## 5. 当前代码替换点

拿到齐为 / 通联正式资料后，只需要优先替换以下组件：

| 顺序 | 组件 | 当前状态 | 外部资料到位后需要做什么 |
| --- | --- | --- | --- |
| 1 | `AllinpayDirectRequestFactory` | 已可按动作生成内部调用上下文 | 回填三个动作的真实 `serviceCode` |
| 2 | `AllinpayDirectProtocolSerializer` | skeleton | 将内部信封映射为真实 XML / 表单请求 |
| 3 | `AllinpayDirectRequestPreparer` | 可签名，但签名口径仍是通用骨架 | 按真实签名串规则调整待签名原文 |
| 4 | `AllinpayDirectTransportMapper` | skeleton | 明确签名字段、header、body 或 form 的最终落点 |
| 5 | `AllinpayDirectHttpExecutor` | 已有通用 HTTP 执行器 | 若协议无需特殊传输逻辑，可直接复用 |
| 6 | `AllinpayDirectResponseSignatureResolver` | 默认按响应头提取 | 改成真实签名值提取规则 |
| 7 | `AllinpayDirectResponseVerificationStage` | skeleton | 接入真实响应验签口径 |
| 8 | `AllinpayDirectResponseParser` | skeleton | 将已验签响应映射到三个业务 DTO |

补充说明：

1. 默认协议 Bean 已由 `AllinpayDirectConfiguration` 统一提供。
2. 后续优先通过替换 Spring Bean 覆盖 skeleton 组件，不建议直接改 `AllinpayDirectQwBenefitClient` 主流程。
3. `AllinpayDirectRequestFactory`、`AllinpayDirectPayloadMapperRegistry`、`AllinpayDirectEnvelopeFactory` 也已纳入统一装配，后续可继续通过替换 Bean 覆盖默认实现。

当前不要改的部分：

1. `BenefitOrderServiceImpl.createOrder()`
2. `BenefitOrderServiceImpl.getExerciseUrl()`
3. `NotificationServiceImpl.handleGrant()`
4. `QwBenefitClient` 的三个业务方法定义

## 6. 外部阻塞项

当前影响进入真实联调的阻塞项如下：

| 阻塞项 | 需要谁确认 | 不确认的直接影响 |
| --- | --- | --- |
| 三个业务动作对应的真实接口 / `serviceCode` | 齐为 / 通联 | 不能发真实业务请求 |
| 请求协议格式和字段定义 | 齐为 / 通联 | 不能生成正确请求报文 |
| 签名串规则与签名字段位置 | 齐为 / 通联 | 不能通过请求签名校验 |
| 响应签名来源和验签原文规则 | 齐为 / 通联 | 不能完成回包可信校验 |
| 响应成功码 / 失败码 / 数据节点结构 | 齐为 / 通联 | 不能解析业务返回 |
| `public-rsa.cer` / `allinpay-pds.cer` 的实际用途 | 齐为 / 通联 | 容易用错证书导致误判 |
| 双向 TLS 要求 | 齐为 / 通联 | 无法确认 HTTPS 握手配置是否正确 |

## 7. 推荐联调执行顺序

建议按以下顺序推进，不要跳步：

1. 先用 `docs/plan/20260331_齐为通联直连外部确认清单.md` 对齐外部协议资料。
2. 再用 `docs/plan/20260331_通联直连联调字段待填模板.md` 回填真实字段、样例和签名规则。
3. 先替换 `RequestFactory + ProtocolSerializer + TransportMapper`，完成真实请求出站。
4. 再替换 `ResponseSignatureResolver + ResponseVerificationStage + ResponseParser`，完成真实回包处理。
5. 最后切 `ALLINPAY_DIRECT` 做建单、联登、放款通知三个动作的端到端联调。

## 9. 联调完成判定

后续真实联调时，至少要补齐以下证据：

| 检查项 | 完成标准 | 证据 |
| --- | --- | --- |
| 建单请求成功发出 | 请求报文与样例一致 | 请求日志 / 报文截图 |
| 建单响应通过验签并可解析 | 本地拿到有效业务字段 | 响应日志 / DTO 输出 |
| 联登链接可获取 | 返回真实可访问的链接或 token | 响应日志 / 页面跳转记录 |
| 放款通知可回执 | 下游返回明确成功回执 | 响应日志 / 通知状态记录 |
| 失败样例可识别 | 失败码、错误信息可分类记录 | 错误日志 / 对方样例 |
| 数据库主链不回归 | MySQL 回归仍通过 | 测试结果 |
