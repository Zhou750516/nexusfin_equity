# 2026-04-10 科技平台对接会议纪要

## 1. 会议主题

围绕科技平台（小花）、云卡、艾博生三方在借款、用信审批、行权、还款、联合登录、短信与联调排期等事项，统一当前业务结论、待确认问题和后续行动项。

## 2. 核心结论

### 2.1 导流与页面配置

1. 拒件导流页中的出资机构配置由科技平台侧负责。
2. 试算页本期仅允许借 `3` 期，后期再支持灵活返回期数。
3. 试算页不展示“放款资方”信息，相关展示项本期去掉。
4. 借款详情与协议点击后进入同一个页面，该页面已提供。
5. 除现有页面外，还需要补充一份借款协议。
6. 协议页面需要支持半弹窗展示，并要求强制阅读。

### 2.2 联合登录与协议交互

1. 小花科技跳转艾博生页面时，需要采用联合登录技术方案。
2. 当前艾博生给出的技术思路为：
   - 小花科技跳转到艾博生指定 `URL`
   - 跳转时携带一个 `token`
   - 艾博生后端收到 `token` 后，调用小花科技接口校验用户有效性，并查询用户四项核心信息
   - 当前已明确的核心字段包括：`uid`、姓名、身份证、银行卡、手机号
   - 若校验有效，则由艾博生侧 `setCookie`，维持登录态
3. 该联合登录方案仍需小花技术同学确认是否支持及具体接口形式。
4. 试算页中所有需要展示的小花协议，原则上由小花以接口形式提供，不走静态页面硬编码。

### 2.3 用信审批与绑卡

1. 用信审批流程上，先购买权益。
2. 权益成功后，再给云卡，再由云卡给到小花侧。
3. 用信开通服务协议流程中存在绑卡环节。
4. 绑卡环节需要短信验证。
5. 用信审批结果方面，小花侧可以同时提供查询能力和回调能力。

### 2.4 行权与活体数据

1. 用户购买权益后，小花侧需要提供用户行权入口。
2. 活体数据是否需要在用信阶段由小花侧接收，目前尚未定。
3. 如果需要活体数据，则需要从上游开始推送，再由云卡转给小花侧。

### 2.5 还款页面与银行卡处理

1. 还款场景从小花试算页中的第三方订单进入。
2. 小花侧当前已有试算能力，可先在内部完成试算，再跳转到艾博生还款页。
3. 该跳转过程同样需要考虑联合登录问题。
4. 还款页需要展示用户银行卡信息。
5. 云卡需要通过 `uid` 查询小花侧的已绑卡列表。
6. 若还款所用银行卡此前未签约，则需要增加短信验证与签约流程。

### 2.6 短信、金额与排期

1. 当前短信触达默认携带小花签名。
2. 短信签名是否需要修改，需与淏哥确认，且需覆盖贷后场景。
3. 借款金额是否允许修改，当前仍待张蒙确认。
4. 开发、联调、上线的大致时间点，以及联调环境部署时间，仍需小花侧给出明确排期。

## 6. 风险与建议

### 6.1 当前风险

1. 若联合登录方案不尽快确认，借款页、协议页、还款页三条链路都会受阻。
2. 若协议仍不能以接口形式提供，前端页面展示方式与强制阅读交互难以定版。
3. 若还款补签约和卡列表查询方案未定，贷后还款主链无法闭环。
4. 若联调环境部署时间不明确，开发排期与联调窗口无法可靠评估。

### 6.2 建议下一步

1. 优先以“联合登录 + 协议接口 + 还款卡列表 / 签约 + 联调排期”四项作为对外首轮确认重点。
2. 建议将本纪要同步给小花、淏哥、张蒙，并要求按责任项逐条回复。
3. 小花侧回复后，再回填到 `docs/plan/20260404_科技平台接口与改造需求_修订版.md` 与对外问题清单中。
