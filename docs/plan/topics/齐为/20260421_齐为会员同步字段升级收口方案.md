# 2026-04-21 齐为会员同步字段升级收口方案

## 1. 文档目的

本文档用于在已完成“齐为签约后端骨架”基础上，进一步收口 `abs.push.order` 会员同步接口的字段升级方案。

重点收口以下问题：

1. 齐为最新文档新增的 `mobile`、`username`、`accountNo`、`cardNo` 如何映射；
2. 现有主链仍在使用的 `payProtocolNo` 如何处理；
3. 当前阶段哪些可以先做，哪些不宜立即改代码；
4. 后续 H5 绑卡页接入后，会员同步应如何平滑切换。

---

## 2. 已确认事实

### 2.1 来自齐为 2026-04-21 文档

会员同步接口 `abs.push.order` 已明确 / 新增：

1. `mobile`
2. `username`
3. `accountNo`
4. `cardNo`

其中：

1. `accountNo` 表示签约银行卡号；
2. `cardNo` 表示会员卡号；
3. `cardNo` 为空视为新卡，不为空视为续期。

### 2.2 来自现有项目真实联调结论

当前项目里，齐为主链已经有明确证据表明：

1. `payProtocolNo` 是当前权益购买成功的关键字段；
2. 之前真实问题根因收敛为 `payProtocolNo` 错误，而不是 `orderNo / cardNo` 缺失；
3. 当前生产化闭环中，`payProtocolNo` 不能被轻易移除。

### 2.3 来自本轮已完成实现

当前已经完成：

1. 齐为签约查询 / 申请 / 确认后端骨架；
2. 签约确认成功后，本地落 `member_payment_protocol`；
3. 但当前本地还**没有稳定的银行卡号持久化模型**可供 `abs.push.order` 直接读取。

---

## 3. 当前矛盾点

当前会员同步字段升级的真正矛盾，不是 DTO 怎么改，而是**字段语义冲突**：

### 3.1 齐为新文档强调 `accountNo`

从最新接口文档看，齐为希望在会员同步里拿到：

1. 手机号
2. 姓名
3. 银行卡号 `accountNo`
4. 会员卡号 `cardNo`

### 3.2 现有艾博生主链强调 `payProtocolNo`

从现有真实联调结论看，当前主链实际上是围绕：

1. `payProtocolNo`
2. 支付协议有效性
3. 协议号可扣款

来驱动权益购买成功。

### 3.3 当前不能直接做“替换”

因此，现阶段不能简单把：

- `payProtocolNo`

直接替换成：

- `accountNo`

原因是：

1. 两者语义并不等价；
2. `payProtocolNo` 已被证明确实影响齐为主链成功率；
3. `accountNo` 当前在本地没有稳定数据来源；
4. 一旦直接替换，可能回退已经跑通的权益购买主链。

---

## 4. 收口结论

### 4.1 结论一：本轮不直接修改生产代码里的会员同步主链字段

当前建议：

1. **保留现有 `QwMemberSyncRequest` 与 `payProtocolNo` 主链不动；**
2. **不在本轮直接把 `accountNo` 强塞进现有权益购买主链；**
3. **先把“签约能力”与“银行卡号来源”收口出来，再推进会员同步字段升级。**

这是当前风险最低、对已跑通主链影响最小的方式。

### 4.2 结论二：会员同步字段升级按“两阶段”推进

建议采用“两阶段”方案。

#### 阶段 A：方案收口，不改主链

当前阶段只做：

1. 输出方案；
2. 明确字段来源；
3. 明确后续落库模型；
4. 等 H5 绑卡页与银行卡信息来源落定。

#### 阶段 B：在银行卡来源稳定后，再升级代码

待后续条件满足后，再推进代码升级：

1. H5 绑卡页提供；
2. 银行卡号能在艾博生本地稳定持久化；
3. 齐为联调确认 `accountNo` 是否会替代、补充还是并行于 `payProtocolNo`；
4. 会员卡号 `cardNo` 回写链路可用。

---

## 5. 推荐技术方案

## 5.1 字段来源建议

建议将字段来源收口为下表：

| 字段 | 当前建议来源 | 备注 |
|---|---|---|
| `mobile` | `member_info.mobile_encrypted` 解密 | 当前已具备 |
| `username` | `member_info.real_name_encrypted` 解密 | 当前已具备 |
| `accountNo` | 后续绑卡页签约成功后本地落库的银行卡号 | 当前尚未稳定 |
| `cardNo` | 齐为开卡成功后回传并持久化的会员卡号 | 当前尚未稳定 |
| `payProtocolNo` | 本地真实有效支付协议号 | 当前已存在真实业务意义 |

## 5.2 本地模型建议

当前建议不要把 `accountNo` 临时塞到无关字段中，而是后续明确增加“银行卡事实来源”。

推荐方向：

1. 优先新增独立银行卡签约 / 绑卡记录模型；
2. 或在现有协议模型旁边增加银行卡事实表；
3. 不建议把 `accountNo` 直接混写到 `member_payment_protocol.protocol_no`。

原因：

1. `protocolNo` 与 `accountNo` 语义不同；
2. 一个是支付协议号，一个是银行卡号；
3. 混写后容易导致后续查询、联调、补偿链路混乱。

## 5.3 DTO 调整建议

当进入下一阶段代码改造时，建议做下面的 DTO 演进：

### 目标态 DTO

`QwMemberSyncRequest` 建议最终支持：

1. `mobile`
2. `username`
3. `accountNo`
4. `cardNo`

同时对 `payProtocolNo` 的处理，建议采用以下策略之一：

### 方案 A：兼容过渡期双字段并存

短期内：

1. 对内 DTO 同时保留 `payProtocolNo` 与 `accountNo`；
2. 对外齐为请求根据联调结论决定是否：
   - 只传 `accountNo`
   - 或 `accountNo + payProtocolNo` 做兼容扩展

优点：

1. 风险最低；
2. 对已跑通主链影响最小；
3. 最适合作为下一阶段的默认技术方案。

### 方案 B：一步切到最新文档字段

直接删除 `payProtocolNo`，改为：

1. `accountNo`
2. `cardNo`

不推荐当前采用，原因：

1. 与当前真实联调结论冲突；
2. 容易影响已经跑通的权益购买链路；
3. 缺少稳定本地数据来源。

### 结论

当前更推荐：

> **方案 A：兼容过渡期双字段并存。**

## 6. 推荐实施顺序

建议按下面顺序推进：

### Step 1：保持当前签约后端骨架不动

当前已完成的签约能力先稳定下来，包括：

1. 签约查询；
2. 签约申请；
3. 签约确认；
4. 本地协议记录。

### Step 2：补银行卡事实落库

下一步应先补：

1. 银行卡号本地记录模型；
2. 会员与银行卡的关联关系；
3. 最近有效银行卡的查询口径。

### Step 3：补会员卡号 `cardNo` 持久化

后续若齐为返回：

1. `orderNo`
2. `cardNo`

则应在本地模型中可追踪保存，供续期场景使用。

### Step 4：再升级 `QwMemberSyncRequest`

待上面两类来源都稳定后，再推进：

1. DTO 调整；
2. `BenefitOrderServiceImpl.buildQwMemberSyncRequest(...)` 调整；
3. `AllinpayMemberSyncPayloadMapper` 调整；
4. 相关测试回归。
