# 2026-04-19 异步补偿运维与人工处理说明

## 1. 适用范围

本文档面向交付、联调、运维和后续接手同学，说明当前艾博生侧异步消息补偿机制的：

1. 数据模型职责
2. 任务状态流转
3. 分片 worker / supervisor 边界
4. 重试退避与死任务判定
5. 人工排查与介入方式
6. 当前运行边界、上线前提与不可承诺事项

当前文档对应的实现范围，仅覆盖两条补偿主链路：

1. `QW_BENEFIT_PURCHASE_RETRY`
2. `YUNKA_LOAN_APPLY_RETRY`

不覆盖：

1. 退款链路异步补偿
2. 绑卡链路异步补偿
3. 权益状态回推异步补偿

## 2. 当前实现定位

### 2.1 当前属于什么

当前实现属于：

1. **应用内异步补偿骨架**
2. **本地 MySQL 持久化任务模型**
3. **分片 worker + supervisor 监控机制**

也就是说：

1. 任务有数据库落库
2. 有按 `partition` 处理的 worker 模型
3. 有 supervisor 负责心跳、过期租约回收、积压识别和死任务统计
4. 已完成 MySQL 真库回归验证

### 2.2 当前不属于什么

当前**尚未展开**以下能力：

1. 独立部署进程
2. 多实例调度编排
3. 跨服务统一消息平台
4. 统一运维控制台
5. 自动人工审批流或补单后台

因此当前对外应统一表述为：

> 已具备应用内异步补偿能力和数据库持久化能力，但尚未进入独立中间件 / 独立消息平台形态。

## 3. 数据模型说明

## 3.1 `async_compensation_task`

用途：**任务主表**，记录“要不要补偿、补偿什么、当前做到哪里”。

核心字段说明：

| 字段 | 含义 | 说明 |
| --- | --- | --- |
| `task_id` | 任务主键 | 内部唯一任务号 |
| `task_type` | 任务类型 | 当前支持 `QW_BENEFIT_PURCHASE_RETRY`、`YUNKA_LOAN_APPLY_RETRY` |
| `biz_key` | 业务幂等键 | `(task_type, biz_key)` 唯一，用于重复入队保护 |
| `biz_order_no` | 业务单号 | 用于日志、排查、人工检索 |
| `partition_no` | 分片号 | 决定由哪个 worker 处理 |
| `task_status` | 当前状态 | `INIT / RETRY_WAIT / PROCESSING / SUCCESS / CANCELLED / DEAD` |
| `target_code` | 目标系统 | 当前支持 `QW`、`YUNKA` |
| `request_path` | 请求路径 | 外部调用的目标 path |
| `request_payload` | 请求体快照 | 补偿重放时的关键输入 |
| `response_payload` | 响应快照 | 成功后保留返回结果 |
| `retry_count` | 已重试次数 | 每失败一次递增 |
| `max_retry_count` | 最大重试次数 | 当前默认 `5` |
| `next_retry_ts` | 下次可重试时间 | 用于退避控制 |
| `last_error_code` | 最近错误码 | 失败留痕 |
| `last_error_message` | 最近错误信息 | 失败留痕 |
| `lease_owner` | 当前租约持有者 | 标识哪一个 worker 正在处理 |
| `lease_expire_ts` | 租约到期时间 | 过期后可由 supervisor 回收 |
| `success_ts` | 成功时间 | 成功后记录 |

### 3.2 `async_compensation_attempt`

用途：**尝试日志表**，记录“每一次处理尝试发生了什么”。

核心作用：

1. 记录第几次尝试
2. 记录由哪个 worker 处理
3. 记录本次请求体、响应体、失败码和失败信息
4. 作为后续排查“到底失败过几次、每次为什么失败”的证据链

### 3.3 `async_compensation_partition_runtime`

用途：**分片运行时表**，记录每个 `partition` 当前由谁处理、是否存活。

核心作用：

1. 记录当前 `workerId`
2. 记录当前 `workerStatus`
3. 记录 `currentTaskId`
4. 记录 `lastHeartbeatTs`
5. 用于识别离线分片和僵死 worker

## 4. 任务状态流转

当前任务主状态流转如下：

1. `INIT`
   - 新任务刚入队
   - 尚未被 worker 领取
2. `PROCESSING`
   - worker 已领取任务
   - 当前持有租约
3. `SUCCESS`
   - 任务执行成功
   - 已写入 `response_payload`
4. `RETRY_WAIT`
   - 本次执行失败
   - 尚未达到最大重试次数
   - 等待 `next_retry_ts` 到点后重试
5. `DEAD`
   - 达到最大重试次数
   - 进入死任务状态
   - 不再自动重试
6. `CANCELLED`
   - 预留状态
   - 当前第一阶段未形成正式人工取消流程

建议理解为：

```text
INIT -> PROCESSING -> SUCCESS
INIT -> PROCESSING -> RETRY_WAIT -> PROCESSING -> SUCCESS
INIT -> PROCESSING -> RETRY_WAIT -> ... -> DEAD
PROCESSING(租约过期) -> RETRY_WAIT
```

## 5. 分片、worker 与 supervisor 职责

### 5.1 分片规则

分片通过 `bizKey.hashCode()` 对 `partitionCount` 取模得到。

当前默认配置：

1. `partition-count = 8`

含义：

1. 同一 `bizKey` 始终落到同一分片
2. 同一业务键不会并发落到多个分片处理
3. 多分片可用于未来多进程 / 多实例扩展

### 5.2 worker 职责

worker 负责：

1. 按分片领取 `INIT / RETRY_WAIT` 且已到可执行时间的任务
2. 把任务状态改为 `PROCESSING`
3. 写入租约信息
4. 调用对应 `executor`
5. 成功则改 `SUCCESS`
6. 失败则改 `RETRY_WAIT` 或 `DEAD`
7. 写入 `attempt` 尝试日志

### 5.3 supervisor 职责

supervisor 负责：

1. 写入分片 `heartbeat`
2. 回收超时未完成的租约
3. 识别离线分片
4. 汇总各分片积压任务数
5. 统计死任务数

建议理解为：

1. worker 解决“执行”
2. supervisor 解决“监控与回收”

## 6. 重试退避与死任务判定

### 6.1 当前默认配置

当前默认配置如下：

| 配置项 | 默认值 | 含义 |
| --- | --- | --- |
| `partition-count` | `8` | 分片数 |
| `max-retry-count` | `5` | 最大重试次数 |
| `lease-seconds` | `60` | worker 领取任务后的租约时长 |
| `retry-initial-delay-seconds` | `60` | 初始重试延迟 |
| `retry-max-delay-seconds` | `900` | 最大重试延迟 |
| `worker-heartbeat-timeout-seconds` | `120` | 超过该时间未心跳视为离线 |

### 6.2 退避节奏

当前重试采用**指数退避**：

1. 第 1 次失败后：`60s`
2. 第 2 次失败后：`120s`
3. 第 3 次失败后：`240s`
4. 第 4 次失败后：`480s`
5. 第 5 次失败后：达到上限，进入 `DEAD`

说明：

1. 实际计算结果受 `retry-max-delay-seconds` 限制
2. 当前最大延迟上限为 `900s`

### 6.3 死任务判定

当前判定规则：

1. `retry_count + 1 >= max_retry_count`
2. 最后一次失败后直接置为 `DEAD`
3. `DEAD` 状态不再自动重试

## 7. 如何识别积压、死任务与离线分片

### 7.1 识别积压任务

关注：

1. 状态为 `INIT`
2. 或状态为 `RETRY_WAIT` 且 `next_retry_ts <= now()`

示例 SQL：

```sql
select partition_no, count(*) as backlog_count
from async_compensation_task
where task_status in ('INIT', 'RETRY_WAIT')
  and (next_retry_ts is null or next_retry_ts <= now())
group by partition_no
order by partition_no;
```

如果某个分片长期积压，需要重点排查：

1. 该分片 worker 是否在跑
2. 是否有大量同类外部异常
3. 是否存在租约未回收

### 7.2 识别死任务

示例 SQL：

```sql
select task_id, task_type, biz_order_no, retry_count, last_error_code, last_error_message, updated_ts
from async_compensation_task
where task_status = 'DEAD'
order by updated_ts desc;
```

死任务意味着：

1. 系统已按当前配置自动重试到上限
2. 需要人工判断是“继续重试”还是“放弃 / 补单”

### 7.3 识别离线分片

示例 SQL：

```sql
select partition_no, worker_id, worker_status, current_task_id, last_heartbeat_ts
from async_compensation_partition_runtime
where last_heartbeat_ts < date_sub(now(), interval 120 second)
order by partition_no;
```

如果出现离线分片，需要重点排查：

1. 对应 worker 是否退出
2. 应用实例是否卡死
3. 心跳逻辑是否未被调度

## 8. 如何区分“外部异常”和“本地数据异常”

### 8.1 更像外部异常的特征

通常具备以下特征：

1. 同一时间多个任务一起失败
2. `last_error_message` 类似超时、网关不可用、连接异常、对方返回系统忙
3. 同类型任务失败集中在同一外部系统
4. 重新执行后偶发成功

典型处理：

1. 保持自动重试
2. 观察是否在下一轮退避后恢复
3. 若大面积失败，优先排查外部环境

### 8.2 更像本地数据异常的特征

通常具备以下特征：

1. 只影响个别业务单
2. 每次重试都稳定失败
3. 错误信息指向数据缺失、状态不一致、映射记录缺失、payload 无法解析
4. 其他同类任务正常

典型处理：

1. 先核查本地业务单状态
2. 核查补偿任务 `request_payload`
3. 必要时人工修正数据后再决定是否重试

## 9. 人工介入建议

### 9.1 何时建议继续重试

适用于：

1. 外部系统短时不可用
2. 网络抖动或网关超时
3. 任务本身业务状态仍未成功
4. 其他同类任务已有恢复迹象

### 9.2 何时建议放弃自动重试

适用于：

1. 本地业务状态已人工确认成功
2. 继续调用外部可能造成重复下单 / 重复提交
3. 外部已经明确拒绝且无恢复可能

### 9.3 何时建议人工补单

适用于：

1. 用户侧状态已经承接，但外部单据未成功创建
2. 自动重试已到 `DEAD`
3. 本地数据与外部状态已发生偏差，自动补偿无法收敛

### 9.4 当前阶段人工处理原则

当前建议采用：

1. **先查业务状态，再决定是否重试**
2. **先区分外部异常还是本地异常**
3. **达到 `DEAD` 后不直接删任务**
4. **保留任务和尝试日志，作为人工处理证据**

## 10. 当前上线前提

若要把当前能力视为“可上线候选”，至少需要满足：

1. MySQL 表结构已部署完成
2. 默认参数与运行环境已明确
3. worker / supervisor 触发方式已明确
4. 运维侧知道如何看：
   - 积压
   - 死任务
   - 离线分片
5. 联调侧知道不同场景下的前端承接口径
6. 外部系统调用失败时的日志与 traceId 可追溯

## 11. 当前触发方式与运行事实

### 11.1 当前已确认事实

当前代码中已经具备：

1. `AsyncCompensationEnqueueService`
2. `AsyncCompensationWorkerService`
3. `AsyncCompensationSupervisorService`
4. 对应实现类、数据库模型与测试

但当前**尚未发现**以下自动运行机制：

1. `@Scheduled`
2. `CommandLineRunner`
3. `ApplicationRunner`
4. `SmartLifecycle`
5. 专门的异步补偿调度线程 / 启动器

### 11.2 当前含义

这意味着当前阶段的真实结论是：

1. **补偿能力已经具备“可被调用”的服务能力**
2. **但尚未形成“应用启动后自动持续扫描并执行”的正式调度入口**

因此当前不能对外表述为：

1. “补偿任务已经全自动常驻运行”
2. “worker 会自动拉起并持续消费”

更准确的表述应为：

> 当前已经具备应用内异步补偿模型、执行服务与监控服务，但正式自动调度 / 常驻运行方式仍需后续单独补齐。

### 11.3 对上线前提的影响

若后续要进入更接近生产的形态，至少还需要明确：

1. 谁负责定时触发 `worker`
2. 谁负责定时触发 `supervisor`
3. 分片与进程实例之间的绑定方式
4. 应用重启、宕机、长时间暂停后的恢复策略
5. 运维如何确认补偿是否处于“正在运行”状态

## 12. 当前不可承诺事项

当前阶段**不可承诺**：

1. 已具备独立消息平台能力
2. 已具备跨实例调度一致性能力
3. 已具备统一后台人工处理界面
4. 已具备退款 / 绑卡 / 状态回推全链路补偿
5. 已具备正式生产级常驻进程部署方案

## 13. 一句话结论

当前异步补偿机制已经具备“任务可落库、执行可重试、失败可留痕、运行可观测、问题可人工接手”的第一阶段能力，但仍属于**应用内补偿骨架**，后续若进入生产化承接，还需要补运维和部署层面的正式方案。
