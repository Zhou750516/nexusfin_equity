# 2026-04-19 异步补偿人工处理 SQL 与操作模板

## 1. 适用范围

本文档用于给联调、交付、排查同学提供一份**直接可执行的 SQL / 操作模板**，便于：

1. 查询任务状态
2. 查询失败原因
3. 查询积压与死任务
4. 查询分片存活情况
5. 对单个业务单进行人工排查

说明：

1. 本文档以当前 MySQL 表结构为准
2. 当前只提供**查询模板**
3. 不直接提供批量修改状态的自动化脚本

原因：

1. 当前第一阶段重点是“先查清楚”
2. 避免在没有人工确认业务状态前，误操作任务状态

## 2. 按业务单排查

### 2.1 根据业务单号查任务主记录

```sql
select
  task_id,
  task_type,
  biz_key,
  biz_order_no,
  partition_no,
  task_status,
  retry_count,
  max_retry_count,
  next_retry_ts,
  last_error_code,
  last_error_message,
  lease_owner,
  lease_expire_ts,
  success_ts,
  created_ts,
  updated_ts
from async_compensation_task
where biz_order_no = '替换为业务单号'
order by created_ts desc;
```

适用：

1. 已知 `benefitOrderNo`
2. 已知 `applicationId`
3. 想快速看当前任务走到哪里

### 2.2 根据业务幂等键查任务

```sql
select *
from async_compensation_task
where biz_key = '替换为业务幂等键';
```

示例：

1. `BENEFIT_PURCHASE:ordxxxx`
2. `LOAN_APPLY:APP-xxxx`

## 3. 查询尝试日志

### 3.1 查看某个任务的全部尝试记录

```sql
select
  attempt_id,
  task_id,
  task_type,
  partition_no,
  worker_id,
  attempt_no,
  result_status,
  error_code,
  error_message,
  started_ts,
  finished_ts
from async_compensation_attempt
where task_id = '替换为任务ID'
order by attempt_no asc;
```

用途：

1. 看失败了几次
2. 每次失败原因是否一致
3. 是否存在偶发成功 / 偶发失败

### 3.2 看最近失败的尝试

```sql
select
  attempt_id,
  task_id,
  task_type,
  worker_id,
  attempt_no,
  error_code,
  error_message,
  started_ts,
  finished_ts
from async_compensation_attempt
where result_status = 'FAILED'
order by started_ts desc
limit 50;
```

## 4. 查询积压任务

### 4.1 按分片看当前可执行积压

```sql
select
  partition_no,
  count(*) as backlog_count
from async_compensation_task
where task_status in ('INIT', 'RETRY_WAIT')
  and (next_retry_ts is null or next_retry_ts <= now())
group by partition_no
order by partition_no;
```

解读：

1. 数量越大，不代表一定异常
2. 若某个分片长期不下降，需要重点关注该分片 worker 是否有调度

### 4.2 看具体哪些任务处于积压状态

```sql
select
  task_id,
  task_type,
  biz_order_no,
  partition_no,
  task_status,
  retry_count,
  next_retry_ts,
  last_error_code,
  last_error_message,
  updated_ts
from async_compensation_task
where task_status in ('INIT', 'RETRY_WAIT')
  and (next_retry_ts is null or next_retry_ts <= now())
order by updated_ts asc, created_ts asc
limit 200;
```

## 5. 查询死任务

### 5.1 查看全部死任务

```sql
select
  task_id,
  task_type,
  biz_order_no,
  partition_no,
  retry_count,
  max_retry_count,
  last_error_code,
  last_error_message,
  updated_ts
from async_compensation_task
where task_status = 'DEAD'
order by updated_ts desc;
```

### 5.2 统计死任务数量

```sql
select count(*) as dead_count
from async_compensation_task
where task_status = 'DEAD';
```

## 6. 查询分片与心跳

### 6.1 查看所有分片运行时

```sql
select
  partition_no,
  worker_id,
  worker_status,
  current_task_id,
  current_task_started_ts,
  last_heartbeat_ts,
  updated_ts
from async_compensation_partition_runtime
order by partition_no;
```

### 6.2 查询疑似离线分片

```sql
select
  partition_no,
  worker_id,
  worker_status,
  current_task_id,
  last_heartbeat_ts
from async_compensation_partition_runtime
where last_heartbeat_ts < date_sub(now(), interval 120 second)
order by partition_no;
```

说明：

1. `120` 秒来自当前默认 `worker-heartbeat-timeout-seconds`
2. 若后续配置变更，这里的阈值也要一起调整

## 7. 查询正在处理中的任务

### 7.1 看当前 `PROCESSING` 任务

```sql
select
  task_id,
  task_type,
  biz_order_no,
  partition_no,
  lease_owner,
  lease_expire_ts,
  updated_ts
from async_compensation_task
where task_status = 'PROCESSING'
order by lease_expire_ts asc;
```

### 7.2 看疑似租约过期但未回收的任务

```sql
select
  task_id,
  task_type,
  biz_order_no,
  partition_no,
  lease_owner,
  lease_expire_ts,
  updated_ts
from async_compensation_task
where task_status = 'PROCESSING'
  and lease_expire_ts < now()
order by lease_expire_ts asc;
```

## 8. 人工排查步骤模板

## 8.1 场景一：用户反馈“已经提交，但结果一直不变”

建议步骤：

1. 先查业务单对应的 `async_compensation_task`
2. 看状态是：
   - `INIT`
   - `RETRY_WAIT`
   - `PROCESSING`
   - `DEAD`
3. 再看 `partition_runtime` 是否存在离线分片
4. 再看 `attempt` 表中最近失败原因
5. 最后再判断是：
   - 外部超时
   - 本地数据异常
   - 调度未触发

## 8.2 场景二：补偿任务到了 `DEAD`

建议步骤：

1. 查主表确认 `retry_count` 和最近错误
2. 查尝试日志确认是否每次都相同错误
3. 查本地业务状态，看是否其实已经成功
4. 若本地未成功，再判断：
   - 是否可人工补单
   - 是否需要恢复任务执行

## 8.3 场景三：怀疑 worker 没有工作

建议步骤：

1. 查 `async_compensation_partition_runtime`
2. 若没有心跳或心跳过旧，先确认当前是否存在正式调度入口
3. 再查任务表是否有大量 `INIT / RETRY_WAIT`
4. 若存在积压但无心跳，应先判断是：
   - 当前阶段本就未挂自动调度
   - 还是后续接入调度后运行异常

## 9. 当前不建议直接执行的 SQL 操作

当前阶段，不建议在没有人工确认前直接做：

1. 批量改 `task_status`
2. 批量清空 `retry_count`
3. 直接删除 `DEAD` 任务
4. 手工覆盖 `lease_owner`
5. 直接伪造 `SUCCESS`

原因：

1. 容易破坏排查证据链
2. 容易导致重复外部调用
3. 容易让业务状态与补偿状态再次失真

## 10. 建议的人工处理顺序

建议统一按以下顺序处理：

1. **先查主表**
2. **再查尝试日志**
3. **再查分片运行时**
4. **再核对本地业务状态**
5. **最后决定是否重试、放弃或人工补单**

## 11. 一句话结论

当前异步补偿机制已经具备较完整的数据库排查入口；在正式调度与后台工具到位前，建议所有人工处理都遵循“先查再动、先证据后操作”的原则。
