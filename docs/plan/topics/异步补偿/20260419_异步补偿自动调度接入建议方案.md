# 2026-04-19 异步补偿自动调度接入建议方案

## 1. 目标

在当前“异步补偿第一阶段能力已经完成”的基础上，补一版面向后续实现和上线承接的**自动调度接入建议方案**，明确：

1. 现有能力缺口是什么
2. 最小可上线形态如何落
3. 后续可扩展形态如何演进
4. 当前阶段不建议做什么

本文档重点解决的问题不是“补偿机制是否需要”，而是：

> 当前补偿机制已经有任务、worker、supervisor 和 MySQL 模型，但后续究竟该怎么真正跑起来。

## 1.1 对应实施拆解

本文档用于回答“推荐怎么接入”；如果要继续进入编码实施，请直接配合以下拆解文档推进：

两者关系如下：

1. 本文档负责说明**为什么推荐方案 A**
2. 拆解文档负责说明**下一步具体改哪些文件、按什么顺序实现**

## 2. 当前现状

### 2.1 当前已经具备的能力

当前代码已具备：

1. 任务入队能力：`AsyncCompensationEnqueueService`
2. 任务执行能力：`AsyncCompensationWorkerService`
3. 监控与回收能力：`AsyncCompensationSupervisorService`
4. 分片能力：按 `bizKey` 进行 `partition`
5. 重试能力：指数退避
6. 留痕能力：
   - 主任务表
   - 尝试日志表
   - 分片运行时表

### 2.2 当前缺失的能力

当前尚未看到自动调度入口，因此还缺：

1. 应用启动后自动驱动 worker 的机制
2. 应用启动后自动驱动 supervisor 的机制
3. 分片和进程实例之间的正式绑定方式
4. 健康检查与运行状态暴露方式
5. 正式部署和运维操作规范

## 3. 设计原则

建议后续自动调度接入继续遵循以下原则：

### 3.1 先最小可运行，再演进

不要一开始就引入：

1. 分布式任务平台
2. MQ 中间件
3. 独立控制台
4. 跨服务统一调度中心

先把当前应用内补偿机制跑起来，证明：

1. 调度逻辑稳定
2. 分片模型可用
3. 运维排查链路可承接

### 3.2 保持“任务状态机”仍以数据库为准

无论后续调度方式怎么演进，都建议保持：

1. 任务状态以 `async_compensation_task` 为准
2. 尝试历史以 `async_compensation_attempt` 为准
3. 分片存活信息以 `async_compensation_partition_runtime` 为准

原因：

1. 方便人工排查
2. 方便多实例后续接入
3. 不把运行状态只留在内存中

### 3.3 先保证“可停、可看、可恢复”

自动调度第一阶段最重要的不是吞吐量，而是：

1. 进程启动后知道自己该做什么
2. 运维能知道它是不是在运行
3. 宕机后可以恢复
4. 不会重复打外部请求

## 4. 推荐接入方案

## 4.1 方案 A：应用内 `@Scheduled` 驱动（推荐作为第一阶段正式接入）

### 方案描述

在当前 Spring Boot 应用内增加两类调度器：

1. `worker scheduler`
2. `supervisor scheduler`

其中：

1. `worker scheduler`
   - 周期性遍历本实例负责的 `partition`
   - 调用 `processNext(workerId, partitionNo)`
2. `supervisor scheduler`
   - 周期性调用：
     - `heartbeat(...)`
     - `recycleExpiredLeases()`
     - `collectReadyTaskBacklogByPartition()`
     - `countDeadTasks()`

### 推荐原因

1. 与当前代码结构最贴合
2. 改动最小
3. 不需要引入新中间件
4. 最容易先形成可运行闭环

### 第一阶段建议形态

建议增加配置项，例如：

1. `enabled`
2. `worker-enabled`
3. `supervisor-enabled`
4. `worker-poll-interval-ms`
5. `supervisor-interval-ms`
6. `owned-partitions`
7. `worker-id`

这样可以支持：

1. 本地单机运行
2. 测试环境指定某实例只跑 worker
3. 某实例只跑 supervisor
4. 手工控制分片归属

### 适用阶段

适合：

1. 当前项目第一阶段上线承接
2. 单实例或少量实例部署
3. 补偿任务量不大的场景

## 4.2 方案 B：应用内专用补偿 runner + 独立 profile

### 方案描述

在当前仓库中保留同一套代码，但通过独立 profile 或启动参数，区分：

1. Web/API 实例
2. 补偿 runner 实例

即：

1. Web 实例主要提供业务接口
2. Runner 实例专门负责：
   - worker 调度
   - supervisor 调度

### 优点

1. 不影响主 API 实例吞吐
2. 更利于后续调优
3. 运维上更容易看清责任边界

### 缺点

1. 比方案 A 多一层部署复杂度
2. 需要更清楚地管理配置

### 建议定位

建议作为：

1. **方案 A 之后的第二阶段演进形态**
2. 当前不必直接一步到位

## 4.3 方案 C：独立补偿服务 / 独立调度系统（当前不建议）

### 方案描述

后续若补偿范围继续扩大，再考虑：

1. 独立补偿服务
2. 统一任务中心
3. MQ / 分布式任务平台

### 当前不建议的原因

1. 当前补偿链路只有两条
2. 当前任务量和运行压力未知
3. 当前最缺的不是平台能力，而是先把现有机制稳定跑起来

## 5. 推荐的最小上线形态

建议当前优先采用：

> **方案 A：应用内 `@Scheduled` 驱动**

并控制为如下最小上线形态：

### 5.1 角色划分

1. 一个实例负责 `supervisor`
2. 一个或多个实例负责 `worker`
3. 单实例阶段也可由同一实例同时承担

### 5.2 分片归属方式

第一阶段建议**显式配置分片归属**，不要一开始做复杂自动抢占。

例如：

1. 实例 A：`0,1,2,3`
2. 实例 B：`4,5,6,7`

这样有几个好处：

1. 排查简单
2. 不容易出现多实例抢同一分片
3. 与现有 `partition_runtime` 模型容易对齐

### 5.3 调度频率建议

建议起步值：

1. `worker poll interval = 3s ~ 5s`
2. `supervisor interval = 10s ~ 30s`

原因：

1. 当前补偿场景不是毫秒级实时性
2. 过快轮询没有明显收益
3. 过快轮询反而增加数据库压力

### 5.4 启停方式建议

建议保证：

1. 启动时打印：
   - 当前 `workerId`
   - 当前拥有的 `partition`
   - 当前调度是否启用
2. 停机时打印：
   - 当前实例退出
   - 最后处理的任务信息

## 6. worker 与 supervisor 的建议职责拆分

## 6.1 worker 调度器建议职责

调度器只负责：

1. 定时轮询本实例拥有的分片
2. 对每个分片调用一次 `processNext(...)`
3. 不在调度层写业务逻辑

业务逻辑仍放在：

1. `AsyncCompensationWorkerServiceImpl`
2. `AsyncCompensationRouterExecutor`
3. 各具体 `executor`

## 6.2 supervisor 调度器建议职责

调度器只负责：

1. 周期性更新本实例 / 分片心跳
2. 周期性回收过期租约
3. 周期性统计积压与死任务
4. 打日志输出运行摘要

不建议把：

1. 人工补单
2. 批量任务修复
3. 自动删死任务

直接塞进 supervisor。

## 7. 配置项建议

建议后续新增或补齐以下配置：

| 配置项 | 示例 | 作用 |
| --- | --- | --- |
| `nexusfin.async-compensation.enabled` | `true` | 总开关 |
| `nexusfin.async-compensation.worker-enabled` | `true` | 是否启用 worker 调度 |
| `nexusfin.async-compensation.supervisor-enabled` | `true` | 是否启用 supervisor 调度 |
| `nexusfin.async-compensation.worker-id` | `async-worker-a` | 当前实例 worker 标识 |
| `nexusfin.async-compensation.owned-partitions` | `0,1,2,3` | 当前实例负责的分片 |
| `nexusfin.async-compensation.worker-poll-interval-ms` | `5000` | worker 轮询间隔 |
| `nexusfin.async-compensation.supervisor-interval-ms` | `15000` | supervisor 执行间隔 |

## 8. 日志与可观测性建议

自动调度接入后，建议至少补以下日志：

1. 应用启动时打印调度配置摘要
2. 每次 worker 拉取到任务时打印：
   - `taskId`
   - `taskType`
   - `bizOrderNo`
   - `partitionNo`
3. supervisor 每轮打印：
   - 回收了多少过期租约
   - 各分片积压数
   - 当前死任务数
4. 发现离线分片时打印明确告警日志

## 9. 不建议当前阶段直接做的事情

当前不建议直接做：

1. 自动抢分片
2. 自动迁移分片归属
3. 自动重置死任务
4. 自动补单
5. 引入 MQ 再重构整套补偿模型
6. 一开始就拆独立微服务

原因：

1. 当前最重要的是稳定
2. 当前阶段任务量和故障模式仍在观察
3. 过早复杂化会抬高维护成本

## 10. 推荐实施顺序

建议后续按以下顺序实施：

### 第一步：补调度配置与开关

1. 增加 `enabled / worker-enabled / supervisor-enabled`
2. 增加 `worker-id`
3. 增加 `owned-partitions`

### 第二步：补 `@Scheduled` 调度器

1. 新增 worker scheduler
2. 新增 supervisor scheduler
3. 仅调用现有 service，不下沉业务逻辑

### 第三步：补运行日志与最小健康检查

1. 启动日志
2. 轮询日志
3. supervisor 摘要日志

### 第四步：补回归测试

1. 调度器启停测试
2. 分片配置测试
3. 单实例最小调度验证

### 第五步：再决定是否进入独立 runner 形态

1. 观察任务量
2. 观察与主 API 的资源竞争
3. 再决定是否升级到方案 B

## 11. 一句话结论

当前最推荐的自动调度接入方式，是先在现有 Spring Boot 应用内通过 `@Scheduled + 显式分片归属配置` 把 worker / supervisor 跑起来，先形成**最小可上线形态**，再根据任务量和运维反馈决定是否演进到独立 runner 或更重的调度架构。
