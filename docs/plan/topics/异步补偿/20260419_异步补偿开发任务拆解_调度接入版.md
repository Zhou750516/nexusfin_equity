# 2026-04-19 异步补偿开发任务拆解（调度接入版） Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把当前“已有入队、执行、监督能力，但仍需手工触发”的异步补偿骨架，推进到“应用启动后可按配置自动驱动 worker / supervisor”的可上线形态。

**Architecture:** 在现有 `AsyncCompensationWorkerService`、`AsyncCompensationSupervisorService`、MySQL 三张补偿表的基础上，增加应用内调度层与运行配置层。调度层只负责实例角色开关、分片归属、轮询频率、心跳和运行摘要，不承接具体补偿业务逻辑；任务状态机仍以数据库为准，不引入 MQ，不做动态抢占与自动 rebalance。

**Tech Stack:** Java 17, Spring Boot 3.2, Spring Scheduling, MyBatis-Plus, MySQL 8.0, H2, JUnit 5, Mockito

---

## Scope Decision

### 本次计划包含

1. 扩展补偿调度配置项，支持启停、角色、分片和轮询频率
2. 在应用内启用 `@Scheduled` 驱动 worker / supervisor
3. 为每个 owned partition 建立固定 worker 调度循环
4. 输出启动摘要、运行摘要和空跑日志，便于排查
5. 补配置、调度、边界行为的测试
6. 补当天计划 / checklist / 说明文档，形成可执行拆解

### 本次计划不包含

1. 独立 runner 进程或独立补偿服务
2. 自动抢占分片、动态扩缩容、自动 rebalance
3. 退款 / 绑卡 / 权益回推等新补偿场景
4. Prometheus / Actuator 指标接入
5. 外部任务平台或 MQ 接入

## Design Constraints

1. **调度层不写业务逻辑**：只调用既有 `AsyncCompensationWorkerService` / `AsyncCompensationSupervisorService`
2. **数据库状态为准**：不把任务状态、重试状态仅保存在内存
3. **分片显式配置**：通过 `ownedPartitions` 控制实例职责，不做自动争抢
4. **单实例 / 多实例均可落地**：同一实例可同时承担 worker + supervisor
5. **无任务时允许空跑**：空轮询属于正常状态，但要避免刷屏式日志
6. **保持当前业务口径不变**：本次只补“自动跑起来”，不改前端返回语义

- [ ] **Step 1：启用 Spring Scheduling**

在应用启动类上增加调度能力，使后续 `@Scheduled` Bean 可以生效。

要求：

1. 不新增额外框架
2. 不引入复杂线程池之前，先沿用 Spring 默认调度机制

- [ ] **Step 2：抽出调度协调器**

新增 `AsyncCompensationSchedulerCoordinator`，统一提供以下能力：

1. `isSchedulingEnabled()`

1. 配置判断

### Task 3：接入 worker 自动轮询

**Files:**
- Create: `src/main/java/com/nexusfin/equity/schedule/AsyncCompensationWorkerScheduler.java`
- Create: `src/test/java/com/nexusfin/equity/schedule/AsyncCompensationWorkerSchedulerTest.java`
- Modify: `src/test/java/com/nexusfin/equity/service/AsyncCompensationWorkerServiceTest.java`

- [ ] **Step 1：为 worker 增加固定频率调度入口**

新增 `AsyncCompensationWorkerScheduler`，按 `workerPollIntervalMs` 轮询。

每次 tick 的最小行为：

1. 判断总开关和 worker 开关
2. 读取 `ownedPartitions`
3. 依次对每个 partition 调用 `coordinator.runWorkerTick(partitionNo)`

- [ ] **Step 2：心跳与状态上报口径收口**

在 `runWorkerTick(partitionNo)` 中，对 `AsyncCompensationSupervisorService.heartbeat(...)` 做标准化调用：

1. tick 开始前上报 `IDLE`
2. 若分片命中任务并开始执行，则上报 `RUNNING`
3. tick 结束后恢复 `IDLE`

说明：

1. 当前 `AsyncCompensationWorkerService.processNext(...)` 只返回 `boolean`
2. 第一阶段可以先按“处理到任务 / 未处理到任务”区分简化上报
3. 不要求在本次就补充更细粒度任务 ID 回填，但要预留接口收口点

- [ ] **Step 3：控制空跑日志与异常日志**

要求：

1. 没有任务时不每个 tick 打 info 日志
2. 真正处理到任务时打 info
3. 调度入口捕获运行时异常并打 warn / error，避免定时线程被打断

- [ ] **Step 4：补 worker scheduler 测试**

覆盖以下场景：

1. `enabled=false` 时不调用 worker service
2. `workerEnabled=false` 时不调用 worker service
3. `ownedPartitions` 为空时不调用 worker service
4. `ownedPartitions=[0,1,2]` 时，会按顺序触发三个 partition
5. 单个 partition 抛异常时，不影响下一个 partition 继续执行

- [ ] **Step 5：执行验证**

Run: `mvn -q -Dtest=AsyncCompensationWorkerSchedulerTest,AsyncCompensationWorkerServiceTest test`  
Expected:

1. PASS
2. 不破坏既有 worker service 单元测试

## Risks To Watch

1. 默认开启调度会污染现有集成测试，因此必须默认关闭
2. worker tick 若未兜底异常，可能导致定时线程失活
3. 多分片顺序轮询可能拉长单次 tick，第一阶段先接受该约束
4. `heartbeat(...)` 当前粒度偏粗，若后续要看“正在处理哪个 task”，需再补上下文透传
5. 若未来进入多实例部署，需要再补“实例与分片映射”的运维文档
