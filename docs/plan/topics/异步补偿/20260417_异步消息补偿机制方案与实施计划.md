# 异步消息补偿机制方案与实施计划

> 目标：基于艾博生本地 MySQL，实现“同步失败或超时后的后台补偿”机制，覆盖齐为权益购买失败补偿、云卡/小花用信请求失败补偿，并支持 supervisor + 多 partition 常驻进程模型。

## 1. 已确认业务口径

### 1.1 同步接口返回口径

已确认：

1. 用户点击前端页面后，由艾博生前端调用艾博生后端接口
2. 如果是**齐为权益购买**场景，同步调用齐为超时或报错，当前仍按失败口径处理，异步补偿只作为后台自愈机制
3. 如果是**用信提交**场景，艾博生后端内部调用云卡、云卡再透传给小花科技时，如果同步调用超时，则艾博生后端需要落本地补偿任务
4. 在上述用信提交超时场景下，前端页面应显示**已提交**或者**审核中**
5. 因此，用信提交场景不再采用“同步超时即页面失败”的口径，而是采用“页面已提交 / 审核中，后台继续补偿”的口径

### 1.2 本轮覆盖场景

需要纳入异步补偿的场景：

1. 用户购买权益时，调用齐为接口超时或报错
2. 用户提交用信时，艾博生调用云卡、云卡透传给小花科技，如果同步请求超时或报错

## 2. 方案选型

### 方案 A：独立补偿任务表 + Supervisor + Partition Worker（推荐）

**做法**

1. 新建独立补偿任务主表，保存待补偿任务
2. 新建补偿任务执行日志表，保存每次尝试记录
3. 按 `partition_no` 分片
4. Supervisor 常驻扫描任务，监控分片积压和心跳
5. 每个 partition 由一个独立常驻进程消费

**优点**

1. 语义清晰，不污染现有业务表
2. 支持多任务类型、多目标地址、多次重试
3. 天然支持监控、补偿、人工排查
4. 后续可以平滑替换为真正 MQ，而不改业务入队接口

**缺点**

1. 需要新增 2~3 张表
2. 需要补常驻进程和调度治理能力

### 方案 B：复用现有业务日志表承载补偿任务

例如尝试复用：

1. `notification_receive_log`
2. `benefit_status_push_log`
3. 其他业务流水表

**优点**

1. 初期表结构改动少

**缺点**

1. 语义混乱，接收日志和补偿任务不是同一类对象
2. 很难统一支持不同任务类型、分片、租约、重试策略
3. 后续维护成本高，容易把业务状态和调度状态耦合在一起

**结论**

不推荐。

### 方案 C：直接引入外部 MQ

**优点**

1. 标准消息能力更强
2. 天然支持多消费者和重试

**缺点**

1. 不符合“先用本地 MySQL 落地”的当前约束
2. 需要额外基础设施和运维链路

**结论**

当前阶段不采用。

## 3. 推荐方案

已确认：**采用方案 A：独立补偿任务表 + Supervisor + Partition Worker**。

这意味着本方案后续实施时，统一按以下边界推进：

1. 使用独立补偿任务表，不复用现有日志表
2. 使用独立 attempt 日志表记录每次重试
3. 使用 partition runtime 表承接 supervisor 监控
4. 每个 partition 一个独立常驻 worker 进程
5. supervisor 负责监控、回收、告警，不直接发送
6. 用信提交超时场景返回前端“已提交 / 审核中”

核心原则：

1. **同步链路只负责判定失败并入队**
2. **异步链路负责补偿重试**
3. **业务状态与调度状态分离**
4. **所有补偿任务必须可追踪、可重试、可取消**
5. **分片固定，单 partition 单进程串行处理**

## 4. 总体架构

## 4.1 组件划分

### A. CompensationEnqueueService

职责：

1. 在同步调用失败时创建补偿任务
2. 负责幂等入队
3. 按业务主键计算 partition

### B. CompensationSupervisor

职责：

1. 常驻运行
2. 监控任务表中的待处理消息
3. 监控每个 partition worker 的心跳
4. 发现分片积压、分片掉线、超长未处理任务时告警
5. 必要时把卡死任务从 `PROCESSING` 回收为 `RETRY_WAIT`

### C. CompensationPartitionWorker

职责：

1. 每个 partition 一个独立常驻进程
2. 只消费本 partition 的任务
3. 加锁、发送、更新状态、记录尝试日志
4. 根据任务类型路由到不同发送器

### D. CompensationTaskExecutor

职责：

1. 按任务类型执行真正的 HTTP 请求
2. 不同任务类型配置不同地址
3. 请求参数从任务表 payload 中读取

### E. CompensationTaskResolver

职责：

1. 根据 `task_type` 决定调用哪个客户端
2. 根据 `target_code` / `request_path` / `http_method` 组装请求

## 4.2 数据流

### 同步失败 / 超时入队

#### A. 齐为权益购买

1. 业务服务发起同步请求
2. 请求超时 / 报错
3. 业务服务记录同步失败日志
4. 调用 `CompensationEnqueueService.enqueue(...)`
5. 写入补偿任务表
6. 同步接口仍返回失败

#### B. 云卡 / 小花用信提交

1. 前端发起用信提交请求
2. 艾博生后端调用云卡
3. 云卡透传请求到小花科技
4. 如果调用超时，艾博生后端记录“本次同步结果未知”
5. 调用 `CompensationEnqueueService.enqueue(...)`
6. 写入补偿任务表
7. 同时在本地申请映射或申请状态表中写入 `SUBMITTED` / `PENDING_REVIEW` 类状态
8. 同步接口对前端返回“已提交”或“审核中”

> 这里的关键差异是：齐为权益购买仍是“失败 + 后台补偿”，而用信提交超时是“页面显示已提交 / 审核中 + 后台补偿”。

### 异步补偿执行

1. Supervisor 监控到有待处理任务
2. Partition Worker 轮询本分片任务
3. Worker 领取任务并加租约
4. 按任务类型调用目标接口
5. 成功则置 `SUCCESS`
6. 失败则置 `RETRY_WAIT` 并计算 `next_retry_ts`
7. 达到最大次数则置 `DEAD`

## 5. 表设计

## 5.1 主表：`async_compensation_task`

建议新增一张独立主表，用于保存补偿任务。

### 表字段建议

1. `task_id`：任务主键
2. `task_type`：任务类型
3. `biz_key`：业务幂等键
4. `biz_order_no`：业务单号，如 `benefitOrderNo` / `applicationId`
5. `partition_no`：分片号
6. `task_status`：任务状态
7. `target_code`：目标系统，如 `QW` / `YUNKA`
8. `request_path`：目标请求地址
9. `http_method`：请求方法
10. `request_headers`：请求头快照，敏感信息可脱敏后保存
11. `request_payload`：请求参数 JSON
12. `response_payload`：最近一次响应体
13. `retry_count`：已重试次数
14. `max_retry_count`：最大重试次数
15. `next_retry_ts`：下次重试时间
16. `last_error_code`：最近失败码
17. `last_error_message`：最近失败原因
18. `lease_owner`：当前持有任务的 worker
19. `lease_expire_ts`：租约过期时间
20. `created_ts`
21. `updated_ts`
22. `success_ts`

### 索引建议

1. 主键：`task_id`
2. 唯一索引：`uk_task_type_biz_key`
3. 普通索引：`idx_partition_status_next_retry`
4. 普通索引：`idx_status_updated_ts`
5. 普通索引：`idx_lease_expire_ts`

### 状态建议

1. `INIT`：刚创建
2. `RETRY_WAIT`：等待重试
3. `PROCESSING`：处理中
4. `SUCCESS`：成功
5. `CANCELLED`：取消，不再重试
6. `DEAD`：达到最大次数，死信

## 5.2 执行日志表：`async_compensation_attempt`

建议新增执行日志表，记录每次发送尝试。

### 表字段建议

1. `attempt_id`
2. `task_id`
3. `task_type`
4. `partition_no`
5. `worker_id`
6. `attempt_no`
7. `request_payload`
8. `response_payload`
9. `result_status`
10. `error_code`
11. `error_message`
12. `started_ts`
13. `finished_ts`

### 作用

1. 排查问题
2. 审计补偿历史
3. 支撑 supervisor 告警和报表

## 5.3 运行状态表：`async_compensation_partition_runtime`

建议新增分片运行状态表，用于 supervisor 监控。

### 表字段建议

1. `partition_no`
2. `worker_id`
3. `worker_status`
4. `last_heartbeat_ts`
5. `current_task_id`
6. `current_task_started_ts`
7. `updated_ts`

### 作用

1. 监控每个 partition 是否活着
2. 发现某个分片卡死
3. 给 supervisor 做“掉线/积压”告警

## 6. 分片设计

## 6.1 分片规则

推荐固定分片总数，例如：

1. `partition_count = 8` 或 `16`

分片算法：

1. `partition_no = hash(biz_key) % partition_count`

### `biz_key` 建议

1. 齐为权益购买补偿：`BENEFIT_PURCHASE:{benefitOrderNo}`
2. 用信申请补偿：`LOAN_APPLY:{applicationId}`

这样可以确保：

1. 同一业务主键永远落到同一分片
2. 同一业务的补偿串行执行，避免并发重入

## 6.2 分片进程模型

按你当前要求，采用：

1. **每个 partition 一个独立进程**

例如：

1. `worker-0` 负责 `partition=0`
2. `worker-1` 负责 `partition=1`
3. ...

### 好处

1. 模型简单
2. 串行保证更强
3. 某一分片异常不会直接影响其他分片

### 约束

1. `partition_count` 不能频繁变
2. 分片扩缩容需要额外迁移策略

当前阶段可以先固定分片数，不做动态 rebalance。

## 7. Supervisor 设计

## 7.1 角色定位

Supervisor 是常驻进程，但**不直接代替 worker 发送消息**。

它主要负责：

1. 扫描待处理任务规模
2. 监控 partition worker 心跳
3. 发现新任务和积压
4. 回收超时租约
5. 触发告警

## 7.2 为什么不让 Supervisor 直接发送

如果让 Supervisor 自己发送，就会和 partition worker 的职责冲突：

1. 既监控又消费，边界会乱
2. 分片串行模型会被破坏
3. 未来扩容困难

因此推荐：

1. Supervisor 做监控和治理
2. Worker 做实际消费和发送

### 对“发现有新消息就触发发送”的理解

落地上可以采用：

1. Supervisor 扫描到新任务后，更新运行指标和唤醒信号
2. Worker 仍然通过短轮询/条件轮询拉取本分片任务

因为当前只用 MySQL，不额外引入本地 IPC / MQ，**短轮询 + 租约控制** 是最稳妥实现。

## 7.3 Supervisor 检查项

1. 是否存在 `RETRY_WAIT` 且 `next_retry_ts <= now` 的任务
2. 是否存在 `PROCESSING` 且 `lease_expire_ts < now` 的任务
3. 某个 partition 的 `last_heartbeat_ts` 是否超时
4. 某个 partition 的积压是否超过阈值
5. 是否存在 `DEAD` 任务超过阈值

## 8. Worker 处理流程

## 8.1 领任务

每个 worker 只拉取本 partition：

1. 查询 `partition_no = 当前分片`
2. 查询 `task_status in (INIT, RETRY_WAIT)`
3. 查询 `next_retry_ts <= now`
4. 按 `created_ts asc` / `next_retry_ts asc`
5. `for update skip locked`

## 8.2 执行前检查

发送前必须做业务前置检查，避免重复补偿：

### 齐为权益购买补偿

检查：

1. `benefit_order` 是否已经是成功状态
2. 是否已经存在对应成功扣费 / 开通结果
3. 若已成功，则直接将任务置 `CANCELLED` 或 `SUCCESS`

### 用信申请补偿

检查：

1. `loan_application_mapping` 是否已经存在成功映射
2. 是否已经能通过查询接口查到申请结果
3. 如果本地已是 `SUBMITTED` / `PENDING_REVIEW`，则继续按“优先查询、必要时补发”的顺序处理
4. 若已成功，则直接关闭任务
5. 若已明确失败，则将任务置 `CANCELLED` 或 `DEAD`

## 8.3 执行结果处理

### 成功

1. 写 `attempt` 日志
2. 更新任务为 `SUCCESS`
3. 清空租约

### 失败但可重试

1. `retry_count + 1`
2. 更新 `next_retry_ts`
3. 置 `RETRY_WAIT`
4. 保存最近错误信息

### 失败且不可重试

例如：

1. 参数非法
2. 业务已关闭
3. 上游明确返回永久失败

处理：

1. 置 `DEAD` 或 `CANCELLED`
2. 记录错误原因

## 9. 重试策略

## 9.1 第一阶段已落地策略

第一阶段采用**配置化指数退避 + 最大间隔封顶**，避免固定间隔在上游恢复前形成持续压力。

配置项：

1. `nexusfin.async-compensation.max-retry-count`：最大失败重试次数，默认 `5`
2. `nexusfin.async-compensation.retry-initial-delay-seconds`：首次失败后的重试间隔，默认 `60`
3. `nexusfin.async-compensation.retry-max-delay-seconds`：指数退避上限，默认 `900`

计算公式：

```text
next_retry_delay_seconds =
  min(retry_initial_delay_seconds * 2^(retry_count_after_failure - 1),
      retry_max_delay_seconds)
```

默认配置下：

| 失败后 retry_count | next_retry_ts 间隔 | 任务状态 |
|---:|---:|---|
| 1 | 60 秒 | `RETRY_WAIT` |
| 2 | 120 秒 | `RETRY_WAIT` |
| 3 | 240 秒 | `RETRY_WAIT` |
| 4 | 480 秒 | `RETRY_WAIT` |
| 5 | 达到阈值，不再调度 | `DEAD` |

处理规则：

1. Worker 执行失败时，先记录 `async_compensation_attempt`
2. 若 `retry_count_after_failure < max_retry_count`，任务置为 `RETRY_WAIT` 并写入 `next_retry_ts`
3. 若 `retry_count_after_failure >= max_retry_count`，任务置为 `DEAD`
4. 成功后任务置为 `SUCCESS`，写入 `success_ts`，并清空租约字段

## 9.2 可按任务类型单独配置

1. 齐为权益购买补偿：重试更积极
2. 云卡/小花用信申请补偿：重试次数可更少，但每次前置查询更严格，并且优先查询而不是直接补发

> 当前第一阶段先使用全局重试配置；后续如需要按 `task_type` 差异化，可在配置中扩展 `task-type-policies`，不需要改变主表结构。

## 9.3 租约回收后的重试口径

Supervisor 发现 `PROCESSING` 任务的 `lease_expire_ts < now` 时，认为 worker 进程可能已卡死或退出。

回收规则：

1. 任务从 `PROCESSING` 回收为 `RETRY_WAIT`
2. 清空 `lease_owner` 和 `lease_expire_ts`
3. `next_retry_ts` 按 `retry_initial_delay_seconds` 推后
4. 不增加 `retry_count`，因为这不是一次明确的业务调用失败
5. 后续仍由对应 partition worker 重新领取并执行

## 10. 幂等设计

## 10.1 入队幂等

同一业务失败重复入队时，必须保证只保留一条有效任务：

唯一键建议：

1. `task_type + biz_key`

例如：

1. `QW_BENEFIT_PURCHASE_RETRY + BENEFIT_PURCHASE:BEN-xxx`
2. `YUNKA_LOAN_APPLY_RETRY + LOAN_APPLY:APP-xxx`

## 10.2 执行幂等

发送前必须做“状态确认”：

1. 如果业务已经成功，则不再重复发
2. 如果上游支持幂等号，则复用原始业务幂等号
3. 若上游不支持幂等号，则本地必须先查再发

### 10.2.1 第一阶段已落地口径

当前代码已落地以下执行前守卫：

1. `QW_BENEFIT_PURCHASE_RETRY`
   - 通过 `benefitOrderNo` 查询本地 `benefit_order`
   - 若 `sync_status = SYNC_SUCCESS`，直接返回 `SKIPPED_ALREADY_SYNCED`
   - 若本地未成功，再执行齐为同步补偿

2. `YUNKA_LOAN_APPLY_RETRY`
   - 通过 `applyId` 查询本地 `loan_application_mapping`
   - 若 `mapping_status = ACTIVE`，直接返回 `SKIPPED_ALREADY_ACTIVE`
   - 若 `mapping_status = PENDING_REVIEW`，先调用 `/loan/query`
   - 若查询已经能拿到有效结果，则直接以查询结果结束本次补偿，不再补发 `/loan/apply`
   - 只有“本地不是 ACTIVE，且查询未拿到有效结果”时，才补发 `/loan/apply`

这意味着第一阶段已经实现了“**先查后发**”的最小闭环，可降低重复补发风险。

### 10.2.2 用户重试与异步补偿并发的第一阶段兜底

针对“同步失败后用户手工重试，同时后台异步补偿仍在执行”的场景，第一阶段补了两层兜底：

1. **请求级重放**
   - `BenefitOrderService.createOrder(...)` 在正式创建前，先按 `requestId` 查询本地 `benefit_order`
   - 如果已经存在同 `requestId` 的订单，即使 `idempotency_record` 还没来得及写入，也直接回放已有订单
   - 这可以覆盖“首次请求已插入订单，但齐为超时导致主链路返回失败，用户再次点击重试”的窗口期

2. **补偿成功后立即回写本地业务状态**
   - 齐为补偿成功后，回写 `benefit_order.sync_status = SYNC_SUCCESS`
   - 云卡补偿成功或查询到上游有效结果后，回写 `loan_application_mapping.mapping_status = ACTIVE`
   - 后续无论是用户再次触发，还是 worker 再次扫描任务，都能先基于本地状态跳过重复执行

因此，第一阶段已把“同步失败 + 用户重试 + 异步补偿”最容易出现的重复执行窗口压缩到最小。

## 10.3 第一阶段已落地监控能力

当前代码已提供以下 supervisor 基础监控查询能力：

1. `findOfflinePartitions()`
   - 基于 `async_compensation_partition_runtime.last_heartbeat_ts`
   - 超过 `worker-heartbeat-timeout-seconds` 的分片会被识别为离线

2. `collectReadyTaskBacklogByPartition()`
   - 统计 `INIT` / `RETRY_WAIT` 且 `next_retry_ts <= now` 的待处理任务
   - 返回按 `partition_no` 聚合后的积压数量

3. `countDeadTasks()`
   - 统计 `DEAD` 状态任务数量

当前阶段这些能力先作为服务层查询能力提供，后续可接定时巡检、日志告警或监控面板。

## 11. 与现有表的关系

## 11.1 不建议复用现有表

现有表中：

1. `notification_receive_log` 是接收入站通知日志
2. `benefit_status_push_log` 是状态回推日志

二者都不适合作为通用补偿任务主表。

## 11.2 业务表仍作为真相来源

补偿任务表只负责“调度状态”，不负责替代业务状态：

1. 权益购买真实状态仍看 `benefit_order`
2. 用信映射真实状态仍看 `loan_application_mapping`

## 12. 本轮建议任务类型

## 12.1 齐为权益购买补偿

### 任务类型

1. `QW_BENEFIT_PURCHASE_RETRY`

### 请求信息

1. `target_code = QW`
2. `request_path = 齐为购买权益接口地址`
3. `request_payload = 原购买权益请求参数`

### biz_key

1. `BENEFIT_PURCHASE:{benefitOrderNo}`

## 12.2 云卡/小花用信补偿

### 任务类型

1. `YUNKA_LOAN_APPLY_RETRY`

### 请求信息

1. `target_code = YUNKA`
2. `request_path = 云卡 gateway 地址`
3. `request_payload = 原用信请求参数`
4. `request_payload` 内部保留科技平台 path

### biz_key

1. `LOAN_APPLY:{applicationId}`

## 13. 风险与边界

## 13.1 风险

1. 齐为权益购买场景下，同步接口仍返回失败，用户可能手工重试，异步补偿也会同时发生，必须靠幂等控制
2. 用信提交场景下，页面会显示“已提交 / 审核中”，但如果上游其实未落单，补偿链路必须继续推进并定期查询
3. 如果上游已经成功但本地没收到响应，补偿时容易重复下单，因此必须先查后发
4. 分片数一旦上线，不宜频繁变更
5. 只用 MySQL 轮询时，会有一定查询压力

## 13.2 边界

本轮不进入：

1. 外部 MQ 引入
2. 动态分片迁移
3. Supervisor 自动拉起 OS 级进程
4. 全量任务类型泛化到所有下游场景

## 14. 推荐实施顺序

### Phase 1：表结构和基础状态机

1. 新增 `async_compensation_task`
2. 新增 `async_compensation_attempt`
3. 新增 `async_compensation_partition_runtime`
4. 定义状态枚举、任务类型枚举、目标系统枚举

### Phase 2：入队能力

1. 实现 `CompensationEnqueueService`
2. 在齐为权益购买失败处入队
3. 在云卡/小花用信超时处入队，并同步回写 `SUBMITTED` / `PENDING_REVIEW`
4. 补幂等入队测试

### Phase 3：Worker 执行能力

1. 实现 partition worker 拉取任务
2. 实现租约和超时回收
3. 实现按任务类型路由到不同 executor
4. 补成功 / 可重试失败 / 永久失败测试

### Phase 4：Supervisor 治理能力

1. 实现心跳写入
2. 实现分片健康检查
3. 实现积压检查
4. 实现卡死租约回收

### Phase 5：联调与运维口径

1. 输出任务排查文档
2. 输出人工重放 / 人工取消说明
3. 输出死信处理流程

## 15. 建议的实现 plan

1. 先做独立补偿任务表，不复用现有日志表
2. 先支持两类任务：
  - `QW_BENEFIT_PURCHASE_RETRY`
  - `YUNKA_LOAN_APPLY_RETRY`
3. 先固定 `partition_count`
4. 每个 partition 一个独立 worker 进程
5. supervisor 只做监控、告警、回收，不直接发送
6. 所有请求参数完整落库
7. 所有补偿前必须做业务状态确认，避免重复执行

## 16. 一句话结论

> 这套机制本质上是“基于 MySQL 的补偿任务队列”：对齐为权益购买链路，作用是“同步失败后后台自愈”；对云卡/小花用信提交流程，作用是“同步超时后页面先显示已提交 / 审核中，后台继续补偿与查询”，最终把两条关键链路都做成可恢复、可追踪、可运维的后台自愈能力。
