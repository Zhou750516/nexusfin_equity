# 2026-04-26 云卡状态查询与 H5 状态映射方案

## 1. 文档目标

本文档用于解决当前借款链路中的一个核心问题：

> 审批中、审批结果、拒绝落地和分发等页面，不能继续混用“云卡内部状态”“小花最终放款结果”“H5 页面展示状态”。

本方案的目标是：

1. 先把三层状态拆开；
2. 给出一版推荐的本地状态枚举；
3. 给出状态到 H5 页面阶段的映射表；
4. 明确 `approval-status` / `approval-result` / 分发上下文接口建议字段；
5. 区分哪些字段由艾博生本地维护，哪些必须依赖云卡接口。

---

## 2. 为什么需要单独做状态映射

当前如果继续把状态混在一起，会有几个明显问题：

1. “云卡订单已创建”不等于“小花已开始放款”；
2. “小花放款处理中”不等于“H5 应直接展示审批结果页”；
3. “放款失败”与“用户放弃权益导致关单”虽然都不是成功，但页面承接不一定相同；
4. “拒绝导流页”“行权分发页”“审批中页”所需的上下文字段不同，不能只靠一个简单状态码硬扛。

因此建议后续统一按三层状态思路处理：

1. **本地业务阶段状态**
2. **云卡订单状态**
3. **小花最终放款结果状态**

H5 页面则在这三层之上做一层**展示态映射**。

---

## 3. 三层状态模型

## 3.1 第一层：艾博生本地业务阶段状态

这一层是艾博生最适合维护的状态，用于表达当前流程推进到哪一步。

建议枚举：

1. `ORDER_CREATED`
2. `WAIT_BENEFIT_PURCHASE`
3. `WAIT_SIGN`
4. `SIGN_COMPLETED`
5. `WAIT_TRIGGER_LENDING`
6. `LENDING_TRIGGERED`
7. `LENDING_PROCESSING`
8. `LENDING_SUCCESS`
9. `LENDING_FAILED`
10. `ORDER_CLOSED`

这一层的特点：

1. 最贴近 H5 页面承接；
2. 最适合记录本地流程是否已经过权益 / 签约 / 触发放款；
3. 不依赖云卡是否已经给出最终完整状态机。

## 3.2 第二层：云卡订单状态

这一层表示云卡对放款订单本身的判断。

当前建议先按概念层预留：

1. `CREATED`
2. `WAIT_BENEFIT`
3. `WAIT_SIGN`
4. `WAIT_LENDING`
5. `LENDING`
6. `SUCCESS`
7. `FAILED`
8. `CLOSED`

注意：

1. 这不是最终字段定义；
2. 真实名称和语义仍要等待云卡接口定版；
3. 当前只用于本地建立映射框架。

## 3.3 第三层：小花最终放款结果状态

这一层表示小花最终放款语义。

基于当前已知小花 `4.22` 口径，建议先统一理解为：

1. `7001 = SUCCESS`
2. `7002 = PROCESSING`
3. `7003 = FAILED`

注意：

1. 这层只在真正触发放款后才有意义；
2. 在建单、权益、签约阶段，可能根本还没有小花最终结果。

---

## 4. H5 页面展示态建议

H5 不建议直接消费云卡状态或小花状态，而建议统一消费**页面展示态**。

建议枚举：

1. `TO_BENEFIT_PAGE`
2. `TO_SIGN_PAGE`
3. `TO_LOADING_PAGE`
4. `TO_APPROVAL_PENDING_PAGE`
5. `TO_APPROVAL_RESULT_SUCCESS_PAGE`
6. `TO_APPROVAL_RESULT_FAILURE_PAGE`
7. `TO_REJECT_LANDING_PAGE`
8. `TO_DISPATCH_PAGE`
9. `TO_CLOSED_PAGE`

说明：

1. 页面展示态的目的不是替代底层状态；
2. 而是让前端明确“现在应该展示哪个页面 / 阶段”。

---

## 5. 推荐映射表

## 5.1 本地业务状态 -> H5 页面态

| 本地业务状态 | 建议 H5 页面态 | 说明 |
| --- | --- | --- |
| `ORDER_CREATED` | `TO_BENEFIT_PAGE` | 云卡订单已创建，待进入权益页 |
| `WAIT_BENEFIT_PURCHASE` | `TO_BENEFIT_PAGE` | 待完成权益购买 |
| `WAIT_SIGN` | `TO_SIGN_PAGE` | 待完成短信签约 |
| `SIGN_COMPLETED` | `TO_LOADING_PAGE` | 签约完成，待进入放款触发 / 状态承接 |
| `WAIT_TRIGGER_LENDING` | `TO_LOADING_PAGE` | 已完成前置条件，待触发放款 |
| `LENDING_TRIGGERED` | `TO_APPROVAL_PENDING_PAGE` | 已触发放款，待后续查询 |
| `LENDING_PROCESSING` | `TO_APPROVAL_PENDING_PAGE` | 小花 / 云卡仍在处理中 |
| `LENDING_SUCCESS` | `TO_APPROVAL_RESULT_SUCCESS_PAGE` | 放款成功 |
| `LENDING_FAILED` | `TO_APPROVAL_RESULT_FAILURE_PAGE` | 放款失败，需视失败原因决定是否转拒绝页 |
| `ORDER_CLOSED` | `TO_CLOSED_PAGE` / `TO_REJECT_LANDING_PAGE` | 关单原因不同，页面承接可能不同 |

## 5.2 云卡状态 + 小花结果 -> H5 页面态

| 云卡状态 | 小花结果 | 建议 H5 页面态 | 说明 |
| --- | --- | --- | --- |
| `CREATED` | 无 | `TO_BENEFIT_PAGE` | 已建单但未触发放款 |
| `WAIT_BENEFIT` | 无 | `TO_BENEFIT_PAGE` | 等待权益前置完成 |
| `WAIT_SIGN` | 无 | `TO_SIGN_PAGE` | 等待短信签约 |
| `WAIT_LENDING` | 无 | `TO_LOADING_PAGE` | 前置完成，待触发放款 |
| `LENDING` | `PROCESSING` / 无 | `TO_APPROVAL_PENDING_PAGE` | 已进入处理中 |
| `SUCCESS` | `SUCCESS` | `TO_APPROVAL_RESULT_SUCCESS_PAGE` | 最终成功 |
| `FAILED` | `FAILED` | `TO_APPROVAL_RESULT_FAILURE_PAGE` | 最终失败 |
| `CLOSED` | 无 / `FAILED` | `TO_REJECT_LANDING_PAGE` / `TO_CLOSED_PAGE` | 要看关闭原因 |

## 6. 关键页面承接说明

## 6.1 权益页

适用阶段：

1. `ORDER_CREATED`
2. `WAIT_BENEFIT_PURCHASE`

页面要表达的是：

1. 借款流程已创建；
2. 当前需要完成权益前置动作；
3. 并不是借款已经失败或已经进入最终审批。

## 6.2 签约页

适用阶段：

1. `WAIT_SIGN`

页面要表达的是：

1. 用户已进入权益签约环节；
2. 签约成功前不应触发放款。

## 6.3 loading 页

适用阶段：

1. `SIGN_COMPLETED`
2. `WAIT_TRIGGER_LENDING`

页面要表达的是：

1. 权益和签约已完成；
2. 当前系统正在承接下一步，不是结果页。

## 6.4 审批中页

适用阶段：

1. `LENDING_TRIGGERED`
2. `LENDING_PROCESSING`

页面要表达的是：

1. 已进入放款处理；
2. 当前结果未最终确定；
3. 前端可以轮询状态。

## 6.5 审批结果页

适用阶段：

1. `LENDING_SUCCESS`
2. `LENDING_FAILED`

页面要表达的是：

1. 当前已经拿到明确结果；
2. 成功和失败使用不同承接文案。

## 6.6 拒绝落地页 / 分发页

适用阶段：

1. `ORDER_CLOSED`
2. 某些 `LENDING_FAILED` 的细分类

页面要表达的是：

1. 当前不是普通“审批失败提示”，而是要引导到其他承接路径；
2. 需要更多上下文字段支持导流和分发。

## 7. 接口建议字段

## 7.1 `GET /api/loan/approval-status/{id}` 建议字段

建议返回：

1. `applicationId`
2. `yunkaOrderNo`
3. `localStageStatus`
4. `displayStatus`
5. `yunkaOrderStatus`
6. `xiaohuaResultStatus`
7. `canPoll`
8. `nextAction`
9. `message`
10. `updatedAt`

用途：

1. 供审批中页 / loading 页使用；
2. 供前端决定是否继续轮询；
3. 供前端决定下一步跳转。

## 7.2 `GET /api/loan/approval-result/{id}` 建议字段

建议返回：

1. `applicationId`
2. `yunkaOrderNo`
3. `localStageStatus`
4. `displayStatus`
5. `resultCode`
6. `resultMessage`
7. `approvedAmount`
8. `benefitOrderNo`
9. `redirectType`
10. `redirectContext`

用途：

1. 供结果页、拒绝导流页、分发页统一承接；
2. 供后续导流 / 分发页面读取上下文。

## 7.3 分发上下文接口建议字段

建议至少包括：

1. 用户标识
2. 权益订单号
3. 云卡订单号
4. 当前分发原因
5. 当前可展示权益 / 产品信息
6. 页面来源

---

## 8. 哪些字段本地维护，哪些依赖云卡

## 8.1 本地优先维护

1. 本地业务阶段状态 `localStageStatus`
2. 页面展示态 `displayStatus`
3. 权益购买状态
4. 签约状态
5. 本地映射主键：
   - `applicationId`
   - `benefitOrderNo`
   - `yunkaOrderNo`

## 8.2 依赖云卡返回

1. 云卡真实订单状态
2. 是否允许关单
3. 放款是否已触发
4. 云卡对小花最终结果的映射
5. 某些分发 / 拒绝原因

## 8.3 依赖小花最终结果

1. 最终放款成功 / 处理中 / 失败语义
2. 失败原因细分
3. 可能的最终金额 / 结果说明

---

## 9. 当前最需要避免的错误

1. 把 `7003` 继续当成成功；
2. 把“云卡已建单”误当成“已进入审批结果阶段”；
3. 把“签约完成”误当成“放款已触发”；
4. 把“订单关闭”和“普通审批失败”完全等同；
5. 让前端直接消费底层状态码而没有页面展示态。

---

## 10. 建议下一步

1. 将本方案和“用信新链路改造方案”一起作为后续接口设计前置材料；
2. 在 Shared Foundation 中优先预留：
   - `localStageStatus`
   - `displayStatus`
   - `yunkaOrderStatus`
   - `xiaohuaResultStatus`
3. 等云卡状态机接口定版后，再把本映射表从“概念稿”升级为“真实接口映射表”。

---

## 11. 一句话结论

> 后续状态设计的关键，不是再增加一个新的状态码，而是把“本地业务阶段、云卡订单状态、小花最终结果、H5 页面态”拆成四层，再用映射关系串起来。
