# 2026-03-31 云卡完整交互独立排期

## 1. 文档目标

本文档只用于规划 `specs/004-benefit-loan-flow` 这条独立功能主线，即：

1. 艾博生权益页补充接口
2. 云卡借款确认与审核状态相关能力

说明：

1. 本文档 **不** 把齐为联调作为同一排期主线。
2. 本文档 **不** 用“齐为参数是否到位”作为云卡功能开发的唯一节奏判断。
3. 本文档只回答“云卡完整交互这条线还差什么、怎么排、预计多久能收口”。

## 4. 当前关键待确认事项

这条主线虽然不和齐为联调混在一起，但仍有几项独立待确认事项，会影响开发节奏。

### 4.1 借款试算 / 确认借款的真实调用方向

根据 `docs/third-part/科技平台/科技平台接口-业务流程分析合理性评审_20260330.md`，当前文档之间仍存在视角冲突：

1. PRD/流程文档里有“云卡调用科技平台”的表述
2. 标准接口文档里对应接口又更像“平台调机构”

**影响**

1. 直接影响 `T023`、`T024` 的 client 设计边界
2. 如果方向不先澄清，后续 service/controller 很容易建立在错误职责上

### 4.2 云卡借款页到底是谁承载

当前仍需确认：

1. 是云卡自有页面
2. 还是科技平台承载页 / 中转页
3. 或者是艾博生只提供入口参数与回跳能力

**影响**

1. 影响 `Y2` 需要实现的接口到底是“页面后端接口”还是“给外部系统的编排接口”

### 4.3 还款身份验证方案

当前仍需确认：

1. 是否复用科技平台短信验证码能力
2. 是否需要单独的身份认证服务
3. 还款前的发起 / 校验 / 超时口径

**影响**

1. 直接影响 `T030`、`T031`、`T032`
2. 若不确认，`Y3` 很难进入稳定实现

### 4.4 结清证明责任方

当前 spec 已明确这是待确认项。

**影响**

1. 不阻塞 `Y1/Y2` 主体开发
2. 但会影响 `Y3` 之后的页面闭环定义和验收范围

## 5. 对交付流程的影响

这条主线对整体交付的影响，应单独表述为：

1. 即使齐为联调推进顺利，**云卡完整交互主线** 仍可能因为页面职责、接口方向、还款鉴权方案未确认而延后。
2. 反过来，即使云卡完整交互开发推进顺利，也不应被用来替代齐为联调完成的判断。
3. 后续汇报应明确拆成两条交付视图：
   - 交付视图 A：当前 ABS 主链 + 齐为联调
   - 交付视图 B：云卡完整页面与借还款交互

## 6. 建议执行顺序

建议按以下顺序推进：

1. 先完成 `Y1`，把艾博生权益页补齐
2. 再完成 `Y2`，让借款确认和审核状态形成闭环
3. 最后完成 `Y3`，补齐还款页和还款同步
4. 结束后统一进入 `Y4`

原因：

1. 权益页是最前置的页面入口
2. 借款确认与审核是中段主链
3. 还款链路依赖更多业务定义和鉴权口径
