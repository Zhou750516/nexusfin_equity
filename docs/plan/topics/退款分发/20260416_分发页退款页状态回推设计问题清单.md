# 2026-04-16 艾博生分发页 / 退款页 / 状态回推设计问题清单

## 1. 背景

根据 `2026-04-15` 小花科技沟通结论：

1. 放款成功后的可行权权益提醒 push 由科技平台触发
2. push 可从科技平台 App 跳转到艾博生分发页面
3. 艾博生分发页面再跳转到齐为供应商页面
4. 权益订单页可直接跳转艾博生页面
5. 权益包订单退款流程也跳转艾博生页面
6. 权益状态变更需要推送给科技平台
7. 联合登录采用“科技平台携带 token 跳转艾博生，艾博生后端调小花后端”方案

在小花联合登录正式文档尚未补齐前，先把艾博生侧需要设计的问题收集起来。

---

## 2. 分发页设计问题

### 2.1 入口类型

需要明确艾博生分发页是否统一承接以下入口：

| 入口 | 来源 | 目标 |
|---|---|---|
| 可行权提醒 push | 科技平台 App | 艾博生分发页 -> 齐为供应商页面 |
| 权益订单页行权 | 科技平台权益订单页 | 艾博生页面 / 分发页 |
| 权益包退款入口 | 科技平台权益包订单 | 艾博生页面 / 退款页 |

待确认问题：

1. 三类入口是否复用同一个 H5 路由？
2. 是否通过 `scene` / `bizType` 区分场景？
3. 是否需要不同落地页 UI？
4. 是否需要兜底页处理参数缺失 / 状态异常？

### 2.2 跳转参数

待确认参数：

1. `token`
2. `orderNo` / `benefitOrderNo`
3. `benefitPackageId` / `productCode`
4. `scene`：`push` / `exercise` / `refund`
5. `redirectUrl` 或供应商页面标识
6. 时间戳、nonce、签名

待确认问题：

1. 哪些参数由科技平台直接携带？
2. 哪些参数由艾博生后端通过 token 换取？
3. 是否允许前端直接拼供应商跳转地址？
4. 是否所有跳转都必须经过艾博生后端确认？

### 2.3 登录态与用户上下文

待确认问题：

1. 分发页打开时是否一定携带小花 token？
2. 艾博生是否需要建立本地 `NEXUSFIN_AUTH` Cookie？
3. token 失效时展示重试页、错误页，还是跳回科技平台？
4. 是否需要支持同一用户多次进入同一权益订单？

### 2.4 跳齐为供应商页面

待确认问题：

1. 齐为供应商页面地址由谁生成？
2. 是否需要艾博生后端调用齐为获取行权 URL？
3. 是否需要记录跳转流水？
4. 供应商页面打不开时，艾博生页面如何展示？

## 3. 退款页设计问题

### 3.1 页面职责

需要明确退款入口跳转到艾博生页面后，该页面是：

1. 退款说明页
2. 退款申请页
3. 退款状态页
4. 仅中转到齐为 / 供应商退款页

待确认问题：

1. 用户是否在艾博生页面点击“申请退款”？
2. 是否需要展示可退金额、服务状态、退款规则？
3. 是否需要展示退款进度？
4. 退款失败是否需要允许重试？

### 3.2 接口边界

待确认问题：

1. 退款申请由艾博生调齐为，还是跳转齐为页面完成？
2. 退款结果由齐为回调艾博生，还是艾博生主动查询？
3. 退款状态是否要同步给科技平台？
4. 是否需要新增 H5 接口：
   - `GET /api/refund/info/{orderNo}`
   - `POST /api/refund/apply`
   - `GET /api/refund/result/{refundId}`

## 4. 权益状态回推科技平台设计问题

### 4.1 事件清单

初步建议覆盖以下事件：

| 事件 | 是否需要回推 | 说明 |
|---|---|---|
| 权益订单创建 | 待确认 | H5 借款申请中会先创建权益订单 |
| 权益开通成功 | 待确认 | 用户主动开通惠选卡 |
| 放款成功 | 待确认 | 科技平台自身会触发 push，但艾博生也可能需回推状态 |
| 行权成功 | 是 | 状态发生实质变化 |
| 行权失败 | 是 | 需要科技平台感知失败原因 |
| 退款申请 | 待确认 | 是否需要实时展示退款中 |
| 退款成功 | 是 | 权益包状态变化 |
| 退款失败 | 是 | 需要展示失败原因或后续动作 |

### 4.2 回推接口问题

待确认问题：

1. 科技平台提供回调地址，还是艾博生提供接口由科技平台拉取？
2. 回推是否需要签名？
3. 回推失败是否重试？重试几次？间隔多久？
4. 是否需要幂等键？建议使用 `eventId` 或 `orderNo + status + eventTime`
5. 是否需要记录回推日志表？

### 4.3 回推字段问题

建议字段：

1. `eventId`
2. `eventType`
3. `externalUserId`
4. `benefitOrderNo`
5. `productCode`
6. `statusBefore`
7. `statusAfter`
8. `eventTime`
9. `reason` / `remark`
10. `sourceSystem`

待小花 / 科技平台确认最终字段。

5. 测试环境与联调样例
