# 2026-04-18 退款页联调检查清单

## 1. 适用范围

本文用于联调退款页当前已落地的最小闭环，验证范围包括：

1. `/api/refund/info/{benefitOrderNo}`
2. `/api/refund/apply`
3. `/api/refund/result/{refundId}`
4. H5 `joint-refund.logic.ts`

---

## 2. 联调前检查

### 后端

1. 确认后端已启动
2. 确认退款三个接口可访问
3. 确认访问受保护接口时可以带 `NEXUSFIN_AUTH` Cookie

### H5

1. 确认 H5 本地开发环境已启动
2. 确认 `/joint-refund-entry` 可打开
3. 确认 `/api/*` 代理正常

---

## 3. 手工执行步骤

### 场景 A：退款前置信息

1. 请求：

```text
GET /api/refund/info/{benefitOrderNo}
```

2. 检查返回字段：
   - `benefitOrderNo`
   - `refundable`
   - `refundStatus`
   - `refundableAmount`

### 场景 B：退款申请

1. 请求：

```text
POST /api/refund/apply
```

2. 请求体：

```json
{
  "benefitOrderNo": "BEN-20260418-001",
  "reason": "USER_REQUEST"
}
```

3. 检查返回字段：
   - `refundId`
   - `status`
   - `message`

### 场景 C：退款结果查询

1. 请求：

```text
GET /api/refund/result/{refundId}
```

2. 检查返回字段：
   - `refundId`
   - `status`
   - `message`

### 场景 D：H5 退款状态机

1. 当：
   - `refundable=true`
   - `refundStatus=NONE`
2. 检查前端逻辑是否进入：

```text
apply
```

3. 当：
   - `refundable=false`
4. 检查前端逻辑是否进入：

```text
blocked
```

5. 当：
   - `refundStatus=PROCESSING`
6. 检查前端逻辑是否进入：

```text
processing
```

---

## 4. 关键检查项

1. `benefitOrderNo` 与 `refundId` 是否在链路中可追踪
2. 退款申请接口是否能返回稳定受理编号
3. 退款结果接口是否能返回处理中状态
4. 日志是否带：
   - `traceId`
   - `bizOrderNo`

---

## 5. 当前说明

本轮仅验证“退款页骨架”。

当前仍未包含：

1. 齐为真实退款协议
2. 退款申请真实落库
3. 退款失败重试与客服引导
