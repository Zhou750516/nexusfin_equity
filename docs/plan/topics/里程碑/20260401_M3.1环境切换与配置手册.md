# 2026-04-01 M3.1 环境切换与配置手册

## 1. 文档目的

本文档用于把 `M3.1` 出站接口真实联调前需要完成的本地环境切换、配置覆盖和启动检查固定下来，避免拿到外部参数后还要临场查代码判断该改哪些配置。

说明：

1. 本文档只覆盖 **齐为联调主线**。
2. 当前代码默认仍以 `MOCK` 口径启动，真实联调必须显式覆盖运行时配置。
3. 若当天仍未拿到真实参数，可先用本手册完成“待填模板”和启动动作预演。

## 2. 当前代码中的关键事实

### 2.1 齐为出站链路已接入的本地入口

1. `BenefitOrderServiceImpl.createOrder()` -> `abs.push.order`
2. `BenefitOrderServiceImpl.getExerciseUrl()` -> `abs.token.get`
3. `NotificationServiceImpl.handleGrant()` -> `abs.lending.notify`

### 2.2 当前默认配置特点

从 `src/main/resources/application.yml` 可见：

1. `QW_MODE` 默认值为 `MOCK`
2. `QW_BASE_URL`、`QW_METHOD_PATH`、`QW_PARTNER_NO`、`QW_SIGN_KEY`、`QW_AES_KEY_BASE64` 都有占位默认值
3. `TECH_PLATFORM_API_MODE` 默认值也为 `MOCK`
4. `TECH_PLATFORM_BASE_URL` 与 `TECH_PLATFORM_USER_ME_PATH` 仍使用本地占位值

因此结论是：

1. 代码主链已就绪
2. 真实联调默认不会自动发生
3. 必须通过环境变量或启动参数覆盖默认配置

## 3. `QW_MODE` 选择规则

当前代码支持以下模式：

1. `MOCK`
2. `HTTP`
3. `QWEIMOBILE_HTTP`
4. `ALLINPAY_DIRECT`

建议使用口径如下：

### 3.1 如果今天走齐为标准 HTTP 联调

使用：

- `QW_MODE=HTTP`

或：

- `QW_MODE=QWEIMOBILE_HTTP`

原因：

1. 在当前路由实现中，`MOCK`、`HTTP`、`QWEIMOBILE_HTTP` 都走 `QwBenefitClientImpl`
2. 其中只有 `MOCK` 会返回本地 mock 数据
3. 因此只要不是 `MOCK`，且参数已真实覆盖，就会命中真实 HTTP 请求

### 3.2 如果今天走通联直连模式

使用：

- `QW_MODE=ALLINPAY_DIRECT`

但要注意：

1. 该模式依赖证书、验签证书、serviceCode 等一整套配置
2. 缺任一关键项都会抛配置缺失异常
3. 若当前联调目标仍是齐为标准 HTTP 网关，不应误切到该模式

## 4. `M3.1` 最小必填配置

### 4.1 齐为 HTTP 联调最小配置

以下配置拿齐后，才具备真实 `M3.1` 启动条件：

```bash
export QW_ENABLED=true
export QW_MODE=HTTP
export QW_BASE_URL='待齐为确认'
export QW_METHOD_PATH='/api/abs/method'
export QW_PARTNER_NO='待齐为确认'
export QW_VERSION='v1.0'
export QW_SIGN_KEY='待齐为确认'
export QW_AES_KEY_BASE64='待齐为确认'
export QW_CONNECT_TIMEOUT_MS=3000
export QW_READ_TIMEOUT_MS=5000
```

补充说明：

1. `QW_METHOD_PATH`、`QW_VERSION` 虽然当前有默认值，但联调前仍应以齐为最终口径为准
2. 超时时间可按联调当天网络情况调整，但不建议继续使用过低默认值

### 4.2 科技平台真实闭环最小配置

若当天要同时验证真实用户信息链路，至少补齐：

```bash
export TECH_PLATFORM_BASE_URL='待科技平台确认'
export TECH_PLATFORM_USER_ME_PATH='/api/users/me'
export TECH_PLATFORM_API_ENABLED=true
export TECH_PLATFORM_API_MODE=HTTP
export TECH_PLATFORM_API_BASE_URL='待科技平台确认'
export TECH_PLATFORM_API_CHANNEL_ID='待科技平台确认'
export TECH_PLATFORM_API_VERSION='待科技平台确认'
export TECH_PLATFORM_API_SIGN_SECRET='待科技平台确认'
export TECH_PLATFORM_API_AES_KEY_BASE64='待科技平台确认'
```

说明：

1. `auth.tech-platform-base-url` 影响认证侧 `/api/users/me` 拉取
2. `nexusfin.third-party.tech-platform` 影响主动通知科技平台的三类 outbond client
3. 两组配置不要混淆

## 5. 通联直连模式额外配置

如果后续切到 `ALLINPAY_DIRECT`，除第 4 节外，还至少需要：

```bash
export QW_MODE=ALLINPAY_DIRECT
export QW_DIRECT_BASE_URL='待确认'
export QW_DIRECT_PROCESS_PATH='/aipg/ProcessServlet'
export QW_DIRECT_PKCS12_PATH='待提供'
export QW_DIRECT_PKCS12_PASSWORD='待提供'
export QW_DIRECT_VERIFY_CERT_PATH='待提供'
export QW_DIRECT_MEMBER_SYNC_SERVICE_CODE='待提供'
export QW_DIRECT_EXERCISE_URL_SERVICE_CODE='待提供'
export QW_DIRECT_LENDING_NOTIFY_SERVICE_CODE='待提供'
```

若缺以下任一项，将无法启动真实调用：

1. `pkcs12Path`
2. `pkcs12Password`
3. `verifyCertPath`
4. 每个接口对应 `serviceCode`

## 6. 本地启动建议步骤

### 6.1 启动前检查

- [ ] 齐为真实参数已拿到
- [ ] 科技平台真实地址已拿到
- [ ] 若走真实用户链路，token 样例已拿到
- [ ] 当前终端已导入真实环境变量
- [ ] 确认未误使用默认占位值

### 6.2 启动方式建议

如果本地直接运行 Spring Boot：

```bash
mvn spring-boot:run
```

如果使用打包方式启动：

```bash
mvn clean package -DskipTests
java -jar target/nexusfin-equity.jar
```

## 7. 联调前自查清单

启动后建议至少确认以下点：

1. 启动日志中未出现配置缺失异常
2. `QW_MODE` 已不再是 `MOCK`
3. 真实请求将发往齐为确认后的 `QW_BASE_URL + QW_METHOD_PATH`
4. 日志可检索 `traceId`
5. 下单后日志可串联 `traceId + bizOrderNo`

## 8. 当前建议结论

截至 `2026-04-01` 当前仓库状态，建议统一执行口径如下：

1. 先拿齐真实外部参数，再切换运行时配置
2. 默认优先走 `QW_MODE=HTTP`
3. 没有明确直连证书与 serviceCode 前，不切 `ALLINPAY_DIRECT`
4. 参数未齐时不要误判为“代码还没准备好”，当前主要差的是运行时外部条件
