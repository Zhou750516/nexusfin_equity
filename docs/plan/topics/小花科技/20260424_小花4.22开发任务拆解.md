# 2026-04-24 小花 4.22 开发任务拆解

> 基于 `plan/20260424_小花4.22现有实现差异分析与开发计划.md`，将下一步开发工作拆为可以直接安排的任务组。

## 1. 任务目标

本轮任务目标不是从零重写 H5 接口，而是在现有代码基础上完成以下三件事：

1. 把“艾博生通过云卡调用小花”的共享网关层补齐
2. 把权益 / 借款 / 还款 / 回调链路按小花 `4.22` 文档对齐
3. 把 `2026-04-23` 会议新增的链路变化纳入后续借款主链设计

---

## 2. 任务分组总览

| 任务组 | 优先级 | 是否阻塞 | 目标 |
| --- | --- | --- | --- |
| Task Group A：共享基础收口 | P0 | 是 | 统一 Yunka 路径、DTO、facade、回调模型 |
| Task Group B：权益页动态化 | P1 | 否，依赖 A | 接协议列表、绑卡列表、权益同步 |

## 3. 任务明细

### Task Group A：共享基础收口

**目标**

把当前散落在 `LoanServiceImpl`、`RepaymentServiceImpl` 中的 Yunka 调用抽成统一模型层，避免后续权益 / 借款 / 还款继续各自拼 JSON。

**建议文件**

1. `src/main/java/com/nexusfin/equity/config/YunkaProperties.java`
2. `src/main/resources/application.yml`
3. `src/test/resources/application.yml`
4. `src/main/java/com/nexusfin/equity/service/XiaohuaGatewayService.java`
5. `src/main/java/com/nexusfin/equity/service/impl/XiaohuaGatewayServiceImpl.java`
6. `src/main/java/com/nexusfin/equity/thirdparty/yunka/` 下新增小花 `4.22` DTO
7. `src/test/java/com/nexusfin/equity/service/XiaohuaGatewayServiceTest.java`

**任务**

1. 扩展 `YunkaProperties.Paths`，补齐以下路径：
   - `protocolQuery`
   - `userToken`
   - `userQuery`
   - `loanRepayPlan`
   - `cardSmsSend`
   - `cardSmsConfirm`
   - `cardUserCards`
   - `creditImageQuery`
   - `benefitSync`
2. 为以下能力建立 typed request/response：
   - 协议列表
   - 用户鉴权
   - 用户信息查询
   - 用户绑卡列表
   - 借款试算 / 申请 / 查询 / 还款计划
   - 还款试算 / 申请 / 查询
   - 验证码发送 / 确认
   - 图片查询
   - 权益订单同步
3. 新建 `XiaohuaGatewayService`，由它统一封装：
   - 请求路径选择
   - Yunka 调用
   - 响应解包
   - 错误码转换
4. 为放款回调、还款回调定义新的 DTO，明确“小花通过云卡回调艾博生”的字段口径
5. 为共享 facade 补单测

**完成标准**

1. 业务 service 不再直接拼装原始 Yunka JSON
2. 小花 `4.22` 的主要接口在代码中有明确类型模型
3. 回调模型不再混用旧字段语义

### Task Group B：权益页动态化

**目标**

让权益页从“本地静态展示”为主，升级到“本地展示 + 小花动态协议 / 绑卡数据 + 权益同步”模式。

**建议文件**

1. `src/main/java/com/nexusfin/equity/service/impl/BenefitsServiceImpl.java`
2. `src/main/java/com/nexusfin/equity/controller/BenefitsController.java`
3. `src/main/java/com/nexusfin/equity/dto/response/` 下权益页响应 DTO
4. `src/test/java/com/nexusfin/equity/service/BenefitsServiceTest.java`
5. `src/test/java/com/nexusfin/equity/controller/BenefitsControllerIntegrationTest.java`

**任务**

1. 接入 `2.1 /protocol/queryProtocolAggregationLink`
2. 接入 `2.13 /card/userCards`
3. 调整权益页返回结构，支持：
   - 动态协议列表
   - 默认绑卡摘要
   - 本地展示字段与动态字段共存
4. 在权益开通成功后补 `2.15 权益订单同步`
5. 增加协议勾选 / 阅读完成约束的 service 校验
6. 为权益页接口补单测和集成测试

**完成标准**

1. `GET /api/benefits/card-detail` 能返回动态协议和卡信息
2. `POST /api/benefits/activate` 成功后能触发权益同步
3. 未满足协议条件时，后端能阻断开通

## 4. 开发任务分配建议

如果按 3 名后端同学并行推进，建议按边界分配：

### 开发 A：共享基础 + 权益页

负责范围：

1. Task Group A
2. Task Group B

原因：

1. 权益页动态化最依赖共享 facade
2. 协议列表、绑卡列表、权益同步属于同一批对外模型

### 开发 B：借款链路

负责范围：

1. Task Group C
2. Task Group E 中与借款状态映射相关的设计与预留

原因：

1. 借款链路改动最集中在 `LoanServiceImpl`
2. 状态码修正与新链路改造要由同一人控边界，避免前后矛盾

### 开发 C：还款链路 + 回调

负责范围：

1. Task Group D
2. Task Group F 中回调测试与文档回填

原因：

1. 还款短信、还款回调、绑卡列表更适合一条链收口
2. 当前回调骨架已存在，适合和还款链路一起改

## 5. 建议执行顺序

推荐顺序如下：

1. 先完成 **Task Group A**
2. 然后并行推进：
   - Task Group B
   - Task Group C
   - Task Group D
3. 云卡新接口定稿后，再推进 **Task Group E**
4. 最后统一做 **Task Group F**

不建议的顺序：

1. 直接先改 `LoanServiceImpl` 主链
2. 在没有共享 facade 的前提下继续往 service 里堆 JSON 处理
3. 在云卡“建单 / 状态查询 / 关单”接口没定前就重写借款主链
