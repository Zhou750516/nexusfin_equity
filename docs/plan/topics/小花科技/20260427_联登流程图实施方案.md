## 一、流程图解读

`docs/小花科技与艾博生联合登录方案流程图.png` 是一张 6 泳道时序图：


| 泳道           | 角色            | 关键动作                                                  |
| ------------ | ------------- | ----------------------------------------------------- |
| 用户           | 终端用户          | 点击科技平台资源位                                             |
| 客户端及前端       | 科技平台 App / H5 | 把"小花 token"拼到艾博生 H5 落地 URL 上                          |
| 三方（艾博生）前端    | 艾博生 H5        | 内部接口请求，把 token 透传给艾博生后端                               |
| 三方（艾博生）后端    | 艾博生 Java      | **token 有效性鉴权**（向小花 thor）→ 有效则继续业务接口流转、展示页面；无效则提示会话失效 |
| 小花 thor 流量平台 | 小花网关          | **鉴权缓存**：命中即返回；未命中则下钻到 oidc，再缓存结果                     |
| oidc         | 身份提供方         | 真正的鉴权下游                                               |


> **关于 "thor"**：流程图原文如此。经确认，thor 是小花科技服务端网关，承载小花的全部对外能力（鉴权、协议、用户、绑卡、影像、还款计划、短信、权益同步等），**鉴权只是其中之一**。本方案后文统一称"小花网关"，以与艾博生侧已有的 `XiaohuaGatewayService` facade 对齐；保留此处一处 "thor" 字眼仅作为流程图原文出处。

**核心语义**

1. token 在客户端 → H5 URL 流动；艾博生 H5 不解析、不验签，**只透传给艾博生后端**。
2. 艾博生后端是唯一调用小花网关鉴权的角色。
3. 鉴权缓存在 **小花网关侧**——艾博生不需要自己实现 oidc 调用或本地 token 缓存。
4. 鉴权成功后"后续内部接口流转"——这部分依赖艾博生本地登录态（JWT cookie），不再每次跟小花网关通信。

## 二·补充 · 关于三层抽象的判断

当前涉及小花调用的代码里有 3 层抽象，并非一次设计出来，是 3 个时期叠加的产物：


| 层                       | 干的事                                                                                                    | 输入 / 输出                             | 真实价值                                                              |
| ----------------------- | ------------------------------------------------------------------------------------------------------ | ----------------------------------- | ----------------------------------------------------------------- |
| `YunkaGatewayClient`    | HTTP transport，组 `requestId/timestamp/sign/data` envelope，调云卡 `proxy`                                  | 任意 path + payload → `JsonNode` 原始返回 | **必要**：通信细节、签名、超时统一                                               |
| `XiaohuaGatewayService` | 把 9 个小花网关能力做 typed 包装：`validateUserToken / queryUserCards / queryLoanRepayPlan / …`，每个方法对应 4.22 文档一个接口 | typed 4.22 DTO                      | **有价值**：4.22 协议字段集中，避免业务 service 满地 `JsonNode.path("...")`        |
| `XiaohuaAuthClient`     | 把 `UserTokenResponse` 进一步映射成业务领域对象 `JointIdentity`                                                     | `String token` → `JointIdentity`    | **冗余**：实质 5 行字段映射 + 一个 skeleton 兜底，与 `XiaohuaGatewayService` 职责重叠 |


### 历史成因

1. `YunkaGatewayClient` 最早建，做通用 proxy。
2. 4-24 对齐 4.22 文档时，发现业务 service 直接拼 `JsonNode` 越来越乱，于是补了 `XiaohuaGatewayService`。但 `LoanServiceImpl` / `RepaymentServiceImpl` 至今**仍在直连 `YunkaGatewayClient`**，未迁移。
3. 4-17 设计联合登录骨架时，又在最上面加了 `XiaohuaAuthClient`，想把"业务领域身份"和"外部协议"分开。当时 `XiaohuaGatewayService` 还没建，所以这层抽象当时是合理的；但现在两者**职责重叠**，差别只在一个返回 `UserTokenResponse`、一个返回 `JointIdentity`。

### 决定

- `YunkaGatewayClient` —— **保留**（transport 不可省）。
- `XiaohuaGatewayService` —— **保留**（9 个 4.22 接口 typed 化集中是值得的；后续 `LoanService / RepaymentService` 也应迁过来，属于 `20260427_重构降摩擦.md` Phase 2 的工作，不在本方案范围内）。
- `XiaohuaAuthClient` —— **移除**。理由：
  - 与 `XiaohuaGatewayService` 职责重叠；
  - 实质映射只有 5 行（`UserTokenResponse → JointIdentity`），可以放在 `JointLoginServiceImpl` 内部一个 private 方法；
  - `SkeletonXiaohuaAuthClient` 的"未对接"兜底意义不大——真正的"未启用"开关应在 `yunkaProperties.paths().userToken` 是否配置上判断，而不是在最业务的那层加 skeleton；
  - 删掉以后，全项目剩两层：业务 service → 小花网关 facade → 云卡 transport。

### 移除 `XiaohuaAuthClient` 后的最终结构

```
JointLoginServiceImpl
   │ 业务编排 + 内嵌 5 行 Response → Identity 映射
   ▼
XiaohuaGatewayService.validateUserToken(...)   ← 小花网关 facade（thor 在艾博生侧的对应物）
   │ proxy
   ▼
YunkaGatewayClient.proxy(paths().userToken(), ...) ← 云卡 transport
   │ HTTP
   ▼
小花网关（鉴权 / 协议 / 用户 / 卡 / 影像 / 还款计划 / 短信 / 权益同步 …）
```

## 三、实施方案

按依赖顺序拆 5 个阶段，每阶段的产出可独立 PR。

### Phase A · 切到现有 `XiaohuaGatewayService` + 移除冗余的 `XiaohuaAuthClient` 层（1–2 天，**主路径**）

**目的**：让 `JointLoginServiceImpl` 直接调用已经存在的 `XiaohuaGatewayService.validateUserToken(...)`，并删除 `XiaohuaAuthClient` 这层冗余抽象。完成后流程图的"小花网关鉴权"那一格在艾博生侧从 stub 变为可用，且全项目调用层级减少一层。

**动作**

1. **校对 `UserTokenRequest` / `UserTokenResponse`** 字段与 `docs/third-part/科技平台/科技平台-小花接口文档-艾博生4.22_整理分析.md` §2.2 一致；缺口或歧义写到该文件 §2.15 待确认区。重点校对：
  - 入参 `token` 字段名（小花文档原文）；
  - 出参 `externalUserId / phone / realName / idCard / channelCode` 五个字段是否齐全、是否需要解密；
  - 错误码字段（`code` / `msg` / `message`）。
2. **改造 `JointLoginServiceImpl`**：
  - 把构造函数依赖 `XiaohuaAuthClient xiaohuaAuthClient` 改为 `XiaohuaGatewayService xiaohuaGatewayService`；
  - `login(...)` 内部把 `xiaohuaAuthClient.exchange(token)` 改为：
    ```java
    String requestId = "XA-" + BizIds.newCompactUuid();
    UserTokenResponse response = xiaohuaGatewayService.validateUserToken(
        requestId, request.benefitOrderNo(),
        new UserTokenRequest(token, /* channelId 等 */));
    JointIdentity identity = mapToIdentity(response);
    ```
  - `mapToIdentity(...)` 是私有方法，做 5 行字段映射（仍可复用 record `JointIdentity`，挪到 `JointLoginServiceImpl` 内嵌 record，或单独留在 `service/` 包下）；
  - 错误码翻译见 Phase B。
3. **删除冗余文件**：
  - `src/main/java/com/nexusfin/equity/service/XiaohuaAuthClient.java`
  - `src/main/java/com/nexusfin/equity/service/impl/SkeletonXiaohuaAuthClient.java`
  - 顺带搜索 `grep -rn "XiaohuaAuthClient\|JointIdentity" src/` 确认所有引用已清理；
  - `JointIdentity` 若决定保留，从 `XiaohuaAuthClient` 移到 `JointLoginServiceImpl` 内嵌或 `service/` 顶层。
4. **配置项**：
  - `application.yml` 中 `nexusfin.yunka.paths.userToken` 路径需配置（参考 `4.22_整理分析.md` §2.2 路径）；
  - 不需要 `nexusfin.xiaohua.client` 这种切换开关——是否启用直接由 `paths().userToken()` 是否非空 + 集成测试覆盖来保证。

**完成判据**

- `grep -rn "XiaohuaAuthClient" src/` 返回 0 条；
- `mvn -Dtest=JointLoginServiceTest,JointLoginControllerIntegrationTest test` 全绿（用 `@MockBean XiaohuaGatewayService` 替换原来的 `@MockBean XiaohuaAuthClient`）；
- 流程图中"无缓存→鉴权→oidc→缓存"那条线我们不实现，但要在 `JointLoginServiceImpl.mapToIdentity(...)` 旁注释写明"小花网关侧已自带鉴权缓存，本侧不要再加本地缓存"，避免后续误加。

**净变化**：删 2 个文件（`XiaohuaAuthClient.java` ≈ 15 行 + `SkeletonXiaohuaAuthClient.java` ≈ 14 行），`JointLoginServiceImpl` 减少一层 mock、增加 5 行映射。整体 -25 行左右。

### Phase B · 错误语义 / 日志 / 幂等加固（1–2 天，可与 A 并行）

**目的**：把流程图右下角"是否有效→无效→提示会话失效"那条分支落到具体错误码，统一前后端语义。

**动作**

1. 新增 `exception/ErrorCodes.java` 中三个小花网关相关错误码（如已有则核对）：
  - `JOINT_LOGIN_TOKEN_INVALID`（小花网关明确返回 token 无效/过期）
  - `JOINT_LOGIN_UPSTREAM_TIMEOUT`（小花网关超时）
  - `JOINT_LOGIN_UPSTREAM_FAILED`（小花网关 5xx / 业务异常）
2. **在 `JointLoginServiceImpl` 内部**捕获 `XiaohuaGatewayService.validateUserToken(...)` 抛出的异常，把云卡返回 code 翻译到上述三个错误码，**不要让 raw 云卡 code 漏到 H5**。如发现错误码翻译逻辑被多个 service 复用，再下沉到 `XiaohuaGatewayServiceImpl` 统一处理（暂不抢跑）。
3. `JointLoginServiceImpl.login(...)` 里加 begin/success/warn 三段日志（日志字段与 `20260427_重构降摩擦.md` Phase 1 里的 `YunkaCallTemplate` 对齐：`traceId / bizOrderNo / requestId / scene / token-fingerprint`），其中 `token-fingerprint = sha256(token).substring(0,8)`，**不要打印明文 token**。
4. 加幂等：用 `IdempotencyService`（已存在）在"相同 token + scene"窗口（建议 60s）内复用上一次 `JointLoginResult`，避免短时间内 H5 重新挂载触发重复小花网关调用。

**完成判据**

- `H5/.../JointEntryPage.tsx` 里 `PageError` 文案分三类：参数缺失 / 会话失效 / 系统繁忙，分别对应 H5 i18n 的 `jointEntry.missingParams / sessionExpired / systemBusy`；
- 日志中无 raw token 出现；`grep -n "request.token()" src/main` 仅在 controller / service 入口出现一次。

### Phase C · H5 承接页对齐流程图（1 天）

**目的**：让 H5 行为与流程图右路（鉴权失败 → 会话失效提示）严格一致。

3. `App.tsx` 把 `/joint-entry` 维持为 H5 的"实际入口"，确认 `LandingPage`、`Home.tsx` 是否还需要——现在两套并存（已在重构 plan Phase 4 提及），本阶段只做最小动作：在 `App.tsx` 顶部注释清楚"`/calculator` 仅本地调试用，生产入口为 `/joint-entry`"。
4. `auth-api.ts` `jointLogin(...)` 加超时 / 重试一次的策略（小花网关偶发超时不至于直接到错误页）。

**完成判据**

- 前端单测 `joint-entry.logic.test.ts` 三类错误分支全部覆盖；
- 手工测试：

  - `?scene=push`（缺 token）→ 参数缺失页

### Phase D · 与本地 JWT 登录态的衔接（1 天）

**目的**：确保流程图中"后续内部接口流转 → 展示页面信息"那一段小花 token 不会被反复消耗。

**动作**

1. 明确小花 token 与本地 JWT 的关系（写到 `JointLoginServiceImpl` 类注释 + `AGENTS.md` 后端章节）：
  - 小花 token 一次性消费，仅用于换取 `JointIdentity`；
  - 之后所有 H5 业务接口（`/api/loan/`*、`/api/repayment/`* 等）只认本地 `NEXUSFIN_AUTH` cookie；
  - JWT 过期时 `JwtAuthenticationFilter` 返回 401 + `code = AUTH_SESSION_EXPIRED`，H5 拦截后跳回 `/joint-entry?token=...`（前提是 H5 还能拿到原始小花 token；不能则跳兜底）。
2. 在 `JwtAuthenticationFilter` 的 401 响应里加上 `redirectHint: "joint-entry"`，让 H5 全局拦截器据此决定是否回跳。
3. 在 `apiRequest`（`H5/client/src/lib/api.ts`）的 axios 拦截器里识别该字段，统一回跳。

**完成判据**

- `JwtAuthenticationFilterTest` 加一条"401 响应包含 redirectHint"用例；
- H5 端任意业务页在 cookie 过期时，自动回到联登入口（不需要每个页面单独处理）。
