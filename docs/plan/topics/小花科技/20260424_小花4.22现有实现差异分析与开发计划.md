- **艾博生通过云卡调用小花**
- **小花通过云卡回调艾博生**

- **不是艾博生直接提交到小花**
- **而是艾博生先调云卡建放款订单，权益成功后再由云卡触发小花放款**

## 2. 当前代码已具备的主线能力

结合现有代码，以下能力已经存在，不应再按“从零开始”理解：

| 能力项 | 当前状态 | 说明 |
| --- | --- | --- |
| H5 借款页 API 骨架 | 已实现 | 已有 `LoanController`，暴露试算、申请、审批状态、审批结果接口 |
| H5 权益页 API 骨架 | 已实现 | 已有 `BenefitsController`，暴露卡详情、权益开通接口 |
| H5 还款页 API 骨架 | 已实现 | 已有 `RepaymentController`，暴露还款信息、提交、结果接口 |
| 艾博生 -> 云卡基本调用 | 已实现 | `LoanServiceImpl`、`RepaymentServiceImpl` 已通过 `YunkaGatewayClient` 调 `loan/trail`、`loan/apply`、`loan/query`、`repay/*` |
| 小花旧通知类接口 client | 已实现但非主线 | `TechPlatformClient` 目前只有 `notifyCreditStatus`、`notifyLoanInfo`、`notifyRepaymentInfo` 三个 legacy notify 方法 |
| 回调入口骨架 | 已实现 | 已有 `NotificationCallbackController` 与 `NotificationServiceImpl`，支持放款/还款 forward callback 基础处理 |
| H5 页面本地展示配置 | 已实现 | 权益卡详情、借款年化、页面文案等已有本地配置和 H5 适配逻辑 |

结论：

1. 当前仓库**不是完全没有做**，H5 主入口和基础编排已经存在。
2. 但当前实现更像是“先让 H5 主链跑通”的第一版，而不是“小花 `4.22` 文档 fully aligned”的正式版。
3. 下一步重点不是再造一套新入口，而是把现有 H5 API、Yunka 调用、回调处理**按小花 `4.22` + `2026-04-23` 新链路**重新对齐。

借款链路最关键的问题有两个：

1. **状态码映射错误**
   - 当前代码把 `7003` 映射成 `approved`
   - 但小花 `4.22` 文档明确：
     - `7001 = 放款成功`
     - `7002 = 放款中`
     - `7003 = 放款失败`
2. **主链顺序已变化**
   - 当前代码更接近“借款申请直接透传”
   - 但 `2026-04-23` 会议已确认应该改成：
     - 艾博生调用云卡建放款订单
     - 用户完成权益购买 / 签约
     - 云卡再触发小花放款

因此借款链路不能只做字段补齐，还需要评估**是否要重排入口语义**。

## 4. 未实现部分清单

结合上表，当前真正未实现或需要明显补强的部分如下。

## 4.1 P0：共享基础缺口

1. Yunka 路径配置未覆盖以下能力：
   - `/protocol/queryProtocolAggregationLink`
   - `/user/token`
   - `/user/query`
   - `/loan/repayPlan`
   - `/card/smsSend`
   - `/card/smsConfirm`
   - `/card/userCards`
   - `/credit/image/query`
   - `2.15 权益订单同步` 对应路径
2. 缺少一层统一的 **小花经云卡 facade/service**
3. 缺少面向小花 `4.22` 的 typed DTO，而不是在 service 里直接拼装 JSON
4. 缺少明确的“小花经云卡回调 DTO + handler”

## 4.2 P1：权益链路缺口

1. 权益页协议列表仍是本地静态语义，未接 `2.1`
2. 权益页卡信息未接 `2.13 userCards`
3. 权益开通成功后未做 `2.15 权益订单同步`
4. 协议强制阅读 / 阅读完成状态未形成后端约束闭环

## 4.3 P1：借款链路缺口

1. `loan/apply` 字段模型不完整
2. `loan/query` 状态码映射与文档冲突
3. 缺少 `loan/repayPlan`
4. 缺少小花放款结果回调模型与处理
5. 需要结合 `2026-04-23` 会议，判断当前 `/api/loan/apply` 是保留兼容还是重构为“云卡建单”语义

## 4.4 P1：还款链路缺口

1. 缺少 `smsSend`
2. 缺少 `smsConfirm`
3. 缺少 `userCards` 对接
4. 还款结果字段和查询语义需按小花文档收口
5. 缺少小花还款回调模型与处理

## 4.5 P2：待外部确认缺口

1. `2.15 权益订单信息同步` 的接口路径未给出
2. `2.12 smsConfirm` 中 `loanId` 的传参规则不清楚
3. `imageInfo` 结构和必填范围需要确认
4. Yunka 是否会对小花回调做统一 envelope 转换，需要和云卡明确
5. `2026-04-23` 会议新增的“云卡建放款订单 / 云卡状态查询 / 放弃权益关单”接口尚未定版

## 7. 当前建议

当前最合理的推进方式不是立刻大改全部主链，而是：

1. 先做**共享基础 + 权益页动态化**
2. 再根据云卡“建放款订单 / 状态查询”接口定稿情况推进借款链路
3. 还款链路与回调链路并行收口

如果直接按旧 `004` 任务继续做，风险主要有三点：

1. 把旧状态码和旧主链继续固化进代码
2. 借款主链在“云卡先建单”方案定稿前发生返工
3. 回调 DTO 和日志主键继续沿用旧语义，后续联调难以对账
