# 2026-04-26 H5 自动埋点方案

## 1. 文档目标

本文档用于先把 H5 自动埋点方案收口清楚，再决定后续如何进入实际开发。

本轮只做方案，不直接启动 H5 全量代码改造。

目标：

1. 明确埋点覆盖页面；
2. 明确关键事件清单；
3. 明确事件命名、公共字段和业务主键；
4. 明确隐私边界与脱敏要求；
5. 明确上报失败、重试、降级处理原则。

---

## 2. 设计原则

建议本轮埋点方案遵循以下原则：

1. **先覆盖主流程关键节点，再逐步细化**；
2. **先统一事件模型，再决定接入方式**；
3. **先保证业务可追踪，再扩展营销分析维度**；
4. **默认不采集敏感明文**；
5. **埋点失败不能阻塞主业务流程**。

---

## 3. 埋点覆盖页面

建议第一批先覆盖以下页面：

1. 试算页 `CalculatorPage`
2. 审批中页 `ApprovalPendingPage`
3. 权益页 `BenefitsCardPage`
4. 审批结果页 `ApprovalResultPage`
5. 确认还款页 `ConfirmRepaymentPage`
6. 还款成功页 `RepaymentSuccessPage`
7. 拒绝落地页（待新增）
8. 签约页（待新增 / 待接入）
9. loading 页（待新增）
10. 行权分发页（待新增）

说明：

1. 已有页面要覆盖真实用户关键动作；
2. 待新增页面先在方案中预埋事件位，后续落地时不需要重新设计一轮。

## 5. 事件命名规范

建议采用：

`业务域_动作`

示例：

1. `loan_apply_click`
2. `benefit_activate_success`
3. `sign_sms_send_click`
4. `status_poll_timeout`
5. `repayment_confirm_click`

规范要求：

1. 全小写；
2. 使用下划线；
3. 动词尽量固定，如 `view / click / success / fail / start / timeout`；
4. 不直接把页面文案写入事件名。

---

## 6. 公共字段建议

所有事件建议统一带以下公参：

1. `traceId`
2. `pageCode`
3. `eventTime`
4. `appVersion`
5. `deviceType`
6. `osType`
7. `language`
8. `channel`
9. `sessionId`
10. `userId`（如可得）

说明：

1. `traceId` 用于串联本地日志；
2. `pageCode` 用于页面维度统计；
3. `sessionId` 用于串联用户一次访问路径；
4. `language` 用于多语言场景分析；
5. `channel` 用于区分来源渠道。

---

## 7. 业务主键建议

根据不同事件类型，建议按需补充：

1. `applicationId`
2. `benefitOrderNo`
3. `yunkaOrderNo`
4. `loanId`
5. `repaymentId`
6. `productCode`
7. `amount`
8. `term`
9. `displayStatus`
10. `resultCode`

原则：

1. 只带能支持后续分析和排障的最小必要字段；
2. 不为了“多而全”把所有业务字段都塞进事件。

---

## 8. 隐私边界与脱敏要求

禁止直接上报以下敏感明文：

1. 身份证号
2. 手机号
3. 银行卡号
4. 详细地址
5. 用户姓名
6. 图片原文 / OCR 原文

建议处理方式：

1. 手机号只保留后四位或不上传；
2. 银行卡只保留后四位；
3. 用户姓名不上传明文；
4. 身份证不上传明文；
5. 如确需标识用户，优先使用内部脱敏 ID / hash ID。

---

## 9. 失败重试与降级策略

## 9.1 基本原则

1. 埋点失败不能阻塞页面主流程；
2. 埋点接口超时不能影响借款、权益、签约、还款主动作；
3. 上报失败优先本地忽略或轻量重试，不做重度补偿。

## 9.2 建议策略

1. 页面曝光类事件：
   - 允许单次失败丢失
   - 不做强补偿
2. 关键按钮点击事件：
   - 可做轻量重试
   - 失败后不影响主动作继续执行
3. 关键结果事件：
   - 优先后端日志兜底
   - 前端埋点只作为辅助分析

## 9.3 超时与异常处理

建议：

1. 设短超时；
2. 不阻塞 UI；
3. 异常只记录调试日志，不弹业务错误给用户。

---

## 10. 建议接入方式

建议后续采用统一埋点封装，而不是每个页面零散调用：

1. 统一事件上报入口；
2. 统一公共字段注入；
3. 统一错误吞吐与重试策略；
4. 统一页面编码管理。

这样做的好处：

1. 页面接入成本低；
2. 多语言 / 多页面新增时不易失控；
3. 后续事件名和字段变更可集中调整。

---

## 11. 第一批最值得先埋的事件

若要控制范围，建议优先落以下事件：

1. `page_view`
2. `loan_apply_click`
3. `benefit_activate_click`
4. `sign_confirm_click`
5. `status_poll_start`
6. `loan_result_view`
7. `repayment_confirm_click`
8. `repayment_result_view`

原因：

1. 这批事件已覆盖主流程关键转折点；
2. 足以支撑基础漏斗分析和问题回看；
3. 不会一上来就把埋点做成大工程。

---

## 12. 当前建议

建议后续分两步推进：

### 第一步：方案确认

1. 确认页面清单；
2. 确认事件清单；
3. 确认公参与业务主键；
4. 确认脱敏规则。

### 第二步：最小实现

1. 先接统一埋点入口；
2. 先给关键页面和关键按钮接入；
3. 先跑通一轮基础漏斗和排障验证。

---

## 13. 一句话结论

> H5 自动埋点本轮最重要的不是“尽快把所有页面都埋上”，而是先把页面、事件、公参、脱敏和失败策略统一起来，再做最小范围接入。
