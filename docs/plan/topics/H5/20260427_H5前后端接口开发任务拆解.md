## 1. 结论先行

这份对接清单可以直接拆成 **4 组任务**，推荐按下面顺序推进：

1. **P0：借款用途 `purpose` 字段闭环**
2. **P1：权益开通短信验证闭环**
3. **P2：资方匹配状态轮询 + 协议详情页/接口**
4. **P3：mock 端点补全**

其中：

- **P0 必须先做**，这是最小真实业务缺口，前后端都存在实际缺失
- **P1 业务价值高，但后端改造面明显大于前端**，应拆成“契约/数据模型/控制器”和“真实扣款串联”两步
- **P2 不是阻塞主链上线的第一批任务**，应放在 P1 之后
- **P3 只影响本地联调体验，不应与真实主链任务抢优先级**

## 2. 文档纠偏

在分配任务前，需要先纠正这份清单里的 3 个认知偏差，避免重复开发。

### 2.1 第 5 节不是“前端未实现”，而是“mock server 未补”

仓库内以下前端 API 客户端已经存在：

- `H5/client/src/lib/auth-api.ts`
- `H5/client/src/lib/benefit-dispatch-api.ts`
- `H5/client/src/lib/refund-api.ts`

因此第 5 节应理解为：

- **真实 Java 后端接口已存在**
- **前端 API 客户端已存在**
- **缺的是 `H5/mock-server/index.ts` 里的 mock 路由**

这部分只能分配给 **H5/mock 任务**，不要误拆成 FE/BE 主链任务。

### 2.2 `purpose` 是真实缺口

当前状态已确认：

- `CalculatorPage.tsx` 有 `purposeKey` 状态
- `ApplyParams` 还没有 `purpose`
- `applyLoan()` 调用没有把用途上传
- `LoanApplyRequest` 也没有 `purpose` 字段

因此这项是标准的 **FE + BE + DB + test 联动 P0**。

### 2.3 协议不是“完全没有”，而是“缺详情接口”

当前仓库里已经有：

- `BenefitsController#card-detail`
- `BenefitsCardDetailResponse.protocols`
- 对应 service / integration test

所以协议相关的真实缺口不是“协议链接从零开始”，而是：

- H5 页面没有协议详情页
- 后端没有统一的 `GET /api/protocols/{key}` 详情接口

这项应拆成 **协议详情能力补齐**，不是重做卡详情接口。

## 3. 推荐任务分组

## Task Group A · P0 借款用途 `purpose` 字段闭环

### 目标

让试算页选中的借款用途从 H5 一直传到后端、完成校验并可被后续审批结果回显使用。

### 负责人建议

- **H5 前端**：1 人
- **Java 后端**：1 人
- **DB / 测试**：由后端同人完成

### FE 任务

1. 修改 `H5/client/src/types/loan.types.ts`
   - 为 `ApplyParams` 新增 `purpose`
   - 增加 `LoanPurpose` 类型
2. 修改 `H5/client/src/pages/CalculatorPage.tsx`
   - `handleSubmit()` 将 `purposeKey` 映射为 `shopping | rent | education | travel`
   - 调 `applyLoan()` 时带上 `purpose`
3. 如有需要，补 `H5/mock-server/index.ts`
   - `POST /api/loan/apply` 的 mock 可接收 `purpose`，便于本地调试

### BE 任务

1. 修改 `src/main/java/com/nexusfin/equity/dto/request/LoanApplyRequest.java`
   - 新增 `purpose`
   - 加枚举白名单校验
2. 修改 loan apply 业务链路
   - 将 `purpose` 写入申请实体/落库对象
3. 新增数据库迁移
   - `loan_application.purpose`
4. 修改审批状态/审批结果回显 DTO 与 service
   - 若当前响应有申请概览，补回 `purpose`

### 测试任务

1. H5 侧
   - 提交 payload 单测或页面逻辑测试
2. Java 侧
   - controller integration test：`purpose=rent` 提交成功
   - 参数校验 test：缺失 / 非法值失败
   - repository / service 断言：字段落库正确

### 验收口径

- 用户在 H5 选择的用途可进入 `/api/loan/apply`
- 非法值被拒绝
- 合法值落库成功

### 依赖关系

- **无前置依赖**
- 推荐作为第一批开发与联调闭环

## Task Group B · P1 权益开通短信验证闭环

### 目标

把审批等待页“确认订单 → 获取验证码 → 确认支付 → 权益激活”从纯前端假流程，改成真实接口闭环。

### 负责人建议

- **Java 后端主责**：1 人
- **H5 前端配合**：1 人
- **联调 / 验证**：前后端共同

### 为什么后端主责

这组任务的核心成本在后端，不在前端：

1. 新接口定义
2. 验证码发送与确认
3. 频控 / 过期 / 错误次数限制
4. 激活权益后的业务串联
5. 与现有 `BenefitsController#activate` 的关系梳理

如果先让前端开工，最终很容易被后端契约反复打回。

### BE 子任务 B1 · 契约与最小通路

1. 定义请求/响应 DTO
   - `BenefitsSmsSendRequest/Response`
   - `BenefitsSmsConfirmRequest/Response`
2. 定义 controller
   - 新增 `BenefitsSmsController` 或扩展 `BenefitsController`
3. 复用现有 `RepaymentController` 的短信模式
4. 先打通最小 happy path
   - 发送成功
   - 确认成功
   - 返回 `activationId/status/message`

### BE 子任务 B2 · 业务串联与状态控制

1. 串接权益订单激活流程
2. 增加频控、过期、错误次数限制
3. 设计 `smsToken` 的生命周期
4. 明确与既有 `POST /api/benefits/activate` 的兼容关系

### FE 子任务

1. 扩展 `H5/client/src/lib/loan-api.ts` 或新增 `benefits-api.ts`
2. 修改 `H5/client/src/pages/ApprovalPendingPage.tsx`
   - “获取验证码” 调 `sms-send`
   - 倒计时改为以后端 `expiresIn` 为准
   - “确认支付” 调 `sms-confirm`
3. 成功后关闭确认抽屉并进入“匹配中”态

### 测试任务

1. Java integration test
   - `sms-send` 成功 / 频控
   - `sms-confirm` 成功 / 错误码 / 过期
2. H5 页面逻辑测试
   - 倒计时状态
   - 成功关闭弹窗
   - 错误提示

### 验收口径

- 审批等待页不再是纯前端假验证码流程
- 验证码错误、过期、重复发送有明确返回
- 权益激活成功后前端能进入下一步状态

### 依赖关系

- 建议依赖 Task Group A 完成后再启动联调
- 后端可先行开发，前端等待契约稳定后接入

## Task Group C · P2 资方匹配状态轮询

### BE 任务

## Task Group D · P2 协议详情页/接口

### 目标

补齐 H5 协议阅读能力，让协议项点击后能进入正文详情，而不是只在前端记“已读”。

### 负责人建议

- **H5 前端**：1 人
- **Java 后端**：1 人

### FE 任务

1. 新增协议详情页
2. 从试算页协议抽屉跳转到详情页
3. 返回后保留“已读”状态

### BE 任务

1. 新增 `GET /api/protocols/{key}`
2. 定义协议详情 DTO
3. 确认协议内容来源
   - 配置
   - DB
   - 模板表

### 建议拆法

先分两步：

1. **Step D1**：接口返回静态/配置协议正文，先闭环 H5
2. **Step D2**：如确有需要，再升级为模板表 + 阅读记录

不要第一轮就同时做：

- `protocol_template`
- `user_protocol_read`
- 多版本生效逻辑

否则范围会失控。

## 9. 不建议这样拆

1. 不要把 B、C、D 三组一起作为“前端一口气全接完”的大任务
2. 不要把 E 组 mock 补全误分配给 Java 后端
3. 不要把 D 组第一轮直接升级成“协议模板表 + 阅读记录 + 多版本生效”完整平台能力
4. 不要在 A 组未闭环前，就让多人并行改审批结果、匹配状态、协议详情

## 10. 最终建议

如果只允许本周先启动一轮开发，我建议直接分配为：

1. **本周必做**
   - A 组：`purpose` 字段闭环
   - B1 组：权益短信最小闭环
   - E 组：mock 补全
2. **下轮再做**
   - B2 组：权益短信完整状态控制
   - C 组：匹配状态轮询
   - D 组：协议详情页/接口

这样拆的好处是：

- 第一轮就能交付真实业务价值
- 不会把任务面一次性铺得过宽
- 联调节奏最稳
