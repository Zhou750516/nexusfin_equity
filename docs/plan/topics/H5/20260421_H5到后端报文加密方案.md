# 2026-04-21 艾博生 H5 -> 后端报文加密方案

## 1. 文档目标

本文档用于承接 `2026-04-21` 与小花科技沟通中新增的结论：

> 艾博生 H5 页面调用艾博生后端接口时，需要纳入报文加密方案。

本次目标是：

1. 给出可评估的技术方案；
2. 明确前后端职责；
3. 给出改造范围与成本判断；
4. 暂不进入开发，只作为后续是否启动改造的决策输入。

---

## 2. 已确认事实

当前已确认以下事实：

1. 小花侧提出“艾博生 H5 -> 艾博生后端”需要做报文加密；
2. 当前项目后端已经具备部分安全基础能力：
   - `SignatureInterceptor`：支持请求头签名校验；
   - `SignatureUtil`：支持 `HMAC-SHA256` 签名；
   - `SensitiveDataCipher`：支持本地敏感字段 `AES/GCM` 加密；
3. 当前 `SignatureInterceptor` 仅保护 `/api/callbacks` 等外部入口，并未覆盖 H5 主业务接口；
4. 当前 H5 主链已经大量依赖现有 `/api/loan/*`、`/api/benefits/*`、`/api/repayment/*`、`/api/bank-card/*`；
5. 当前项目默认登录态依赖 `NEXUSFIN_AUTH` Cookie。

因此可以明确：

1. 这不是一项“纯前端改造”；
2. 也不是“仅后端增加一个拦截器”就能闭环；
3. 它会同时影响 H5 请求封装、后端解密验签、异常处理、接口兼容和联调方式。

---

## 3. 目标与边界

## 3.1 目标

本方案希望解决：

1. H5 发给后端的敏感请求是否需要做机密性保护；
2. 后端如何识别请求来源、防止篡改、防止重放；
3. 如何尽量复用现有签名与加密基础能力；
4. 如何控制改造影响面，避免一次性拖动全部页面与接口。

## 3.2 边界

本次方案只讨论：

1. H5 -> 艾博生后端请求；
2. 浏览器内 H5 与 Java 后端之间的协议封装；
3. 请求报文加密、签名、时间戳、nonce、防重放。

本次不讨论：

1. 艾博生后端 -> 小花后端的真实外部协议；
2. 浏览器静态资源传输安全；
3. Cookie 本身替换；
4. 页面级 CSP、XSS、SRI 等前端安全治理；
5. 生产密钥托管的最终平台选型。

---

## 4. 方案选项

## 4.1 方案 A：全量接口统一做“请求体加密 + 签名 + 防重放”

### 做法

对所有 H5 调后端接口统一改为以下请求结构：

```json
{
  "alg": "AES_GCM",
  "data": "Base64密文",
  "timestamp": "1710000000",
  "nonce": "随机串",
  "sign": "HMAC-SHA256签名"
}
```

后端统一：

1. 校验时间戳；
2. 校验 nonce；
3. 验签；
4. 解密 `data`；
5. 反序列化为原始业务请求。

### 优点

1. 口径最统一；
2. 安全模型最完整；
3. 后续新接口接入规则一致。

### 缺点

1. 改造面最大；
2. 所有 H5 接口调用代码都要改；
3. 所有后端 Controller / Filter / 解析链路都要适配；
4. 对现有调试、抓包、自测、联调影响最大；
5. 与当前项目现状相比，属于一次较重的协议升级。

### 成本判断

> 成本：高

---

## 4.2 方案 B：仅对敏感接口做“分级保护”，普通接口保留现状

### 做法

按接口分级：

#### L1：普通读接口

例如：

1. 试算配置；
2. 权益详情；
3. 非敏感展示型查询。

处理方式：

1. 保持现有 Cookie + HTTPS；
2. 可选增加签名；
3. 不强制请求体加密。

#### L2：敏感写接口 / 核心身份接口

例如：

1. 联合登录入口；
2. 借款申请；
3. 还款提交；
4. 绑卡签约申请 / 确认；
5. 涉及实名、银行卡、身份证、验证码的接口。

处理方式：

1. 强制加签；
2. 强制时间戳 + nonce；
3. 对请求体 `data` 做 AES 加密；
4. 后端做验签、解密、防重放。

### 优点

1. 满足“敏感链路要加密”的核心要求；
2. 不需要把全部 H5 页面一次性拖入大改；
3. 更容易先做 POC 和成本评估；
4. 对现有借款、还款、绑卡、联合登录等主链可以逐步接入；
5. 与当前项目的安全能力更容易衔接。

### 缺点

1. 需要先定义“哪些接口属于敏感接口”；
2. 前后端都要支持“明文模式 + 加密模式”双栈；
3. 文档和联调口径要更清晰，否则容易混乱。

### 成本判断

> 成本：中

---

## 4.3 方案 C：先只做签名 / 时间戳 / nonce，不做请求体加密

### 做法

H5 每次请求统一带：

1. `X-App-Id`
2. `X-Timestamp`
3. `X-Nonce`
4. `X-Signature`

请求体仍保持当前 JSON 明文。

后端：

1. 校验签名；
2. 校验时效；
3. 校验 nonce；
4. 不解密业务 body。

### 优点

1. 改造成本最低；
2. 与现有 `SignatureInterceptor` 最接近；
3. 适合先行低成本试点。

### 缺点

1. 只能解决防篡改、防重放；
2. 不满足“报文机密性”要求；
3. 若会议口径中的“加密”是硬性要求，则该方案不足以单独作为最终方案。

### 成本判断

> 成本：低

---

## 5. 推荐结论

当前更推荐：

> **方案 B：仅对敏感接口做分级保护。**

推荐原因：

1. 它同时满足“需要报文加密”的方向性要求；
2. 又不会把当前所有 H5 接口一次性拖进高风险重构；
3. 最适合先做技术方案、再做改造成本评估；
4. 与现有项目能力最匹配：
   - 已有签名工具；
   - 已有敏感数据加密工具；
   - 已有拦截器扩展点；
   - 已有明确的敏感业务接口集合。

---

## 6. 推荐技术设计

## 6.1 推荐协议模型

建议对敏感接口使用统一 envelope：

```json
{
  "data": "Base64(AES-GCM(ciphertext))",
  "timestamp": "1710000000",
  "nonce": "6f7b9c2d0a4e",
  "sign": "hex(HMAC-SHA256(appId|timestamp|nonce|data))"
}
```

配套请求头：

| Header | 说明 |
|---|---|
| `X-App-Id` | H5 应用标识 |
| `X-Timestamp` | 秒级时间戳 |
| `X-Nonce` | 随机串 |
| `X-Signature` | 签名结果 |
| `X-Encrypt-Mode` | 可选，标记 `PLAIN` / `AES_GCM` |

说明：

1. `data` 为业务 JSON 的密文；
2. 签名建议把 `data` 纳入签名范围；
3. 与当前 `SignatureInterceptor` 相比，需要从“只签头”升级为“头 + body 密文”参与签名。

---

## 6.2 前端职责

H5 侧建议承担：

1. 识别目标接口是否属于敏感接口；
2. 敏感接口将业务 JSON 序列化后加密为 `data`；
3. 生成 `timestamp`；
4. 生成 `nonce`；
5. 对 `appId|timestamp|nonce|data` 做签名；
6. 发请求时统一透传 Cookie。

H5 不建议承担：

1. 保存长期主密钥；
2. 动态生成服务端可验证的复杂密钥派生逻辑；
3. 与小花后端直接做任何加解密交互。

---

## 6.3 后端职责

后端建议新增一层“入站安全协议适配”：

1. 判断请求是否属于敏感接口；
2. 对敏感接口执行：
   - 验签
   - 时间戳校验
   - nonce 去重
   - 解密 `data`
   - 反序列化业务请求
3. 对非敏感接口仍沿用现状；
4. 保持 Controller 层不承担协议解密逻辑。

推荐新增的位置：

1. Filter / Interceptor + RequestWrapper
2. 或新增统一的 `InboundSecurePayloadResolver`

不建议：

1. 在每个 Controller 手写解密；
2. 在每个 DTO 单独兼容密文格式。

---

## 6.4 密钥管理建议

当前仅做方案建议，不进入最终落地。

推荐原则：

1. H5 与后端之间不要使用硬编码长期明文密钥；
2. 若后续必须实现前后端对称加密，建议采用“短期会话密钥”或“按环境配置的应用级密钥 + 轮换机制”；
3. 测试环境可使用静态密钥；
4. 生产环境必须支持轮换和失效。

当前建议先按两层评估：

### 低成本评估口径

1. 测试环境静态 `appId + secret + aesKey`
2. 后续再评估生产密钥轮换

### 生产化口径

1. 会话级或版本化密钥
2. 配套 `keyId`
3. 灰度轮换

---

## 6.5 防重放建议

建议后端对敏感接口增加：

1. `timestamp` 时效控制，例如 300 秒；
2. `nonce` 去重存储，TTL 与时效一致；
3. 对重复 `nonce + appId` 请求直接拒绝；
4. 对时钟偏差过大的请求拒绝。

可复用基础：

1. 现有 `SignatureProperties.maxSkewSeconds`

后续若进入开发，推荐：

1. 本地缓存 / Redis 存 nonce；
2. 敏感写接口默认开启防重放；
3. 读接口可按需要降级。

---

## 7. 推荐敏感接口范围

若采用方案 B，建议第一批纳入以下接口：

### P0：必须纳入

1. `/api/auth/joint-login`
2. `/api/loan/apply`
3. `/api/repayment/submit`
4. `/api/bank-card/sign-apply`
5. `/api/bank-card/sign-confirm`

### P1：建议纳入

1. `/api/loan/calculate`
2. `/api/loan/approval-status/*`
3. `/api/loan/approval-result/*`
4. `/api/repayment/info/*`
5. `/api/repayment/result/*`

### P2：可暂不纳入

1. `/api/loan/calculator-config`
2. `/api/benefits/card-detail`
3. 纯展示类、无敏感入参、无实名影响的接口

---

## 8. 对现有项目的改造影响

## 8.1 前端改造点

1. 新增统一请求封装器；
2. 新增敏感接口白名单或分级配置；
3. 在 `axios` / `fetch` 层支持加密 envelope；
4. 调试与日志输出需要增加“明文 / 密文模式”区分；
5. 本地联调工具与 stub 也可能需要适配。

## 8.2 后端改造点

1. `SignatureInterceptor` 需要升级；
2. 需要新增 body 参与签名校验；
3. 需要新增敏感接口解密层；
4. 需要新增 nonce 去重机制；
5. 需要兼容加密与非加密接口共存；
6. 需要新增错误码与异常口径。

## 8.3 测试改造点

1. Controller 集成测试要支持加密请求构造；
2. H5 本地联调脚本要支持带签名和密文；
3. 人工测试文档要补“如何生成签名 / 时间戳 / nonce”；
4. 现有 curl / stub 调试方式要更新。

---

## 9. 改造成本评估

## 9.1 粗粒度成本判断

| 方案 | 成本 | 风险 | 推荐度 |
|---|---|---|---|
| 方案 A：全量接口统一加密 | 高 | 高 | 中 |
| 方案 B：敏感接口分级保护 | 中 | 中 | 高 |
| 方案 C：只签名不加密 | 低 | 中 | 中 |

## 9.2 主要成本来源

真正的成本不在算法本身，而在：

1. 前后端双端协议兼容；
2. 联调、测试、排错复杂度上升；
3. 本地调试工具全部要适配；
4. 生产密钥管理若要做规范化，会显著增加工程复杂度。

## 9.3 当前建议

当前建议先做两步：

### Step 1：保留方案，不开发

先把本方案作为待决策文档，不进入开发。

### Step 2：若后续决定启动评估

优先评估以下问题：

1. 小花对“加密”的强制程度：
   - 是否必须全量接口
   - 是否仅限敏感接口
2. 是否必须做请求体机密性保护，还是签名即可
3. H5 是否允许持有应用级密钥
4. 后端是否接受双栈兼容期
5. 测试与联调成本是否可接受

---

## 10. 当前不建议立即开发的原因

当前不建议直接进入开发，原因如下：

1. 小花只给出“需要加密”的方向性要求，未给出正式协议；
2. 加密范围是否全量、是否只针对敏感接口，尚未最终定死；
3. 若现在直接开发，后续很可能因正式口径变化返工；
4. 该改造会同时影响 H5、后端、联调工具、测试文档，不是轻量小改。

因此更合理的顺序应是：

1. 先形成方案；
2. 再评估改造成本；
3. 再决定是否开发以及从哪个范围开始。

---

## 11. 建议下一步

如果后续要继续推进，建议按以下顺序：

1. 先让小花明确“加密范围”和“最低协议要求”；
2. 再做一版“改造成本评估清单”；
3. 若决定启动，优先选取：
   - `/api/auth/joint-login`
   - `/api/bank-card/sign-apply`
   - `/api/bank-card/sign-confirm`
   做最小 POC；
4. POC 验证通过后，再决定是否扩到借款 / 还款主链。

---

## 12. 一句话结论

> 当前更推荐把“艾博生 H5 -> 后端报文加密”按**敏感接口分级保护**来设计，先形成方案、后评估成本、再决定是否进入开发，而不是立即对全部 H5 接口做一次性重构。
