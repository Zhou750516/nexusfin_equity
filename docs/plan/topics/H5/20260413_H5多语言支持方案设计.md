# 2026-04-13 H5 多语言支持方案设计

## 1. 文档目的

本文档用于沉淀 `H5` 多语言支持的已确认方案，作为后续前后端实现、协议内容治理和联调的统一依据。

当前确认范围包括：

1. `H5` 页面静态文案
2. 后端返回的提示语、协议标题、富文本内容
3. 协议类长文案与页面内展示的长文本内容

本文档当前聚焦 **方案设计与实施边界**，不直接展开代码实现任务。

## 2. 已确认的产品决策

### 2.1 支持语言

本期支持以下 4 种语言：

1. `zh-CN`：简体中文
2. `zh-TW`：繁体中文
3. `en-US`：英文
4. `vi-VN`：越南文

### 2.2 语言选择规则

进入页面时，默认按浏览器语言自动匹配；同时允许用户手动切换，并持久化用户选择。

最终优先级按以下顺序执行：

1. 用户手动选择的语言
2. 预留的外部显式语言参数（如未来接入 URL 参数、宿主 App 参数）
3. 浏览器语言自动识别结果
4. 默认回退 `zh-CN`

### 2.3 覆盖范围

本次多语言不是只做前端页面标题与按钮，而是完整覆盖以下内容：

1. `H5` 页面静态文案
2. 后端接口返回的业务提示语
3. 协议标题
4. 富文本 / 长文案内容
5. 页面内展示的权益说明、还款提示、审批提示等业务描述

## 3. 方案结论

本次采用 **方案 C：统一文案资源中心**。

一句话定义：

> 前端、后端、协议长文案均围绕统一的语言编码和统一的文案资源管理方式组织，实现“自动识别 + 手动切换 + 一处治理、多处消费”。

选择该方案的原因：

1. 能同时覆盖 `H5 + 后端 + 协议/富文本`
2. 结构清晰，便于先文件化治理、后续再升级到数据库配置化
3. 可避免前端一套、后端一套、协议再单独散落维护
4. 对当前项目推进最稳妥，不会引入过重的内容平台建设成本

## 4. 语言识别与回退规则

### 4.1 浏览器语言映射

建议按以下规则映射浏览器语言：

| 浏览器语言 | 归一化语言 |
| --- | --- |
| `zh` / `zh-CN` / `zh-Hans` | `zh-CN` |
| `zh-TW` / `zh-HK` / `zh-Hant` | `zh-TW` |
| `en` / `en-US` / `en-GB` | `en-US` |
| `vi` / `vi-VN` | `vi-VN` |
| 其他全部值 | `zh-CN` |

### 4.2 缺失文案回退规则

1. 当前语言存在文案时，直接返回当前语言内容
2. 当前语言缺失时，回退到 `zh-CN`
3. 若 `zh-CN` 仍缺失，则返回安全默认值，并记录日志，便于补齐资源

### 4.3 持久化规则

用户一旦在页面上手动切换语言，需持久化本次选择；下次进入页面优先使用该语言，而不是再次用浏览器语言覆盖。

## 5. 总体架构设计

### 5.1 前端侧

前端负责：

1. 浏览器语言识别
2. 用户语言切换
3. 当前语言持久化
4. 静态文案的本地渲染
5. 请求后端时透传当前语言

建议前端设计：

1. 新增统一 `i18n` Provider
2. 提供统一 `t(key)` 文案访问方式
3. 为 `H5` 页面、弹窗、错误提示、按钮文案统一接入
4. 在请求头中透传当前语言，例如使用 `Accept-Language`

建议前端资源组织方式：

1. 短文案、按钮、标签、提示语：采用本地资源文件
2. 页面常量文案：统一改为按 key 读取
3. 不在页面组件内继续硬编码长文案和多语言字符串

### 5.2 后端侧

后端负责：

1. 解析当前请求语言
2. 返回语言相关的提示语和状态描述
3. 返回协议标题和长文案资源
4. 统一处理找不到文案时的回退

建议后端设计：

1. 增加统一的语言解析机制
2. 优先从请求头读取 `Accept-Language`
3. 将解析结果归一化为 `zh-CN`、`zh-TW`、`en-US`、`vi-VN`
4. 在 Service 层按当前语言返回展示文案
5. 业务状态码仍保持稳定，变化的仅是展示文案

说明：

1. 数值、状态码、业务枚举尽量保持语言无关
2. 仅将面向用户显示的标题、说明、提示、协议内容做多语言化

### 5.3 协议与富文本内容

协议与富文本不建议一开始就散落在前端代码中维护，而应由后端统一管理并下发。

本期建议：

1. 协议标题走后端多语言资源
2. 长文案 / 富文本内容以资源文件方式管理
3. 后端按语言读取对应内容并返回给前端

这样做的好处：

1. 协议内容变更时，不必依赖前端单独改包
2. 后续如需法务修订、运营替换文案，更便于集中治理
3. 可与未来数据库化的内容中心平滑衔接

## 6. 文案资源分层设计

### 6.1 短文案资源

适用范围：

1. 按钮
2. 标签
3. 页面标题
4. 空状态提示
5. Toast / Dialog 提示语
6. 业务状态描述

建议存放方式：

1. 前端本地资源文件承载纯页面静态文案
2. 后端资源文件承载接口返回提示语和服务端生成文案

### 6.2 长文案 / 协议资源

适用范围：

1. 协议标题
2. 协议正文
3. 权益说明
4. 风险提示
5. 还款说明
6. 审批过程长提示文案

建议存放方式：

1. 首期采用后端资源文件统一承载
2. 每种语言一份独立内容资源
3. 前端仅负责展示，不负责维护正文

### 6.3 后续可演进方向

若后续出现以下需求，再考虑升级为数据库内容中心：

1. 频繁改协议
2. 运营侧需要后台配置
3. 同一语言需要多版本切换
4. 需要灰度、版本化或生效时间控制

当前阶段不建议一步到位做数据库 CMS，以免超出本期实现边界。

## 7. 接口口径建议

### 7.1 请求侧

`H5` 请求后端时，建议统一携带语言信息：

1. 优先使用 `Accept-Language`
2. 后续如宿主系统要求，也可扩展 `X-Locale`

### 7.2 响应侧

接口响应建议遵循以下原则：

1. 业务字段、状态码、金额字段保持稳定，不因语言改变结构
2. 展示文案、提示语、协议标题、长文案内容按语言返回
3. 若接口返回的是“页面展示模型”，其中描述类字段应已是当前语言

### 7.3 协议内容返回方式

本期建议优先采用：

1. 后端直接返回协议标题与正文内容
2. `H5` 直接展示

暂不优先采用：

1. 前端自行拼接协议正文
2. 让前端自己决定不同语言的协议资源路径

## 8. 实施边界

### 8.1 本期必须完成的设计边界

1. 统一语言编码
2. 明确自动识别与手动切换规则
3. 明确前后端职责划分
4. 明确协议 / 富文本内容由后端统一治理
5. 明确资源缺失回退规则

### 8.2 本期暂不展开的能力

1. 数据库化文案管理平台
2. 多语言运营后台
3. 文案灰度发布
4. 按国家 / 地区 / 渠道做更细颗粒度内容分流

## 9. 对现有 Phase 9 的影响

本方案不会改变 `Phase 9` 接口清单本身，但会对后续实现提出新增要求：

1. `H5` 页面硬编码文案需要统一抽离
2. 后端接口中返回给页面的描述类字段需要支持语言化
3. 协议相关接口 / 内容返回需要纳入多语言资源治理
4. 联调时需覆盖 4 种语言下的页面与接口表现

## 10. 推荐落地顺序

建议按以下顺序实施：

1. 先完成多语言方案与资源组织设计
2. 再完成前端静态文案抽离与语言切换能力
3. 再补后端消息与协议内容多语言化
4. 最后完成四种语言联调与回归

## 11. 本次确认结论

截至 `2026-04-13`，以下事项已确认：

1. 支持语言为 `zh-CN`、`zh-TW`、`en-US`、`vi-VN`
2. 首次进入按浏览器语言自动匹配
3. 允许用户手动切换语言
4. 用户手动切换后需要持久化
5. 覆盖范围包括 `H5 + 后端 + 协议/富文本内容`
6. 采用 **方案 C：统一文案资源中心**
7. 协议标题与长文案优先由后端统一治理

## 12. 一句话结论

> `H5` 多语言支持方案已经完成设计：本期采用“统一文案资源中心”，覆盖前端页面、后端提示和协议富文本，按“浏览器自动识别 + 用户手动切换 + 默认回退简中”的统一规则推进。
