## 1. P0 — 借款用途字段未上传(必修)

### 现状
试算页 `H5/client/src/pages/CalculatorPage.tsx` 维护 `purposeKey` 状态(4 选 1:`shopping / rent / education / travel`),但点击 `确认借款` 调用 `applyLoan()` 时**未将该字段提交后端**,用户的选择丢失。

### FE 需要的契约变更

`H5/client/src/types/loan.types.ts`

```ts
export type LoanPurpose = "shopping" | "rent" | "education" | "travel";

export interface ApplyParams {
  amount: number;
  term: number;
  receivingAccountId: string;
  agreedProtocols: string[];
  purpose: LoanPurpose;            // 新增,必填
}
```

`H5/client/src/pages/CalculatorPage.tsx#handleSubmit` 内将 `purposeKey` 末段(去掉 `calculator.loanPurpose.` 前缀)作为 `purpose` 传入。FE 这一段我们已经准备好,只需 BE 侧接住。

### BE 任务(Codex)

1. **DTO**:在 `LoanApplyRequest`(或同类入参 DTO)新增 `purpose` 字段,类型 `String`,枚举校验:`shopping | rent | education | travel`,`@NotBlank`。
2. **持久化**:`loan_application` 表新增 `purpose` 字段(`VARCHAR(32) NOT NULL DEFAULT 'shopping'`),迁移脚本同步;MyBatis-Plus 实体补字段。
3. **校验**:不在白名单的取值返回 `code: 4001`(参数错误)。
4. **回显**:`approval-status` / `approval-result` 响应若包含申请概览,需要把 `purpose` 透出(枚举原值,前端再 i18n)。

### 验收标准
- `POST /api/loan/apply` 用包含 `purpose: "rent"` 的 JSON 提交,落库正确。
- 不传 `purpose` 返回 4001。
- 既有调用未带 `purpose` 的旧客户端不破坏(可在迁移期通过 default 兜底,但 FE 必传是目标)。

## 2. P1 — 权益开通短信验证闭环(必修)

### 现状
审批等待页点击 `开通服务，极速借款` → 弹出"确认订单"抽屉(`ApprovalPendingPage.tsx`),内含:
- `获取验证码` 按钮(60s 倒计时,**当前纯前端 setTimeout,未调任何接口**)
- 验证码输入框(限 6 位数字)
- `确认支付` 按钮(**当前直接 `setBenefitDismissed(true)` 关掉,未调接口**)
- 之后展示"正在为您匹配"30s 弹窗

业务上这一步是「权益服务费扣款」,必须有真实短信验证。后端 `/api/repayment/sms-*` 是还款短信流程,**需要为权益开通新增同样的两个接口**(或扩展现有)。

### 建议契约

#### 2.1 `POST /api/benefits/sms-send` — 发送权益开通短信验证码

**请求**

```json
{
  "applicationId": "app-001",
  "cardType": "huixuan_card",
  "amount": 99,
  "bankCardId": "acc-001"
}
```

**响应**

```json
{
  "code": 0,
  "data": {
    "smsToken": "tok_xxx",
    "expiresIn": 60,
    "maskedMobile": "138****1234"
  }
}
```

错误码:`6001` 频控(60s 内重发);`6002` 银行卡不可用;`6003` 申请单状态非法。

#### 2.2 `POST /api/benefits/sms-confirm` — 确认验证码并完成扣款 + 激活权益

**请求**

```json
{
  "smsToken": "tok_xxx",
  "verificationCode": "123456"
}
```

**响应**(权益激活成功)

```json
{
  "code": 0,
  "data": {
    "activationId": "act-001",
    "status": "activated",
    "benefitOrderNo": "BO20260427001",
    "matchingTaskId": "match-xyz",
    "estimatedSeconds": 30,
    "message": "权益服务已开通"
  }
}
```

错误码:`6101` 验证码错误;`6102` 验证码过期;`6103` 扣款失败;`6104` 已激活。

#### 2.3 `GET /api/benefits/matching-status/{matchingTaskId}` — 资方匹配进度(可选,见 §3)

### FE 任务(我们这边)

- 新增 `benefits-api.ts` 或扩展 `loan-api.ts`,加 `sendBenefitsSms` / `confirmBenefitsSms`。
- `获取验证码` 点击 → 调 `sms-send`,成功后启 60s 倒计时(用 `expiresIn`)。
- `确认支付` 点击 → 调 `sms-confirm`,成功后:
  - 关确认订单抽屉,清验证码状态
  - 用响应的 `estimatedSeconds` 启动匹配弹窗倒计时(代替当前写死的 30s)
  - 倒计时结束:或停留并轮询 `matching-status`(P2),或直接 `setBenefitDismissed(true)`

### BE 任务(Codex)

1. 新增 `BenefitsSmsController` 或在 `BenefitsController` 加上述 2 个接口。
2. 复用项目已有 SMS 通道(参考 `RepaymentController` 的实现)。
3. **核心业务串联:验证通过 → 触发扣款 → 成功后异步触发资方匹配 → 写入 `benefit_order` 表 → 同步推送 YunKa(参考现有放款回调 `PaymentCallbackController`)**。
4. 验证码 5 分钟过期、错误次数限制 5 次/单。
5. `BenefitsCardPage` 已有的 `POST /api/benefits/activate` 是「无 SMS 直开」路径,保留向后兼容,内部可走相同的扣款逻辑但不要求验证码(走免密)。

### 验收标准
- 调 `sms-send`,数据库 `sms_verification` 表写入记录;短信触达手机号。
- 60s 内再次调 `sms-send` 同 token 返回 6001。
- 错误码连错 5 次后 token 作废。
- 验证通过后 `benefit_order` 状态 `ACTIVATED`,`benefitOrderNo` 返回前端。

## 3. P2 — 资方匹配状态轮询(锦上添花)

### 现状
"正在为您匹配"弹窗只是前端 30s 静默倒计时,无后端联动。真实场景里资方匹配是异步任务,可能 10s 完成也可能 60s 都没有结果。

### 建议契约

`GET /api/benefits/matching-status/{matchingTaskId}`

**响应**

```json
{
  "code": 0,
  "data": {
    "status": "matching" | "matched" | "failed" | "timeout",
    "matchedSupplier": "蒙商消费金融",
    "remainingSeconds": 18,
    "message": "正在匹配最优资方"
  }
}
```

### FE 任务
匹配弹窗期间每 3s 轮询一次,`status === "matched"` 立即关弹窗;`failed` / `timeout` 提示并允许用户继续等待。

### BE 任务
匹配任务表 `benefit_matching_task` 设计;后台调度器 + DB 状态机;接口只读不改状态。

### 验收
轮询期间 status 流转 `matching → matched`;前端正确感知。

## 4. P2 — 协议详情页/接口

### 现状
试算页"借款及相关协议"抽屉中的 5 项协议(`借款合同 / 个人信息授权书 / 用户服务协议 / 隐私政策 / 委托扣款协议`)点击后只在前端标记"已读",**没有跳转协议正文**。

### 建议契约

`GET /api/protocols/{key}?applicationId=xxx`

`key` 枚举:`loan_contract | privacy_auth | user_service | privacy | payment`

**响应**

```json
{
  "code": 0,
  "data": {
    "key": "loan_contract",
    "title": "《借款合同》",
    "version": "v2026.04",
    "format": "html",
    "content": "<html>...</html>",
    "renderedAt": "2026-04-27T10:00:00+08:00"
  }
}
```

### FE 任务
新增协议详情子页(`/protocol-detail?key=...`),点击协议项跳过去全屏渲染,返回时仍保留"已读"计数。

### BE 任务
- `protocol_template` 表(key, version, format, content, effective_at);
- 接口按 `applicationId` 上下文返回当前生效版本;
- 用户阅读记录可记入 `user_protocol_read`。

## 6. 字段层小修(顺手做掉)

### 6.1 `BenefitsCardDetail` 增加 `protocols` 必填

FE 当前默认 `protocols: []`,但 §4 上线后建议在 `card-detail` 同时返回:

```json
"protocols": [
  { "key": "user_service", "name": "《用户服务协议》", "url": "/protocol-detail?key=user_service" }
]
```

### 6.2 `ApprovalResult.steps` 命名对齐

后端目前可能返回 `智能审批 / 放款准备` 等多种命名;前端设计稿固定为:

- `提交申请` / `申请已提交成功`
- `审批完成` / `资质审核已通过`
- `准备放款` / `资金将在30分钟内到账`

后端按这个口径返回,前端不再做映射。

### 6.3 `RepaymentResult.bankCard.lastFour` 与 `bankName` 必返

当前 mock 给的 `招商银行` + `8648`,FE 用 `formatBankCard(bankName, lastFour)` 拼 "招商银行 8648",无 `accountId` 在该响应。BE 保持现状即可,确认非空。
