# 2026-04-12 H5 Phase 9 实施决策分析

## 1. 文档目的

本文档用于把已经明确的 `Phase 9` 实施决策单独整理出来，便于你确认：

1. 是否接受这套实现路径。
2. 是否按这套路径进入开发。

本文档不直接替代 `H5/PLAN.md`，而是作为“执行前确认稿”。

## 2. 背景结论

当前已明确的前提是：

1. H5 页面已完成。
2. `Phase 9` 的重点不再是重新定义全部接口，而是按照既定决策把 10 个 H5 后端接口落地。
3. 后端实现分为三类：
  - 本地配置型
  - 齐为接口适配型
  - 云卡 gateway 转发型
4. `loan/apply` 是唯一一个需要串联 **齐为购买权益 + 云卡借款申请** 的编排接口。

## 4. 推荐实施拆分

### 4.1 第一组：最简单、最先打通

建议先做以下 3 个接口：

1. `GET /api/loan/calculator-config`
2. `GET /api/benefits/card-detail`
3. `POST /api/benefits/activate`

原因：

1. 这 3 个接口不依赖完整借款链路。
2. 其中 `calculator-config` 仅依赖本地配置，交付风险最低。
3. `benefits/`* 两个接口只需要适配齐为，不会一开始就把问题扩散到云卡 / 科技平台链路。

### 4.2 第二组：云卡转发但不含复杂编排

建议第二批做以下 5 个接口：

1. `POST /api/loan/calculate`
2. `GET /api/loan/approval-status/{applicationId}`
3. `GET /api/loan/approval-result/{applicationId}`
4. `GET /api/repayment/info/{loanId}`
5. `POST /api/repayment/submit`

原因：

1. 这些接口都属于“单次转发 + 结果映射”模式。
2. 可以优先把云卡 gateway 客户端、统一 `path + data` 封装、响应映射机制沉淀出来。
3. 一旦这一层稳定，后续新增同类接口成本会明显降低。

### 4.3 第三组：需要编排或跳转参数承接的接口

建议最后做以下 2 个接口：

1. `POST /api/loan/apply`
2. `GET /api/repayment/result/{repaymentId}`

原因：

1. `loan/apply` 需要先齐为、后云卡，涉及顺序编排和失败回滚 / 补偿判断。
2. `repayment/result/{repaymentId}` 虽不需要艾博生本地生成 `repaymentId`，但依赖科技平台跳转参数承接和还款链路整体联调结果，建议放在还款主链稳定后再收口。

## 5. 推荐技术实现口径

### 5.1 本地配置接口

`GET /api/loan/calculator-config`

建议：

1. 用 `@ConfigurationProperties` 或固定配置类承载金额范围、默认金额、期数选项、收款账户。
2. 当前直接固定 3 期，不在本阶段扩展多期动态能力。
3. 保持返回结构和 `H5/API_SCHEMA.md` 一致。

### 5.2 齐为接口适配

`GET /api/benefits/card-detail`、`POST /api/benefits/activate`

建议：

1. 新增一个对 H5 暴露的 `BenefitsController`。
2. 内部复用或封装齐为客户端能力。
3. 对 H5 返回结构做单独映射，不把齐为原始字段直接暴露给前端。

### 5.3 云卡 gateway 转发

涉及接口：

- `POST /api/loan/calculate`
- `GET /api/loan/approval-status/{applicationId}`
- `GET /api/loan/approval-result/{applicationId}`
- `GET /api/repayment/info/{loanId}`
- `POST /api/repayment/submit`
- `GET /api/repayment/result/{repaymentId}`

建议：

1. 统一封装云卡请求客户端。
2. 所有请求统一构造成：
  - gateway URI
  - `requestId`
  - `path`
  - `bizOrderNo`
  - `data`
3. 所有云卡响应统一先解包，再映射为 H5 需要的 DTO。
4. `approval-status` 与 `approval-result` 虽共用 `/loan/query`，但对外返回 DTO 应分开定义。

### 5.4 复合编排接口

`POST /api/loan/apply`

建议：

1. 输入为 H5 的借款申请参数。
2. 第一步调用齐为购买权益。
3. 第二步仅在权益成功后调用云卡 `/loan/apply`。
4. 返回给 H5 的 `applicationId` 应明确来源：
  - 优先使用上游已有稳定标识
  - 若无稳定标识，则需本地生成并维护映射

## 6. 执行前确认项与确认结论

以下事项已完成确认，可直接作为执行口径：

### 6.1 借款试算路径

- **确认结论**：使用配置项，默认值为 `/loan/trail`
- **执行口径**：
  1. 不在代码中写死 `/loan/trail`
  2. 通过配置项承载上游路径
  3. 后续若科技平台确认应改为 `/loan/trial`，仅改配置，不改业务代码

### 6.2 `approval-status` 和 `approval-result` 是否都用 `/loan/query`

- **确认结论**：是，两个接口都走 `/loan/query`
- **执行口径**：
  1. `GET /api/loan/approval-status/{applicationId}` → 云卡 gateway → 上游 `/loan/query`
  2. `GET /api/loan/approval-result/{applicationId}` → 云卡 gateway → 上游 `/loan/query`
  3. 两个 H5 接口共用同一个上游查询接口，但输出 DTO 分开映射

### 6.3 `applicationId` 来源

- **确认结论**：允许艾博生本地生成并维护映射
- **执行口径**：
  1. `applicationId` 由艾博生生成并返回给 H5
  2. 本地需要保存 `applicationId` 与齐为权益订单、云卡 / 科技平台查询参数之间的映射关系
  3. 后续 `approval-status`、`approval-result` 根据该映射恢复上游查询参数

### 6.4 `repaymentId` 来源

- **确认结论**：`repaymentId` 由科技平台跳转参数携带，不由艾博生本地生成
- **执行口径**：
  1. 艾博生不为 `repaymentId` 单独生成本地编号
  2. `GET /api/repayment/result/{repaymentId}` 直接以该参数作为查询入口
  3. 如仍需补充 `loanId`、`swiftNumber` 等上游查询条件，则通过跳转参数或既有上下文恢复，不再依赖“本地生成 repaymentId”

### 6.5 `loan/apply` 的部分成功处理策略

当前主路径明确为：

1. 先齐为购买权益
2. 权益成功后再调用云卡借款申请

- **确认结论**：需要返回“放款失败”，但要明确“权益购买成功”
- **执行口径**：
  1. 当齐为成功、云卡失败时，对 H5 返回失败结果
  2. 失败结果中必须清楚表明“权益购买成功、借款 / 放款失败”
  3. 当前阶段不做自动补偿 / 自动撤销，后续按人工处理或专门补偿流程承接
  4. 本地需保留足够的状态字段，区分：
     - 权益购买成功
     - 借款申请失败 / 放款失败

## 7. 建议你确认后的执行入口

既然上述事项已确认，建议按以下顺序推进：

1. 先确认路径与标识问题：
  - 将 `/loan/trail` 抽成配置项
  - 落 `applicationId` 本地映射设计
  - 确认 `repaymentId` 走跳转参数链路
2. 再先做第一组简单接口：
  - `calculator-config`
  - `benefits/card-detail`
  - `benefits/activate`
3. 再做云卡转发基础设施和第二组接口
4. 最后做 `loan/apply` 编排和 `repayment/result` 映射

## 8. 已确认结论摘要

本次已经确认：

1. 10 个接口的实现方式划分可接受。
2. `/loan/trail` 使用配置项承载，默认值先按 `/loan/trail`。
3. `approval-status` / `approval-result` 都使用 `/loan/query`。
4. `applicationId` 由艾博生本地生成并维护映射。
5. `repaymentId` 使用科技平台跳转携带参数，不由艾博生本地生成。
6. `loan/apply` 若出现“齐为成功、云卡失败”，需返回借款失败，但明确提示权益购买成功。
