## 1. 本次沟通结论

### 1.1 可行权权益提醒消息

1. 可行权权益提醒消息的 **push 消息来源**：
   - 放款成功后，由 **科技平台侧自行触发**。
2. push 消息支持的跳转链路：
   - 从科技平台 App 跳转到 **艾博生的分发页面**；
   - 再由艾博生分发页面继续跳转到 **齐为供应商页面**。

**当前结论**

- 放款成功后的权益提醒 push，不由艾博生侧主动发起；
- 艾博生侧需要提供可承接科技平台跳转的分发页面，并负责继续路由到齐为供应商页面。

### 1.2 权益行权展示

1. 权益的行权展示入口：
   - 从 **权益订单页** 直接跳转到 **艾博生页面**。

**当前结论**

- 权益行权链路的页面入口在艾博生侧；
- 艾博生页面需具备承接订单上下文并继续分发 / 展示的能力。

### 1.3 权益退款流程

1. 权益包订单的退款流程入口：
   - 跳转到 **艾博生页面**。

**当前结论**

- 退款流程的页面入口也由艾博生承接；
- 后续需要进一步明确：
  - 退款页是纯展示页、提交申请页，还是中转分发页；
  - 艾博生与齐为在退款流程中的职责边界。

### 1.4 权益状态变更推送

1. 权益状态发生变化后：
   - 需要 **推送给科技平台**。

**当前结论**

- 艾博生侧后续需要补充“权益状态变更 -> 科技平台通知 / 回调 / 推送”的能力；
- 后续需与小花科技进一步明确推送方式、字段口径、幂等策略和重试机制。

### 1.5 联合登录方案

1. 联合登录方案已确认：
   - 从科技平台跳转到艾博生，**携带 token**；
   - 艾博生页面请求艾博生后端；
   - 艾博生后端再调用小花后端。

**当前结论**

- 联合登录整体链路已定：
  - `科技平台前端 -> 艾博生前端 -> 艾博生后端 -> 小花后端`
- 艾博生前端不直接请求小花后端；
- 艾博生后端承担 token 校验 / 用户换取 / 上下文建立等服务端对接职责。
