# 2026-04-17 小花联合登录方案骨架

## 1. 文档目标

本文档基于当前已确认事实，补充一版“艾博生侧推荐实现骨架”，用于统一：

1. 科技平台跳转艾博生 H5 的入口承接方式
2. 艾博生后端对接小花后端的联合登录主流程
3. 本地登录态建立方式
4. 分发页 / 退款页 / 状态回推后续设计的统一前置口径

本文档优先聚焦技术方案，同时补充业务承接说明，作为后续正式详细方案、开发计划和联调计划的输入。

---

## 2. 已确认事实

根据当前沟通结论，已确认以下事实：

1. 科技平台跳转到艾博生页面时，会携带 `token`
2. 艾博生 H5 页面不直接调用小花后端
3. 艾博生 H5 页面先请求艾博生后端
4. 艾博生后端再调用小花后端完成用户身份校验 / 换取 / 上下文建立
5. 放款成功后的可行权权益提醒 push 由科技平台触发
6. push 可跳转到艾博生分发页面，再跳转到齐为供应商页面
7. 权益订单页行权入口跳转到艾博生页面
8. 权益包退款入口跳转到艾博生页面
9. 权益状态变更需要推送给科技平台

基于以上事实，可以先明确：

- 联合登录是艾博生后续所有外部跳转入口的共同前置能力
- 分发页、退款页、状态回推均不宜脱离联合登录独立设计
- 在小花正式调用文档未补齐前，可先固化艾博生侧推荐实现骨架，但不直接定死最终接口字段

---

## 3. 设计目标与边界

### 3.1 设计目标

本方案希望先解决以下问题：

1. 科技平台跳转到艾博生时，H5 如何统一承接入口参数
2. 艾博生后端如何统一承接 `token` 并调用小花后端
3. 艾博生如何建立本地登录态，避免后续每个页面都重复处理联合登录
4. 如何为分发页、退款页和状态回推提供统一的身份与场景上下文
5. 如何在外部契约未完全到位前，先把本侧技术结构和业务承接关系定清楚

### 3.2 本文档边界

本文档当前只定义：

1. 推荐总体链路
2. 前端承接方案
3. 后端联合登录方案
4. 登录态与身份口径
5. 异常与降级处理建议
6. 分发页 / 退款页 / 状态回推的业务承接关系

本文档当前不直接定死：

1. 小花正式接口路径和字段
2. `token` 的最终算法、签名、加密细节
3. 生产环境最终安全策略细则
4. 退款页最终交互形态
5. 状态回推最终事件字段与回调协议

## 4. 推荐总体链路

### 4.1 推荐链路

建议采用以下统一链路：

```text
科技平台 App / 页面
    -> 跳转艾博生 H5（携带 token + scene + 订单上下文）
    -> 艾博生 H5 调用艾博生后端联合登录入口
    -> 艾博生后端调用小花后端完成 token 校验 / 用户身份换取
    -> 艾博生后端建立本地登录态
    -> 艾博生 H5 根据场景进入对应业务承接页
    -> 后续业务接口统一走艾博生后端
```

### 4.2 推荐原因

采用该链路的原因：

1. 把联合登录能力集中到入口阶段，只做一次
2. H5 后续业务页继续沿用艾博生本地登录态和接口体系
3. 前端不直接依赖小花后端，降低前端暴露外部契约和安全细节的风险
4. 后续即使小花接口变化，也主要影响艾博生后端适配层，不会同时拖动 H5 业务页大改

---

## 5. 前端承接方案

### 5.1 推荐页面定位

建议艾博生 H5 增加一个“联合登录承接入口页”或统一承接路由，用于处理外部跳转。

该入口页不承担具体业务展示，而是承担：

1. 解析跳转参数
2. 调用联合登录入口接口
3. 根据结果跳转到对应业务页
4. 在异常情况下展示统一错误页

### 5.2 推荐承接场景

建议先统一承接以下场景：

| 场景 | 含义 | 后续去向 |
|---|---|---|
| `push` | 放款成功后的可行权提醒消息 | 分发页 / 齐为供应商页面 |
| `exercise` | 权益订单页行权入口 | 分发页 / 行权页 |
| `refund` | 权益包退款入口 | 退款页 / 退款承接页 |

### 5.3 推荐前端入参口径

建议前端统一按“最小必需参数”承接：

1. `token`
2. `scene`
3. `orderNo` 或 `benefitOrderNo`
4. 可选 `productCode`
5. 可选 `packageId`
6. 可选 `channelCode`
7. 可选 `timestamp`
8. 可选 `sign`

说明：

- 当前仅建议先定义推荐接收口径
- 最终是否全部由科技平台直接携带，仍待小花 / 科技平台正式文档确认

### 5.4 前端入口页职责

前端入口页进入后，只做以下动作：

1. 读取 URL 参数
2. 校验最小参数是否存在
3. 调用艾博生后端联合登录入口接口
4. 根据返回结果决定进入：
   - 分发页
   - 退款页
   - 统一异常页

前端不负责：

1. 解密 `token`
2. 判断用户真实身份是否合法
3. 直接调用小花后端
4. 直接拼接齐为敏感跳转地址

### 5.5 前端需要预留的页面态

建议至少预留以下页面态：

1. 加载中
2. 登录成功，准备跳转
3. 参数缺失
4. `token` 失效
5. 小花服务异常
6. 场景暂不支持

## 6. 后端联合登录方案

### 6.1 推荐后端职责

艾博生后端应承担以下职责：

1. 接收 H5 联合登录请求
2. 校验基础入参
3. 调用小花后端完成 `token` 校验与用户身份换取
4. 结合场景和订单上下文构建本地业务承接上下文
5. 建立艾博生本地登录态
6. 返回前端后续跳转所需的最小信息

### 6.2 推荐服务拆分

建议后端按如下职责拆分：

#### `JointLoginController`

职责：

1. 接收 H5 请求
2. 做基础参数校验
3. 调用联合登录主服务
4. 统一返回 `Result<T>`

#### `JointLoginService`

职责：

1. 编排联合登录主流程
2. 组织小花调用、本地用户恢复、本地登录态建立
3. 决定前端后续目标页类型

#### `XiaohuaAuthClient`

职责：

1. 封装艾博生后端到小花后端的调用
2. 统一处理请求头、签名、时间戳、防重放等外部协议细节
3. 屏蔽小花接口变化对上层业务的影响

#### `JointLoginSessionService`

职责：

1. 根据换取结果恢复或初始化本地用户身份
2. 建立 `NEXUSFIN_AUTH` Cookie 或等效本地登录态
3. 处理会话续期、重复进入和登录态刷新问题

#### `SceneResolver`

职责：

1. 根据 `scene + orderNo + 用户上下文` 计算目标承接页
2. 区分是否进入分发页、退款页、兜底页
3. 为后续业务扩展保留统一场景路由能力

### 6.3 推荐后端返回内容

建议联合登录成功后的统一返回至少包含：

1. `loginSuccess`
2. `scene`
3. `targetPage`
4. `orderNo` / `benefitOrderNo`
5. `localUserReady`
6. 页面初始化所需的最小展示 / 路由信息

这样前端只需根据后端返回进行跳转，不需要自己推断业务去向。

## 7. 登录态与身份口径

### 7.1 推荐身份分层

建议统一管理以下三类身份 / 上下文：

1. 科技平台 / 小花侧用户唯一标识
2. 艾博生本地用户标识
3. 当前业务场景标识

### 7.2 推荐处理方式

建议处理顺序如下：

1. 艾博生后端通过小花接口换取科技平台用户身份
2. 根据该身份尝试匹配本地用户
3. 若本地已存在，则恢复本地身份
4. 若本地不存在，则创建或初始化最小本地上下文
5. 建立本地登录态，供后续 H5 普通接口继续使用

### 7.3 推荐原因

这样处理的好处是：

1. 联合登录只在入口处处理一次
2. H5 后续页面无需每次重复携带外部 `token`
3. 后续分发页、退款页、状态回推都可以复用同一套本地身份口径

---

## 8. 业务承接说明

### 8.1 分发页承接说明

分发页建议承接以下入口：

1. 放款成功后的可行权提醒 push
2. 权益订单页行权入口

建议分发页当前承担：

1. 根据场景识别当前入口类型
2. 根据订单状态和用户上下文决定：
   - 是否直接跳齐为供应商页面
   - 是否先展示中间说明页
   - 是否进入异常提示页

### 8.2 退款页承接说明

退款页当前建议先定义为“退款入口承接页”，不急于定死最终页面形态。

后续可演进为以下一种或多种角色：

1. 退款说明页
2. 退款申请页
3. 退款状态页
4. 退款中转页

在小花和齐为最终契约未明确前，当前只建议先保留统一承接入口和场景路由能力。

### 8.3 状态回推说明

权益状态回推科技平台并不属于联合登录接口本身，但联合登录会决定：

1. 使用哪套用户唯一标识
2. 使用哪套订单上下文
3. 状态回推时如何统一识别当前用户与订单

因此建议后续状态回推方案复用联合登录阶段确定的：

1. 用户标识口径
2. 订单号口径
3. 场景来源口径

避免多套身份体系并存。

## 9. 异常与降级设计

### 9.1 参数异常

当最小必需参数缺失时：

1. 前端直接进入参数异常页
2. 不调用后端业务链路
3. 保留返回科技平台或重试入口

### 9.2 `token` 失效

当小花侧返回 `token` 无效或过期时：

1. 后端统一返回标准错误
2. 前端进入登录失效页
3. 页面提示用户返回科技平台重新进入

### 9.3 小花接口失败

当小花后端不可用、超时或返回系统异常时：

1. 后端记录完整日志、追踪号和请求上下文
2. 前端展示系统繁忙页
3. 如后续需要，可补充重试或联系客服引导

### 9.4 本地登录态建立失败

若外部换取成功但本地登录态建立失败：

1. 视为联合登录失败
2. 后端统一返回错误
3. 前端不自行推断原因

### 9.5 场景暂不支持

若 `scene` 已识别但当前业务尚未支持：

1. 不直接返回 500
2. 建议进入兜底说明页
3. 记录当前不支持场景，便于后续补齐

---

## 10. 推荐实施原则

建议后续正式方案与实现继续遵循以下原则：

1. 联合登录能力只在入口阶段集中处理，不分散到各业务页面
2. H5 不直接依赖小花后端
3. 所有外部协议细节集中在后端适配层封装
4. 分发页、退款页、状态回推统一复用联合登录后的本地身份口径
5. 在正式文档未补齐前，只做骨架设计，不抢跑最终实现

---

## 11. 待确认项

当前仍需外部补齐或进一步确认的事项包括：

1. 科技平台跳转艾博生时的完整入参清单
2. `token` 参数名、有效期、是否一次性、是否签名 / 加密
3. 艾博生后端调用小花后端的正式接口文档
4. 小花返回的用户唯一标识字段及其稳定性
5. `token` 失效、用户异常、系统失败的标准错误码
6. 是否存在测试环境、白名单、防重放和签名机制
7. 分发页是否直接跳齐为，还是先展示中间说明页
8. 退款页最终是说明页、申请页、状态页，还是中转页
9. 权益状态回推的事件清单、字段、签名、幂等和重试策略

---

## 12. 当前结论

在当前信息条件下，艾博生侧推荐采用“技术先行、业务承接补充”的联合登录骨架方案：

1. 科技平台负责跳转并携带 `token` 与场景参数
2. 艾博生 H5 只负责承接参数和调用本地后端
3. 艾博生后端负责对接小花后端、恢复用户身份、建立本地登录态
4. 后续分发页、退款页、状态回推统一复用该登录入口与身份口径

该骨架可以先作为艾博生侧内部统一方案入口；待小花正式文档补齐后，再继续细化为正式技术方案和开发计划。
