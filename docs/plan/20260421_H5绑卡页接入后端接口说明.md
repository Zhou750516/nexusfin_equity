# 2026-04-21 H5 绑卡页接入后端接口说明

## 1. 文档目的

本文档用于给后续 H5 绑卡页面提供后端接口接入说明。

当前背景：

1. 齐为已补充签约查询、签约申请、签约确认 3 个接口；
2. 艾博生后端已先行完成对应接口骨架；
3. H5 绑卡页面尚未提供；
4. 当前文档只定义 H5 后续接入方式，不要求立即开发 H5 页面。

---

## 2. 当前接入结论

H5 后续绑卡页只需要调用艾博生后端，不直接调用齐为。

调用链路：

```text
H5 绑卡页
  -> 艾博生后端 /api/bank-card/*
    -> 齐为 abs.sign.query / abs.sign.apply / abs.sign.confirm
      -> 艾博生后端记录本地签约成功事实
```

设计原则：

1. H5 不直接传手机号、姓名、身份证号；
2. H5 只传用户输入的银行卡号和短信验证码；
3. 艾博生后端从登录态识别用户；
4. 艾博生后端从本地会员信息中解密手机号、姓名、身份证号；
5. 签约成功后由后端写入本地记录。

---

## 3. 前置条件

H5 调用以下接口前，需要满足：

1. 用户已完成联合登录；
2. 浏览器已持有 `NEXUSFIN_AUTH` Cookie；
3. 后端可通过登录态解析出 `memberId`；
4. `member_info` 已存在用户手机号、姓名、身份证号的加密信息；
5. 用户已在页面输入银行卡号。

若未登录：

1. 后端返回 HTTP `401`；
2. H5 应引导用户重新走联合登录或返回上游登录页。

---

## 4. 接口总览

| 场景 | 方法 | 接口 | H5 输入 | 后端处理 |
|---|---|---|---|---|
| 查询银行卡是否已签约 | `GET` | `/api/bank-card/sign-status` | `accountNo` | 调齐为 `abs.sign.query` |
| 申请绑卡短信 | `POST` | `/api/bank-card/sign-apply` | `accountNo` | 调齐为 `abs.sign.apply` |
| 确认绑卡签约 | `POST` | `/api/bank-card/sign-confirm` | `accountNo`、`verificationCode` | 调齐为 `abs.sign.confirm`，成功后本地落库 |

---

## 5. 接口详情

## 5.1 查询银行卡是否已签约

### 请求

```http
GET /api/bank-card/sign-status?accountNo=6222020202020208
```

### Query 参数

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `accountNo` | `string` | 是 | 用户输入的银行卡号 |

### 成功响应

```json
{
  "code": 0,
  "message": "OK",
  "data": {
    "accountNo": "6222020202020208",
    "signed": true,
    "status": "SIGNED"
  }
}
```

### 字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| `accountNo` | `string` | 原样返回本次查询的银行卡号 |
| `signed` | `boolean` | 是否已签约 |
| `status` | `string` | `SIGNED` 已签约；`UNSIGNED` 未签约 |

### H5 建议处理

1. 用户输入银行卡号后，点击下一步或离焦时调用；
2. 若 `signed=true`，可直接展示“银行卡已签约”，允许继续后续权益 / 借款流程；
3. 若 `signed=false`，进入短信申请步骤；
4. 若接口返回 `401`，跳转登录；
5. 若接口失败，提示“暂时无法查询签约状态，请稍后重试”。

## 5.2 申请绑卡短信

### 请求

```http
POST /api/bank-card/sign-apply
Content-Type: application/json
```

```json
{
  "accountNo": "6222020202021234"
}
```

### 请求字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `accountNo` | `string` | 是 | 用户输入的银行卡号 |

### 成功响应

```json
{
  "code": 0,
  "message": "OK",
  "data": {
    "requestNo": "mock-sign-apply-1234",
    "status": "SMS_SENT"
  }
}
```

### 字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| `requestNo` | `string` | 本次签约申请流水；当前只用于展示、日志或后续排查，不要求 H5 强依赖 |
| `status` | `string` | 当前固定为 `SMS_SENT`，表示短信申请已受理 |

### H5 建议处理

1. 调用成功后进入验证码输入态；
2. 启动短信验证码倒计时；
3. `requestNo` 可选保存到页面状态，用于日志排查；
4. 当前确认接口不要求 H5 回传 `requestNo`；
5. 若失败，提示“短信发送失败，请稍后重试”。

## 5.3 确认绑卡签约

### 请求

```http
POST /api/bank-card/sign-confirm
Content-Type: application/json
```

```json
{
  "accountNo": "6222020202025678",
  "verificationCode": "123456"
}
```

### 请求字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `accountNo` | `string` | 是 | 用户输入的银行卡号 |
| `verificationCode` | `string` | 是 | 用户收到的短信验证码 |

### 成功响应

```json
{
  "code": 0,
  "message": "OK",
  "data": {
    "requestNo": "mock-sign-confirm-5678",
    "signed": true,
    "status": "SIGNED"
  }
}
```

### 字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| `requestNo` | `string` | 签约确认流水 |
| `signed` | `boolean` | 是否签约成功 |
| `status` | `string` | 当前成功时返回 `SIGNED` |

### 后端本地记录

确认成功后，后端会写入 `member_payment_protocol`：

| 字段 | 当前写入规则 |
|---|---|
| `provider_code` | `QW_SIGN` |
| `protocol_no` | `QW-SIGN-{requestNo}` |
| `protocol_status` | `ACTIVE` |
| `sign_request_no` | `requestNo` |
| `channel_code` | 用户最近一条渠道绑定记录 |
| `signed_ts` | 后端当前时间 |

### H5 建议处理

1. 若 `signed=true`，展示绑卡 / 签约成功；
2. 成功后按页面来源决定跳转：
   - 若来自权益开通流程，返回权益开通或继续借款确认；
   - 若来自独立绑卡入口，进入绑卡成功页或返回上一页；
3. 若失败，提示验证码错误、签约失败或稍后重试；
4. 当前后端尚未细分齐为错误码，H5 第一版可以统一展示通用失败提示。

## 6. 推荐 H5 页面流程

建议 H5 绑卡页按以下状态组织：

```text
进入绑卡页
  -> 输入银行卡号
  -> 调 GET /api/bank-card/sign-status
    -> 已签约：展示已签约，允许继续
    -> 未签约：展示短信验证区域
      -> 调 POST /api/bank-card/sign-apply
      -> 输入短信验证码
      -> 调 POST /api/bank-card/sign-confirm
        -> 成功：展示签约成功并返回业务流程
        -> 失败：提示失败，允许重试
```

---

## 7. 页面状态建议

| 页面状态 | 触发条件 | 用户可见表现 |
|---|---|---|
| `INPUT_CARD` | 初始进入页面 | 输入银行卡号 |
| `CHECKING_STATUS` | 查询签约状态中 | 按钮 loading |
| `SIGNED` | 已签约 | 展示已签约，可继续 |
| `UNSIGNED` | 未签约 | 展示发送验证码按钮 |
| `SMS_SENDING` | 申请短信中 | 发送按钮 loading |
| `SMS_SENT` | 短信发送成功 | 显示验证码输入框与倒计时 |
| `CONFIRMING` | 提交验证码中 | 确认按钮 loading |
| `SIGN_SUCCESS` | 签约成功 | 展示成功态并跳转 |
| `SIGN_FAILED` | 签约失败 | 展示失败提示，允许重试 |

## 8. 前端校验建议

第一版建议只做基础校验：

| 字段 | 校验建议 |
|---|---|
| `accountNo` | 非空；只允许数字；长度 12-30 位之间 |
| `verificationCode` | 非空；只允许数字；长度 4-8 位之间 |

说明：

1. 银行卡 BIN、支持银行列表、实名一致性暂不在 H5 第一版强校验；
2. 最终是否签约成功以后端和齐为返回为准。

---

## 9. 错误处理建议

| 场景 | 后端表现 | H5 建议 |
|---|---|---|
| 未登录 | HTTP `401`，`code=401` | 跳转登录或提示登录失效 |
| 参数为空 | `code=400` 或失败响应 | 提示用户补充银行卡号 / 验证码 |
| 齐为查询失败 | `code != 0` | 提示“查询失败，请稍后重试” |
| 短信发送失败 | `code != 0` | 提示“短信发送失败，请稍后重试” |
| 验证码错误 / 签约失败 | `code != 0` | 提示“验证码错误或签约失败” |
| 网络超时 | 请求超时 | 提示“网络异常，请稍后重试” |

## 10. 本地 mock 说明

当前后端 mock 规则：

1. `accountNo` 以 `8` 结尾时，签约状态查询返回已签约；
2. 其他卡号查询返回未签约；
3. 签约申请返回 `mock-sign-apply-{银行卡后4位}`；
4. 签约确认返回 `mock-sign-confirm-{银行卡后4位}`；
5. 签约确认成功后会写入本地 `member_payment_protocol`。

本地测试示例：

| 场景 | 测试卡号 |
|---|---|
| 已签约查询 | `6222020202020208` |
| 申请短信 | `6222020202021234` |
| 确认签约 | `6222020202025678` |
