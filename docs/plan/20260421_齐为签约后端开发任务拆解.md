# 2026-04-21 齐为签约后端开发任务拆解

> **目标范围**：仅覆盖艾博生后端、齐为接口接入、本地记录落库；不包含 H5 绑卡页面。

## 1. 任务目标

基于 `plan/20260421_齐为签约后端接入设计方案.md`，将齐为签约能力拆分为可以连续推进的开发任务。

本次需要实现：

1. 会员同步字段升级
2. 齐为签约查询 / 申请 / 确认接入
3. 艾博生后端签约接口
4. 签约成功本地记录
5. 相关测试与文档回填

## 2. 任务分组

### Task Group A：协议与模型收口

**目标**

**任务**

1. 梳理 `QwMemberSyncRequest` 与齐为新文档字段差异
2. 确认 `payProtocolNo` 从会员同步模型中移除或降级处理的改造方式
3. 设计 `QwSignStatusRequest / Response`
4. 设计 `QwSignApplyRequest / Response`
5. 设计 `QwSignConfirmRequest / Response`
6. 设计本侧 `BankCardSign*` 请求 / 返回 DTO
7. 明确签约成功写入 `member_payment_protocol` 的字段口径

**产出物**

1. DTO 列表
2. 字段映射表
3. 本地记录字段写入规则

### Task Group C：签约服务层

**目标**

- 把签约查询、申请、确认的业务编排收口到独立 service

**建议文件**

1. `src/main/java/com/nexusfin/equity/service/BankCardSignService.java`
2. `src/main/java/com/nexusfin/equity/service/impl/BankCardSignServiceImpl.java`

**任务**

1. 新增 `BankCardSignService`
2. 实现签约查询逻辑：
   - 参数校验
   - 调齐为 `abs.sign.query`
   - 映射为本侧返回
3. 实现签约申请逻辑：
   - 参数校验
   - 调齐为 `abs.sign.apply`
   - 返回“短信已发送 / 申请已受理”类最小口径
4. 实现签约确认逻辑：
   - 参数校验
   - 调齐为 `abs.sign.confirm`
   - 成功后触发本地协议记录
5. 保持所有业务逻辑不进入 controller

**完成标准**

1. Service 能独立单测
2. 签约确认成功后能正确进入本地记录逻辑

## 5. 当前结论

截至当前，齐为签约后端接入已经具备启动开发任务拆解的条件。

当前最合理的推进方式是：

1. 先按本文件拆出的 7 个任务组执行
2. 先完成后端闭环，不等待 H5 页面
3. 先复用 `member_payment_protocol` 做签约成功记录
4. 暂不把状态机、重试、幂等扩到第一版范围

一句话结论：

> 本轮可正式启动“齐为签约后端接入”开发，建议按“协议接入 -> 业务闭环 -> 测试验证”的顺序推进，并将签约成功本地记录纳入第一版必做范围。
