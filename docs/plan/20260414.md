**现象**

- 真实联调环境需使用持久化数据库，避免重启后链路上下文丢失

#### 问题 2：语言切换后，当前页已加载的接口文案不会立即重刷

**现象**

- 在 `repayment-success` 页从 `en-US` / `zh-TW` / `vi-VN` 手动切换语言时
- 页面静态文案会立即切换
- 但当前页已经加载完成的接口返回字段，需要页面刷新后才会与目标语言完全一致

**复现口径**

1. 进入还款成功页
2. 先切换到英文
3. 再切换繁中或越南文
4. 不刷新页面时，部分已加载的接口文案仍保留上一个语言
5. 刷新后恢复正确

**初步判断**

- 当前语言切换已驱动前端静态文案切换
- 但页面未在语言变化时统一触发接口重拉

**建议**

- 将“语言切换”纳入接口依赖条件
- 在语言变化时重新请求当前页依赖接口，确保富文本 / 提示语 / 银行名称等动态字段同步刷新

#### 问题 4：`loan_failed` 场景下，后端仍返回 `applicationId`，导致前端误查结果接口

**复现路径**

1. 在首页以本地联调金额 `4200` 发起借款申请
2. Yunka stub 返回借款申请失败
3. 后端 `POST /api/loan/apply` 返回：
   - `status = loan_failed`
   - `benefitsActivated = true`
   - `message = 权益购买成功，借款申请失败：...`
   - **同时仍返回 `applicationId`**
4. 前端收到后跳转 `/approval-result`
5. `ApprovalResultPage` 因检测到存在 `applicationId`，会主动请求：
   - `GET /api/loan/approval-result/{applicationId}`
6. 由于该场景下并未保存 `loan_application_mapping`
7. 最终页面报错：
   - `application mapping not found`

**已确认事实**

1. 前端 fallback 页面本身是正确的  
   当手动清空 `sessionStorage` 中的 `applicationId` 后，再访问 `/approval-result`，页面可正确展示：
   - 标题：`借款申请失败`
   - 提示：`权益购买成功，借款申请失败：...`
2. 因此当前问题不是 `ApprovalResultPage` 视觉层错误
3. 根因是 **后端在 `loan_failed` 场景返回了不应继续查询结果详情的 `applicationId`**

**建议修复方向**

建议二选一，优先采用方案 A：

1. **方案 A：后端修正返回口径**
   - `loan_failed` 场景下，`POST /api/loan/apply` 返回 `applicationId = null`
   - 前端继续沿用现有 fallback 逻辑
2. **方案 B：前端增加防御判断**
   - 当 `status === loan_failed` 时，即使存在 `applicationId`
   - `ApprovalResultPage` 也不再调用 `getApprovalResult(applicationId)`
   - 直接展示本地上下文中的 `approvalMessage`

**当前判断**

- 该问题属于“前后端契约不一致”
- 更合理的收口方式是：**后端不在 `loan_failed` 场景返回可查询结果页的 `applicationId`**
