# 2026-04-03 `M3.2` 启动建议模板

## 1. 文档目的

本文档用于在公网测试回调条件确认后，快速生成一版可直接对内同步、对外执行的 `M3.2 入站回调联调` 启动建议。

本模板的定位不是“今天已经可以启动”，而是：

> 把启动前要满足的条件、启动时要给第三方的信息、启动后的验证顺序先固定下来，避免外部条件一旦确认后还要临时补口径。

## 2. 启动前置条件

只有以下 3 项全部确认，才建议将 `M3.2` 标记为“可启动”：

### 2.1 公网测试域名已确认

至少需要明确：

1. 可供第三方访问的测试域名
2. 是否可直接访问
3. 是否已完成 HTTPS 配置

### 2.2 回调路径暴露方式已确认

至少需要明确：

1. `/api/callbacks/**` 是否直接暴露
2. 是否经网关统一转发
3. 是否存在 path rewrite
4. 最终第三方应使用的完整 URL 是什么

### 2.3 白名单策略已确认

至少需要明确：

1. 是否需要来源 IP 白名单
2. 是否需要来源域名白名单
3. 由谁配置
4. 配置生效时间和验证方式

## 3. 可启动后的标准口径

如果以上 3 项已确认，可统一使用如下口径：

> 当前本地回调入口、签名校验、幂等处理和订单状态更新主链已经具备。  
> 公网测试域名、路径暴露方式和白名单策略已确认，可以启动 `M3.2` 首轮入站回调联调。  
> 下一步将整理完整回调 URL 清单给第三方，并按“单接口逐条验证”的方式推进。

## 4. 建议提供给第三方的回调地址清单

在公网条件明确后，建议统一整理并提供以下 URL：

### 4.1 当前主用回调路径

1. `https://<待确认域名>/api/callbacks/grant/forward`
2. `https://<待确认域名>/api/callbacks/repayment/forward`
3. `https://<待确认域名>/api/callbacks/exercise-equity`
4. `https://<待确认域名>/api/callbacks/refund`

### 4.2 扣款类候选路径

若最终仍按齐为历史回调口径执行，再补充：

1. `https://<待确认域名>/api/callbacks/first-deduction`
2. `https://<待确认域名>/api/callbacks/fallback-deduction`

## 5. 建议验证顺序

`M3.2` 启动后，建议不要并发验证所有回调，而是按以下顺序逐条推进：

### 5.1 第一阶段：网络与验签验证

目标：

1. 确认第三方可以从公网访问我方 URL
2. 确认签名头能通过我方校验

建议方式：

1. 先用一条最小回调报文验证可达性
2. 重点观察：
   - 网关日志
   - 应用入口日志
   - 签名校验结果

### 5.2 第二阶段：业务轻链路验证

优先建议验证：

1. `refund`
2. `exercise-equity`

原因：

1. 路径简单
2. 状态影响面相对清晰
3. 更适合先验证“入口 -> 验签 -> 幂等 -> 状态更新”主链

### 5.3 第三阶段：业务主链验证

再验证：

1. `grant/forward`
2. `repayment/forward`

原因：

1. 与主订单状态推进关系更强
2. 对后续链路影响更大

### 5.4 第四阶段：扣款类回调验证

若最终仍保留该类接口，则最后验证：

1. `first-deduction`
2. `fallback-deduction`

原因：

1. 需要结合最终合作方回调口径
2. 容易和当前内部状态机产生耦合，放最后更稳妥

## 6. 建议验证结果留痕格式

每验证一条回调，建议至少记录以下信息：

1. 验证时间
2. 回调路径
3. 完整 URL
4. 请求头摘要
5. 请求体摘要
6. HTTP 返回码
7. 应用日志关键片段
8. 订单状态变化结果
9. 是否通过

## 7. 当前使用方式

今天若网关 / 运维尚未回复，则本模板先作为待启用材料。

后续一旦收到三项条件确认结果，只需：

1. 补齐真实域名
2. 补齐路径暴露说明
3. 补齐白名单结果

即可快速产出正式版启动建议。

## 8. 一句话结论

> `M3.2` 当前是否能启动，不取决于本地代码是否存在，而取决于公网域名、路径暴露和白名单三项是否同时落定。
