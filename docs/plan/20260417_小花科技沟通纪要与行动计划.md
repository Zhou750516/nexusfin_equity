## 2. 会议纪要

### 2.2 本次改动较大的接口

本次需要优先关注“小花提供给艾博生的接口中变动较大的部分”。

当前需要重点追踪：

1. 联合登录 / 用户信息查询相关接口
2. 用信流程中涉及图片获取与回传的接口
3. 后续艾博生调用小花时依赖 `partnerId` 的接口
4. 还款跳转相关页面入口与参数

### 2.3 `userId` / `partnerId` 来源方案

关于小花提供艾博生的接口中 `userId` 的来源，当前有两个方案。

#### 方案 1：艾博生查询小花用户信息时返回 `partnerId`

流程：

1. 艾博生调用小花用户信息查询接口
2. 小花返回用户信息，同时返回 `partnerId`
3. 艾博生后续使用该 `partnerId` 调用小花其他接口

特点：

1. 艾博生侧能在联合登录 / 用户查询阶段拿到后续调用标识
2. 后续调用链路较清晰
3. 需要小花在用户信息查询响应中明确返回 `partnerId`

#### 方案 2：小花内部生成 `partnerId`，返回信息时艾博生获取并使用

流程：

1. 小花科技内部按现有逻辑生成 `partnerId`
2. 艾博生在小花返回信息中获取该 `partnerId`
3. 艾博生后续使用该 `partnerId` 调用小花

特点：

1. 更贴近小花现有系统内部逻辑
2. 艾博生不额外参与 `partnerId` 生成规则
3. 需要小花确认其现有系统是否已有内部生成 `partnerId` 的成熟逻辑

#### 当前决策依据

当前决策依据为：

> 保持和小花现有方案一致。如果小花原来有内部生成 `partnerId` 的逻辑，则采用方案 2；否则采用方案 1。

当前状态：

- `@待小花同学确认`

1. 主动还款链路无需新增“还款前绑卡短信发送 / 确认”步骤
2. 此前齐为绑卡 3 个接口仍作为独立绑卡流程 TODO 维护
3. 还款主流程应继续聚焦云卡 / 科技平台还款试算、还款提交、还款结果查询

1. 用信流程中需要回传图片
2. 图片通过一个接口获取到
3. 艾博生侧再将图片回传过去

当前待明确内容：
