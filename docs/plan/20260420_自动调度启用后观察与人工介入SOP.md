# 2026-04-20 自动调度启用后观察与人工介入 SOP

## 1. 目标

在自动调度第一阶段仍采用“默认关闭、显式开启、显式分片”的前提下，补一版简化版 SOP，帮助联调 / 运维同学在开启后快速回答以下问题：

1. 调度是否真的跑起来了
2. 当前分片是谁在跑
3. 任务是否在被消费
4. 异常时优先看哪里
5. 什么情况需要人工介入

## 2. 启用前检查

在环境内显式开启自动调度前，先确认：

1. `enabled / workerEnabled / supervisorEnabled` 是否按预期配置
2. `workerId` 是否显式配置
3. `ownedPartitions` 是否已明确划分
4. 当前环境是否允许该实例承担 `supervisor`
5. 当前环境是否仍有未完成联调，避免误触发消费

## 3. 启用后第一观察点

### 3.1 启动日志

优先看启动日志中的调度摘要，确认：

1. `enabled`
2. `workerEnabled`
3. `supervisorEnabled`
4. `workerId`
5. `ownedPartitions`

若日志与预期不一致，先不要继续观察任务消费，先修正配置。

### 3.2 分片运行时表

重点看：

1. `async_compensation_partition_runtime`

关注字段：

1. `partition_no`
2. `worker_id`
3. `worker_status`
4. `current_task_id`
5. `last_heartbeat_ts`

判断原则：

1. `last_heartbeat_ts` 持续更新，说明该分片仍有活跃心跳
2. `worker_status = RUNNING` 且 `current_task_id` 非空，说明该分片本次 tick 已认领任务
3. `worker_status = IDLE` 且 `current_task_id = null`，说明当前分片未处于任务处理态

## 4. 正常运行判断

若自动调度正常运行，通常会同时看到：

1. `partition_runtime` 有持续心跳
2. `async_compensation_task` 中 `INIT / RETRY_WAIT` 数量下降
3. `SUCCESS` 数量增加
4. `async_compensation_attempt` 有新的成功 / 失败尝试记录

## 5. 异常场景排查顺序

建议按以下顺序排查：

### 场景 A：看不到任何心跳

先检查：

1. `enabled` 是否为 `true`
2. `workerEnabled / supervisorEnabled` 是否为 `true`
3. `ownedPartitions` 是否为空
4. 应用是否真的重启并加载了新配置

### 场景 B：有心跳，但任务不下降

先检查：

1. `async_compensation_task` 中是否真有 `INIT / RETRY_WAIT`
2. `next_retry_ts` 是否尚未到期
3. `ownedPartitions` 是否覆盖到这些任务所在 partition
4. `async_compensation_attempt` 是否有失败留痕

### 场景 C：失败任务持续增长

先检查：

1. `async_compensation_attempt.error_code`
2. `async_compensation_attempt.error_message`
3. 是否为外部依赖持续失败
4. 是否为本地脏数据 / 缺参 / 映射错误

### 场景 D：`DEAD` 持续增长

先检查：

1. 是否为同一类错误反复失败
2. 是否需要先修正外部依赖再重试
3. 是否需要人工补单或人工关闭

## 6. 人工介入建议

### 可优先人工重试的情况

1. 外部接口已恢复
2. 参数本身无误
3. 同类任务已有成功先例

### 不建议直接重试的情况

1. 映射主数据有问题
2. 业务订单本身已无效
3. 外部依赖仍持续故障

### 需人工补单 / 人工确认的情况

1. 权益订单与补偿任务状态不一致
2. 外部状态已成功，但本地仍停留失败
3. 已进入 `DEAD` 且业务要求强一致恢复

## 7. 当前建议动作模板

### 模板 A：发现 backlog 不下降

1. 查 `partition_runtime`
2. 查 backlog 任务所属 partition
3. 对照 `ownedPartitions`
4. 查 `attempt` 是否有失败记录
5. 再判断是配置问题还是外部失败

### 模板 B：发现 `DEAD` 增长

1. 查最近一批 `DEAD` 任务
2. 归类 `errorCode / errorMessage`
3. 判断是否可统一修复
4. 需要时通知业务确认是否人工补单

### 模板 C：怀疑调度没生效

1. 看启动日志摘要
2. 看 `partition_runtime.last_heartbeat_ts`
3. 看 `task / attempt` 是否有新增变化
4. 若三者都无变化，再回到配置排查

## 8. 当前边界

本 SOP 只覆盖当前第一阶段自动调度场景，不覆盖：

1. 多实例自动抢占
2. 长任务持续心跳
3. 外部 MQ / 调度平台
4. 退款 / 绑卡 / 权益状态回推补偿

## 9. 一句话结论

> 自动调度启用后，优先看“启动日志摘要 + partition runtime + task/attempt 留痕”三件套；先判断是配置没生效、任务未到期，还是外部依赖失败，再决定是否人工介入。
