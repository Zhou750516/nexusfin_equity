# 2026-04-02 齐为主流程落地边界说明

## 1. 目的

本文档用于基于 `2026-04-02` 当天齐为三条主动接口真实联调结果，明确：

1. 哪些能力已经可以视为主流程已落地
2. 哪些能力当前只能保留为测试联调配置
3. 后续正式上线前还需要补什么

## 2. 结论先行

当前建议统一使用如下口径：

### 2.1 已可纳入主流程

以下能力已经完成代码接线并经过真实联调验证，可继续保留在主流程中：

1. `abs.push.order`
2. `abs.token.get`
3. `abs.lending.notify`

这里的“可纳入主流程”指的是：

1. 主链路接线方式已经正确
2. 请求字段映射当前口径成立
3. 标准 HTTP 协议适配层已经可用

### 2.2 只能保留为测试联调配置

以下能力当前不能视为正式主流程方案，只能保留为联调辅助能力：

1. `QW_MEMBER_SYNC_PAY_PROTOCOL_NO_OVERRIDE`
2. 齐为提供的 mock 协议号

原因是：

1. mock 协议号不是真实支付协议号
2. override 只是为了在当前未接通联时继续验证齐为链路
3. 正式上线必须使用真实通联协议号，而不是测试占位值

当前位于下单主链：

1. 协议层已打通
2. 用 mock 协议号时可完成真实联调
3. 后续真正需要替换的不是“是否保留此接口”，而是 `payProtocolNo` 的真实来源

### 3.2 `abs.token.get`

## 4. 当前不能直接固化到正式主流程的点

当前不能直接说“齐为链路已全部生产就绪”，原因不在接口本身，而在支付协议号来源。

核心问题是：

1. 当前项目尚未对接通联
2. 因此当前项目还拿不到真实用户级 `payProtocolNo`
3. 所以 `abs.push.order` 虽然链路已通，但目前是靠 mock 协议号继续验证

这意味着：

1. 可以保留齐为三接口的主链接线
2. 不能把测试期 override 当成正式实现

## 5. 当前建议的落地动作

### 5.1 现在就可以做的

1. 保留当前 `abs.push.order`、`abs.token.get`、`abs.lending.notify` 主流程接线
2. 保留 `QW_MEMBER_SYNC_PAY_PROTOCOL_NO_OVERRIDE`，但明确只允许测试环境使用
3. 将“齐为三接口已真实联调”的结论同步到部署说明与日报口径

### 5.2 正式上线前必须补的

1. 对接通联，获取真实 `payProtocolNo`
2. 明确真实协议号的落库字段与取值优先级
3. 将 `abs.push.order` 切换到真实协议号来源
4. 生产环境清理 `QW_MEMBER_SYNC_PAY_PROTOCOL_NO_OVERRIDE`
