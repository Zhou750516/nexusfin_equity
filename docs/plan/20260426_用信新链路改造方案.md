# 2026-04-26 用信新链路改造方案

## 1. 文档目标

本文档用于把 `2026-04-23` 已确认的借款主链调整，进一步细化为一版可讨论、可承接、可继续拆任务的方案文档。

本轮只讨论**艾博生侧**的主链改造口径，不替云卡定义内部状态机实现，不替小花定义真实放款协议细节。

核心目标：

1. 说明旧链路与新链路的根本差异；
2. 明确新链路各阶段的后端责任边界；
3. 明确本地最少需要维护的核心标识与状态；
4. 标注失败 / 超时 / 放弃权益 / 关单等关键分支；
5. 标出哪些内容本侧可以先定，哪些必须等待云卡接口定版。

---

## 2. 背景与核心变化

### 2.1 旧链路理解

当前第一版实现更接近以下思路：

1. 艾博生侧接收 H5 借款申请；
2. 艾博生通过云卡直接透传借款申请；
3. 云卡继续调用小花；
4. 艾博生主要通过申请结果、审批状态和回调来承接页面流程。

该模式的问题在于：

1. 权益购买与借款触发耦合过紧；
2. 无法清晰表达“权益已完成，但尚未真正触发放款”的阶段；
3. 审批中 / 审批结果页容易混用云卡内部状态与小花最终结果；
4. 用户放弃权益、签约 loading、未结清限制再借等新增规则难以自然接入。

### 2.2 新链路理解

根据 `2026-04-23` 联动会议，后续主链需调整为：

1. 艾博生提交用信时，**先调用云卡创建放款订单**；
2. 云卡先不真正触发小花放款；
3. 艾博生完成权益购买与短信签约；
4. 权益完成后，再由艾博生调用云卡触发放款；
5. 云卡再继续调用小花；
6. 艾博生侧通过“云卡状态 + 小花结果”承接审批中、审批结果、拒绝导流等页面。

该调整意味着：借款主链已经从“单次申请调用”升级为“分阶段推进的状态型流程”。

---

## 3. 新链路总体分段

建议将新主链拆为六段：

1. **阶段 A：创建云卡放款订单**
2. **阶段 B：权益购买前置准备**
3. **阶段 C：权益购买与短信签约**
4. **阶段 D：签约完成后的 loading 承接**
5. **阶段 E：触发云卡放款**
6. **阶段 F：状态查询与结果承接**

这样拆分的好处是：

1. 页面阶段和后端阶段可以对齐；
2. 能单独表达“已建单未放款”“已签约待触发”“放款处理中”等中间态；
3. 方便放弃权益、关单、超时补偿、结果回调等分支接入。

---

## 4. 六段主路径说明

## 4.1 阶段 A：创建云卡放款订单

### 目标

先建立本次借款对应的云卡侧订单，而不是直接触发小花放款。

### 艾博生侧责任

1. 接收 H5 借款发起动作；
2. 完成基础资格校验：
   - 用户身份完整性
   - 权益状态
   - 未结清借据限制
   - 必要协议阅读状态
3. 调用云卡创建放款订单接口；
4. 记录本地与云卡的映射关系；
5. 向前端返回下一阶段所需上下文。

### 建议输出

返回给 H5 的不是最终放款结果，而是：

1. 是否建单成功；
2. 云卡放款订单号；
3. 当前阶段状态；
4. 后续应跳转的页面或流程阶段。

### 当前待确认

1. 云卡建单接口路径与字段；
2. 云卡建单成功 / 失败 / 幂等重复的返回规则；
3. 建单失败是否仍保留本地申请记录。

---

## 4.2 阶段 B：权益购买前置准备

### 目标

在真正进入权益购买和短信签约之前，把页面承接所需的动态数据准备好。

### 艾博生侧责任

1. 拉取权益页动态协议；
2. 拉取默认绑卡摘要 / 用户卡信息；
3. 读取本地展示配置；
4. 向 H5 返回可直接渲染的权益页聚合结果。

### 关键说明

这一阶段不应再把“借款申请已经触发放款”当作默认前提，而应表达为：

> 借款流程已创建云卡订单，但还未完成权益前置条件。

---

## 4.3 阶段 C：权益购买与短信签约

### 目标

完成权益购买，并通过齐为短信签约完成权益生效前置动作。

### 艾博生侧责任

1. 接收权益开通请求；
2. 执行协议校验与必要参数校验；
3. 发起权益购买；
4. 触发齐为短信签约流程；
5. 在签约成功后记录本地状态；
6. 如需后续法大大签章，记录后续触发入口。

### 关键说明

此阶段结束后，系统应能明确区分：

1. 权益未购买；
2. 权益已购买但签约未完成；
3. 权益购买与签约均完成，但还未真正触发放款。

---

## 4.4 阶段 D：签约完成后的 loading 承接

### 目标

避免签约页完成后，前端直接跳到结果页或出现空白等待。

### 艾博生侧责任

1. 为 H5 提供一个清晰的 loading 状态承接；
2. 告诉前端当前是“签约完成，等待触发放款 / 查询状态”阶段；
3. 如后端已触发放款，则进入轮询状态；
4. 如后端尚未触发放款，也要有明确中间态。

### 关键说明

loading 页不是结果页，而是连接“权益完成”和“放款处理中”之间的中间页。

---

## 4.5 阶段 E：触发云卡放款

### 目标

在权益完成后，由艾博生侧调用云卡触发真正放款。

### 艾博生侧责任

1. 校验本地权益和签约状态已满足触发条件；
2. 调用云卡触发放款接口；
3. 更新本地借款主状态；
4. 记录触发时间、触发结果、必要错误信息；
5. 如调用超时或失败，决定是否进入补偿或重试体系。

### 关键说明

该阶段才是“真正触发小花放款”的前置入口，因此：

1. 当前异步补偿逻辑是否需要迁移到此节点，要单独评估；
2. 不能再把旧 `loan/apply` 的同步超时语义简单平移。

### 当前待确认

1. 云卡触发放款接口路径与字段；
2. 触发放款是否同步返回处理中状态；
3. 超时 / 幂等重复 / 已触发过的处理规则；
4. 是否需要本地单独记录触发放款流水。

---

## 4.6 阶段 F：状态查询与结果承接

### 目标

通过查询和回调，向 H5 正确展示审批中、成功、失败、拒绝导流等页面。

### 艾博生侧责任

1. 先查询云卡放款订单状态；
2. 再根据云卡状态决定是否映射小花最终结果；
3. 给 H5 返回适合页面承接的阶段状态；
4. 接收放款结果回调并更新本地状态；
5. 为审批结果、拒绝落地页、分发页提供统一上下文。

### 关键说明

后续接口设计上，必须清晰区分：

1. **云卡内部订单状态**
2. **小花最终放款结果**
3. **H5 页面展示状态**

三者不能继续混用。

---

## 5. 关键状态建议

建议本地先按业务阶段维护一组中间状态，便于页面承接与后续扩展：

1. `ORDER_CREATED`：云卡放款订单已创建
2. `WAIT_BENEFIT_PURCHASE`：待权益购买
3. `WAIT_SIGN`：待短信签约
4. `SIGN_COMPLETED`：签约已完成
5. `WAIT_TRIGGER_LENDING`：待触发放款
6. `LENDING_TRIGGERED`：已触发放款
7. `LENDING_PROCESSING`：放款处理中
8. `LENDING_SUCCESS`：放款成功
9. `LENDING_FAILED`：放款失败
10. `ORDER_CLOSED`：订单关闭

说明：

1. 这组状态是**艾博生本地业务状态建议**；
2. 并不等于云卡真实状态机字段；
3. 最终仍需映射表来对齐云卡和小花结果。

---

## 6. 本地最少需要维护的核心标识

建议后续至少维护以下主键或业务标识：

1. 艾博生本地借款申请号 / 业务号
2. 云卡放款订单号
3. 本地权益订单号
4. 齐为签约流水号 / 签约结果标识
5. 小花最终放款结果标识（如有）
6. 当前本地业务阶段状态

建议后续至少维护以下关键状态：

1. 权益状态
2. 签约状态
3. 放款触发状态
4. 云卡订单状态
5. 小花最终结果状态
6. 页面承接状态

---

## 7. 关键异常分支

## 7.1 建单失败

处理建议：

1. 前端不进入权益页主链；
2. 返回明确失败口径；
3. 视错误类型决定是否允许重试。

## 7.2 权益购买失败

处理建议：

1. 保留云卡订单上下文；
2. 不触发放款；
3. 前端停留在权益相关阶段，允许用户重试或退出。

## 7.3 签约失败 / 超时

处理建议：

1. 视为权益前置未完成；
2. 不触发放款；
3. 给出重新签约或后续重试承接。

## 7.4 用户放弃权益

处理建议：

1. 艾博生通知云卡；
2. 云卡判断是否允许关单；
3. 前端根据返回结果进入关闭、拒绝或其他承接页。

## 7.5 触发放款超时

处理建议：

1. 本地进入处理中或待补偿状态；
2. 不应直接给前端“成功”；
3. 需结合现有异步补偿体系评估是否复用。

## 7.6 放款失败

处理建议：

1. 页面进入失败 / 拒绝导流承接；
2. 记录失败原因；
3. 明确是否允许重新申请或进入分发页。

---

## 8. 本侧可先定与待外部确认项

## 8.1 本侧可先定

1. 本地阶段状态分层方式；
2. H5 页面承接分段；
3. 本地主键与映射关系设计；
4. 接口返回中“页面态”和“业务态”的拆分；
5. 哪些异常分支进入 loading、失败页、拒绝页、分发页。

## 8.2 必须等待外部确认

1. 云卡建放款订单接口；
2. 云卡触发放款接口；
3. 云卡状态查询接口；
4. 放弃权益 / 关单判断接口；
5. 云卡状态机字段与语义；
6. 小花最终结果经云卡返回或回调的 envelope 规则。

---

## 9. 建议下一步

建议后续按以下顺序继续推进：

1. 单独形成“云卡状态查询与 H5 状态映射方案”；
2. 单独形成“H5 自动埋点方案”；
3. 明确 Shared Foundation 第一批落地项；
4. 在云卡接口进一步定版后，再把本方案下沉为接口设计和实现任务。

---

## 10. 一句话结论

> 新借款主链的关键，不是把旧 `/api/loan/apply` 再补更多字段，而是把“建单、权益、签约、loading、触发放款、状态查询”拆成清晰阶段，并让本地状态、云卡状态、小花结果三层语义彻底分开。
