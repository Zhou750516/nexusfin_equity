1. 创建权益订单接口缺少 request-level 幂等，存在重复创建风险。

## 3. Bug 明细

### 3.1 创建权益订单接口缺少请求级幂等

**发现时间**

- 2026-03-23

**问题现象**

- 创建权益订单接口原先没有 `requestId` 字段，也没有重复请求回放逻辑。
- 同一请求在重试场景下可能产生重复订单或重复业务副作用。

**影响**

1. 违反基线规格中对 request-level idempotency 的要求。
2. 订单创建在网络重试、客户端重放等场景下存在重复入库风险。
3. 后续排障时难以准确关联一次请求和一笔业务订单。

**处理动作**

1. 在 `CreateBenefitOrderRequest` 中新增 `requestId`。
2. 在 `BenefitOrderServiceImpl.createOrder()` 前置查询 `idempotency_record`。
3. 已处理请求直接回放历史 `benefit_order_no`，不再重复创建业务对象。
4. 创建成功后写入 `benefit_order.request_id` 和 `idempotency_record`。

**验证结果**

1. 单元测试、集成测试通过。
2. H2 测试通过。
3. MySQL 回归通过。

**对应提交 / 文档**

1. `140a0d9 Fix order idempotency and callback audit handling`
2. `docs/CODE_MODIFY_20260323.md`

### 3.2 回调接口在缺单/异常场景下审计状态不清晰

**发现时间**

- 2026-03-23

**问题现象**

- 行权、退款、还款等回调在订单不存在或处理异常时，虽然写入了通知日志，但默认可能被标记为 `PROCESSED`。
- 审计结果看起来像“成功处理”，与真实处理结果不一致。

**影响**

1. 缺单场景在审计层失真，不利于人工排障。
2. 对账时难以区分“已成功处理”和“收到消息但处理失败”。
3. 异步通知链路的可信性不足。

**处理动作**

1. 统一将回调接收状态先记录为 `RECEIVED`。
2. 业务处理成功后更新为 `PROCESSED`。
3. 订单不存在或运行时异常时更新为 `FAILED`。
4. 还款回调在订单不存在时只记录失败审计，不误写幂等成功记录。

**验证结果**

1. `NotificationServiceTest`、回调集成测试通过。
2. MySQL 回调链路回归通过。

**对应提交 / 文档**

1. `140a0d9 Fix order idempotency and callback audit handling`
2. `docs/CODE_MODIFY_20260323.md`

### 3.3 业务接口身份来源不可信，认证边界不清晰

**发现时间**

- 2026-03-24

**问题现象**

- 旧链路依赖静默注册入口和外部传入的 `memberId` 语义。
- 浏览器态业务接口与系统态回调接口的鉴权边界不够清晰。

**影响**

1. 会员身份来源不够可信，业务接口安全边界偏弱。
2. 旧注册入口与新登录入口并存时，容易形成双轨语义。
3. 业务接口和回调接口混用同一套身份逻辑会增加后续维护风险。

**处理动作**

1. 新增 `GET /api/auth/sso-callback`，以科技平台 token 换取本地登录态。
2. 新增 `GET /api/users/me`，供前端判定本地登录态。
3. 引入本地 JWT + Cookie 认证过滤器。
4. `/api/equity/**` 改为从认证上下文获取当前会员，不再接收显式 `memberId`。
5. `/api/callbacks/**` 保持签名鉴权。
6. 下线旧的 `/api/users/register` 静默注册入口。

**验证结果**

1. `mvn -q test` 通过。
2. `mvn -q clean package -DskipTests` 通过。
3. `mvn -q checkstyle:check` 通过。
4. MySQL 写库与回调回归通过。

**对应提交 / 文档**

1. `48e9ad6 feat: implement cross-domain auth flow`
2. `docs/日报/20260324.md`
3. `docs/CODE_MODIFY_20260323.md`

### 3.4 本地 MySQL schema 漂移导致真实回归失败

**发现时间**

- 2026-03-24

**问题现象**

- H2 和编译通过，但真实 MySQL 回归首次执行失败。
- 本地复用的 `nexusfin_equity.member_info` 缺少 `tech_platform_user_id` 列。

**影响**

1. 说明本地存量数据库与当前代码 schema 演进不同步。
2. 仅依赖 H2 或编译无法发现真实环境兼容性问题。
3. 影响真实写库回归和交付可信度。

**处理动作**

1. 根据失败 SQL 锁定根因。
2. 在 `MySqlRoundTripIntegrationTest` 中增加测试前置 schema 对齐逻辑。
3. 在不重建本地数据库的前提下，增量补齐缺失列和唯一索引。
4. 重新执行 MySQL 集成测试确认回归通过。

**验证结果**

1. `MYSQL_IT_ENABLED=true MYSQL_IT_DATABASE=nexusfin_equity mvn -Dtest=MySqlRoundTripIntegrationTest,MySqlCallbackFlowIntegrationTest test` 通过。

### 3.5 SSO 建档敏感信息处理不合规

**发现时间**

- 2026-03-26

**问题现象**

- 登录建档链路中的手机号、身份证、姓名等敏感字段此前未按正式加密方案处理。
- 文档明确指出原方案存在 Base64 伪加密问题。

**影响**

1. 敏感信息安全保护不足。
2. 与“敏感字段必须加密存储、查询使用 hash 索引”的项目要求不一致。
3. 若继续沿用，会形成明显的安全合规缺口。

**处理动作**

1. 新增 `CryptoProperties` 和 `SensitiveDataCipher`。
2. 采用 `AES/GCM/NoPadding` 对手机号、身份证、姓名进行加密存储。
3. 仍保留手机号和身份证的 SHA-256 hash 作为检索索引。
4. 对上游字段按 `channelCode` 做入站解码，并补格式校验。
5. `AuthServiceImpl` 切换到正式加密链路，不再使用 Base64 伪加密。

**验证结果**

1. `SensitiveDataCipherTest` 通过。
2. 聚焦链路测试通过。
3. 全量测试、MySQL 回归、打包、checkstyle 均通过。

**对应提交 / 文档**

1. `8e8e2e2 feat: integrate qw upstream and harden auth flows`

### 3.6 状态字段与 Provider 命名耦合单一供应商，且对存量库不兼容

**发现时间**

- 2026-03-26

**问题现象**

- 代码和数据库中存在 `qw_first_deduct_status`、`qw_fallback_deduct_status`、`qw_exercise_status` 等强绑定齐为的命名。
- `payment_record.channel_name = "QW"` 也把供应商编码写进了通用领域语义。

**影响**

1. 核心领域模型与单一供应商耦合，不利于后续接入多家权益方。
2. Java 模型、API 输出、数据库物理列命名不够中性。
3. 若直接切换 ORM 到新字段，对复用已有本地数据库的场景会产生兼容问题。

**处理动作**

1. 将状态字段收敛为中性命名：
   - `firstDeductStatus`
   - `fallbackDeductStatus`
   - `exerciseStatus`
2. 将订单来源渠道字段改为 `sourceChannelCode`。
3. 将支付流水执行方字段改为 `providerCode`。
4. 在初始化脚本中统一使用新物理列名。
5. 新增 `SchemaMigrationRunner`，启动时对旧列进行自动迁移，兼容存量库。

**验证结果**

1. 聚焦链路测试通过。
2. 全量测试通过。
3. MySQL 回归通过。
4. 打包和 checkstyle 通过。

**对应提交 / 文档**
