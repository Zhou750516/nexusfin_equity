截至 `2026-04-02`，对 `515 会员卡未生效` 的最合理判断如下：

2. 该返回并不表示开卡失败
3. 该返回也不表示 `abs.token.get` 查询参数本身错误
4. 该返回是齐为业务语义下对“首扣未成功”的外层提示
5. 首扣未成功的核心根因是：我方传入的 `payProtocolNo` 不正确

可以概括为：

> `515` 是业务结果提示，根因不在卡状态本身，而在首扣使用的支付协议号。

## 5. 当前项目内的影响点

今天同时确认，当前项目内部也存在一个关键设计问题：

1. 项目尚未正式接通通联签约流程
2. 因而当前没有真实稳定的 `payProtocolNo` 来源
3. 如果继续把内部 `agreementNo`、占位值或测试 override 混作支付协议号，就会导致齐为首扣失败

因此该问题已经从“第三方联调异常”上升为“主流程生产闭环缺口”。

影响点主要包括：

1. 下单主流程 `abs.push.order` 的 `payProtocolNo` 取值来源
2. 用户维度真实支付协议号的存储模型
3. 订单维度协议号快照留痕
4. 测试 override 与正式环境边界隔离

## 6. 今日已采取措施

围绕该问题，今天已完成以下处理：

1. 新增用户支付协议号模型与仓储能力，承接真实支付协议号
2. 新增 `PaymentProtocolService`，统一解析主流程协议号来源
3. 改造下单主流程，优先读取真实协议号
4. 在订单中增加：
   - `pay_protocol_no_snapshot`
   - `pay_protocol_source`
5. 增加测试 override 的环境边界保护，避免正式环境误用 mock 口径
6. 新增真实协议号写入入口：
   - `POST /api/callbacks/payment-protocol`
