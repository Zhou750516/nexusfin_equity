## 3. 当前代码与协议口径

### 3.1 当前实现的关键行为

当前标准 HTTP 模式由 `QwBenefitClientImpl` 承担，关键行为如下：

1. `timestamp` 使用 `System.currentTimeMillis()` 生成
2. `sign` 按 `MD5(method + partnerNo + timestamp + version + signKey)` 计算
3. `requestBody` 先将业务对象序列化为 JSON
4. 再使用 `AES/GCM/NoPadding` 加密
5. 最终将 `iv + ciphertext` 做 Base64，作为 `requestBody`

### 3.2 当前文档中已经明确的不确定项

仓库现有文档已明确指出，齐为标准 HTTP 协议仍存在未完全确认的实现细节：

1. `partnerNo` 的正式值和测试值仍需要确认
2. AES 的具体密钥口径、加密模式、IV 规则、填充方式，材料并未给全
3. 齐为测试网关地址、`methodPath`、协议版本口径仍需以对方最终确认为准

因此，当前代码虽然已经具备真实出站能力，但并不能说明已经与齐为最终协议完全对齐。

## 4. 对 `504 数据协议非法` 的判断

本次 `504` 不是网络错误，更像是齐为在协议解析阶段直接拒绝了请求。

原因是：

1. 如果网络不通，通常不会拿到齐为标准 JSON 响应
2. 如果只是普通业务前置条件不满足，更常见的是“订单不存在”“用户不存在”“未开卡”等业务类提示
3. 现在返回的是“数据协议非法”，更接近：
   - 签名不通过
   - AES 解密失败
   - 请求体结构不符合齐为要求
   - 时间戳或版本字段口径不符合要求

## 5. 高概率原因排序

### 5.1 原因一：AES 报文口径不一致（最高概率）

这是当前最值得优先怀疑的点。

原因如下：

1. 当前实现固定使用 `AES/GCM/NoPadding`
2. 本次真实请求也是按 `AES/GCM/NoPadding` 构造的
3. 但齐为测试材料只给出了：
   - `md5密钥`
   - `AES密钥：FbRW7iaiwcEKk2kY`
4. 材料没有说明：
   - AES 是 `GCM`、`ECB` 还是 `CBC`
   - IV 是否需要单独传输
   - 密文是否只传 ciphertext，还是 `iv + ciphertext + tag`
   - 密文编码是 Base64 还是十六进制

只要齐为服务端采用的解密口径与我方不同，就极容易返回“数据协议非法”。

### 5.2 原因二：`requestBody` 外层封装格式不对（高概率）

即使 AES 密钥本身没错，也可能是密文外层格式不符合齐为要求。

当前我方发送的是：

1. 业务 JSON 明文
2. AES-GCM 加密
3. `iv + encryptedPayload` 拼接
4. Base64 编码

但齐为材料并未明确说明：

1. 是否允许这样拼接 IV
2. 是否需要把 IV 放在单独字段
3. 是否只传密文部分
4. 是否使用固定字段名或固定排序后的原文串参与加密

若外层格式与对方协议不一致，也会在解析层被直接拒绝。

### 5.3 原因三：签名口径不一致（中等概率）

当前签名算法为：

```text
MD5(method + partnerNo + timestamp + version + signKey)
```

这与我方整理文档中的签名规则是一致的。

但仍然存在以下潜在问题：

1. `partnerNo=abs` 是否确认为当前环境最终值
2. `signKey=h2Zh53keYy8eksijfj7HfPCncmEMCBXt` 是否确认为当前网关对应环境的正式测试值
3. `version=v1.0` 是否仍是当前环境有效值
4. 齐为侧是否对大小写、空格、编码方式有额外要求

若签名校验失败，也可能统一归类为“数据协议非法”。

### 5.4 原因四：`timestamp` 格式不符合要求（中低概率）

当前实现与本次真实调用都使用了毫秒级时间戳字符串。

但现有材料只写了“时间戳”，没有明确：

1. 是否必须是秒级时间戳
2. 是否必须是 10 位而非 13 位
3. 是否存在固定时间窗校验
4. 是否必须使用日期时间格式字符串

如果齐为按不同时间戳格式解析，也可能在协议层直接报错。

### 5.5 原因五：`abs.token.get` 的业务前置条件未满足（较低概率）

`abs.token.get` 往往依赖前置订单、会员或开卡状态已在齐为侧存在。

本次调用使用的是临时构造值：

1. `uniqueId=debug-user-20260401-001`
2. `partnerOrderNo=debug-ord-20260401-001`

这些值未经过 `abs.push.order` 建单链路。

因此，从业务视角看，请求本身大概率也不满足接口前置条件。

但若只是业务数据不存在，更常见的应该是业务类报错，而不是“数据协议非法”。

因此该项不是第一嫌疑。

## 6. 当前最合理的判断

综合日志事实、现有代码和仓库文档，当前最合理的判断是：

1. 请求已经成功到达齐为测试网关
2. 齐为已收到并解析外层 HTTP 请求
3. 失败点大概率出在“签名/加密/请求体封装”这一层
4. 其中最可疑的是 AES 模式和 `requestBody` 编码/拼接规则

可以概括为：

> 当前更像是“协议没有对齐”，而不是“业务数据值本身不对”。

## 7. 建议的验证顺序

建议按以下顺序继续排查：

### 7.1 先向齐为确认 4 个关键点

1. AES 模式到底是 `GCM`、`ECB` 还是 `CBC`
2. `AES密钥` 是原始字符串还是 Base64
3. `requestBody` 要求使用 Base64 还是十六进制
4. `timestamp` 的具体格式与容忍窗口

### 7.2 联调顺序先打 `abs.push.order`

建议不要直接拿 `abs.token.get` 做第一枪长期排障，原因是：

1. `abs.token.get` 依赖前置订单和会员上下文
2. 先打 `abs.push.order` 更容易区分“协议问题”和“业务问题”
3. 待建单成功后，再打 `abs.token.get`，更利于判断问题是否进入业务层

### 7.3 不要默认把 `QW_AES_KEY_BASE64` 当成已确认事实

当前配置名里使用了 `QW_AES_KEY_BASE64`，但齐为测试材料给出的更像是原始 16 位字符串，而不是 Base64 密钥。

这意味着：

1. 变量命名本身带有我方假设
2. 若直接把“原始字符串”和“Base64 字符串”混用，会导致解密口径完全错误

## 13. 对我方代码的直接提示

当前代码中，`payProtocolNo` 的来源是：

1. 优先使用 `agreementNo`
2. 否则回退为 `proto-` + `benefitOrderNo`

而 `agreementNo` 当前是我方本地生成的内部协议号，不一定等于齐为可识别的真实支付协议号。

因此后续若要修复，最优先不是继续改 `abs.token.get`，而是：

1. 确认真实支付协议号来自哪里
2. 确认当前 `agreementNo` 是否被误传
3. 在 `abs.push.order` 请求阶段修正 `payProtocolNo`
