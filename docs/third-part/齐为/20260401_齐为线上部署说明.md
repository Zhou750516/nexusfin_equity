# 2026-04-01 齐为线上部署说明

## 1. 文档目的

本文档用于基于当前仓库内的齐为接入实现和 `2026-04-01` 的真实联调结果，整理一份后续可用于测试环境切换、预发验证、线上部署的参考说明。

本文档只覆盖当前已经进入代码主链的齐为标准 HTTP 模式，不覆盖以下内容：

1. 通联证书直连模式的正式投产切换
2. 齐为回调我方的完整线上联调说明
3. 云卡完整页面交互、借还款编排等独立业务主线

## 2. 当前结论摘要

截至 `2026-04-01`，齐为标准 HTTP 模式的阶段性事实如下：

1. 标准 HTTP 协议已打通，不再返回 `504 数据协议非法`
2. `abs.push.order` 已在齐为测试环境真实成功
3. `abs.token.get` 当前仍返回 `515 会员卡未生效`
4. 当前主要问题已从“协议不通”转为“业务生效时序待确认”

本轮真实联调关键结论：

1. `abs.push.order` 成功返回齐为侧：
   - `orderNo=94260401229898`
   - `cardNo=98260401557343`
   - `openFlag=0`
   - `productCode=abs001`
2. 同一组 `uniqueId + partnerOrderNo` 紧接着调用 `abs.token.get`，返回：
   - `code=515`
   - `msg=会员卡未生效`

因此：

> 目前代码已经具备上线所需的基础协议能力，但 `abs.token.get` 的业务前置状态仍需在上线前与齐为确认。

补充说明：

截至 `2026-04-02`，齐为已进一步确认：

1. `openFlag=0` 表示开卡成功
2. `abs.token.get` 只传 `uniqueId + partnerOrderNo` 即可
3. 不需要额外传 `orderNo / cardNo`
4. 当前 `515 会员卡未生效` 的真正原因，不是时序等待，而是：
   - 扣款失败
   - 扣款失败原因是支付协议号错误

因此当前上线前待解决的重点，已进一步收敛为：

> **确保 `abs.push.order` 里的 `payProtocolNo` 使用真实可扣款的支付协议号，而不是内部占位协议号。**

## 3. 当前接入范围

当前项目齐为接入的主动调用接口有 3 个：

1. `abs.push.order`：会员同步 / 建单 / 开卡
2. `abs.token.get`：获取联登地址
3. `abs.lending.notify`：放款通知

当前代码主路径如下：

1. 标准 HTTP client：`src/main/java/com/nexusfin/equity/thirdparty/qw/QwBenefitClientImpl.java`
2. 配置模型：`src/main/java/com/nexusfin/equity/config/QwProperties.java`
3. 调用路由：`src/main/java/com/nexusfin/equity/thirdparty/qw/RoutingQwBenefitClient.java`
4. 下单链路接线：`src/main/java/com/nexusfin/equity/service/impl/BenefitOrderServiceImpl.java`

当前模式说明：

1. `MOCK`：本地模拟
2. `QWEIMOBILE_HTTP`：当前已真实联调的标准 HTTP 模式
3. `ALLINPAY_DIRECT`：通联直连骨架模式，当前不作为本说明的线上部署主路径

补充说明：

1. 截至 `2026-04-03`，已新增通联正式对接与上线切换说明：
   - `docs/third-part/20260403_通联服务对接流程与上线切换说明.md`
2. 后续凡涉及真实 `payProtocolNo` 来源、回写入口、订单快照、override 下线策略，应以该文档为准

## 4. 当前确认可用的协议口径

结合 Demo 分析、代码修正和真实联调结果，当前已验证通过的齐为标准 HTTP 口径如下。

### 4.1 请求地址

测试环境地址：

```text
https://t-api.test.qweimobile.com/api/abs/method
```

### 4.2 请求结构

统一使用：

```json
{
  "requestHead": {
    "partnerNo": "abs",
    "timestamp": 1775048122861,
    "method": "abs.push.order",
    "version": "v1.0",
    "sign": "md5签名"
  },
  "requestBody": "十六进制密文"
}
```

### 4.3 签名规则

签名原文：

```text
method + partnerNo + timestamp + version + signKey
```

签名算法：

```text
MD5
```

输出格式：

```text
32位小写十六进制串
```

### 4.4 加密规则

当前已通过真实联调验证的加密口径为：

1. AES 模式：`AES/ECB/PKCS5Padding`
2. AES 密钥口径：原始 16 位字符串
3. 密文编码：十六进制字符串

这也是当前线上部署应采用的默认协议口径。

## 5. 线上部署所需配置

### 5.1 推荐环境变量

若后续切换到真实非 MOCK 模式，建议至少配置以下变量：

```bash
export QW_ENABLED='true'
export QW_MODE='QWEIMOBILE_HTTP'
export QW_BASE_URL='待齐为提供正式线上地址'
export QW_METHOD_PATH='/api/abs/method'
export QW_PARTNER_NO='待齐为提供正式线上partnerNo'
export QW_VERSION='v1.0'
export QW_SIGN_KEY='待齐为提供正式线上md5密钥'
export QW_AES_KEY='待齐为提供正式线上AES密钥'
export QW_AES_KEY_ENCODING='RAW'
export QW_AES_ALGORITHM='AES/ECB/PKCS5Padding'
export QW_CIPHERTEXT_ENCODING='HEX'
export QW_CONNECT_TIMEOUT_MS='2000'
export QW_READ_TIMEOUT_MS='3000'
export QW_DEFAULT_PAY_PROTOCOL_PREFIX='proto-'
```

### 5.2 配置项与当前代码映射

当前默认配置入口位于：

- `src/main/resources/application.yml`

当前代码已经支持的关键配置项如下：

1. `partner-no`
2. `sign-key`
3. `aes-key`
4. `aes-key-encoding`
5. `aes-algorithm`
6. `ciphertext-encoding`
7. `connect-timeout-ms`
8. `read-timeout-ms`
9. `default-pay-protocol-prefix`

### 5.3 当前默认值说明

当前仓库默认值偏联调口径，不应直接视为线上正式值：

1. `partner-no=abs`
2. `aes-key-encoding=RAW`
3. `aes-algorithm=AES/ECB/PKCS5Padding`
4. `ciphertext-encoding=HEX`

其中以下内容在上线前必须替换成齐为正式提供的生产参数：

1. `QW_BASE_URL`
2. `QW_PARTNER_NO`
3. `QW_SIGN_KEY`
4. `QW_AES_KEY`

## 6. 业务链路上的部署影响

### 6.1 下单链路

当前业务上，齐为会员同步会挂在本地下单成功之后。

对应主入口：

- `POST /api/equity/orders`

当前下单后对齐为传输的重点字段包括：

1. `uniqueId`
2. `partnerOrderNo`
3. `payAmount`
4. `productCode`
5. `productName`
6. `mobile`
7. `username`
8. `payProtocolNo`
9. `shareFlag`

其中：

1. `partnerOrderNo` 建议稳定使用本地 `benefitOrderNo`
2. `payProtocolNo` 当前默认来自协议号；若为空，则按 `proto-` 前缀拼接本地订单号
3. 测试联调阶段已补充一个临时 override：
   - `QW_MEMBER_SYNC_PAY_PROTOCOL_NO_OVERRIDE`
   - 仅用于联调验证齐为链路，不能当成正式上线方案

需要特别注意：

1. 这个当前默认口径已经被真实联调证明存在风险
2. 齐为已明确反馈：当前扣款失败原因是支付协议号错误
3. `2026-04-02` 已使用齐为提供的 mock 协议号完成真实联调，`abs.token.get` 已返回可用 token
4. 因此上线前不能继续默认把本地 `agreementNo` 或 `proto-` 前缀值直接视为真实支付协议号
5. 正式上线前必须切回“真实通联协议号来源”，不能继续依赖 override

上线切换建议：

1. 联调阶段可暂时保留 `QW_MEMBER_SYNC_PAY_PROTOCOL_NO_OVERRIDE`
2. 预发 / 生产切换前，应先完成通联真实协议号回写接入
3. 预发 / 生产环境不得继续通过 override 或 mock 协议号支撑齐为首扣

### 6.2 联登链路

当前业务上，联登地址获取走：

- `abs.token.get`

但根据当前前一轮真实联调结果，开卡成功后立即请求联登曾遇到：

```text
515 会员卡未生效
```

截至 `2026-04-02`，该问题已经进一步收口：

1. `515` 的根因不是时序，也不是 `orderNo / cardNo`
2. 根因是 `abs.push.order` 阶段使用了错误的 `payProtocolNo`
3. 在切换为齐为提供的 mock 协议号后，`abs.token.get` 已真实返回 `200 / OK`
4. 当前参数仍可继续使用 `uniqueId + partnerOrderNo`

### 6.3 放款通知链路

当前代码已经具备 `abs.lending.notify` 主动调用入口，且截至 `2026-04-02` 已完成首轮真实联调验证。

本轮真实联调结果已确认：

1. `abs.lending.notify` 已真实打到齐为测试环境
2. 返回 `HTTP 200`
3. 原始响应：
   - `{"code":200,"msg":"OK","data":"8c82f7602d98cbb7a6b8a9beaacffadd5cebf0a221a82696ec7435de571c1b7f"}`
4. 解密后：
   - `{"orderNo":"94260401229898"}`

因此在正式上线前，仍建议继续补：

1. 失败样例联调
2. 超时 / 异常日志留痕验证
3. 与真实放款主流程联动的一次完整链路留痕

## 7. 推荐部署步骤

建议按以下顺序推进。

### 7.1 第一步：完成正式参数确认

上线前先向齐为确认正式环境信息：

1. 正式网关地址
2. 正式 `partnerNo`
3. 正式 `signKey`
4. 正式 `AES_KEY`
5. `methodPath` 与版本是否仍为当前口径
6. `abs.token.get` 的生效时序要求

### 7.2 第二步：先在测试 / 预发环境切到真实 HTTP 模式

建议不要直接从 `MOCK` 跳正式生产。

推荐顺序：

1. 测试环境切 `QWEIMOBILE_HTTP`
2. 预发环境重复验证
3. 生产环境灰度切换

### 7.3 第三步：按接口顺序做冒烟

推荐线上前冒烟顺序：

1. `abs.push.order`
2. `abs.token.get`
3. `abs.lending.notify`

原因：

1. `abs.push.order` 是业务起点
2. `abs.token.get` 依赖前置开卡状态
3. `abs.lending.notify` 通常在更后业务节点触发

### 7.4 第四步：保留完整证据

每次环境切换至少保留：

1. 请求关键报文
2. 响应关键报文
3. 本地业务单号
4. 齐为侧返回 `orderNo` / `cardNo`
5. 应用日志中的 `traceId + bizOrderNo`

## 9. 回滚建议

若线上切换后出现异常，建议按以下优先级回滚：

1. 优先将 `QW_MODE` 切回 `MOCK`
2. 或将 `QW_ENABLED=false`，避免影响主业务链继续放大
3. 保留出错时的完整请求、响应和 traceId
4. 不建议在未保留证据的情况下直接覆盖配置
