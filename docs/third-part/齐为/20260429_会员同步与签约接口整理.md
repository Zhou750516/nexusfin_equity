# 2026-04-29 齐为会员同步与签约接口整理

> 本文件由 `20260421_会员同步与签约接口整理.md` 增量更新而来，对应齐为 `2026-04-29` 版本接口文档。0421 版本仅作历史快照参考。

## 1. 文档目的

本文档用于从 `third-part/齐为/艾博生 & 齐为权益接口文档-20260429.docx` 中提取本轮需要重点关注的接口内容，整理为一份独立接口文档，供艾博生侧后续方案评估、字段映射和接口承接使用。

本次整理范围仅包括：

1. 会员同步接口
2. 签约查询接口
3. 签约申请接口
4. 签约确认接口

## 1.1 本次相对 0421 版本的变更概要

齐为侧 `2026-04-24` 调整了签约查询、签约申请、签约确认、会员同步接口字段，主要变化：

1. **会员同步接口**：删除 `mobile / username / accountNo` 三个四要素字段，改为以 `userSignId`（签约记录 id）作为关联键，**强依赖签约确认成功**。
2. **签约查询接口**：请求新增 `merchantId`（支付渠道商户号）。
3. **签约申请接口**：请求新增 `merchantId`；业务返回参数明确为 `userSignId / applyTime`。
4. **签约确认接口**：请求字段大幅精简，仅保留 `userSignId + verCode`；业务返回参数明确为 `userSignId / agreementNo`。
5. **签约号口径反转**：0421 版本曾约定"本侧不显式使用签约号"，0429 版本由于 `userSignId` 成为会员同步入参的必填项，**本侧必须持久化 `userSignId` 与 `agreementNo`**。
6. **流程编排**：签约确认成功后，可直接以 `userSignId` 触发会员同步，形成"签约确认 → 会员同步"串联（详见 §7.4）。

### 7.3 本次已确认口径

#### A. 返回结构口径

签约查询、签约申请、签约确认接口的外层返回结构统一参考齐为公共返回结构：

```json
{
  "msg": "成功",
  "code": 200,
  "data": "AES加密后的业务数据"
}
```

`data` 解密后的字段：

- 签约查询：`status`（`0` 未签约，`1` 已签约）
- 签约申请：`userSignId`、`applyTime`
- 签约确认：`userSignId`、`agreementNo`

失败时外层结构不变，`code` 与 `data` 会变化，具体失败码与业务 `data` 结构待联调补充。

#### B. 签约号口径（相对 0421 版本反转）

> ⚠️ 0421 版本约定 "本次不要求显式使用签约号 / 协议号 / 申请流水号"。0429 版本由于 `userSignId` 成为会员同步接口的必填入参，本口径已反转。

当前口径：

1. 签约申请成功后，本侧需缓存或持久化齐为返回的 `userSignId`（用于签约确认入参）。
2. 签约确认成功后，本侧需持久化齐为返回的 `userSignId` 与 `agreementNo`。
3. 后续会员同步以 `userSignId` 作为关联键。

#### C. 本地记录口径

签约确认成功后，本侧持久化齐为返回的关键字段：

| 字段 | 来源 | 用途 |
|---|---|---|
| `userSignId` | 签约确认返回 | 会员同步入参；后续幂等、补偿依据 |
| `agreementNo` | 签约确认返回 | 协议号本地留痕，对账与排查使用 |

仍待确认：

1. 记录到现有 `member_payment_protocol`，还是新增独立签约记录模型；
2. 若记录到 `member_payment_protocol`，字段映射关系如何确定；
3. 记录时点：以"签约确认成功"为准；
4. 后续查询接口是否以本地记录作为辅助判断依据。

### 7.4 流程编排：签约确认 → 会员同步

`0429` 版本将 "签约确认 → 会员同步" 通过 `userSignId` 串联起来。本侧采用 **同业务流程内串行调用，不纳入 DB 事务**：

```
签约确认成功（返回 userSignId, agreementNo）
        │
        ▼
本地独立小事务：落库 userSignId、agreementNo
        │
        ▼
触发会员同步（abs.push.order，以 userSignId 关联）
        │
        ▼
会员同步成功 / 失败 → 由本地记录承接补偿、重试
```

口径说明：

1. **不**将外部 HTTP（齐为会员同步）置于 DB 事务内：避免事务时长不可控，避免"齐为已签约成功但本地无记录"的不可恢复态。
2. 签约确认本地落库与会员同步调用在**同一业务流程**内串行执行；即使会员同步失败，本地已持有 `userSignId / agreementNo`，可异步补偿。
3. 会员同步失败的具体补偿策略作为 TODO（参见 §7.3 D），本轮先保证字段链路与编排骨架。

## 8. 当前结论

本次从齐为 `2026-04-29` 版本接口文档中可明确确认：

1. 会员同步接口字段调整：
   - **删除**：`mobile`、`username`、`accountNo`
   - **新增**：`userSignId`（签约记录 id，必填）
2. 签约 3 个接口字段调整：
   - 签约查询、签约申请：请求新增 `merchantId`
   - 签约申请：返回字段明确为 `userSignId / applyTime`
   - 签约确认：请求精简为 `userSignId + verCode`，返回字段明确为 `userSignId / agreementNo`
3. 签约号口径相对 0421 版本反转：本侧必须持久化 `userSignId` 与 `agreementNo`，作为会员同步与本地补偿的依据。
4. 流程编排：签约确认成功后，在同一业务流程内串行触发会员同步；外部 HTTP 不放在 DB 事务内。
5. 状态流转、重试、幂等、细粒度错误码、会员同步失败补偿策略仍作为 TODO，不影响先做字段映射与接口承接设计。
