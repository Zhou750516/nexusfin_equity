# 2026-04-01 齐为调用 Demo 分析

## 1. 文档目的

本文档用于详细分析以下 Demo 目录中的齐为调用实现：

- `docs/third-part/qw-demo/pom.xml`
- `docs/third-part/qw-demo/src/main/java/org/example/Main.java`
- `docs/third-part/qw-demo/src/main/java/org/example/common/AESHelper.java`
- `docs/third-part/qw-demo/src/main/java/org/example/common/Response.java`
- `docs/third-part/qw-demo/src/main/java/org/example/common/ValidateUtil.java`
- `docs/third-part/qw-demo/src/main/java/org/example/pojo/DemoParams.java`
- `docs/third-part/qw-demo/src/main/java/org/example/pojo/ReqData.java`
- `docs/third-part/qw-demo/src/main/java/org/example/pojo/ReqHead.java`

本文档目标不是复述源码，而是抽取对当前项目真正有用的结论：

1. Demo 实际是如何构造齐为请求的
2. Demo 使用的签名、加密、编码、响应解析口径是什么
3. Demo 与当前项目实现有哪些关键差异
4. 后续若要参考 Demo 修正当前项目，应优先改哪些点

## 2. Demo 目录结构与角色

Demo 目录本质上是一个独立的 Maven 小工程，而不是当前 Spring Boot 主工程的一部分。

目录中的文件职责如下：

| 文件 | 作用 |
| --- | --- |
| `pom.xml` | 定义 Demo 依赖，主要使用 `hutool`、`fastjson`、`lombok`、`hibernate-validator` |
| `Main.java` | 演示一次完整的请求构造、发送、响应解析流程 |
| `AESHelper.java` | 提供 AES 加解密能力 |
| `Response.java` | 定义齐为通用响应对象 |
| `ValidateUtil.java` | 提供 Bean Validation 校验能力 |
| `DemoParams.java` | 定义业务请求体示例字段 |
| `ReqData.java` | 定义外层请求结构：`requestHead + requestBody` |
| `ReqHead.java` | 定义请求头：`partnerNo / timestamp / method / version / sign` |

结论：

**这个 Demo 已经完整体现了齐为标准 HTTP 协议的一条最小调用链，真正值得参考的是协议口径，而不是其依赖选型。**

## 3. Demo 的真实调用流程

从 `Main.java` 可以还原出如下调用步骤：

### 3.1 准备固定参数

Demo 在代码中直接写死了：

1. 接口地址 `API_URL`
2. `MD5_KEY`
3. `AES_KEY`
4. 示例业务参数 `DemoParams`
5. 示例请求头 `ReqHead`

这说明 Demo 的目标是“最短路径演示如何调通”，而不是“工程化落地”。

### 3.2 组装业务参数

`DemoParams` 中定义了 4 个字段：

1. `uniqueId`
2. `payAmount`
3. `productCode`
4. `phone`

这些字段会先被序列化成 JSON 字符串，随后整体加密。

注意：

1. Demo 中提供了 `checkParams()` 能力
2. 但 `Main.java` 实际并没有调用 `checkParams()`

这意味着：

**校验能力只是配套工具，真实调用链里并未强制执行。**

### 3.3 组装请求头

`ReqHead` 包含以下字段：

1. `partnerNo`
2. `timestamp`
3. `method`
4. `version`
5. `sign`

其中签名规则在源码注释和 `Main.java` 中都写得很明确：

```text
Md5(String.format("%s%s%s%s%s", method, partnerNo, timestamp, version, key))
```

因此签名顺序是固定的：

1. `method`
2. `partnerNo`
3. `timestamp`
4. `version`
5. `MD5_KEY`

这个结论与当前项目已有整理文档是一致的。

### 3.4 加密业务体

Demo 通过 `AESHelper.EncryptByKey(AES_KEY, JSON.toJSONString(params))` 对业务 JSON 进行加密。

`AESHelper` 中的实际实现非常关键：

1. AES 模式：`AES/ECB/PKCS5Padding`
2. 密钥要求：**原始 16 位字符串**
3. 输出编码：**十六进制字符串**

这里有 3 个必须强调的结论：

1. Demo **不是** `AES/GCM`
2. Demo **不是** Base64 输出
3. Demo 的密钥 **不是** Base64 再解码，而是直接用 UTF-8 字节作为 AES key

换句话说，Demo 的 `requestBody` 实际是：

```text
AES-ECB-PKCS5Padding(业务JSON) -> hex 字符串
```

这点对当前项目后续修正非常关键。

### 3.5 组装外层请求

最终请求对象 `ReqData` 只有两个字段：

1. `requestHead`
2. `requestBody`

符合齐为公共协议的“外层头 + 加密体”结构。

### 3.6 发送 HTTP 请求

Demo 通过 `hutool` 发起：

1. `POST`
2. `Content-Type: application/json`
3. body 为整个 `ReqData` 的 JSON

Demo 中没有增加其他 header，也没有额外签名头。

这说明标准 HTTP 口径下，齐为更像是：

**所有鉴权信息都在 JSON body 的 `requestHead` 中，而不是 HTTP header 中。**

### 3.7 解析响应

Demo 将响应反序列化为：

1. `code`
2. `msg`
3. `data`

当且仅当：

1. `code == 200`
2. `data` 非空

才会进一步调用 `AESHelper.DecryptByKey(AES_KEY, data)` 解密 `data`。

这说明 Demo 的响应处理策略是：

1. 先看业务码
2. 成功时才尝试解密 `data`
3. 解密口径与请求体一致，仍然是 `AES/ECB/PKCS5Padding + hex`

## 4. Demo 中体现出的协议口径

综合源码，可以提炼出以下协议事实。

### 4.1 外层协议

请求结构：

```json
{
  "requestHead": {
    "partnerNo": "...",
    "timestamp": 1234567890,
    "method": "...",
    "version": "v1.0",
    "sign": "md5..."
  },
  "requestBody": "十六进制密文"
}
```

响应结构：

```json
{
  "code": 200,
  "msg": "success",
  "data": "十六进制密文"
}
```

### 4.2 签名协议

签名算法：

- `MD5`

签名原文：

```text
method + partnerNo + timestamp + version + md5Key
```

输出形式：

- 32 位小写 MD5 十六进制串

### 4.3 加密协议

加密算法：

- `AES/ECB/PKCS5Padding`

密钥口径：

- 原始 16 位字符串

密文编码：

- hex

这三个点一起构成了 Demo 的真实协议语义，不能拆开理解。

## 5. Demo 的限制与注意事项

虽然 Demo 很有参考价值，但它本身也有明显限制。

### 5.1 它更像协议示例，不是生产实现

主要原因：

1. 密钥写死在 `Main.java`
2. URL 写死在代码里
3. 没有抽象 client 接口
4. 没有统一异常模型
5. 没有日志、traceId、bizOrderNo 等业务可观测性

因此不能直接把 Demo 整体搬进当前主工程。

### 5.2 它调用的是 Demo 网关，不是当前项目正在使用的正式测试地址

`Main.java` 里的地址是：

```text
http://t-api.test.qweimobile.com/api/partner/demo/method
```

而当前艾博生测试环境文档里给出的地址是：

```text
https://t-api.test.qweimobile.com/api/abs/method
```

这说明 Demo 有两个特殊点：

1. path 是 `/api/partner/demo/method`
2. 协议是 `http`，不是 `https`

因此不能把 Demo 的 URL 原样当成艾博生当前联调地址。

### 5.3 它演示的方法和商户号也不是当前项目口径

Demo 中用的是：

1. `partnerNo = jdm`
2. `method = jdm.active.notify`

而当前艾博生项目使用的是：

1. `partnerNo = abs`
2. `abs.push.order`
3. `abs.token.get`
4. `abs.lending.notify`

因此真正该参考的是：

1. 头部字段结构
2. 签名规则
3. AES 模式
4. 密文编码方式

而不是业务字段名本身。

## 6. 与当前项目实现的关键差异

这是本文最重要的部分。

### 6.1 当前项目已经具备的正确方向

当前项目 `QwBenefitClientImpl` 已经做对了以下几件事：

1. 已抽象出统一的 `QwBenefitClient`
2. 已把 `requestHead + requestBody` 外层结构封装起来
3. 已按 `MD5(method + partnerNo + timestamp + version + signKey)` 生成签名
4. 已按接口动作封装 `syncMemberOrder / getExerciseUrl / notifyLending`
5. 已把响应统一解析为 `code / msg / data`

也就是说：

**当前项目在“协议外形”和“调用分层”上，方向基本是对的。**

### 6.2 当前项目与 Demo 最大的差异：AES 口径完全不同

当前项目 `QwBenefitClientImpl` 现在使用的是：

1. `AES/GCM/NoPadding`
2. key 通过 `Base64.getDecoder().decode(aesKeyBase64)` 获取
3. `requestBody` 输出为 `Base64(iv + ciphertext)`
4. 响应解密时按同样的 `Base64 + GCM` 口径处理

而 Demo 使用的是：

1. `AES/ECB/PKCS5Padding`
2. key 直接取 16 位原始字符串字节
3. `requestBody` 输出为 hex
4. 响应 `data` 也按 hex 解密

这不是“小差异”，而是**协议不兼容**。

只要对方服务端按 Demo 口径实现，而我方按当前 `GCM + Base64` 口径发，就会在协议层直接失败。

### 6.3 当前项目的 `QW_AES_KEY_BASE64` 命名与 Demo 口径不一致

当前配置名是：

- `QW_AES_KEY_BASE64`

但 Demo 的密钥口径是：

- 原始 16 位字符串，例如 `FCiA8fNy6zjH4cHt`

艾博生测试环境文档中给的也是这种形式：

- `AES密钥：FbRW7iaiwcEKk2kY`

这意味着：

1. 当前项目配置名自带“Base64 假设”
2. 但对方资料和 Demo 都更像“plain 16-char key”
3. 如果继续按 `Base64` 理解，极容易造成解密失败或“数据协议非法”

### 6.4 当前项目对响应 `data` 的解密策略需要兼容 Demo 口径

当前项目会在 `code == 200` 后强制读取并解密 `data`。

这个大方向与 Demo 一致，但具体处理上还应注意：

1. `data` 是 hex，不是 Base64
2. `AESHelper` 使用的是 ECB，不是 GCM
3. 只有 `data` 非空时才应继续解密

因此，当前项目的响应解密逻辑应按 Demo 口径收敛。

### 6.5 当前项目的 URL 需要和环境、模式一起配置，不应把 Demo URL 当成正式地址

Demo 地址只适合说明“调用方式”，不适合作为艾博生正式测试地址。

当前项目应继续保留：

1. `baseUrl`
2. `methodPath`
3. `partnerNo`
4. `method`

这些作为配置或业务映射项，而不是硬编码成 Demo 值。

## 7. 对当前项目代码修改的直接启发

基于 Demo，后续若要修正当前项目，建议优先考虑以下方向。

### 7.1 优先修正 AES 策略

这是第一优先级。

建议：

1. 把 `QwBenefitClientImpl.encodeBusinessData()` 从当前 `GCM + Base64` 调整为可配置实现
2. 优先支持：
   - `AES/ECB/PKCS5Padding`
   - raw 16-char key
   - hex 输出
3. `decodeBusinessData()` 同步按相同口径实现

否则即使签名正确，也大概率会继续报协议错误。

### 7.2 把 AES key 配置从“Base64 假设”改成“支持原始字符串”

建议至少二选一：

1. 把 `QW_AES_KEY_BASE64` 更名为更中性的配置
2. 或同时支持两种来源：
   - `plainTextAesKey`
   - `base64AesKey`

但从 Demo 和现有测试材料看，**原始 16 位字符串**更像真实口径。

### 7.3 明确密文编码方式

当前项目必须显式支持：

1. `HEX`
2. `BASE64`

并将其做成配置项，而不要继续把“Base64”写死在实现里。

对齐 Demo 时，应该使用：

- `HEX`

### 7.4 保留现有调用分层，不要把 Demo 的 `Main.java` 风格直接搬进主链

当前项目已经有更合理的结构：

1. `QwBenefitClient`
2. `QwBenefitClientImpl`
3. `QwProperties`
4. `RoutingQwBenefitClient`

所以推荐做法是：

1. 参考 Demo 的协议口径
2. 修改当前 `QwBenefitClientImpl`
3. 不要引入 `hutool + fastjson` 作为主链依赖
4. 不要把 `Main.java` 式硬编码调用搬进生产代码

### 7.5 增加一个“Demo 协议兼容模式”会更安全

从演进角度看，最稳妥的方式不是一次性硬改，而是：

1. 在 `QwProperties` 中显式增加：
   - AES 模式
   - key 类型
   - 密文编码
2. 或者单独增加一个 `QWEIMOBILE_DEMO_COMPAT` 风格的兼容模式
3. 让当前 `HTTP/QWEIMOBILE_HTTP` 在需要时切到 Demo 口径

这样便于联调期间快速切换，而不会反复改业务代码。

## 8. 建议的代码修改落点

如果后续要按本文修正当前项目，优先改这些位置：

| 文件 | 建议修改点 |
| --- | --- |
| `src/main/java/com/nexusfin/equity/thirdparty/qw/QwBenefitClientImpl.java` | 调整 `encodeBusinessData()`、`decodeBusinessData()`、`buildSecretKey()` 的实现口径 |
| `src/main/java/com/nexusfin/equity/config/QwProperties.java` | 补充 AES 模式、密钥类型、密文编码等配置，不要把 Base64 假设写死 |
| `src/main/resources/application.yml` | 提供与 Demo 协议兼容的配置模板 |
| `src/test/java/com/nexusfin/equity/thirdparty/qw/*` | 增加 Demo 口径的加解密与请求构造测试，避免再次偏离 |

如果只允许做最小修改，则应优先处理：

1. AES 模式
2. AES key 口径
3. 密文编码

## 9. 当前最重要的结论

结合 Demo 与当前项目实现，最重要的结论是：

1. 当前项目的请求头结构和签名方向基本正确
2. 当前项目最可疑、也最需要优先修正的是 **AES 加解密与密文编码口径**
3. Demo 明确展示的是：
   - `AES/ECB/PKCS5Padding`
   - 16 位原始字符串 key
   - hex 密文
4. 当前项目如果继续使用：
   - `AES/GCM/NoPadding`
   - Base64 key
   - Base64 密文
   
   就很可能与齐为真实实现不兼容

可以概括为一句话：

**当前项目后续若要参考 Demo 修正代码，重点不是改签名，而是把加密口径从“GCM + Base64”收敛到 Demo 所体现的“ECB + 原始16位key + hex”。**

## 10. 后续建议

建议后续按以下顺序推进：

1. 先以本分析文档为依据，确认当前项目是否要兼容 Demo 协议口径
2. 若确认兼容，先改 `QwProperties` 和 `QwBenefitClientImpl`，不要先改业务层
3. 先补单测，覆盖：
   - Demo 示例请求体加密结果格式
   - Demo 示例响应体解密
   - 请求头签名规则
4. 再进行一次真实联调调用，验证是否还会出现“数据协议非法”

如果需要继续推进，下一步最值得做的是：

**基于本文把当前 `QwBenefitClientImpl` 改成“兼容 Demo 协议”的实现，并补对应测试。**
