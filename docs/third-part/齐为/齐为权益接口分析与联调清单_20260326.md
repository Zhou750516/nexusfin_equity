# 齐为权益接口分析与联调清单

文档日期：2026-03-26

来源文档：`docs/third-part/齐为/齐为权益接口文档.pdf`

适用范围：仅整理当前 `nexusfin-equity` 项目本次需要使用、或需要为联调预留的齐为接口，不把 PDF 中的内容原样全量搬运。

## 1. 结论

基于当前项目代码边界与齐为 PDF，本次项目需要关注的齐为接口分为两类：

1. 我方主动调用齐为的接口，共 3 个。
2. 齐为回调我方的接口，共 3 个。

本次真正需要纳入联调清单的接口如下：

| 方向 | 齐为接口/能力 | method | 是否本次需要 | 说明 |
| --- | --- | --- | --- | --- |
| 我方 -> 齐为 | 会员同步接口 | `abs.push.order` | 是 | 用于下单后把会员、产品、支付协议等信息同步到齐为，并获取齐为侧订单号/卡号 |
| 我方 -> 齐为 | 获取联登 URL | `abs.token.get` | 是 | 用于用户跳转齐为行权页 |
| 我方 -> 齐为 | 放款通知 | `abs.lending.notify` | 是 | 用于放款成功/失败后通知齐为扣卡费或停止权益 |
| 齐为 -> 我方 | 扣款通知 | 无单独 method，属于齐为回调 | 是 | 用于通知首扣或兜底扣款结果 |
| 齐为 -> 我方 | 用户行权通知 | 无单独 method，属于齐为回调 | 是 | 用于通知用户在齐为侧已行权 |
| 齐为 -> 我方 | 退款通知 | 无单独 method，属于齐为回调 | 是 | 用于通知退款处理结果 |

补充判断：

- 当前项目自己的 `SSO` 登录回调来自科技平台，不属于齐为接口范围。
- 当前项目的 `grant/forward`、`repayment/forward` 是内部业务回调入口，不是齐为 PDF 中定义的合作方回调接口。
- 齐为 PDF 末页只明确列出了 3 个“权益平台侧接口方法枚举”，因此我方主动调用齐为时，目前以这 3 个接口为主。

## 2. 齐为公共调用协议

无论调用哪个齐为主动接口，都需要遵循同一套公共协议。

### 2.1 请求结构

请求体外层结构：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `requestBody` | 是 | 业务参数 JSON 字符串做 AES 加密后的密文 |
| `requestHead.partnerNo` | 是 | 商户编号 |
| `requestHead.timestamp` | 是 | 时间戳 |
| `requestHead.method` | 是 | 接口方法名 |
| `requestHead.version` | 是 | 固定值 `v1.0` |
| `requestHead.sign` | 是 | 签名值 |

### 2.2 签名规则

签名原文：

```text
Md5(String.format("%s%s%s%s%s", method, partnerNo, timestamp, version, key))
```

关键点：

- 签名字段顺序固定为 `method -> partnerNo -> timestamp -> version -> key`
- 响应报文不做签名校验
- 响应只需要解密 `data`

### 2.3 响应结构

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `code` | 是 | 成功 `200`，失败 `500`，其余业务码见附录 |
| `msg` | 是 | 中文描述 |
| `data` | 否 | AES 加密后的业务数据 |

### 2.4 当前项目与齐为协议的差异

当前项目还没有落地齐为专用协议适配层，存在以下差异：

1. 当前回调验签是 `X-App-Id / X-Timestamp / X-Nonce / X-Signature` 头部模式，不是齐为 PDF 中的 `requestHead + requestBody` 模式。
2. 当前项目内部接口请求体是普通 JSON DTO，不是 AES 包裹报文。
3. 当前代码里还没有齐为专用 HTTP Client，也没有对 `partnerNo`、`key`、AES 加解密报文做统一封装。

结论：

- 联调前必须补齐“齐为协议适配层”，不能直接拿现有内部 DTO 与齐为测试地址对接。

## 3. 本次需要主动调用齐为的接口

## 3.1 会员同步接口

接口用途：

- 合作方同步会员信息到齐为权益平台。
- 按当前项目业务，这个接口本质上不仅是“同步会员”，还承担“下游开卡/建单”的职责。

齐为测试地址：

```text
https://t-api.test.qweimobile.com/api/abs/method
```

调用方式：

| 项目 | 内容 |
| --- | --- |
| `method` | `abs.push.order` |
| HTTP | `POST` |
| Content-Type | `application/json` |
| 报文格式 | 外层 `requestHead + requestBody` |

业务入参重点字段：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `uniqueId` | 是 | 会员唯一标识 |
| `partnerOrderNo` | 是 | 合作方订单号，要求唯一 |
| `payAmount` | 是 | 支付金额，单位分 |
| `productCode` | 是 | 产品编码 |
| `productName` | 否 | 产品名称 |
| `mobile` | 否 | 会员手机号 |
| `username` | 是 | 用户姓名，用于扣除权益卡费用 |
| `payProtocolNo` | 是 | 支付渠道协议号，用于扣除权益卡费用 |
| `cardNo` | 否 | 为空视为新卡，不为空视为续期 |
| `shareFlag` | 是 | 是否清分，`0` 否，`1` 是 |
| `partnerMerchantNo` | 条件必填 | `shareFlag=1` 时必填 |
| `partnerShareAmount` | 条件必填 | `shareFlag=1` 时必填，单位分 |
| `shareMerchantNo` | 条件必填 | `shareFlag=1` 时必填 |
| `shareAmount` | 条件必填 | `shareFlag=1` 时必填，单位分 |

业务返回重点字段：

| 字段 | 说明 |
| --- | --- |
| `orderNo` | 齐为系统订单号 |
| `cardNo` | 用户卡号 |
| `openFlag` | 开卡结果，`0` 成功，`1` 退款，`2` 开通中 |
| `productCode` | 产品编码 |
| `productName` | 产品名称 |
| `productType` | 产品类型，如 `independence` / `renew` |
| `cardCreatedDate` | 开通日期 |
| `cardExpiryDate` | 失效日期 |

结合当前项目的使用结论：

1. 该接口应映射到本地“创建权益订单”后的下游开卡/建单动作。
2. 当前代码中最接近的本地业务入口是 `POST /api/equity/orders`。
3. 当前 `BenefitOrderServiceImpl.createOrder` 只完成本地落单、协议任务、幂等记录，还没有真正调用齐为。
4. 后续对接时，应把本地 `benefitOrderNo` 作为 `partnerOrderNo` 的核心来源之一，保证双方都能对账。

## 3.2 获取联登 URL

接口用途：

- 用户行权前，我方先向齐为申请联登 URL，再把用户跳转到齐为页面。

齐为测试地址：

```text
https://t-api.test.qweimobile.com/api/abs/method
```

调用方式：

| 项目 | 内容 |
| --- | --- |
| `method` | `abs.token.get` |
| HTTP | `POST` |
| Content-Type | `application/json` |

业务入参重点字段：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `uniqueId` | 是 | 会员唯一标识 |
| `partnerOrderNo` | 是 | 订单流水号，要求唯一 |

业务返回重点字段：

| 字段 | 说明 |
| --- | --- |
| `memberFlag` | 是否已开卡，`0` 是，`1` 否，`2` 开通中 |
| `redirectUrl` | 行权页面地址 |
| `token` | 用户 token |
| `cardCreatedDate` | 开通日期 |
| `cardExpiryDate` | 失效日期 |

结合当前项目的使用结论：

1. 该接口应映射到本地“获取行权 URL”能力。
2. 当前代码中最接近的本地业务入口是 `GET /api/equity/exercise-url/{benefitOrderNo}`。
3. 当前 `BenefitOrderServiceImpl.getExerciseUrl` 只是返回了一个拼接的 mock URL，还没有调用齐为真实联登接口。
4. 联调时应使用订单主键、会员唯一标识去换取齐为 `redirectUrl`，不能继续返回本地假地址。

## 3.3 放款通知

接口用途：

- 当用户放款成功或失败时，我方通知齐为，触发齐为扣除权益卡费用或停止权益卡。

齐为测试地址：

```text
https://t-api.test.qweimobile.com/api/abs/method
```

调用方式：

| 项目 | 内容 |
| --- | --- |
| `method` | `abs.lending.notify` |
| HTTP | `POST` |
| Content-Type | `application/json` |

业务入参重点字段：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `uniqueId` | 是 | 会员唯一标识 |
| `partnerOrderNo` | 是 | 合作方订单号 |
| `status` | 是 | 放款结果，`0` 失败，`1` 成功 |

业务返回重点字段：

| 字段 | 说明 |
| --- | --- |
| `orderNo` | 权益订单号，唯一 |

结合当前项目的使用结论：

1. 该接口应映射到本地“接收放款结果后，再通知齐为”的出站动作。
2. 当前代码中最接近的本地业务入口是 `POST /api/callbacks/grant/forward`。
3. 当前 `NotificationServiceImpl.handleGrant` 只更新本地订单状态、必要时触发兜底代扣，还没有出站调用齐为。
4. 后续接入时，要在“内部放款通知处理成功”后，再调用齐为 `abs.lending.notify`，避免本地状态未落稳就对外发起通知。

## 4. 本次需要接收的齐为回调接口

齐为 PDF 没有给出固定的“齐为回调 URL”，因为这类地址应由我方测试环境提供给齐为。当前项目需要预留或适配以下回调能力。

## 4.1 扣款通知

齐为含义：

- 齐为在收到放款成功通知后开始扣款。
- 若扣款失败，齐为会继续重试，直到成功或卡过期。
- 齐为扣款成功/失败后通知合作方。

齐为回调业务字段重点：

| 字段 | 说明 |
| --- | --- |
| `uniqueId` | 会员唯一标识 |
| `partnerOrderNo` | 合作方订单号 |
| `orderNo` | 齐为系统订单号 |
| `payAmount` | 支付金额，单位分 |
| `productCode` | 产品编码 |
| `payStatus` | `SUCCESS` / `FAIL` |
| `payDoneTime` | 支付完成时间 |
| `payChannelRespSeqno` | 通道流水号 |
| `payTransSeqNo` | 支付流水号 |

我方当前代码对应能力：

| 当前本地接口 | 说明 |
| --- | --- |
| `POST /api/callbacks/first-deduction` | 接收首扣结果 |
| `POST /api/callbacks/fallback-deduction` | 接收兜底代扣结果 |

落地注意点：

1. 齐为 PDF 只有一个“扣款通知”定义，没有天然区分首扣还是兜底扣款。
2. 当前项目内部状态机区分 `FIRST_DEDUCT` 和 `FALLBACK_DEDUCT` 两条路径。
3. 因此联调时必须补一个路由判断机制，常见做法有两种：
   - 通过我方传给齐为的业务上下文，让齐为回调时带回扣款阶段标识。
   - 或者我方收到通知后，按订单当前状态判断这笔回调属于首扣还是兜底。

## 4.2 用户行权通知

齐为含义：

- 用户在齐为平台行权后，齐为回调我方，通知具体权益消费内容。

齐为回调业务字段重点：

| 字段 | 说明 |
| --- | --- |
| `uniqueId` | 会员唯一标识 |
| `partnerOrderNo` | 合作方订单号 |
| `orderNo` | 齐为平台订单号 |
| `rightName` | 权益名称 |
| `rightNo` | 权益编码 |
| `rightPrice` | 官方原价，单位分 |
| `payValue` | 用户支付价格，单位分 |
| `discountedPrice` | 优惠价格，单位分 |
| `buyTime` | 购买时间 |

我方当前代码对应能力：

| 当前本地接口 | 说明 |
| --- | --- |
| `POST /api/callbacks/exercise-equity` | 接收行权回调并更新订单行权状态 |

落地注意点：

1. 当前本地 `ExerciseCallbackRequest` 字段明显比齐为 PDF 中的字段少。
2. 如果后续需要保留完整行权明细用于对账、客服或资损分析，则需要扩展 DTO 与日志留痕结构。

## 4.3 退款通知

齐为含义：

- 可由齐为通知我方退款结果，也可由我方通知齐为退款结果。
- 从本项目当前代码看，主要先关注“齐为回调我方退款结果”这一侧。

齐为回调业务字段重点：

| 字段 | 说明 |
| --- | --- |
| `uniqueId` | 会员唯一标识 |
| `partnerOrderNo` | 订单流水号 |
| `partnerRefundNo` | 合作方退款流水号 |
| `refundDoneTime` | 退款完成时间 |
| `refundAmount` | 退款金额，单位分 |
| `orderCost` | 退款成本，单位分 |
| `fullRefund` | 是否全额退款，`1` 是，`0` 否 |
| `agreementRefund` | 是否协商退款，`1` 是，`0` 否 |
| `cardStatus` | `CANCEL` 已退卡，`KEEP` 权益保留 |
| `refundReason` | 退款原因 |
| `shareFlag` | 是否分账订单 |
| `refundNo` | 齐为退款流水号 |
| `refundStatus` | `SUCCESS` / `FAIL` / `REFUNDING` |
| `failedReason` | 失败原因 |

我方当前代码对应能力：

| 当前本地接口 | 说明 |
| --- | --- |
| `POST /api/callbacks/refund` | 接收退款回调并更新本地退款状态 |

落地注意点：

1. 当前本地 `RefundCallbackRequest` 字段比齐为 PDF 中的退款字段少很多。
2. 如果本期只需要更新订单退款状态，可以先做最小字段适配。
3. 如果后续要做分账退款、资损核对、退款追踪，则必须把 `refundNo`、`orderCost`、`cardStatus`、分账字段补进来。

## 5.2 我方需要提供给齐为的回调地址

## 6. 对当前项目的直接落地建议

为了把齐为接口真正接进当前项目，建议按下面的最小闭环落地：

1. 新增齐为协议适配层，统一处理 `AES` 加解密、`MD5` 签名、`requestHead/requestBody` 封装。
2. 在 `POST /api/equity/orders` 成功后，增加对 `abs.push.order` 的出站调用。
3. 把 `GET /api/equity/exercise-url/{benefitOrderNo}` 改为真实调用 `abs.token.get`，不再返回 mock URL。
4. 在 `POST /api/callbacks/grant/forward` 内部处理成功后，增加对 `abs.lending.notify` 的出站调用。
5. 为齐为回调增加专用 DTO 或适配器，不要直接拿当前内部回调 DTO 生搬硬套。
6. 单独确认“扣款通知如何区分首扣和兜底”这一点，否则会直接影响状态机流转准确性。

## 7. 待确认项

当前仍有几项联调前必须确认的信息，PDF 本身没有给全：

1. `partnerNo` 的正式值和测试值。
2. `key` 的正式值和测试值。
3. AES 的具体密钥、模式、IV 规则、填充方式，PDF 只写了 AES，没有给足够实现细节。
4. 齐为回调到我方时，是否也使用同样的 `requestHead/requestBody` 外层包裹。
5. 扣款通知如何区分“首扣”与“兜底代扣”。
6. 我方测试环境给齐为使用的真实域名。

如果以上几项不明确，代码可以先搭框架，但无法进入稳定联调。
