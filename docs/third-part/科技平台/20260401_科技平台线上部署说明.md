# 2026-04-01 科技平台线上部署说明

## 1. 文档目的

本文档用于基于当前仓库内已经落地的科技平台接入代码，整理一份后续可用于测试环境切换、预发验证、线上部署的参考说明。

本文档的定位是：

> 基于 `2026-04-01` 当天对齐结果整理的“阶段性基线版”，方便后续继续改代码时有一个可追溯的起点。

需要特别说明：

1. 科技平台接口今天刚完成一轮对齐，后续仍会继续更新
2. 当前科技平台相关代码不是最终版，后续仍会继续调整
3. 因此本文档是“当前基线版部署说明”，不是最终定版协议
4. 本文档重点说明当前仓库已经具备的配置、调用口径和上线前待确认项

本文档当前覆盖范围：

1. 科技平台 SSO 登录校验链路中对 `/api/users/me` 的服务端调用
2. 科技平台 3 个主动通知接口的 thirdparty client 基线实现
3. 当前代码对应的配置项、冒烟顺序、上线前确认项

本文档当前不覆盖：

1. 科技平台借款 / 还款完整页面流程的最终线上编排
2. 科技平台真实环境完整联调结论
3. 后续尚未落地的新接口、新字段和新签名口径

## 2. 当前结论摘要

截至 `2026-04-01`，结合当前仓库代码、现有文档以及“今天刚完成接口对齐、后续还会继续调整”这一前提，科技平台相关事实如下：

1. 认证侧已经具备通过科技平台 `/api/users/me` 拉取用户信息的能力
2. thirdparty 层已经具备 3 个主动通知接口的 HTTP client 基线实现
3. 主动通知 client 当前仍主要是“能力底座”，尚未完整挂入最终业务主链
4. 当前仓库中没有科技平台真实测试地址、真实 token 样例和正式加签加密口径
5. 科技平台真实线上部署的主要阻塞点，仍然是外部参数、最终协议确认以及后续接口更新，而不是本地代码骨架缺失

因此：

> 目前仓库已经具备科技平台接入的基础代码框架，但当前文档和代码都只能代表“今天对齐后的阶段性版本”，不能直接视为最终线上定论。

## 3. 当前接入范围

当前项目内与科技平台相关的代码，主要分为两块。

### 3.1 认证 / SSO 侧

当前已经接入主链的认证相关代码如下：

1. 认证配置：`src/main/java/com/nexusfin/equity/config/AuthProperties.java`
2. 科技平台用户查询 client：`src/main/java/com/nexusfin/equity/service/impl/TechPlatformUserClientImpl.java`
3. SSO 入口：`src/main/java/com/nexusfin/equity/controller/AuthController.java`
4. 登录编排：`src/main/java/com/nexusfin/equity/service/impl/AuthServiceImpl.java`
5. Cookie / JWT 配置：`src/main/java/com/nexusfin/equity/util/CookieUtil.java`、`src/main/java/com/nexusfin/equity/util/JwtUtil.java`

当前业务语义：

1. 浏览器从科技平台跳转到我方 `GET /api/auth/sso-callback`
2. 我方服务端再调用科技平台 `/api/users/me`
3. 成功后在本地生成 JWT Cookie
4. 后续页面通过我方 `GET /api/users/me` 和受保护接口继续访问

### 3.2 主动通知侧

当前已经实现的科技平台主动通知基线代码如下：

1. 配置模型：`src/main/java/com/nexusfin/equity/config/TechPlatformProperties.java`
2. HTTP client：`src/main/java/com/nexusfin/equity/thirdparty/techplatform/TechPlatformClientImpl.java`
3. 签名 / 加解密：`src/main/java/com/nexusfin/equity/thirdparty/techplatform/TechPlatformPayloadCodec.java`
4. 请求模型：
   - `src/main/java/com/nexusfin/equity/thirdparty/techplatform/CreditStatusNoticeRequest.java`
   - `src/main/java/com/nexusfin/equity/thirdparty/techplatform/LoanInfoNoticeRequest.java`
   - `src/main/java/com/nexusfin/equity/thirdparty/techplatform/RepayInfoNoticeRequest.java`

当前支持的 3 个 path：

1. `/guide/api/creditStatusNotice`
2. `/guide/api/loanInfoNotice`
3. `/guide/api/repayInfoNotice`

### 3.3 当前模式说明

主动通知 client 当前支持两种模式：

1. `MOCK`：本地模拟成功响应
2. `HTTP`：调用真实科技平台地址

注意：

1. 当前 `TechPlatformClientImpl` 主要还是协议底座
2. 代码检索结果显示，它尚未像齐为主链那样完整接入最终业务编排
3. 所以上线文档里的“主动通知”部分，应理解为“当前可部署的协议基线能力”，而不是“已全部投入生产链路”

## 4. 当前确认的代码口径

结合当前代码和测试，科技平台当前基线协议口径如下。

### 4.1 用户信息校验接口口径

认证侧当前调用方式为：

1. 请求方式：`GET`
2. 目标地址：`TECH_PLATFORM_BASE_URL + TECH_PLATFORM_USER_ME_PATH`
3. 鉴权头：`Authorization: Bearer <techToken>`

当前代码对响应结构的兼容方式：

1. 如果响应存在 `data` 节点，则从 `data` 内取用户信息
2. 如果响应没有 `data` 节点，则直接从根节点取用户信息
3. `userId` 是必填字段
4. `phone`、`realName`、`idCard`、`channelCode` 允许为空

当前认证 client 已支持：

1. 连接超时 / 读取超时配置
2. 最小重试能力
3. 失败日志留痕

### 4.2 主动通知请求结构

主动通知接口当前统一发送：

请求头：

1. `channelId`
2. `timestamp`
3. `sign`
4. `version`

请求体：

```json
{
  "param": "加密后的业务报文"
}
```

### 4.3 当前签名规则

当前代码里的签名串为：

```text
timestamp + param
```

签名算法当前支持：

1. `HMAC_SHA256`：当前默认值
2. `MD5`：可配置切换

其中：

1. `HMAC_SHA256` 使用 `sign-secret` 作为密钥
2. `MD5` 使用 `timestamp + param + signSecret` 计算摘要

注意：

1. 这只是当前仓库的工程实现口径
2. 不代表科技平台正式线上最终一定使用该规则
3. 上线前必须再与科技平台确认

### 4.4 当前加密规则

当前代码里的加密口径为：

1. 默认算法：`AES/ECB/PKCS5Padding`
2. AES 密钥配置方式：`base64`
3. 默认密文编码：`base64`

同时保留了以下兼容能力：

1. 若算法包含 `/GCM/`，则走 GCM 路径
2. GCM 模式下支持 `iv-length-bytes`
3. GCM 模式下支持 `gcm-tag-bits`

注意：

1. 当前默认值更适合作为本地开发和联调准备口径
2. 真实生产参数仍需科技平台确认

### 4.5 当前响应判定规则

主动通知接口当前兼容两类响应：

1. 明文响应：

```json
{
  "code": "0",
  "msg": "ok"
}
```

2. 加密响应：

```json
{
  "param": "..."
}
```

当前成功判定：

1. `code=0`
2. 或 `code=000000`

否则会抛出：

```text
TECH_PLATFORM_REJECTED
```

## 5. 线上部署所需配置

### 5.1 认证侧推荐环境变量

若要启用科技平台 SSO 真实校验链路，建议至少配置：

```bash
export TECH_PLATFORM_BASE_URL='待科技平台提供正式地址'
export TECH_PLATFORM_USER_ME_PATH='/api/users/me'
export TECH_PLATFORM_CONNECT_TIMEOUT_MS='3000'
export TECH_PLATFORM_READ_TIMEOUT_MS='5000'
export TECH_PLATFORM_RETRY_MAX_ATTEMPTS='2'
```

如果线上还要同时控制我方本地登录态，还需要同步确认：

```bash
export AUTH_JWT_ISSUER='待确认'
export AUTH_JWT_SECRET='待替换为正式密钥'
export AUTH_COOKIE_NAME='NEXUSFIN_AUTH'
export AUTH_COOKIE_SECURE='true'
```

### 5.2 主动通知侧推荐环境变量

若后续要从 `MOCK` 切到真实科技平台通知模式，建议至少配置：

```bash
export TECH_PLATFORM_API_ENABLED='true'
export TECH_PLATFORM_API_MODE='HTTP'
export TECH_PLATFORM_API_BASE_URL='待科技平台提供正式地址'
export TECH_PLATFORM_API_CHANNEL_ID='待科技平台提供正式channelId'
export TECH_PLATFORM_API_VERSION='待科技平台确认'
export TECH_PLATFORM_API_SIGN_SECRET='待科技平台提供正式签名密钥'
export TECH_PLATFORM_API_SIGN_ALGORITHM='HMAC_SHA256'
export TECH_PLATFORM_API_AES_KEY_BASE64='待科技平台提供正式AES密钥'
export TECH_PLATFORM_API_AES_ALGORITHM='AES/ECB/PKCS5Padding'
export TECH_PLATFORM_API_GCM_TAG_BITS='128'
export TECH_PLATFORM_API_IV_LENGTH_BYTES='12'
export TECH_PLATFORM_API_CONNECT_TIMEOUT_MS='3000'
export TECH_PLATFORM_API_READ_TIMEOUT_MS='5000'
export TECH_PLATFORM_API_CREDIT_STATUS_NOTICE_PATH='/guide/api/creditStatusNotice'
export TECH_PLATFORM_API_LOAN_INFO_NOTICE_PATH='/guide/api/loanInfoNotice'
export TECH_PLATFORM_API_REPAY_INFO_NOTICE_PATH='/guide/api/repayInfoNotice'
```

### 5.3 配置项与当前代码映射

当前主要配置入口位于：

1. `src/main/resources/application.yml`
2. `src/test/resources/application.yml`

两组配置不要混淆：

1. `nexusfin.auth.*`：影响认证链路里的 `/api/users/me` 上游校验
2. `nexusfin.third-party.tech-platform.*`：影响主动通知科技平台的 thirdparty client

### 5.4 当前默认值说明

当前仓库默认值主要是占位和本地测试值，不可直接视为线上正式值。

尤其以下内容上线前必须替换：

1. `TECH_PLATFORM_BASE_URL`
2. `TECH_PLATFORM_API_BASE_URL`
3. `TECH_PLATFORM_API_CHANNEL_ID`
4. `TECH_PLATFORM_API_SIGN_SECRET`
5. `TECH_PLATFORM_API_AES_KEY_BASE64`
6. `AUTH_JWT_SECRET`

## 6. 业务链路上的部署影响

### 6.1 SSO 登录链路

当前主链入口为：

- `GET /api/auth/sso-callback`

部署影响点：

1. 科技平台前端跳转地址必须准确
2. 科技平台 token 必须可用于服务端调用 `/api/users/me`
3. 我方 `redirectWhitelist` 必须与实际前端落地地址一致
4. Cookie 的 `SameSite`、`Secure`、域名策略必须与正式部署域名匹配

### 6.2 登录态用户接口

当前前端依赖我方接口：

- `GET /api/users/me`

部署影响点：

1. 该接口返回的是我方本地登录态，不是直接透传科技平台响应
2. 其前提是 SSO 回调阶段已经成功换取本地 JWT Cookie
3. 若科技平台 `/api/users/me` 无法访问，SSO 登录链路会直接受影响

### 6.3 主动通知链路

当前 thirdparty client 已经具备以下能力：

1. `notifyCreditStatus`
2. `notifyLoanInfo`
3. `notifyRepaymentInfo`

但需要明确：

1. 当前它还不是完整挂主链状态
2. 若后续代码把它接入放款 / 还款 / 授信流程，本文档需要同步更新
3. 所以上线时应先确认“哪些业务节点真的会触发这些调用”

## 7. 推荐部署步骤

建议按以下顺序推进。

### 7.1 第一步：先确认外部正式口径

上线前建议优先向科技平台确认：

1. `/api/users/me` 正式地址
2. `Bearer Token` 的真实传递方式和样例
3. `/api/users/me` 的真实返回结构和字段名
4. 主动通知接口正式 `baseUrl`
5. 正式 `channelId`
6. 正式 `version`
7. 正式签名算法和签名串规则
8. 正式加密算法、AES key、密文编码方式
9. 成功 / 失败响应码口径
10. 3 个通知接口在最终业务语义下是否仍由“我方主动通知科技平台”

### 7.2 第二步：先在测试 / 预发环境做最小闭环

推荐不要直接跳正式生产，建议顺序：

1. 先验证认证侧 `/api/users/me`
2. 再验证通知侧 `creditStatusNotice`
3. 再验证 `loanInfoNotice`
4. 最后验证 `repayInfoNotice`

原因：

1. SSO 登录链路会直接影响用户是否能进入业务页面
2. 主动通知目前还未完全接入主链，更适合逐个接口确认

### 7.3 第三步：每类接口至少保留一条成功样例

建议至少保留以下证据：

1. 请求 URL
2. 请求 header 关键字段
3. 请求 body 脱敏样例
4. 响应 body 原文
5. 应用日志中的 `traceId`
6. 若已进入业务链，再补齐 `bizOrderNo`

### 7.4 第四步：代码更新后及时回填文档

由于科技平台接口今天刚完成一轮对齐、后续一定还会更新，建议每次协议或链路调整后，至少同步回填：

1. 新增 / 下线接口
2. 新的签名规则
3. 新的加密口径
4. 新增的真实联调结论
5. 新的主链挂载点

## 9. 回滚建议

若线上切换后出现问题，建议按影响范围分别处理。

### 9.1 主动通知异常

优先处理方式：

1. 将 `TECH_PLATFORM_API_MODE` 切回 `MOCK`
2. 或将 `TECH_PLATFORM_API_ENABLED=false`
3. 保留出错请求、响应和应用日志

### 9.2 认证链路异常

当前认证侧没有独立的 `enabled` 开关，因此回滚方式更偏“配置回退 / 流量回退”：

1. 回退 `TECH_PLATFORM_BASE_URL`、`TECH_PLATFORM_USER_ME_PATH` 到上一个稳定版本
2. 临时停止科技平台向我方生产 SSO 跳转导流
3. 保留失败时的 `traceId`、请求时间、目标地址和返回报文

## 10. 当前未闭环事项

当前科技平台上线前仍需重点确认以下事项：

1. 科技平台真实测试 / 正式地址尚未在仓库内闭环
2. `/api/users/me` 真实 token 和真实返回样例尚未确认
3. 主动通知的签名、加密正式口径尚未最终确认
4. 3 个通知接口的最终业务方向和是否继续保留，后续可能调整
5. 今天刚完成对齐的接口口径，后续仍可能继续调整字段、路径或调用方向
