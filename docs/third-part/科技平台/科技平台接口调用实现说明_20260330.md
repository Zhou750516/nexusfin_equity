## 4. 请求与响应实现策略

### 4.1 请求头

客户端统一发送：

- `channelId`
- `timestamp`
- `sign`
- `version`

### 4.2 请求体

业务对象先序列化为 JSON，再经过 AES 加密，最终作为：

```json
{
  "param": "..."
}
```

### 4.3 响应体

当前实现兼容两种响应：

1. 明文：

```json
{
  "code": "0",
  "msg": "ok"
}
```

2. 加密：

```json
{
  "param": "..."
}
```

## 5. 当前关键假设

由于标准文档在签名和加密细节上没有给出足够完整的最终约定，本轮实现做了以下工程化假设：

### 5.1 签名

- 默认算法：`HMAC_SHA256`
- 可选算法：`MD5`
- 都通过配置切换

### 5.2 加密

- 默认算法：`AES/ECB/PKCS5Padding`
- 同时保留对 `GCM` 模式的兼容代码路径
- 密钥通过 `base64` 配置

这两项默认值用于：

- 本地开发
- 自动化测试
- 结构性联调准备

它们**不是最终联调参数定论**。正式联调前，需要科技平台侧再确认。

## 6. 为什么这一轮没有直接接入页面流程

原因有两个：

1. 用户要求先完成 `thirdparty` 调用实现。
2. 评审文档已经指出，科技平台标准接口与页面业务流程是两套视角，接口方向必须先澄清，否则页面上层代码容易建立在错误假设上。

所以当前顺序是：

1. 先把科技平台调用封装独立出来
2. 再在后续任务中逐步接入借款状态、还款状态和页面 controller/service

## 7. 本轮 review 建议关注点

建议重点 review 以下几点：

### 7.1 `TechPlatformPayloadCodec`

重点看：

- 签名串是否符合当前假设
- AES 默认模式是否合适
- 对密钥和异常的处理是否足够明确

### 7.2 `TechPlatformClientImpl`

重点看：

- 3 个 path 的划分是否合理
- 是否应该在当前阶段就接入更多科技平台接口
- 日志字段是否足够支撑后续排障

### 7.3 请求模型

重点看：

- 字段命名是否与标准文档一致
- 可选字段是否保留完整
- 是否还有遗漏的通知字段

## 8. 后续建议顺序

建议继续按下面顺序推进：

1. 把 `thirdparty techplatform` 接到放款/还款结果同步链路
2. 再实现艾博生权益页需要的新增接口
3. 再实现借款状态和还款页 controller/service

这样可以保证上层流程依赖的外部协议底座已经稳定。
