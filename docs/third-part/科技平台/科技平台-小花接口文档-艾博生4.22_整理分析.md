## 1. 文档目的

本文档基于小花科技最新提供的 `科技平台-小花接口文档-艾博生4.22.docx`，整理出一版适合艾博生内部评审、对接拆分、接口实现和联调准备使用的 Markdown 版接口文档。

本文档重点做四件事：

1. 提炼原始 `docx` 的接口目录与关键字段。
2. 明确接口调用方向：结合当前 plan 口径，哪些是艾博生通过云卡调用小花，哪些是小花通过云卡回调艾博生。
3. 结合当前项目语义，指出对接时真正需要落地的数据主键、状态和页面映射。
4. 标出原始文档中的歧义、缺口和联调前必须确认的事项。

## 2. 版本结论摘要

### 2.1 本版文档的核心变化

原文档版本记录显示：

- `V1.1.0`：调整加密说明、`2.14` 增加 `repayAmount`、`3.1` 修改入参字段名。

### 2.2 按本项目语义的角色映射

按原文档定义：

- **机构方**：合作接入的信贷机构，如小花。
- **平台方**：流量分发平台，如喜马拉雅。

按当前艾博生对接语义可理解为：

- **小花科技**：资金 / 信贷能力提供方，对应文档中的“机构方”。
- **艾博生**：页面承接与业务编排侧，对应文档中的“平台方”。

结合 `2026-04-23` 之后已经收口的项目 plan 口径，还需要再补一层实际链路解释：

- **艾博生并不直接调用小花**，而是 **艾博生先调用云卡，再由云卡调用小花**。
- **小花也不直接回调艾博生**，而是 **小花先回到云卡，再由云卡回调艾博生**。

因此在本项目里：

- **第 2 章“机构侧接口定义”**：从原始协议语义上看，是“平台调用机构”；落到当前项目实际上应理解为 **艾博生通过云卡调用小花** 的接口能力清单。
- **第 3 章“平台侧接口定义”**：从原始协议语义上看，是“机构通知平台”；落到当前项目实际上应理解为 **小花通过云卡回调艾博生** 的接口能力清单。

### 2.3 当前最重要的总体判断

1. 这份文档已经覆盖借款、还款、验证码、绑卡列表、图片拉取、权益订单同步、放款回调、还款回调等核心能力。
2. 从接口覆盖度看，已经可以支撑艾博生 H5 主链的大部分对接设计。
3. 但原文档仍存在若干关键歧义，不能完全“不确认就直接开写”：
   - 平台环境地址未填写。
   - `2.15 权益订单信息同步` 未给出明确接口路径。
   - `/loan/trail` 很像 `trial` 拼写错误，但原文如此。
   - `msg` / `message`、签名拼接说明、`imageInfo` 结构等处存在不一致。

## 3. 接入总览

### 3.1 基础协议要求

- 统一使用 `HTTPS`
- 编码：`UTF-8`
- 请求方式：`POST`
- 报文格式：`application/json`
- 接口要求保持幂等

### 3.2 加密与签名

原文要求：

1. 对 `data` 字段先做 `AES` 对称加密。
2. 再将加密后的 `data` 与其他参与签名字段按规则拼接。
3. 使用 `RSA` 非对称加密生成 `sign`。
4. 接收方先验签，再对 `data` 解密。
5. 响应报文加密方式与请求保持一致。

### 3.3 公共请求头

所有接口统一要求请求头携带：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `channelId` | `string` | Y | 平台方 id，由机构方分配给平台的唯一标识 |

### 3.4 公共请求体

#### 3.4.1 艾博生通过云卡调用小花（第 2 章）

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `requestId` | `string` | Y | 请求流水，每次请求必须唯一 |
| `timestamp` | `long` | Y | 时间戳 |
| `sign` | `string` | Y | RSA 签名 |
| `data` | `string` | Y | AES 加密后的业务 JSON 字符串 |

#### 3.4.2 小花通过云卡回调艾博生（第 3 章）

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `requestId` | `string` | Y | 请求流水，每次请求必须唯一 |
| `timestamp` | `long` | Y | 时间戳，毫秒值 |
| `sign` | `string` | Y | RSA 签名 |
| `data` | `string` | Y | AES 加密后的业务 JSON 字符串 |

### 3.5 公共响应体

原文同时出现了两套写法：

- 有些地方写 `code + msg + data + timestamp + sign`
- 有些地方写 `code + message + data + timestamp + sign`

建议艾博生实现时：

1. 对 `msg` / `message` 两种字段都做兼容。
2. 对成功判定统一以 `code == 0` 为准。

统一抽象如下：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `code` | `int` | Y | `0` 表示接口调用成功 |
| `msg` / `message` | `string` | N | 响应描述 |
| `data` | `string/object` | Y | 业务数据 |
| `timestamp` | `long` | Y | 时间戳 |
| `sign` | `string` | Y | 响应签名 |

### 3.6 通用错误码

| code | 含义 |
| --- | --- |
| `0` | 接口调用成功 |
| `10001` | 内部错误，接口调用失败 |
| `10003` | 参数错误 |
| `10004` | 签名错误 |

### 3.7 时效约束

原文给出的规则：

- 机构侧接口中，有一个“用户准入类接口”要求响应时间不超过 `4 秒`。
- 其余机构侧接口响应时间不超过 `15 秒`。
- 第 3 章回调接口只通知一次即可；如果超过 `10 分钟` 未通知，平台侧会主动查结果，因此无需做失败重推。

注意：原文这里写的是“`2.1 用户准入接口`”，但 `2.1` 实际标题是“协议列表”，文档本身存在编号描述不一致。

### 3.8 平台环境地址

原文第 `1.6 平台各环境地址` 处只保留了“测试 / 生产”栏目，但**没有填具体地址**。这属于联调前必须补齐的信息。

## 5.1 `2.1 协议列表`

- **路径**：`/protocol/queryProtocolAggregationLink`
- **用途**：统一查询协议展示列表，原文说明覆盖四个环节：绑卡、确认借款、放款后查看已签署协议等。

### 请求参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `userId` | `string` | Y | 用户唯一标识 |
| `loanAmount` | `double` | N | 借款金额，单位元 |
| `loanPeriod` | `int` | N | 借款期数，单位月 |

### 响应参数

返回 `list` 数组，每项包含：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `title` | `string` | Y | 合同名称 |
| `isShow` | `int` | Y | `1-强展示`，`2-非强展示` |
| `url` | `string` | Y | 协议链接 |

### 对接建议

- 这可以作为 H5 协议页 / 协议弹窗的统一入口。
- 如果后续需要“阅读到底才能勾选”，艾博生应在本地补“阅读完成状态”控制，原文档只定义协议列表，不定义阅读进度回写。

## 5.3 `2.3 用户信息查询`

- **路径**：`/user/query`
- **用途**：查询用户资料总表。
- **补充说明**：图片类数据不在此接口直接返回，需要通过 `2.14 图片资源拉取` 获取。

| `cid` | `string` | Y | 小花 `cid` |

### 响应主结构

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `userId` | `string` | Y | 用户唯一标识 |
| `basicInfo` | `object` | Y | 基本信息 |
| `idInfo` | `object` | Y | 实名信息 |
| `contactInfo` | `array` | Y | 紧急联系人 |

#### `basicInfo`

包含教育程度、行业、职位、月收入、居住地址、GPS 地址、经纬度等。

#### `idInfo`

包含姓名、身份证号、性别、身份证住址、签发机关、有效期起止、民族等。

#### `contactInfo`

包含联系人姓名、手机号、关系、排序。  
关系字段依赖附录码表。

#### `supplementInfo`

当前文档中至少包含：

- `occupation`

#### `optionInfo`

当前文档中至少包含：

- `companyName`
- `officeAddressProvince`
- `officeAddressCity`
- `officeAddressDistrict`
- `officeAddressDetail`
- `maritalStatus`

#### `deviceInfo`

包含：

- `ip`
- `deviceModel`
- `osType`
- `osVersion`
- `udid`
- `idfa`
- `imei`
- `idfv`

### 对接建议

- `2.3` 是借款确认 `2.5` 的资料来源之一，因为 `2.5` 要求重新提交完整资料对象。
- 艾博生需要明确哪些字段以小花返回为准，哪些字段允许本地页面补录覆盖。

## 5.4 `2.4 借款试算`

- **路径**：`/loan/trail`
- **用途**：确认借款前进行试算。

> 备注：原文接口路径写的是 `/loan/trail`，从语义上看大概率是 `trial`，但整理文档保留原文写法，联调前需和小花确认。

### 请求参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `userId` | `string` | Y | 用户唯一标识 |
| `loanAmount` | `double` | Y | 借款金额，元 |
| `loanPeriod` | `int` | Y | 借款周期，月 |
| `couponId` | `string` | N | 优惠券 id，无优惠券可不传 |

### 响应参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `receiveAmount` | `double` | Y | 实际到账金额 |
| `repayAmount` | `double` | Y | 应还总金额 |
| `yearRate` | `double` | Y | 年利率，例 `15.4` 表示 `15.4%` |
| `repayPlan` | `array` | Y | 还款计划 |

| `originalRefund` | `double` | N | 减免前应还总额 |

`repayPlan` 每项至少包括：

- `repayPrincipal`
- `repayInterest`

- `repaySvcFee`

### 对接建议

- 对应 H5 试算页的核心接口。
- 原文未定义“综合年化成本”字段，若 H5 页面要展示综合年化，需要单独向小花确认口径。

## 5.5 `2.5 确认借款`

- **路径**：`/loan/apply`
- **用途**：提交借款申请。

### 请求参数

#### 基础字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `userId` | `string` | Y | 用户唯一标识 |
| `loanId` | `int` | Y | 平台借款流水号 |
| `loanAmount` | `double` | Y | 借款金额，元 |
| `loanPeriod` | `int` | Y | 借款周期，月 |
| `loanReason` | `string` | Y | 借款用途，见附录码表 |
| `bankCardNum` | `string` | Y | 银行卡号 |
| `phone` | `string` | Y | 预留手机号 |
| `idno` | `string` | Y | 身份证号 |
| `name` | `string` | Y | 姓名 |
| `platformBenefitOrderNo` | `string` | Y | 平台权益订单号 |

#### 资料对象

原文要求同时提交：

- `basicInfo`
- `idInfo`
- `contactInfo`
- `supplementInfo`
- `optionInfo`
- `imageInfo`

其中说明为“最新信息”，意味着平台不能只传一个 `loanId`，而是要重新把用户资料整体带上。

#### `imageInfo`

原文将 `imageInfo` 标记为 `array`，但紧接着又定义了如下字段：

- `back`
- `front`
- `nature`
- `type`

这说明文档在结构表达上存在不一致，联调前需确认是：

1. `imageInfo` 本身是一个对象；还是
2. `imageInfo` 是数组，但数组项包含 `type + base64`。

### 响应参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `status` | `int` | Y | `4001-确认借款成功` `4002-确认借款失败` `4003-确认借款拒绝` |
| `remark` | `string` | N | 描述 |

### 对接建议

1. `loanId` 是平台生成的主键，循环额度场景下每次用信应生成新的 `loanId`。
2. 原文明确：若请求失败，用户再次发起时 `loanId` 不变。这意味着艾博生需要在重试场景中保持 `loanId` 稳定。
3. 该接口已具备“借款申请 + 资料上送 + 权益订单关联”的业务语义，是主链最重的一个接口。
