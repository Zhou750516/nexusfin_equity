# 2026-04-03 通联服务对接流程与上线切换说明（阶段性基线）

## 1. 文档目的

本文档用于基于 `2026-04-02` 齐为真实联调结果、当前项目已完成的第一轮主流程改造，以及后续正式上线要求，整理一份可直接指导后续开发、联调、预发和生产切换的**阶段性**支付协议号来源对接说明。

本文档聚焦解决一个明确问题：

> 当前已知齐为联调口径中，`abs.push.order` 所需的 `payProtocolNo` 指向通联签约协议号；但由于支付渠道后续**可能调整**，本文仅作为“当前按通联假设整理”的阶段性基线，不视为最终唯一正式方案。

## 2. 当前背景与问题收敛

截至 `2026-04-03`，已确认以下事实：

1. 齐为主动接口三条线已经完成真实联调：
   - `abs.push.order`
   - `abs.token.get`
   - `abs.lending.notify`
2. 齐为已明确：
   - `openFlag=0` 表示开卡成功
   - `abs.token.get` 仍可只传 `uniqueId + partnerOrderNo`
   - `515 会员卡未生效` 的真实根因不是开卡失败，而是首扣失败
   - 首扣失败原因是 `payProtocolNo` 错误
3. 齐为当前联调口径已明确：
   - 当前被要求的 `payProtocolNo` 指向艾博生与第三方支付公司 **通联** 签约后的协议号
   - 一个用户对应一个协议号
4. 但最新项目判断是：
   - 支付渠道后续可能调整
   - 因而“通联 = 最终唯一正式渠道”当前不能直接下最终结论
5. 当前项目仍未正式接入真实稳定的协议号回写来源
6. 因此当前生产级主流程仍缺少最终确认后的正式协议号来源
7. 测试环境为了继续验证齐为链路，已临时引入：
   - `QW_MEMBER_SYNC_PAY_PROTOCOL_NO_OVERRIDE`
   - 齐为提供的 mock 协议号

因此当前问题已经收敛为：

> 需要补齐“最终支付渠道签约成功 -> 真实协议号入库 -> 齐为下单读取 -> 订单快照留痕”的正式生产闭环；当前文档先按通联假设整理。

## 3. 目标链路

后续正式目标链路建议统一为以下 5 步：

### 3.1 第一步：当前假设的支付渠道签约成功

由支付侧、当前假设的渠道接入层或后续补充的第三方服务，在用户签约成功后拿到真实协议号。

此时至少需要具备以下关键信息：

1. `memberId` 或 `externalUserId`
2. `providerCode`
3. `protocolNo`
4. `protocolStatus`
5. `signRequestNo`
6. `signedTs`

### 3.2 第二步：回写当前项目

当前项目已预留正式回写入口：

- `POST /api/callbacks/payment-protocol`

入口目的：

1. 接收外部系统回传的真实协议号
2. 将用户当前有效协议号写入本地
3. 为齐为下单主流程提供稳定来源

### 3.3 第三步：本地维护用户级协议号

当前项目已经新增用户级协议号模型：

- `member_payment_protocol`

该表的定位是：

1. 维护“用户当前有效支付协议号”
2. 保证协议号与订单快照分层
3. 为后续协议失效、换绑、重新签约预留结构

### 3.4 第四步：齐为下单时读取真实协议号

当前主流程已经改成：

1. 下单前优先读取用户当前有效协议号
2. 将其作为 `payProtocolNo` 传给齐为 `abs.push.order`
3. 同时在订单中保留本单实际使用的协议号快照与来源快照

### 3.5 第五步：正式环境关闭测试 override

测试环境可以保留 override 作为联调能力。

正式环境必须保证：

1. 不再依赖 mock 协议号
2. 不再依赖 `proto-` 前缀占位值
3. 不再依赖内部 `agreementNo`
4. `pay_protocol_source` 只出现最终确认渠道值；若当前沿用通联口径，则对应为 `ALLINPAY`

## 4. 当前项目已具备的接入骨架

截至今天，项目已经具备以下正式接入骨架。

### 4.1 回写入口

当前已有回写 Controller：

- `src/main/java/com/nexusfin/equity/controller/PaymentProtocolCallbackController.java`

接口路径：

- `POST /api/callbacks/payment-protocol`

作用：

1. 接收外部回写请求
2. 调用 `PaymentProtocolCallbackService`
3. 记录 `traceId + bizOrderNo`

### 4.2 回写处理逻辑

当前已有回写 Service：

- `src/main/java/com/nexusfin/equity/service/impl/PaymentProtocolCallbackServiceImpl.java`

当前已实现行为：

1. 按 `requestId` 做幂等去重
2. 按 `memberId` 或 `externalUserId` 解析会员
3. 当前默认 `providerCode=ALLINPAY`，这是基于现阶段通联假设的实现口径
4. 当前只接受 `ACTIVE` 状态
5. 将协议号写入支付协议号服务层

当前边界说明：

1. `ACTIVE` 之外的状态目前仍未开放写入
2. 因此后续若支付侧要同步 `EXPIRED / REVOKED`，还需要补第二轮改造

### 4.3 协议号服务层

当前已有服务：

- `src/main/java/com/nexusfin/equity/service/impl/PaymentProtocolServiceImpl.java`

当前已实现行为：

1. 保存用户当前有效协议号
2. 下单时当前优先按 `memberId` 查询 `ALLINPAY` 的 `ACTIVE` 协议号
3. 若按 `memberId` 未找到，再按 `externalUserId` 查询
4. 若仍未找到，则仅在测试环境允许使用 override
5. 若无真实协议号且 override 不可用，则直接失败

### 4.4 测试 override 边界保护

当前已有启动保护：

- `src/main/java/com/nexusfin/equity/config/QwPayProtocolOverrideGuard.java`

对应配置位于：

- `src/main/java/com/nexusfin/equity/config/QwProperties.java`

当前边界为：

1. 若配置了 `QW_MEMBER_SYNC_PAY_PROTOCOL_NO_OVERRIDE`
2. 则必须同时开启 `QW_ALLOW_MEMBER_SYNC_PAY_PROTOCOL_NO_OVERRIDE=true`
3. 且当前 profile 必须命中允许列表
4. 否则应用启动直接失败

这保证了测试 override 不会被无边界地带入正式环境。

## 5. 字段定义

## 5.1 用户级协议号表

当前实体：

- `src/main/java/com/nexusfin/equity/entity/MemberPaymentProtocol.java`

核心字段如下：

| 字段 | 含义 | 当前用途 |
| --- | --- | --- |
| `member_id` | 本地会员 ID | 主流程优先查询键 |
| `external_user_id` | 外部用户标识 | 回写与兜底查询键 |
| `provider_code` | 支付渠道编码 | 当前实现口径固定为 `ALLINPAY`，后续应允许按最终渠道调整 |
| `protocol_no` | 真实支付协议号 | 齐为 `payProtocolNo` 正式来源 |
| `protocol_status` | 协议状态 | 当前仅 `ACTIVE` 可用 |
| `sign_request_no` | 签约请求号 / 外部流水 | 审计与回溯 |
| `channel_code` | 来源渠道 | 来源留痕 |
| `signed_ts` | 签约成功时间 | 审计留痕 |
| `expired_ts` | 协议失效时间 | 后续扩展预留 |
| `last_verified_ts` | 最近一次核验时间 | 后续核验预留 |

## 5.2 订单快照字段

当前订单侧已扩展：

1. `pay_protocol_no_snapshot`
2. `pay_protocol_source`

字段职责如下：

| 字段 | 含义 | 说明 |
| --- | --- | --- |
| `pay_protocol_no_snapshot` | 本单实际使用的协议号 | 用于排障、审计、历史还原 |
| `pay_protocol_source` | 本单协议号来源 | 用于区分真实来源与测试例外 |

当前来源建议值：

1. `ALLINPAY`
2. `TEST_OVERRIDE`

正式环境要求：

1. 订单快照允许保留历史 `TEST_OVERRIDE` 记录
2. 新产生订单应仅出现最终确认渠道值；若当前沿通联方案，则为 `ALLINPAY`

## 6. 状态规则

当前建议统一以下状态语义：

### 6.1 `ACTIVE`

含义：

1. 协议已签约成功
2. 当前可用于齐为 `abs.push.order`

当前处理规则：

1. 回写接口允许写入
2. 主流程允许使用

### 6.2 `INACTIVE`

含义：

1. 协议存在，但未达到可扣款状态

建议处理规则：

1. 当前不用于主流程
2. 是否允许回写，由后续第二轮改造决定

### 6.3 `EXPIRED`

含义：

1. 协议已失效，不可继续扣款

建议处理规则：

1. 当前不用于主流程
2. 后续应支持回写，并同步清理当前有效协议号口径

### 6.4 `REVOKED`

含义：

1. 协议已解约或主动撤销

建议处理规则：

1. 当前不用于主流程
2. 后续应支持回写，并同步失效化

### 6.5 当前代码边界

截至今天，当前代码仅支持：

1. 回写 `ACTIVE`
2. 主流程只读取 `ACTIVE`

因此今天的阶段性结论应明确为：

> 第一轮实现已经满足“真实协议号接入主链”的最小闭环，但尚未覆盖非 `ACTIVE` 状态回写。

## 7. 主流程取值规则

当前主流程取值规则建议统一按以下顺序执行：

### 7.1 正式规则

1. 当前优先按 `memberId + providerCode=ALLINPAY + protocolStatus=ACTIVE` 查询
2. 未查到时，再按 `externalUserId + providerCode=ALLINPAY + protocolStatus=ACTIVE` 查询
3. 查到后，作为本单 `payProtocolNo`
4. 同时写入：
   - `pay_protocol_no_snapshot`
   - `pay_protocol_source=ALLINPAY`（若后续支付渠道调整，这个来源值也应随之调整）

### 7.2 测试环境例外规则

仅当以下条件同时满足时，才允许使用 override：

1. 未查询到 `ACTIVE` 协议号
2. 配置了 `QW_MEMBER_SYNC_PAY_PROTOCOL_NO_OVERRIDE`
3. 显式开启 `QW_ALLOW_MEMBER_SYNC_PAY_PROTOCOL_NO_OVERRIDE=true`
4. 当前 profile 命中允许列表

若使用 override，则：

1. 本单 `pay_protocol_source=TEST_OVERRIDE`
2. 该能力只用于测试环境联调，不得进入正式环境

### 7.3 禁止规则

正式环境禁止：

1. 使用 `agreementNo` 充当 `payProtocolNo`
2. 使用 `proto-` 前缀占位值充当 `payProtocolNo`
3. 使用 mock 协议号替代真实协议号

## 8. 建议的回写报文口径

后续若支付侧或中台回写真实协议号，建议最小字段至少包含：

```json
{
  "requestId": "payproto-20260403-001",
  "memberId": "M202604030001",
  "externalUserId": "U202604030001",
  "providerCode": "ALLINPAY",
  "protocolNo": "AIP211926033187CF73483",
  "protocolStatus": "ACTIVE",
  "signRequestNo": "SIGN202604030001",
  "channelCode": "ALLINPAY",
  "signedTs": "2026-04-03T10:30:00"
}
```

说明：

1. 当前项目代码已支持 `memberId` 与 `externalUserId` 二选一解析会员
2. 当前最重要的是：
   - `requestId`
   - `protocolNo`
   - `protocolStatus`
3. 若 `protocolStatus` 非 `ACTIVE`，当前实现会直接拒绝
