### 3.2 网络与工具要求

- 执行端能直连云服务器 80 端口（已在 `云服务器测试记录-20260429.md` 验证）
- 必须使用 `curl --noproxy '*'` 规避本地代理
- 每条 curl 必须带 `-sS -m <timeout> -i`：`-i` 用于回查响应头，`-m` 防止挂死

### 3.3 标识命名约定

| 类型 | 模板 | 示例 |
|---|---|---|
| `requestId` | `REQ_NO_KJ_<编号>_20260429_<序号>` | `REQ_NO_KJ_B01_20260429_001` |
| `traceId` | `TRACE_NO_KJ_<编号>_20260429_<序号>` | `TRACE_NO_KJ_B01_20260429_001` |
| `bizOrderNo` | `BO_NO_KJ_<编号>_20260429_<序号>` | `BO_NO_KJ_B01_20260429_001` |
| 内部业务键（loanId 等）| `LN_NO_KJ_<编号>_20260429_<序号>` | `LN_NO_KJ_B01_20260429_001` |

`X-Timestamp` 统一使用 ISO 8601 + 时区，例如 `2026-04-29T10:00:00+08:00`。`X-Channel-Code` 固定 `ABS`。`X-Signature` 当前阶段统一占位 `TEST_SIGNATURE`（YK 侧尚未启用真实签名校验）。

## 4. 测试用例集

### 4.1 阶段 B：网关入口校验

**目标**：验证 YK 对 ABS 入站请求的基本防护；KJ 缺位下，B-01 能进入 KJ 出站异常路径并落流水，B-02~B-06 在前置校验阶段被拦截不落流水。

#### B-01 正常头 + KJ 参数占位（待执行）

- **目的**：合法报文进入 YK，YK 尝试转发至 KJ；KJ 不可达导致下游异常。**不能返回系统 500**。
- **请求**

```bash
curl --noproxy '*' -sS -m 30 -i -X POST "http://101.201.236.106/api/gateway/proxy" \
  -H "Content-Type: application/json" \
  -H "X-Trace-Id: TRACE_NO_KJ_B01_20260429_001" \
  -H "X-Request-Id: REQ_NO_KJ_B01_20260429_001" \
  -H "X-Timestamp: 2026-04-29T10:00:00+08:00" \
  -H "X-Channel-Code: ABS" \
  -H "X-Signature: TEST_SIGNATURE" \
  -d '{"requestId":"REQ_NO_KJ_B01_20260429_001","path":"/loan/query","bizOrderNo":"BO_NO_KJ_B01_20260429_001","data":{"loanId":"LN_NO_KJ_B01_20260429_001"}}'
```

- **通过条件**（全部满足）
  1. HTTP `200`
  2. 响应体 `code != "1000"`（不是参数错误）
  3. 响应体 `code != "1001"`（不是路径校验错误）
  4. 响应体 `data.errorType` 体现 KJ 调用层异常（如 `PROVIDER_ERROR / TIMEOUT / NETWORK_ERROR`，具体码值依 YK 实现）
  5. `data.providerMessage` 内含 KJ 不可达 / KJ 参数 / KJ 请求异常字样
  6. **管理端可查到该 `requestId` 的网关日志和审计日志各 1 条**（验收方式见 C-01、C-05）

#### B-02 ~ B-06 复测（已通过，可幂等复跑）

- B-02 缺少 `X-Trace-Id`
- B-03 Header / Body `requestId` 不一致
- B-04 缺少 `data`
- B-05 非白名单路径（`/unknown/path`）
- B-06 KJ 未定义路径（`/auth/token/verify` 或 `/loan/exercise-url`）

**curl 命令与预期**：完全沿用 `云服务器测试记录-20260429.md` §3–§7。复跑时仅把 `requestId / traceId / bizOrderNo` 后缀从 `REMOTE_001` 改为 `20260429_001`。

**通过条件**：

### 4.3 阶段 D：幂等测试

**目标**：相同 `requestId` 重复请求，YK 只产生 1 条主流水，第二次返回历史响应或处理中状态，**不重复调用下游 KJ**。

#### D-01 同 requestId 连续两次请求

- **步骤**

1. **第一次发送**（与 B-01 完全相同的报文，但用独立 requestId 避免与 B-01 串扰）：

```bash
curl --noproxy '*' -sS -m 30 -i -X POST "http://101.201.236.106/api/gateway/proxy" \
  -H "Content-Type: application/json" \
  -H "X-Trace-Id: TRACE_NO_KJ_D01_20260429_001" \
  -H "X-Request-Id: REQ_NO_KJ_D01_20260429_001" \
  -H "X-Timestamp: 2026-04-29T10:10:00+08:00" \
  -H "X-Channel-Code: ABS" \
  -H "X-Signature: TEST_SIGNATURE" \
  -d '{"requestId":"REQ_NO_KJ_D01_20260429_001","path":"/loan/query","bizOrderNo":"BO_NO_KJ_D01_20260429_001","data":{"loanId":"LN_NO_KJ_D01_20260429_001"}}'
```

2. **立即第二次发送**：完全相同的命令再跑一次

3. **查询日志计数**：

```bash
curl --noproxy '*' -sS -m 15 \
  "http://101.201.236.106/api/admin/logs/gateway-api?requestId=REQ_NO_KJ_D01_20260429_001&pageSize=10"
```

- **通过条件**
  1. 两次 HTTP 均 `200`
  2. 两次响应 `code` 与 `data.errorType` 一致（不出现一次成功、一次失败的非确定结果）
  3. 第二次响应应明确体现幂等命中（推荐 YK 在 `data.idempotent=true` 或 `message` 中体现，具体以 YK 实现为准；若未体现，至少响应内容与第一次等价）
  4. 日志查询返回 `data.total == 1`（只一条主流水）

### 4.4 阶段 E：本地 Mock KJ 预联调【依赖 YK 侧前置】

**前置（YK 侧负责）**：

- [ ] `KJ_BASE_URL=http://127.0.0.1:18080`
- [ ] 配置可用的 `KJ_PRIVATE_KEY`
- [ ] `KJ_ENCRYPT_PAYLOAD=false`
- [ ] `KJ_VERIFY_RESPONSE_SIGNATURE=false`
- [ ] 启动本地 Mock KJ 服务（监听 `127.0.0.1:18080`），按场景准备返回报文

**codex 检测方式**：跑 E-01 后若响应 `data.providerMessage` 仍为"KJ 不可达 / 连接拒绝"类，视为前置未就绪，**E 阶段全部 SKIP** 并在结果报告中说明等待 YK 侧确认。

#### E-01 Mock 返回成功

- **YK 侧 Mock 行为**：`POST /loan/query` 返回 KJ 标准成功结构
- **请求**：与 B-01 相同 curl，但 `requestId` 改为 `REQ_NO_KJ_E01_20260429_001`
- **通过条件**
  1. HTTP `200`
  2. `code == "0000"`，`message == "SUCCESS"`
  3. `data` 内含 KJ 业务字段（loanId、status、loanAmount 等）
  4. 网关日志命中 1 条，状态为成功

#### E-02 Mock 返回业务失败

- **YK 侧 Mock 行为**：返回 KJ 业务失败码（如 `code != 0`、`status=FAIL`）
- **请求**：`requestId=REQ_NO_KJ_E02_20260429_001`，其他同 E-01
- **通过条件**
  1. HTTP `200`
  2. YK 透传或映射 KJ 失败码到 `data.providerCode`
  3. `data.errorType` 为业务失败类（非参数错误）
  4. 网关日志命中 1 条，状态为失败

#### E-03 Mock 超时

- **YK 侧 Mock 行为**：sleep > YK 配置的 KJ 超时阈值后再返回
- **请求**：`requestId=REQ_NO_KJ_E03_20260429_001`，其他同 E-01
- **通过条件**
  1. HTTP `200`（不应是 504/网关超时透出）
  2. `data.errorType == "TIMEOUT"` 或等价值
  3. `data.retryable == true`
  4. 网关日志命中 1 条，标记为超时

#### E-04 Mock 返回签名异常（仅当 YK 启用了响应签名校验时）

#### E-05 Mock 返回非法 JSON

- **YK 侧 Mock 行为**：返回 `not-a-json`
- **通过条件**
  1. YK 不应抛出 5xx
  2. `data.errorType` 为下游响应解析错误
  3. 网关日志命中 1 条

## 5. 执行顺序

codex 必须按此顺序执行，**前置失败立即停止后续阶段**：

```
P-01, P-02 (前置探活)
   ↓
B-01 ──→ B-02 ──→ B-03 ──→ B-04 ──→ B-05 ──→ B-06
   ↓ (必须 B-01 成功落流水)
C-01 ──→ C-02 ──→ C-03 ──→ C-04 ──→ C-05 ──→ C-06
   ↓
D-01
   ↓
[E 阶段：检测前置 → 就绪则跑 E-01 ~ E-05；未就绪则全部 SKIP]
   ↓
[F 阶段：检测路径 → 已确认则跑 F-01 ~ F-03；未确认则全部 SKIP]
```

**异常分支**：

- 任意 B 用例 HTTP 返回非 `200`（出现 5xx 或 Tomcat 默认错误页）→ 整轮终止，记录"YK 服务异常"
- B-01 失败但 B-02~B-06 通过 → C 阶段标记为"前置不满足，无法验证"，跳过；D/E/F 仍可单独跑
- 中途网络抖动 → 单个用例最多重试 1 次，仍失败则记录原始响应
