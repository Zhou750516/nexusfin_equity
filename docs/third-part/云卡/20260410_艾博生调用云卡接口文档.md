# 2026-04-10 艾博生调用云卡接口文档

## 1. 文档背景

基于 `2026-04-10` 技术方案调整，三方边界统一为：

1. 所有 H5 页面均由艾博生提供，云卡与科技平台均不再承担面向用户的 H5 页面渲染职责。
2. 艾博生不再直接串联科技平台接口，而是统一调用云卡 gateway。
3. 云卡只承担 gateway 职责：记录请求 / 响应及关键业务数据，并将艾博生请求转发至科技平台。
4. 科技平台继续作为借款、授信、还款等业务能力提供方，但不直接承接艾博生 H5 页面调用。

本文档仅定义 **艾博生 -> 云卡 gateway** 的服务端接口口径，不覆盖：

- 科技平台 -> 云卡 的下游转发细节
- 云卡 -> 艾博生 的异步通知接口
- 艾博生 H5 页面内部接口

## 2. 职责划分

| 参与方 | 职责 |
| --- | --- |
| 艾博生 | 提供全部 H5 页面；维护用户登录态、页面编排、权益订单、支付与还款业务；以服务端方式调用云卡 gateway |
| 云卡 | 作为统一 gateway；记录接口日志、请求报文、响应报文、关键状态；转发艾博生请求至科技平台；向艾博生返回标准化结果 |
| 科技平台 | 提供用户校验、授信、借款、还款、绑卡、卡列表等业务能力；由云卡代为对接 |

## 3. 接口设计原则

### 3.1 统一调用方式

1. 艾博生仅调用云卡，不直接调用科技平台。
2. 云卡对外可以只提供一个统一 URI，由艾博生通过同一个 gateway 入口发起调用。
3. 所有写操作与敏感查询统一使用 `POST`。
4. 所有请求体都必须包含 `path` 字段，用于声明本次需要由云卡转发的科技平台接口地址。
5. 所有接口都必须携带 `requestId`，用于幂等、审计与链路追踪。
6. 所有接口都必须携带 `traceId`，用于跨系统日志关联。
7. 云卡必须落库或至少持久化记录关键请求、响应、重试与下游返回码。

### 3.2 统一 URI 约定

建议云卡仅暴露一个统一 gateway URI，例如：

- `POST /api/gateway/proxy`

说明：

1. 该 URI 只是艾博生调用云卡的统一入口。
2. 实际需要转发到科技平台的具体接口地址，通过请求体中的 `path` 字段传入。
3. `path` 建议传科技平台相对路径，例如：
   - `/auth/token/verify`
   - `/credit/apply`
   - `/credit/query`
   - `/loan/trail`
   - `/loan/apply`
   - `/loan/query`
   - `/repay/trial`
   - `/repay/apply`
   - `/repay/query`

### 3.3 统一请求体结构

建议艾博生调用云卡时采用统一请求体：

```json
{
  "requestId": "REQ202604100001",
  "path": "/loan/trail",
  "bizOrderNo": "BO202604100001",
  "data": {}
}
```

字段说明：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `requestId` | 是 | 请求幂等标识 |
| `path` | 是 | 科技平台接口地址，由云卡按该地址转发 |
| `bizOrderNo` | 否 | 业务单号，便于日志和对账 |
| `data` | 是 | 业务请求体，原样承载具体业务字段 |

### 3.4 推荐 Header

| Header | 是否必填 | 说明 |
| --- | --- | --- |
| `X-Trace-Id` | 是 | 全链路追踪标识 |
| `X-Request-Id` | 是 | 请求幂等标识；同一业务重试保持不变 |
| `X-Biz-Order-No` | 否 | 权益单号 / 还款单号 / 业务单号 |
| `X-Timestamp` | 是 | 请求时间戳 |
| `X-Channel-Code` | 是 | 调用渠道，固定为艾博生渠道编码 |
| `X-Signature` | 是 | 艾博生对请求报文的签名值 |

### 3.5 统一响应结构

建议云卡返回统一网关响应：

```json
{
  "code": "0000",
  "message": "SUCCESS",
  "traceId": "T202604100001",
  "requestId": "REQ202604100001",
  "data": {}
}
```

其中：

- `code/message` 表示云卡 gateway 本次受理结果
- `data` 内部承载科技平台业务结果或云卡标准化后的业务字段
- 如科技平台返回处理中状态，应在 `data.status` 中明确体现，而不是仅靠文案区分

  }

适用场景：

- 云卡已收到科技平台明确失败结果，且失败原因为业务失败。

#### 示例 4：网关异常

```json
{
  "code": "9001",
  "message": "GATEWAY_TIMEOUT",
  "traceId": "TRACE202604100006",
  "requestId": "REQ202604100006",
  "data": {
    "status": "UNKNOWN",
    "errorType": "GATEWAY_TIMEOUT",
    "providerCode": null,
    "providerMessage": null,
    "retryable": true
  }
}
```

适用场景：

- 云卡调用科技平台超时，当前无法确认下游是否已成功受理。
- 艾博生应结合 `requestId` 与后续查询接口做幂等重试或状态补偿。

建议补充约定：

1. `code/message` 体现云卡网关层结果。
2. `data.status` 体现业务处理状态，建议至少支持 `SUCCESS`、`PROCESSING`、`FAILED`、`UNKNOWN`。
3. `providerCode/providerMessage` 原样保留科技平台返回信息，便于排障和对账。
4. `retryable` 用于指导艾博生是否可以安全重试。

## 4. 接口清单

### 4.1 联合登录与用户信息

#### 4.1.1 联合登录校验

- **云卡统一 URI**：`POST /api/gateway/proxy`
- **调用方**：艾博生后端
- **用途**：艾博生 H5 收到上游跳转 token 后，通过云卡校验 token 有效性并获取用户基础信息

**请求体示例**

```json
{
  "requestId": "REQ202604100001",
  "path": "/auth/token/verify",
  "data": {
    "token": "tech-token",
    "scene": "BENEFIT_LOGIN",
    "redirectSource": "TECH_PLATFORM"
  }
}
```

**返回 data 示例**

```json
{
  "verified": true,
  "uid": "KH10001",
  "realName": "张三",
  "idCardNo": "ENC_xxx",
  "phone": "ENC_xxx",
  "bankCardNo": "ENC_xxx"
}
```

**说明**

1. 云卡只做网关转发与记录，不在本接口内自持用户主数据。
2. 艾博生以返回的 `uid` 作为跨系统主关联键之一。
3. 敏感字段建议返回脱敏值或密文，明文使用范围需单独确认。

## 5. 网关侧记录要求

云卡作为 gateway，建议至少记录以下字段：

| 类别 | 字段 |
| --- | --- |
| 链路标识 | `traceId`、`requestId`、`bizOrderNo`、`uid` |
| 接口信息 | `gatewayUri`、`path`、`providerPath`、HTTP 方法、调用时间 |
| 请求数据 | 请求摘要、敏感字段脱敏后的关键入参 |
| 响应数据 | 云卡响应码、科技平台响应码、响应摘要 |
| 处理结果 | 成功 / 失败 / 处理中、失败原因、重试次数 |
| 审计信息 | 创建时间、更新时间、操作来源、签名校验结果 |

## 6. 幂等与异常处理要求

1. 同一 `requestId` 的重复请求，云卡必须返回同一笔业务结果或可重放结果，不得重复落业务效果。
2. 云卡转发科技平台超时后，应明确区分：
   - 云卡未受理
   - 云卡已转发但下游超时
   - 下游已受理处理中
3. 对查询类接口，如果下游状态未知，需返回明确处理中状态，不得伪造成失败。
4. 对写接口，如果云卡已成功转发但未收到明确结果，应返回可供艾博生后续查询的业务标识，例如 `applyId`、`loanId`、`swiftNumber`。

## 7. 已确认口径与剩余待确认项

### 7.1 已确认口径

1. 云卡对外按“统一 gateway URI + 请求体 `path`”模式承接艾博生调用。
2. 借款试算 `path` 由艾博生配置项承载，当前默认值为 `/loan/trail`，后续如科技平台调整，仅改配置。
3. `GET /api/loan/approval-status/{applicationId}` 与 `GET /api/loan/approval-result/{applicationId}` 在艾博生后端均统一转发上游 `/loan/query`。
4. `applicationId` 允许由艾博生本地生成并维护映射关系。
5. `repaymentId` 由科技平台跳转参数携带，不由艾博生本地生成。
6. `POST /api/loan/apply` 若出现“齐为成功、云卡失败”，艾博生对 H5 返回借款 / 放款失败，但明确告知权益购买成功。
7. 云卡服务当前由其他同学并行开发，艾博生现阶段按本文档接口约定完成接入封装；待云卡服务可用后，再按同一口径开展联调。

### 7.2 剩余待确认项

1. 云卡对外基础域名与测试 / 生产环境地址。
2. 云卡统一 URI 的正式路径是什么，例如是否固定为 `/api/gateway/proxy`。
3. `path` 字段要求传相对路径还是完整 URL，是否允许携带 query string。
4. `X-Signature` 的真实签名算法、签名原串与密钥分发方式。
5. 是否全部接口统一 `POST + JSON`，是否存在个别查询接口使用 `GET`。
6. `uid`、`applyId`、`loanId`、`swiftNumber` 的生成责任边界。
7. 云卡 gateway 返回码与科技平台业务码的映射规则。
8. 云卡是否需要对部分查询结果做短暂缓存，以降低科技平台压力。
9. 艾博生与云卡之间是否需要补充异步通知接口，用于减少页面轮询。

## 8. 本轮结论

本轮接口边界以“**艾博生持有全部 H5、云卡只做 gateway、科技平台能力由云卡转发承接**”为准。当前云卡服务由其他同学并行开发，艾博生侧可以先按本文件约定完成接口封装、DTO 映射和调用链路接入，后续待云卡服务环境就绪后再进行真实联调。若科技平台或云卡补充真实接口字段、签名规则、环境地址与状态码口径，应优先更新本文件，并同步回填实现计划与功能规格。
