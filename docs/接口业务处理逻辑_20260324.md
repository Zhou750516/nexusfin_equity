# 艾博生权益分发服务接口业务处理逻辑

## 1. 文档说明

本文档按代码实际实现梳理对外接口的业务处理逻辑，并按照业务调用顺序编排，便于逐接口 Review。

梳理范围基于以下代码入口与实现：

- `controller/*Controller`
- `service/impl/*ServiceImpl`
- `util/OrderStateMachine`
- `config/SignatureInterceptor`
- `src/test/java/com/nexusfin/equity/NexusfinEquityApplicationTests.java` 中的 quickstart 主流程

说明口径：

- 这里描述的是“当前代码真实行为”，不是理想设计。
- 查询类接口本身不推进业务状态，但仍按业务使用场景插入顺序。
- `/api/users/register` 与 `/api/callbacks/**` 当前受签名拦截器保护。
- `/api/equity/**` 当前不受签名拦截器保护，包括下单与查询接口。

## 2. 业务调用顺序总览

### 2.1 主流程（首扣成功）

1. `POST /api/users/register` 静默注册会员
2. `GET /api/equity/products/{productCode}` 查询产品页
3. `POST /api/equity/orders` 创建权益订单
4. `POST /api/callbacks/first-deduction` 接收首扣结果
5. `POST /api/callbacks/grant/forward` 接收放款结果
6. `GET /api/equity/orders/{benefitOrderNo}` 查询订单状态
7. `GET /api/equity/exercise-url/{benefitOrderNo}` 获取行权地址
8. `POST /api/callbacks/exercise-equity` 接收行权结果
9. `POST /api/callbacks/refund` 接收退款结果
10. `POST /api/callbacks/repayment/forward` 接收还款结果通知

### 2.2 条件分支（首扣失败后进入兜底）

1. `POST /api/callbacks/first-deduction` 返回失败
2. `POST /api/callbacks/grant/forward` 返回成功
3. 系统内部调用 `FallbackDeductService.triggerFallback(...)` 发起兜底代扣待处理记录
4. `POST /api/callbacks/fallback-deduction` 接收兜底代扣结果
5. 后续可再走订单状态查询、退款、还款等接口

### 2.3 独立运维接口

1. `GET /api/equity/health` 健康检查

## 3. 接口逐一说明

### 3.1 `POST /api/users/register`

### 接口定位

静默注册入口。把渠道用户映射为平台会员，并落库会员基础信息与渠道绑定关系。

### 代码入口

- Controller: `com.nexusfin.equity.controller.UserRegistrationController`
- Service: `com.nexusfin.equity.service.impl.MemberOnboardingServiceImpl`

### 关键入参

- `requestId`: 幂等请求号
- `channelCode`: 渠道编码
- `userInfo.externalUserId`: 渠道侧用户标识
- `userInfo.mobileEncrypted`
- `userInfo.idCardEncrypted`
- `userInfo.realNameEncrypted`

### 处理步骤

1. Controller 仅做 `@Valid` 校验与 `Result.success(...)` 包装。
2. Service 先按 `channelCode + externalUserId` 查询 `member_channel`，判断是否为同一上游身份重复导流。
3. 同时基于 `requestId` 查询 `idempotency_record`，判断本次注册请求是否已经处理过。
4. 如果命中重复：
   - 优先返回已存在的 `memberId`
   - `registerStatus` 返回 `DUPLICATE`
5. 如果未命中重复：
   - 对手机号、身份证做 SHA-256，生成 `mobileHash`、`idCardHash`
   - 再按 `mobileHash + idCardHash` 查询 `member_info`，判断是否为同一自然人已存在会员
6. 如果 `member_info` 不存在：
   - 新生成 `memberId`
   - 对手机号、身份证、姓名再次加密后写入 `member_info`
   - 会员状态写为 `ACTIVE`
7. 无论会员是否新建，都会新增一条 `member_channel` 绑定记录：
   - 记录 `memberId`
   - 记录 `channelCode`
   - 记录 `externalUserId`
   - `bindStatus = BOUND`
8. 写入 `idempotency_record`：
   - `bizType = REGISTER`
   - `bizKey = memberId`
9. 返回 `memberId` 与 `registerStatus = SUCCESS`

### 读写对象

- 读取：`member_channel`、`idempotency_record`、`member_info`
- 写入：`member_info`、`member_channel`、`idempotency_record`

### 幂等逻辑

- 双重去重：
  - 渠道维度：`channelCode + externalUserId`
  - 请求维度：`requestId`
- 重复请求不会重复生成会员。

### 输出结果

- 成功返回 `memberId`
- 返回值会给后续产品页查询、订单创建接口使用

### 与后续接口关系

- 本接口产出的 `memberId` 是后续 `POST /api/equity/orders` 的必填入参。

### 3.2 `GET /api/equity/products/{productCode}`

### 接口定位

产品页聚合查询接口。返回产品基础信息，并在传入 `memberId` 时补齐会员标识信息。

### 代码入口

- Controller: `com.nexusfin.equity.controller.BenefitOrderController#getProductPage`
- Service: `com.nexusfin.equity.service.impl.BenefitOrderServiceImpl#getProductPage`

### 关键入参

- 路径参数：`productCode`
- 查询参数：`memberId`，可空

### 处理步骤

1. 根据 `productCode` 查询 `benefit_product`。
2. 如果产品不存在，或状态不是 `ACTIVE`，抛出 `PRODUCT_NOT_FOUND`。
3. 如果传入了 `memberId`，额外查询 `member_info`。
4. 组装 `ProductPageResponse`：
   - 返回产品编码、产品名称、费率
   - 当前协议列表固定返回 `EQUITY_AGREEMENT`、`DEFERRED_AGREEMENT`
   - 如果查到会员，则回填 `memberId`、`externalUserId`

### 读写对象

- 读取：`benefit_product`、`member_info`
- 写入：无

### 幂等逻辑

- 查询接口，无幂等记录写入。

### 输出结果

- 提供前端展示所需产品信息
- 给创建订单前的人机交互页面使用

### 与后续接口关系

- 该接口不推进状态，只是为 `POST /api/equity/orders` 做前置展示与确认。

### 3.3 `POST /api/equity/orders`

### 接口定位

权益订单创建接口。负责校验前置条件、落订单主表、生成协议任务与合同归档。

### 代码入口

- Controller: `com.nexusfin.equity.controller.BenefitOrderController#createOrder`
- Service: `com.nexusfin.equity.service.impl.BenefitOrderServiceImpl#createOrder`
- 协议生成：`com.nexusfin.equity.service.impl.AgreementServiceImpl`

### 关键入参

- `requestId`
- `memberId`
- `productCode`
- `loanAmount`
- `agreementSigned`

### 处理步骤

1. 先按 `requestId` 查询 `idempotency_record`。
2. 如果请求已处理过：
   - 用幂等记录里的 `bizKey` 取回已有 `benefit_order`
   - 直接复用旧订单组装返回
3. 如果是首次请求：
   - 通过 `OrderStateMachine.ensureCanCreateOrder(...)` 校验 `agreementSigned` 必须为 `true`
   - 查询 `benefit_product`，要求产品存在且状态为 `ACTIVE`
   - 查询 `member_info`，要求会员存在
   - 查询该会员最近一条 `member_channel`，用于回填渠道信息
4. 构造并写入 `benefit_order`：
   - 生成 `benefitOrderNo`
   - 生成 `agreementNo`
   - 初始化订单状态 `orderStatus = FIRST_DEDUCT_PENDING`
   - 初始化首扣状态 `qwFirstDeductStatus = PENDING`
   - 初始化兜底代扣状态 `qwFallbackDeductStatus = NONE`
   - 初始化行权状态 `qwExerciseStatus = NONE`
   - 初始化退款状态 `refundStatus = NONE`
   - 初始化放款状态 `grantStatus = PENDING`
   - 初始化同步状态 `syncStatus = SYNC_PENDING`
5. 创建订单后，立即调用 `agreementService.ensureAgreementArtifacts(order)`：
   - 为每笔订单补两类协议任务：`EQUITY_AGREEMENT`、`DEFERRED_AGREEMENT`
   - 对应写入 `sign_task`
   - 对应写入 `contract_archive`
   - 当前实现会直接把 `signStatus` 置为 `SIGNED`
6. 写入 `idempotency_record`：
   - `bizType = CREATE_ORDER`
   - `bizKey = benefitOrderNo`
   - `responseBody = orderStatus`
7. 返回创建结果：
   - `benefitOrderNo`
   - `orderStatus`
   - `orderDetailUrl`

### 读写对象

- 读取：`idempotency_record`、`benefit_product`、`member_info`、`member_channel`
- 写入：`benefit_order`、`sign_task`、`contract_archive`、`idempotency_record`

### 幂等逻辑

- 仅按 `requestId` 判重。
- 如果幂等记录存在，但对应订单丢失，会抛 `ORDER_NOT_FOUND`。

### 状态变化

- 新建订单固定进入 `FIRST_DEDUCT_PENDING`

### 与后续接口关系

- 创建完成后，等待支付渠道回调 `POST /api/callbacks/first-deduction`。

### 3.4 `POST /api/callbacks/first-deduction`

### 接口定位

接收首扣回调。记录支付结果，并决定订单走“首扣成功直连路径”还是“首扣失败兜底路径”。

### 代码入口

- Controller: `com.nexusfin.equity.controller.PaymentCallbackController#handleFirstDeduction`
- Service: `com.nexusfin.equity.service.impl.PaymentServiceImpl#handleFirstDeductCallback`
- 状态机：`com.nexusfin.equity.util.OrderStateMachine#applyFirstDeductResult`
- 下游同步：`com.nexusfin.equity.service.impl.DownstreamSyncServiceImpl`

### 关键入参

- `requestId`
- `benefitOrderNo`
- `qwTradeNo`
- `deductStatus`
- `deductAmount`
- `failReason`

### 处理步骤

1. 统一走 `handleCallback(request, FIRST_DEDUCT)`。
2. 先按 `requestId` 查询 `payment_record`，再按 `requestId` 查询 `idempotency_record`，进行重复回调判断。
3. 如果已处理：
   - 优先返回已有 `payment_record`
   - 不重复写库
4. 如果是首次处理：
   - 查询 `benefit_order`
   - 新增一条 `payment_record`
     - `paymentType = FIRST_DEDUCT`
     - `channelName = QW`
     - `channelTradeNo = qwTradeNo`
     - `paymentStatus = SUCCESS/FAIL`
5. 根据 `deductStatus` 调用状态机：
   - 成功时：
     - `orderStatus = FIRST_DEDUCT_SUCCESS`
     - `qwFirstDeductStatus = SUCCESS`
     - `syncStatus = SYNC_SUCCESS`
   - 失败时：
     - `orderStatus = FIRST_DEDUCT_FAIL`
     - `qwFirstDeductStatus = FAIL`
     - `syncStatus = SYNC_PENDING`
6. 无论首扣成功还是失败，都会立即调用 `downstreamSyncService.syncOrder(order)`：
   - 首扣成功时，下游路由标记为 `DIRECT_CONTINUE`
   - 首扣失败时，下游路由标记为 `FALLBACK_ELIGIBLE`
   - 同步成功后会把订单 `syncStatus` 更新为 `SYNC_SUCCESS`
7. 更新 `benefit_order`
8. 写入 `idempotency_record`
   - `bizType = FIRST_DEDUCT`
   - `bizKey = paymentNo`
9. 返回支付结果响应

### 读写对象

- 读取：`payment_record`、`idempotency_record`、`benefit_order`
- 写入：`payment_record`、`benefit_order`、`idempotency_record`

### 幂等逻辑

- 先查支付记录，再查幂等记录
- 重复回调直接回放已有支付结果

### 状态变化

- 成功路径：`FIRST_DEDUCT_PENDING -> FIRST_DEDUCT_SUCCESS`
- 失败路径：`FIRST_DEDUCT_PENDING -> FIRST_DEDUCT_FAIL`

### 与后续接口关系

- 首扣成功后，通常等待放款通知。
- 首扣失败后，仍然等待放款通知；若后续放款成功，会触发兜底代扣分支。

### 3.5 `POST /api/callbacks/grant/forward`

### 接口定位

接收放款结果通知。既更新放款状态，也决定订单是进入行权待处理，还是进入兜底代扣待处理。

### 代码入口

- Controller: `com.nexusfin.equity.controller.NotificationCallbackController#handleGrant`
- Service: `com.nexusfin.equity.service.impl.NotificationServiceImpl#handleGrant`
- 状态机：`com.nexusfin.equity.util.OrderStateMachine#applyGrantResult`
- 兜底代扣：`com.nexusfin.equity.service.impl.FallbackDeductServiceImpl`

### 关键入参

- `requestId`
- `benefitOrderNo`
- `grantStatus`
- `actualAmount`
- `loanOrderNo`
- `timestamp`

### 处理步骤

1. 先按 `requestId` 查 `idempotency_record`，已处理则直接返回。
2. 新增或复用一条 `notification_receive_log`，记录收到放款通知的事实。
3. 查询 `benefit_order`：
   - 若订单不存在，通知日志标记为 `FAILED` 后结束
4. 进入业务处理：
   - 解析 `grantStatus` 是否为成功
   - 调用 `OrderStateMachine.applyGrantResult(order, success, loanOrderNo)`
5. 状态机行为：
   - 无论成功失败，都会更新：
     - `grantStatus`
     - `loanOrderNo`
   - 仅当 `grantStatus = SUCCESS` 且当前订单状态为 `FIRST_DEDUCT_SUCCESS` 时：
     - `orderStatus = EXERCISE_PENDING`
6. 如果 `grantStatus = SUCCESS` 且当前订单状态为 `FIRST_DEDUCT_FAIL`：
   - 调用 `fallbackDeductService.triggerFallback(order, request)`
   - 系统内部新增一条兜底代扣 `payment_record`
   - 订单进入 `FALLBACK_DEDUCT_PENDING`
   - `qwFallbackDeductStatus = PENDING`
7. 更新 `benefit_order`
8. 更新通知日志状态为 `PROCESSED`
9. 写入 `idempotency_record`
   - `bizType = GRANT`
   - `bizKey = benefitOrderNo`

### 读写对象

- 读取：`idempotency_record`、`notification_receive_log`、`benefit_order`
- 写入：`notification_receive_log`、`benefit_order`
- 条件写入：`payment_record`（仅触发兜底时）
- 最终写入：`idempotency_record`

### 幂等逻辑

- 先查 `idempotency_record`
- 再对通知日志做按 `requestId` 复用

### 状态变化

- 主流程：
  - `FIRST_DEDUCT_SUCCESS + GRANT SUCCESS -> EXERCISE_PENDING`
- 兜底分支：
  - `FIRST_DEDUCT_FAIL + GRANT SUCCESS -> FALLBACK_DEDUCT_PENDING`
- 放款失败：
  - 当前代码只更新 `grantStatus = FAIL`
  - `orderStatus` 不会额外切换到独立失败态

### 与后续接口关系

- 主流程下一步通常是查询订单状态和获取行权地址。
- 兜底分支下一步通常是 `POST /api/callbacks/fallback-deduction`。

### 3.6 `POST /api/callbacks/fallback-deduction`

### 接口定位

接收兜底代扣结果回调，完成兜底支付的最终状态落库。

### 代码入口

- Controller: `com.nexusfin.equity.controller.PaymentCallbackController#handleFallbackDeduction`
- Service: `com.nexusfin.equity.service.impl.PaymentServiceImpl#handleFallbackDeductCallback`
- 状态机：`com.nexusfin.equity.util.OrderStateMachine#applyFallbackResult`

### 关键入参

- `requestId`
- `benefitOrderNo`
- `qwTradeNo`
- `deductStatus`
- `deductAmount`
- `failReason`

### 处理步骤

1. 统一走 `handleCallback(request, FALLBACK_DEDUCT)`。
2. 先按 `requestId` 和幂等记录去重。
3. 查询 `benefit_order`。
4. 新增一条 `payment_record`：
   - `paymentType = FALLBACK_DEDUCT`
   - `paymentStatus = SUCCESS/FAIL`
5. 根据结果调用 `OrderStateMachine.applyFallbackResult(order, success)`：
   - 成功时：
     - `orderStatus = FALLBACK_DEDUCT_SUCCESS`
     - `qwFallbackDeductStatus = SUCCESS`
   - 失败时：
     - `orderStatus = FALLBACK_DEDUCT_FAIL`
     - `qwFallbackDeductStatus = FAIL`
6. 更新 `benefit_order`
7. 写入 `idempotency_record`
   - `bizType = FALLBACK_DEDUCT`
   - `bizKey = paymentNo`
8. 返回支付响应

### 读写对象

- 读取：`payment_record`、`idempotency_record`、`benefit_order`
- 写入：`payment_record`、`benefit_order`、`idempotency_record`

### 幂等逻辑

- 与首扣回调同一套实现
- 重复回调返回已有支付结果

### 状态变化

- `FALLBACK_DEDUCT_PENDING -> FALLBACK_DEDUCT_SUCCESS`
- `FALLBACK_DEDUCT_PENDING -> FALLBACK_DEDUCT_FAIL`

### 与后续接口关系

- 处理完成后，可通过订单状态查询接口观察订单最终支付结果。

### 3.9 `POST /api/callbacks/exercise-equity`

### 接口定位

接收云卡行权结果通知，并更新订单行权状态。

### 代码入口

- Controller: `com.nexusfin.equity.controller.NotificationCallbackController#handleExercise`
- Service: `com.nexusfin.equity.service.impl.NotificationServiceImpl#handleExercise`
- 状态机：`com.nexusfin.equity.util.OrderStateMachine#applyExerciseResult`

### 关键入参

- `requestId`
- `benefitOrderNo`
- `exerciseStatus`
- `exerciseTime`
- `exerciseDetail`

### 处理步骤

1. 按 `requestId` 查 `idempotency_record`，已处理则直接返回。
2. 记录 `notification_receive_log`：
   - `notifyType = EXERCISE_RESULT`
3. 查询 `benefit_order`
   - 若不存在，则通知日志标记 `FAILED` 后返回
4. 调用状态机 `applyExerciseResult(order, success)`：
   - 成功时：
     - `qwExerciseStatus = SUCCESS`
     - `orderStatus = EXERCISE_SUCCESS`
   - 失败时：
     - `qwExerciseStatus = FAIL`
     - `orderStatus = EXERCISE_FAIL`
5. 更新 `benefit_order`
6. 通知日志标记 `PROCESSED`
7. 写入 `idempotency_record`
   - `bizType = EXERCISE`
   - `bizKey = benefitOrderNo`

### 读写对象

- 读取：`idempotency_record`、`notification_receive_log`、`benefit_order`
- 写入：`notification_receive_log`、`benefit_order`、`idempotency_record`

### 幂等逻辑

- 按 `requestId` 幂等处理

### 状态变化

- `EXERCISE_PENDING -> EXERCISE_SUCCESS`
- `EXERCISE_PENDING -> EXERCISE_FAIL`

### 3.10 `POST /api/callbacks/refund`

### 接口定位

接收退款结果通知，并更新订单退款状态。

### 代码入口

- Controller: `com.nexusfin.equity.controller.NotificationCallbackController#handleRefund`
- Service: `com.nexusfin.equity.service.impl.NotificationServiceImpl#handleRefund`
- 状态机：`com.nexusfin.equity.util.OrderStateMachine#applyRefundResult`

### 关键入参

- `requestId`
- `benefitOrderNo`
- `refundStatus`
- `refundAmount`
- `refundTime`
- `refundReason`

### 处理步骤

1. 按 `requestId` 判重。
2. 写入 `notification_receive_log`：
   - `notifyType = REFUND_RESULT`
3. 查询 `benefit_order`
   - 若订单不存在，则通知日志标记 `FAILED` 后返回
4. 调用 `applyRefundResult(order, success)`：
   - 成功时：
     - `refundStatus = SUCCESS`
     - `orderStatus = REFUND_SUCCESS`
   - 失败时：
     - `refundStatus = FAIL`
     - `orderStatus = REFUND_FAIL`
5. 更新 `benefit_order`
6. 更新通知日志为 `PROCESSED`
7. 写入 `idempotency_record`
   - `bizType = REFUND`

### 读写对象

- 读取：`idempotency_record`、`notification_receive_log`、`benefit_order`
- 写入：`notification_receive_log`、`benefit_order`、`idempotency_record`

### 幂等逻辑

- 按 `requestId` 去重

### 状态变化

- 退款成功：`orderStatus = REFUND_SUCCESS`
- 退款失败：`orderStatus = REFUND_FAIL`

## 4. 关键内部支撑逻辑

### 4.1 协议生成逻辑

触发点：`POST /api/equity/orders`

处理行为：

1. 为每笔订单创建两类协议任务：
   - `EQUITY_AGREEMENT`
   - `DEFERRED_AGREEMENT`
2. 每类协议都会生成：
   - 一条 `sign_task`
   - 一条 `contract_archive`
3. 当前实现中签约任务默认直接写成 `SIGNED`
4. 合同归档会落 `fileUrl` 与 `fileHash`

### 4.2 下游同步逻辑

触发点：`POST /api/callbacks/first-deduction`

处理行为：

1. 首扣成功或失败后都触发一次 `syncOrder`
2. 根据订单状态选择路由：
   - `FIRST_DEDUCT_SUCCESS -> DIRECT_CONTINUE`
   - `FIRST_DEDUCT_FAIL / FALLBACK_DEDUCT_PENDING -> FALLBACK_ELIGIBLE`
3. 若订单关键字段缺失，更新 `syncStatus = SYNC_FAIL` 并抛异常
4. 若同步成功，写幂等记录 `DOWNSTREAM_SYNC`

### 4.3 自动兜底代扣触发逻辑

触发点：`POST /api/callbacks/grant/forward`

触发条件：

1. `grantStatus = SUCCESS`
2. 当前订单状态是 `FIRST_DEDUCT_FAIL`

处理行为：

1. 查是否已存在 `FALLBACK_DEDUCT` 支付记录
2. 若不存在，则新增一条 `payment_record`
   - `paymentType = FALLBACK_DEDUCT`
   - `paymentStatus = PENDING`
3. 更新订单：
   - `orderStatus = FALLBACK_DEDUCT_PENDING`
   - `qwFallbackDeductStatus = PENDING`

## 6. Review 时建议重点关注的代码观察点

以下内容不是整改建议，只是方便 Review 时快速定位：

1. 订单创建接口当前不受签名拦截保护，而注册与回调受保护。
2. `GET /api/equity/exercise-url/{benefitOrderNo}` 当前只校验订单存在，不校验是否进入行权阶段。
3. 行权、退款、还款回调当前都未校验严格前置状态，只要订单存在即可处理。
4. 放款失败时当前只更新 `grantStatus = FAIL`，不会切换独立订单失败态。
5. 还款回调当前只做通知留痕和幂等记账，不更新订单或还款明细。
