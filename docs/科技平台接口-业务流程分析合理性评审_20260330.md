# 科技平台接口-业务流程分析合理性评审

文档日期：2026-03-30

评审对象：`docs/科技平台接口-业务流程分析.pdf`

对照依据：

- `docs/惠选卡权益与借款流程-20260330-V1.0.pdf`
- `docs/科技平台导流对接标准接口文档.docx`

## 1. 评审结论

`科技平台接口-业务流程分析.pdf` 可以作为一次初步梳理，但目前**不适合作为联调和实现依据直接落地**。

原因不是它完全错误，而是它存在两类问题：

1. 有些判断方向是对的，例如“银行卡信息大概率在导流/进件阶段沉淀到本地”“还款身份验证需要结合鉴权接口”“结清证明在标准接口中没有现成定义”。
2. 但更关键的“谁调用谁”“哪些是同步查询、哪些是异步通知”“哪些接口是流程必选项”存在明显偏差，尤其把第 4 章和第 5 章的接口调用方向混淆了。

因此，综合结论是：

- `科技平台接口-业务流程分析.pdf` **部分合理**；
- 但如果用于后续详细设计、联调清单、接口责任划分，当前版本 **不够合理，需要修订**。

## 2. 三份文档的角色定位

先把三份材料的角色区分清楚，否则很容易把接口方向看反：

### 2.1 `惠选卡权益与借款流程-20260330-V1.0.pdf`

这份是业务流程/产品流程视角，描述的是：

- 用户在哪个页面做什么
- 页面需要展示哪些信息
- 页面点击后期望发生什么
- 艾博生、云卡、科技平台三方在业务上的协作关系

它偏“产品和页面流程”。

### 2.2 `科技平台导流对接标准接口文档.docx`

这份是标准接口协议视角，关键有两个章节：

- 第 4 章：`机构接口列表`
- 第 5 章：`平台接口列表`

结合第 3 章“平台调机构 / 机构调平台”的定义，可以得出：

- 第 4 章列的是 **科技平台调用合作机构** 的接口
- 第 5 章列的是 **合作机构调用科技平台，或者合作机构通知科技平台** 的接口

这一点是本次评审最关键的判断基础。

### 2.3 `科技平台接口-业务流程分析.pdf`

这份是把前两份材料做了一次口语化归纳，但在归纳过程中，部分接口调用方向被简化甚至反转了。

## 3. 当前分析 PDF 中合理的部分

以下内容基本合理，可以保留：

### 3.1 银行卡信息在前置阶段沉淀到机构侧

业务流程文档中提到：

- 艾博生权益页展示的支付卡信息来自本地库
- 数据由科技平台写入

标准接口文档第 `4.2 进件接口 /guide/api/grantCredit` 中，确实包含：

- `bankCardInfo`
- 身份信息
- 联系人
- 公司信息
- 设备信息

因此，“银行卡等资料在前置导流/进件时由科技平台推给机构，后续页面优先查本地”这个判断是成立的。

### 3.2 还款身份验证需要单独设计

业务流程文档中明确写了：

- 用户点击确认还款后，需要先完成身份验证
- 决策点可能是短信验证码或活体认证

标准接口文档中确实存在：

- `4.18 /guide/api/smsApply`
- `4.19 /guide/api/smsVerify`

所以，当前分析 PDF 提出“如果复用科技平台短信验证能力，则还款前需要走鉴权发起/确认”这个判断是合理的。

### 3.3 结清证明在标准文档中没有现成接口

业务流程文档提出了“结清证明”能力，但标准接口文档中没有直接对应的“结清证明查询/下载”接口。

因此，当前分析 PDF 提出“如果要由科技平台提供，需要新增接口或补充约定”，这个判断也是合理的。

## 4. 当前分析 PDF 中不合理的部分

这部分是本次评审的重点。

### 4.1 把借款试算、确认借款的调用方向看反了

当前分析 PDF 的说法是：

- 云卡调用科技平台的借款试算接口 `/guide/api/loantrial`
- 云卡调用科技平台的确认借款接口 `/guide/api/applyloanamount`

这个说法与标准接口文档的章节归属不一致。

标准接口文档中：

- `4.8 借款试算 /guide/api/loantrial`
- `4.9 确认借款 /guide/api/applyloanamount`

都位于第 4 章 `机构接口列表`，而第 4 章的定义是“平台调机构”。

因此更准确的解释应当是：

- 科技平台调用合作机构的借款试算接口
- 科技平台调用合作机构的确认借款接口

也就是说，这两个接口在标准对接语义下，不是“云卡去调科技平台”，而是“科技平台去调机构”。

这是一处核心性错误，因为它会直接影响：

- 接口部署在哪一方
- 谁是请求方，谁是被调方
- 参数生成责任归属
- 联调时的 Mock 和验收方式

### 4.2 把放款结果通知的方向看反了

当前分析 PDF 认为：

- 科技平台会通过 `/guide/api/loanInfoNotice` 主动回调云卡

但标准接口文档中：

- `5.2 接受放款结果通知 /guide/api/loanInfoNotice`

位于第 5 章 `平台接口列表`，语义是“机构通知平台订单放款结果”。

所以正确方向应为：

- 合作机构调用科技平台的 `/guide/api/loanInfoNotice`
- 本质是“机构 -> 平台”的通知接口

而不是“平台 -> 机构”的 Webhook。

这不是小偏差，而是接口职责相反。

### 4.3 把还款试算、确认还款的调用方向也看反了

当前分析 PDF 的说法是：

- 艾博生调用科技平台的 `/guide/api/repayTrial`
- 艾博生调用科技平台的 `/guide/api/repaySubmit`

但标准接口文档中：

- `4.15 还款试算 /guide/api/repayTrial`
- `4.16 确认还款 /guide/api/repaySubmit`
- `4.17 还款结果查询 /guide/api/repayResult`
- `4.18 / 4.19 借款/还款鉴权`

这些都在第 4 章 `机构接口列表` 中，语义仍然是“平台调机构”。

因此，在标准接口定义里，更准确的责任关系是：

- 科技平台调用机构的还款试算接口
- 科技平台调用机构的确认还款接口
- 科技平台调用机构的还款结果查询接口
- 科技平台调用机构的还款鉴权相关接口

当前分析 PDF 把它理解为“艾博生主动去调用科技平台”，与标准接口文档不一致。

### 4.4 漏掉了订单状态查询这个关键接口

业务流程文档的借款审核页说明中有：

- 前端轮询 `/api/loan/status`
- 审核状态展示为 `PENDING / SUCCESS / FAILED`

标准接口文档里与此最接近、且明确存在的接口是：

- `4.10 订单状态查询 /guide/api/getLoanInfo`

当前分析 PDF 没有把这个接口纳入核心清单，而是把审核结果主要归因到“科技平台通过 Webhook 回调云卡”。

这会带来两个问题：

1. 它遗漏了标准文档中已经定义好的订单状态查询能力。
2. 它把“异步通知”和“主动查询”混为一谈。

更合理的表达应当是：

- 借款状态至少需要明确“查询接口”和“异步通知接口”是否同时存在
- 如果按标准接口文档理解，`getLoanInfo` 是正式定义的状态查询能力，不能省略

### 4.5 漏掉了还款结果查询接口

标准接口文档明确定义了：

- `4.17 还款结果查询 /guide/api/repayResult`

但当前分析 PDF 只提了：

- 还款试算
- 确认还款
- 鉴权发起 / 鉴权确认

没有把“还款提交后如何查询结果”纳入流程闭环。

如果后续设计只按当前分析 PDF 推进，会导致还款链路缺少结果确认与状态回查机制。

### 4.6 漏掉了还款入口链接获取接口

业务流程文档写的是：

- 用户从科技平台还款入口跳转到艾博生还款页

标准接口文档中，与“跳转到机构侧还款页”直接相关的接口是：

- `4.13 获取机构地址接口 /guide/api/nwdscenceurl`

该接口用途写得很明确：

- 获取渠道协议、还款等相关链接进行跳转

而当前分析 PDF 没有把这个接口纳入流程主链，这会导致“用户从哪里拿到还款入口 URL”这个关键问题没有被解释清楚。

### 4.7 漏掉了协议与合同展示相关接口

业务流程文档中有两类内容：

- 协议展示
- 电子签章合同存储与展示

标准接口文档中分别存在：

- `4.14 获取协议列表 /guide/api/getAgreement`
- `4.11 放款合同查询 /guide/api/getContract`

当前分析 PDF 没有把这两个接口与业务流程对应起来，导致“页面展示协议/合同的标准接口来源”缺少映射。

### 4.8 没有处理 PRD 与标准接口文档之间的视角冲突

业务流程文档里描述的是：

- 云卡后端调用科技平台试算接口
- 云卡后端调用科技平台确认借款接口
- 艾博生后端调用科技平台还款试算、触发还款接口

但标准接口文档的章节定义又表明：

- 对应的 `/guide/api/loantrial`、`/guide/api/applyloanamount`、`/guide/api/repayTrial`、`/guide/api/repaySubmit`
- 在标准对接语义下属于“平台调机构”

这意味着两份文档之间至少存在一种情况：

1. 要么 PRD 站在“业务编排视角”描述，省略了平台与机构之间的外部对接边界。
2. 要么项目里同时存在“机构内部页面接口”和“科技平台标准外部接口”两套接口体系。
3. 要么当前 PRD 对调用方向本身就还没有完全收敛。

而当前分析 PDF 没有指出这层冲突，直接把 PRD 的表述当成标准接口调用关系，这是不够严谨的。

## 5. 更合理的流程理解

结合两份源文档，建议把科技平台导流流程理解为“两层视角”：

### 5.1 外部对接视角

这是 `科技平台导流对接标准接口文档.docx` 定义的视角：

- 科技平台调用机构的进件、试算、确认借款、订单查询、协议查询、还款试算、还款提交、还款结果查询、短信鉴权等接口
- 机构再通过放款通知、还款通知等接口，把最终结果同步给科技平台

在这个视角下：

- 第 4 章大多是 `科技平台 -> 机构`
- 第 5 章大多是 `机构 -> 科技平台`

### 5.2 页面/产品视角

这是 `惠选卡权益与借款流程-20260330-V1.0.pdf` 的视角：

- 用户先在艾博生权益页操作
- 再进入云卡借款页
- 最终由科技平台提供金融能力

在这个视角下，文档更关心：

- 页面怎么跳
- 页面展示什么
- 用户点击按钮后业务上发生什么

这个视角不等于标准接口调用方向，它更像产品编排说明。

### 5.3 建议的统一理解

比较稳妥的统一方式是：

1. PRD 继续保留用户操作链路和页面编排。
2. 外部接口责任以标准接口文档为准，不要直接拿 PRD 的叙述代替接口方向。
3. 如果项目内部确实要做“云卡后端调用科技平台”的内部聚合能力，需要单独标注为“内部服务编排”，不要与标准对接接口混写。

## 7. 对当前分析 PDF 的修订建议

建议把 `科技平台接口-业务流程分析.pdf` 的结论调整为下面这几个原则：

### 7.1 不要再用“云卡使用的接口 / 艾博生使用的接口”作为主标题

因为这样很容易把“页面所属方”和“接口调用方”混为一谈。

更建议改成：

- `科技平台调用机构的接口`
- `机构通知科技平台的接口`
- `业务流程中需要补充确认的事项`

### 7.2 明确区分“页面流程”和“标准接口方向”

建议新增一句总说明：

> 业务流程文档描述的是页面与业务编排；标准接口文档定义的是外部系统对接方向。两者不是一个视角，不能直接等同。

### 7.3 把遗漏接口补全

至少补上以下接口，否则流程闭环不完整：

- `/guide/api/getLoanInfo`
- `/guide/api/getRepayPlan`
- `/guide/api/nwdscenceurl`
- `/guide/api/getAgreement`
- `/guide/api/getContract`
- `/guide/api/repayResult`
- `/guide/api/repayInfoNotice`

### 7.4 将放款通知方向纠正

把“科技平台回调云卡”改为：

- 机构通知科技平台放款结果

### 7.5 将借款/还款试算与确认的方向纠正

把“机构调用科技平台”改为：

- 在标准接口定义中，这些接口属于科技平台调用机构

如果内部系统仍需要主动调用科技平台，那应在文档里单独注明“内部编排接口”，不能直接拿标准接口路径替代。

## 8. 待确认问题

基于目前三份材料，仍有以下问题需要业务、产品、对接三方统一：

1. 云卡借款页到底是“云卡自有页面”，还是“科技平台承载页/中转页”？
2. PRD 中“前端不调用后端，直接跳转到云卡 URL”与标准接口中的 `applyAmount`、`nwdscenceurl` 如何对应？
3. 还款时的短信鉴权，是由科技平台统一发起，还是机构页面自行完成后只回传结果？
4. 电子签章合同展示，是以机构合同查询接口为准，还是由艾博生本地归档后直接展示？
5. 结清证明最终由哪一方提供？如果由科技平台提供，是否需要新增标准接口？

## 9. 最终判断

最终判断如下：

- `科技平台接口-业务流程分析.pdf` 的出发点是对的，也抓住了银行卡前置沉淀、还款鉴权、结清证明缺口这几个点。
- 但它对标准接口文档的“章节语义”和“调用方向”理解不够准确。
- 因此，这份 PDF **可以保留为阶段性讨论材料**，但**不建议直接作为开发和联调依据**。

更稳妥的做法是：

- 以本评审文档为基础修订原 PDF；
- 明确“产品流程视角”和“标准接口视角”；
- 再据此产出联调版时序图和接口责任矩阵。
