## 3. 当前资金安全机制盘点

| 机制 | 实现位置 | 现状 |
|---|---|---|
| 幂等保护 | `IdempotencyService` | ✅ 用于 BenefitOrderServiceImpl.createOrder（line 102-114）|
| 异步补偿 | `AsyncCompensationWorkerService` | ✅ 仅 BenefitPurchase 用上；签约链路、还款短信链路未接入 |
| 金额单位 | DTO 字段 `loanAmount` 等 | ⚠️ `int`（单位分），需逐字段确认；元↔分混淆是历史财损常见点 |
| 客户端价格校验 | Service 层 | ⚠️ 服务端是否基于 `productCode` 复算金额需要逐接口验证 |
| 状态机守卫 | Service 层状态推进逻辑 | ⚠️ 是否拒绝逆向状态转换（SUCCESS → FAILED）需逐流程验证 |
| 回调幂等 | PaymentCallbackController + IdempotencyService | ⚠️ 架构调研中标注幂等保护"不可见"，需用例 LOS-B.1 验证 |
| 退款流程 | `RefundService` / `RefundController` | ⚠️ 未审核退款金额上限、退款条件、双重退款防护 |
| 用户限额 | — | ❌ 未发现 |
| 对账任务 | — | ❌ 未发现定时对账作业 |
| 资金流追踪 | `payment_record` / `refund_record` 等表 | ⚠️ 字段完整性需确认（callback_request_id、provider_swift_no、amount_cents 等）|

**关键缺口**：

1. **金额单位混淆**：DTO/DB/上游接口的金额单位是否全链路统一为"分"？
2. **客户端价格篡改**：客户端传 `loanAmount=29900`（299 元）能否被改成 `1`（1 分）后下单？
3. **状态机倒退保护**：成功支付被失败回调覆盖的可能性
4. **对账缺位**：发生上游/本地分歧时无主动发现机制
5. **退款金额校验**：能否退款超过原支付金额？

## 4. 验证基础设施

### 4.1 SQL 回查（核心）

```sql
-- 订单与金额（多角度核对）
SELECT id, member_id, biz_order_no, product_code,
       loan_amount, pay_amount,        -- 期望相等
       status, sync_status, created_at, updated_at
  FROM benefit_order WHERE biz_order_no=?;

-- 支付记录（防双扣）
SELECT id, payment_no, biz_order_no, amount_cents, status,
       callback_request_id, provider_swift_no, created_at
  FROM payment_record WHERE biz_order_no=? ORDER BY id;
-- 期望：同 biz_order_no 同 callback_request_id 仅 1 条；金额合计 == 订单金额

-- 退款记录
SELECT id, refund_id, biz_order_no, refund_amount_cents, status, created_at
  FROM refund_record WHERE biz_order_no=? ORDER BY id;
-- 期望：sum(refund_amount_cents) <= benefit_order.pay_amount

-- 借款主单
SELECT id, application_id, loan_id, member_id, benefit_order_no,
       loan_amount, status, created_at
  FROM loan_application WHERE benefit_order_no=?;

-- 还款记录
SELECT id, repayment_id, loan_id, repay_amount_cents, repay_principal_cents,
       repay_interest_cents, status, swift_number, created_at
  FROM repayment_record WHERE loan_id=? ORDER BY id;

-- 异步补偿任务（验证幂等入队）
SELECT task_type, biz_key, biz_order_no, COUNT(*) AS cnt
  FROM async_compensation_task
  WHERE biz_order_no=?
  GROUP BY task_type, biz_key, biz_order_no
  HAVING COUNT(*) > 1;   -- 应为空
```

### 4.2 标识命名约定

| 类型 | 模板 | 示例 |
|---|---|---|
| `requestId` | `REQ_LOS_<编号>_<时间序>` | `REQ_LOS_A_1_001` |
| `bizOrderNo` | `BO_LOS_<编号>_<时间序>` | `BO_LOS_A_1_001` |
| 故障联动 | `REQ_LOS_<编号>_FAULT_<类型>` | `REQ_LOS_B_3_FAULT_TIMEOUT`（用于触发补偿）|

## 5. P0 用例集

### 5.1 LOS-A 金额校验

#### LOS-A.1 单位混淆 — 客户端传"元"伪装"分"

```bash
# 产品标价 299 元 = 29900 分；客户端尝试传 299（即 2.99 元）
curl ... POST "http://127.0.0.1:8080/api/equity/orders" \
  -H "Content-Type: application/json" \
  -d '{"requestId":"REQ_LOS_A_1_001","productCode":"P001","loanAmount":299,"agreementSigned":true}'
```

通过条件：

1. HTTP 200，业务码 = `PRICE_MISMATCH` 或 `PARAM_ERROR`（**服务端必须基于 productCode 复算价格**）
2. SQL 验证：未创建以 299 分支付的订单
3. **若返回 success**：立即升级为 P0 资损 FAIL，记录"金额由客户端决定"严重缺陷

#### LOS-A.2 amount = 0

```bash
curl ... -d '{"requestId":"REQ_LOS_A_2_001","productCode":"P001","loanAmount":0,...}'
```

通过条件：业务码 = `PARAM_ERROR`；DB 无 0 元订单

#### LOS-A.3 amount < 0（负数攻击）

```bash
curl ... -d '{"requestId":"REQ_LOS_A_3_001","loanAmount":-29900,...}'
```

通过条件：

1. 业务码 = `PARAM_ERROR`
2. DB 无负金额订单
3. **关键**：检查所有金额字段（loanAmount / repayAmount / refundAmount）是否都拒绝负数

#### LOS-A.4 amount 超大（int 溢出）

```bash
curl ... -d '{"loanAmount":2147483648,...}'   # > Integer.MAX_VALUE
curl ... -d '{"loanAmount":99999999999,...}'  # 超大
```

通过条件：

1. 业务码 = `PARAM_ERROR` 或上游限额拒绝
2. **关键**：DTO 字段类型若为 `int`，溢出会被 Spring 直接拒为 PARAM_ERROR；若为 `long`，需要业务层有上限校验

#### LOS-A.5 精度攻击（小数 / 科学计数法）

```bash
curl ... -d '{"loanAmount":29900.5,...}'
curl ... -d '{"loanAmount":2.99e2,...}'
curl ... -d '{"loanAmount":"29900.50",...}'
```

通过条件：业务码 = `PARAM_ERROR`；不出现金额被截断/向下取整入库

#### LOS-A.6 服务端价格复算（核心）

```bash
# 客户端传一个错误的 loanAmount，看服务端是否使用产品本身价格
curl ... -d '{"productCode":"P001","loanAmount":1,...}'   # 该产品实际 29900 分
```

通过条件：

1. 服务端基于 `productCode` 查询 `BenefitProduct.price_cents` 作为权威价格
2. 业务码 = `PRICE_MISMATCH` 或服务端忽略 loanAmount，按产品价创建订单
3. SQL 验证：`benefit_order.pay_amount == BenefitProduct.price_cents`，**与 loanAmount 无关**
4. **若服务端按 loanAmount=1 入库**：P0 资损 FAIL

#### LOS-A.7 已下架产品

```bash
curl ... -d '{"productCode":"P_OFFLINE_001",...}'
```

通过条件：业务码 = `PRODUCT_INACTIVE` 或 `PRODUCT_NOT_FOUND`；不创建订单

#### LOS-A.8 还款金额边界

```bash
# 还款额 > 应还
curl ... POST /api/repayment/submit \
  -d '{"loanId":"LN_001","repayAmount":99999999,...}'

# 还款额 = 0
curl ... -d '{"repayAmount":0,...}'

# 部分还款（业务策略待确认）
curl ... -d '{"repayAmount":1,...}'
```

通过条件：

1. 还款额 > 应还 → 业务码 = `REPAY_AMOUNT_EXCEED`
2. 还款额 = 0 → PARAM_ERROR
3. 部分还款若不允许 → 拒绝；若允许 → 验证剩余应还正确递减

### 5.2 LOS-B 重复扣款

#### LOS-B.1 回调重放（同 requestId）

```bash
# 触发支付成功回调两次（同 requestId）
PAYLOAD='{"requestId":"REQ_LOS_B_1_001","bizOrderNo":"BO_LOS_B_1_001","status":"SUCCESS","amount":29900,"providerSwiftNo":"SW001"}'
curl ... POST /api/callbacks/first-deduction -d "$PAYLOAD"
curl ... POST /api/callbacks/first-deduction -d "$PAYLOAD"
```

通过条件：

1. 两次 HTTP 200
2. SQL：`payment_record WHERE callback_request_id='REQ_LOS_B_1_001'` 行数 = 1
3. SQL：`benefit_order.status` 不被二次推进
4. SQL：金额合计 = 29900 分（一次扣款）

#### LOS-B.2 回调重放（同 providerSwiftNo 不同 requestId）

```bash
# 上游可能用同 swift no 重发，但本地 requestId 不同 — 验证是否基于业务唯一性去重
curl ... -d '{"requestId":"REQ_LOS_B_2_A_001","providerSwiftNo":"SW002","bizOrderNo":"BO_LOS_B_2_001",...}'
curl ... -d '{"requestId":"REQ_LOS_B_2_B_001","providerSwiftNo":"SW002","bizOrderNo":"BO_LOS_B_2_001",...}'
```

通过条件：

1. 第二次返回幂等响应或 `DUPLICATE_PROVIDER_SWIFT`
2. SQL：`payment_record WHERE provider_swift_no='SW002'` 行数 = 1
3. **预期可能 FAIL**：当前实现可能仅按 requestId 去重；记录为缺陷

#### LOS-B.3 请求重发 — 创建订单

```bash
# 同 requestId 连发 2 次创建订单
curl ... POST /api/equity/orders \
  -d '{"requestId":"REQ_LOS_B_3_001","productCode":"P001","loanAmount":29900,"agreementSigned":true}'
curl ... POST /api/equity/orders \
  -d '{"requestId":"REQ_LOS_B_3_001","productCode":"P001","loanAmount":29900,"agreementSigned":true}'
```

通过条件：

1. SQL：`benefit_order WHERE biz_order_no` 关联此 requestId 的订单数 = 1
2. 两次返回相同 benefitOrderNo
3. 上游齐为只调用 1 次

#### LOS-B.4 并发请求 — 同用户同时下两单同产品

```bash
# 5 路并发
for i in 1 2 3 4 5; do
  curl ... -d '{"requestId":"REQ_LOS_B_4_'$i'","productCode":"P001",...}' &
done
wait
```

通过条件：

1. 不出现金额合计为 5 倍的扣款
2. **业务策略待确认**：是否允许同用户多笔同产品订单同时存在？若不允许，应拒绝并发；若允许，验证每笔独立

#### LOS-B.5 补偿任务重入

```bash
# 触发会员同步超时（FAULT_TIMEOUT）
curl ... POST /api/equity/orders \
  -d '{"requestId":"REQ_LOS_B_5_FAULT_TIMEOUT","productCode":"P001",...}'

# 立即用同 requestId 重发（IdempotencyService 已记录）
curl ... POST /api/equity/orders \
  -d '{"requestId":"REQ_LOS_B_5_FAULT_TIMEOUT","productCode":"P001",...}'
```

通过条件：

1. 第二次命中幂等，返回相同 benefitOrderNo
2. SQL：`async_compensation_task WHERE biz_key=<orderId>` 行数 = 1（不重复入队）
3. 不二次推进订单状态

#### LOS-B.6 借款 apply 超时后重试（架构已知缺陷验证）

参见 `20260429_艾博生异常case测试方案.md` P0-2.2。本组在资损视角验证：

```bash
curl ... POST /api/loan/apply \
  -d '{"requestId":"REQ_LOS_B_6_FAULT_TIMEOUT","benefitOrderNo":"BO_LOS_B_6_001",...}'
# 重试
curl ... POST /api/loan/apply \
  -d '{"requestId":"REQ_LOS_B_6_NORMAL","benefitOrderNo":"BO_LOS_B_6_001",...}'
```

通过条件：

1. SQL：`benefit_order WHERE biz_order_no='BO_LOS_B_6_001'` 行数 = 1
2. SQL：`loan_application WHERE benefit_order_no='BO_LOS_B_6_001'` 行数 ≤ 1
3. **预期 FAIL**：架构已知缺陷会产生 2 条 benefit_order 或 2 条 loan_application

### 5.3 LOS-C 状态机倒退

#### LOS-C.1 SUCCESS → FAILURE 倒退

```bash
# 先 success
curl ... POST /api/callbacks/first-deduction \
  -d '{"requestId":"REQ_LOS_C_1_A","bizOrderNo":"BO_LOS_C_1","status":"SUCCESS","amount":29900,"providerSwiftNo":"SW101"}'

# 后 failure（不同 requestId 同 bizOrderNo）
curl ... POST /api/callbacks/first-deduction \
  -d '{"requestId":"REQ_LOS_C_1_B","bizOrderNo":"BO_LOS_C_1","status":"FAILURE","reason":"insufficient_funds"}'
```

通过条件：

1. 第二次返回 `STATE_CONFLICT` / `STATE_NOT_TRANSITIONABLE` 或幂等"已终态"
2. SQL：`benefit_order.status` 仍为 SUCCESS（不被 FAILURE 覆盖）
3. SQL：`payment_record` 不增加 FAILURE 记录覆盖原 SUCCESS

#### LOS-C.2 FAILURE → SUCCESS 倒退

```bash
# 先 failure
curl ... -d '{"requestId":"REQ_LOS_C_2_A","status":"FAILURE",...}'
# 后 success
curl ... -d '{"requestId":"REQ_LOS_C_2_B","status":"SUCCESS","amount":29900,"providerSwiftNo":"SW102",...}'
```

通过条件：

1. **业务策略待确认**：上游可能确实"先失败后成功"（如重试成功），是否允许？
2. 若允许：验证最终态为 SUCCESS，且只一次扣款
3. 若不允许：第二次拒绝

#### LOS-C.3 跳过中间态

```bash
# 直接调放款回调，跳过支付成功
curl ... POST /api/callbacks/exercise-equity \
  -d '{"requestId":"REQ_LOS_C_3","bizOrderNo":"BO_NEW_NEVER_PAID",...}'
```

通过条件：业务码 = `STATE_NOT_TRANSITIONABLE` / `PRECONDITION_FAILED`；不推进未支付订单到行权态

#### LOS-C.4 退款触发借款？（业务约束）

```bash
# 退款回调时该订单尚未放款 — 验证状态机不会把"退款失败"算作"放款成功"
curl ... POST /api/callbacks/refund \
  -d '{"requestId":"REQ_LOS_C_4","bizOrderNo":"BO_NEW","status":"SUCCESS","refundAmount":29900}'
```

通过条件：业务码 = `STATE_CONFLICT`；订单状态保持原值

### 6.1 LOS-D 退款一致性

#### LOS-D.1 未支付订单退款

```bash
# 创建订单但未触发回调（仍处于 FIRST_DEDUCT_PENDING）
ORDER=$(curl ... POST /api/equity/orders -d '...' | jq -r .data.benefitOrderNo)
# 立即申请退款
curl ... POST /api/refund/apply \
  -d '{"requestId":"REQ_LOS_D_1","benefitOrderNo":"'${ORDER}'","reason":"test"}'
```

通过条件：业务码 = `REFUND_NOT_ALLOWED` / `ORDER_NOT_PAID`；refund_record 无新增

#### LOS-D.2 退款金额超过原支付金额

```bash
# 已成功支付 29900 分，尝试退 50000 分
curl ... POST /api/refund/apply \
  -d '{"benefitOrderNo":"'${PAID_ORDER}'","refundAmount":50000}'
```

通过条件：业务码 = `REFUND_AMOUNT_EXCEED`；refund_record 无新增

#### LOS-D.3 重复退款

```bash
# 同订单连续两次退款
curl ... POST /api/refund/apply -d '{"requestId":"REQ_LOS_D_3_A","benefitOrderNo":"'${PAID_ORDER}'","refundAmount":29900}'
curl ... POST /api/refund/apply -d '{"requestId":"REQ_LOS_D_3_B","benefitOrderNo":"'${PAID_ORDER}'","refundAmount":29900}'
```

通过条件：

1. 第二次业务码 = `DUPLICATE_REFUND` / `ORDER_REFUNDED`
2. SQL：`refund_record WHERE biz_order_no` 退款合计 ≤ 原支付金额

#### LOS-D.4 退款失败状态回滚

```bash
# 退款流程中触发 FAULT_REJECT
curl ... POST /api/refund/apply \
  -d '{"requestId":"REQ_LOS_D_4_FAULT_REJECT_500","benefitOrderNo":"'${PAID_ORDER}'","refundAmount":29900}'
```

通过条件：

1. HTTP 200，业务码非 0
2. SQL：`refund_record.status` 为失败态（如 `FAILED`），不是 SUCCESS
3. SQL：`benefit_order` 状态保持原值（未被错误标记为已退）

#### LOS-D.5 部分退款累计校验

```bash
# 已支付 29900；多次部分退款累计
curl ... -d '{"refundAmount":10000}'   # 第一次 100 元
curl ... -d '{"refundAmount":10000}'   # 第二次 100 元
curl ... -d '{"refundAmount":10000}'   # 第三次 100 元，累计 30000 > 29900
```

通过条件：第三次拒绝；SQL：累计退款 ≤ 原支付金额；**业务策略待确认**：是否允许部分退款

### 6.2 LOS-E 上游状态分歧

#### LOS-E.1 齐为成功 ABS 失败

模拟方式：齐为 mock 返回成功（默认 happy-path），但 ABS 落库前抛异常（无法直接通过黑盒触发；本用例**通过日志注入异常依赖白盒**，标为 SKIP，仅记录架构关注点）

#### LOS-E.2 ABS 成功 齐为失败（FAULT_REJECT）

```bash
curl ... POST /api/equity/orders \
  -d '{"requestId":"REQ_LOS_E_2_FAULT_REJECT_500","productCode":"P001",...}'
```
