## 6. 关键问题与初步判断

### 问题 1：MySQL 基线缺少 H5 主流程所需权益产品

**现象**

- 首页首次提交借款失败；
- 原因为 `benefit_product` 表中不存在配置产品 `HUXUAN_CARD`。

**影响**

- H5 主流程无法直接起跑；
- 需要人工补种子或初始化脚本。

**结论**

- 这是测试环境基线问题，不是页面调用链本身的问题。

### 问题 2：审批状态链路与数据库映射不一致

**现象**

- `loan_application_mapping` 已成功落库，并保存了上游 `loanId`；
- 审批中页多次轮询仍停留在 `reviewing`；
- 审批结果页最终返回 `rejected`，且 `loanId = null`。

**已确认事实**

1. `POST /api/loan/apply` 已成功返回 `applicationId`；
2. `loan_application_mapping` 已保存 `upstream_query_value=LN-9731edd8858644d691862e6439c4d503`；
3. 权益开通也已成功；
4. 但结果页最终口径为“借款未通过，权益已购买成功”。

**初步判断**

- 需要进一步排查 `approval-status` / `approval-result` 对云卡 `/loan/query` 返回值的映射逻辑；
- 也需要确认当前本地云卡 stub 与本侧状态映射是否存在口径不一致。

> 注：这是基于本轮测试现象的初步判断，不等同于最终根因结论。

### 问题 3：还款试算金额异常为 0

**现象**

- `GET /api/repayment/info/{loanId}` 返回 `repaymentAmount=0.00`；
- 页面展示 `确认支付 ¥0`；
- `POST /api/repayment/submit` 因 `amount=0` 被校验拦截。

**影响**

- 无法自然打通还款提交成功与还款成功页。

**初步判断**

- 需要排查 `repayment/info` 对云卡 `/repay/trial` 返回金额的转换或映射逻辑；
- 也需要确认 `loanId` 对应借款单在本侧后端是否满足还款试算前提条件。

> 注：这是基于本轮测试现象的初步判断，不等同于最终根因结论。

## 10. 根因复核与第二轮复测补充（2026-04-15 17:12）

### 10.1 根因结论

经对 `approval-result` 与 `repayment/info` 两条异常链路做最小化复核，确认本轮异常的**根因不是业务代码逻辑本身**，而是**后端启动参数错误**：

- 当时 MySQL 后端虽然已成功连接真实 MySQL；
- 但启动时**未带上**本地云卡联调参数；
- 实际运行配置仍为默认：`YUNKA_MODE=MOCK`。

在该模式下：

1. `POST /api/loan/apply` 仍可“看起来成功”
   - 因为接口对空云卡响应使用了本地 fallback，能生成 `applicationId`、`loanId`
2. `GET /api/loan/approval-status/{applicationId}`
   - 因拿到的是空 `data`，`status` 被映射为 `reviewing`
3. `GET /api/loan/approval-result/{applicationId}`
   - 因拿到的是空 `data`，最终被判定为 `rejected`
4. `GET /api/repayment/info/{loanId}`
   - 因拿到的是空 `data`，`repayAmount` 被解析为 `0.00`

因此，前一轮报告中的两类异常：

- 审批结果异常；
- 还款金额为 0；

其共同根因均为：

> **后端连接了真实 MySQL，但未切换到本地云卡 REST stub，而是以默认 MOCK 模式运行。**

2. 本轮复测后，`payment_record` 表仍未新增本次 H5 还款记录，说明当前还款页面链路的“提交成功”主要依赖云卡代理返回与页面结果查询，不代表本地账务表已完成落库；
3. 若后续要把“H5 还款成功”与本地账务落库做强一致，还需单独确认 `payment_record` 是否属于该链路的设计范围。

### 10.8 最终口径

> 在正确的本地联调启动参数下，H5 页面维度主流程已跑通，且关键前置与业务映射数据已成功写入 MySQL。前一轮异常的主要根因是运行环境配置错误，而非当前业务代码本身失效。
