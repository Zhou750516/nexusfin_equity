# 2026-05-01 对外联调前技术检查清单

## 1. 适用范围

本清单面向 `2026-05-07` 前的开发侧 / 测试侧联调准备，目标是把“本地已证明的部分”和“外部尚待确认的部分”明确拆开，避免把本地 mock 结论误说成对外真实联调结论。

当前已稳定的本地真值：

1. `joint-login`
2. `POST /api/auth/redrect_benefit_url`
3. `POST /api/loan/calculate`
4. `GET /api/benefits/card-detail`
5. `POST /api/benefits/activate`
6. `GET /api/repayment/info/{loanId}`

当前不在本清单承诺范围内：

1. Yunka 公网真实可用性闭环
2. 电子签章最终方案
3. QW 全量异常矩阵
4. callback 全量对外回归

## 2. 联调前统一口径

联调前先统一 4 条口径，避免讨论时混淆：

1. 本地唯一稳定基线仍是 `Yunka=REST + 本地 stub`、`QW=MOCK`、`tech-user=本地 stub`。
2. `benefiturl` 当前已经接入主链并真实出站，但其运行态来源是 `BenefitRedirectUrlService.generate(...) -> QW exercise redirect URL`。
3. `benefiturl` 当前满足同步报文字段要求，但不自动等价于“所有权益场景通用联登 URL”。
4. `joint-login` 当前边界已拆清：
   - `push` 不要求 `benefitOrderNo`
   - `exercise` 要求 `benefitOrderNo`
   - `refund` 要求 `benefitOrderNo`

## 3. 本地 / 测试环境配置检查

### 3.1 本地回归基线

以下项用于联调前自检，确保本地基线未漂移：

| 检查项 | 期望值 | 说明 |
| --- | --- | --- |
| backend | `http://127.0.0.1:8080` | 本地后端 |
| H5 | `http://127.0.0.1:3000` | 本地前端 |
| Yunka mode | `REST` | 走 `RestYunkaGatewayClient` |
| Yunka baseUrl | `http://127.0.0.1:18081` | 仅本地自检使用 |
| Yunka gatewayPath | `/api/gateway/proxy` | 必须与当前代码配置一致 |
| QW mode | `MOCK` | 当前本地稳定真值 |
| tech-user stub | `http://127.0.0.1:18080` | 本地 SSO / `/api/users/me` 依赖 |
| cookie secure | `false` | 本地 `http` 写 cookie 需要 |

建议重点核对的配置项：

1. `nexusfin.third-party.yunka.mode`
2. `nexusfin.third-party.yunka.base-url`
3. `nexusfin.third-party.yunka.gateway-path`
4. `nexusfin.third-party.yunka.channel-code`
5. `nexusfin.third-party.yunka.signature`
6. `nexusfin.third-party.qw.mode`
7. `nexusfin.auth.tech-platform-base-url`
8. `nexusfin.auth.jwt.cookie-secure`

### 3.2 对外联调切换前提

如果准备切到对外联调环境，以下项必须先有明确值，不能继续使用占位值：

| 检查项 | 当前状态 | 联调前要求 |
| --- | --- | --- |
| Yunka `baseUrl` | 本地为 `127.0.0.1:18081` | 必须替换为对方提供的真实测试地址 |
| Yunka `X-Channel-Code` | 已支持配置 | 需确认外部认可的正式值 |
| Yunka `X-Signature` | 当前支持配置占位值 | 需确认真实签名算法、原串和密钥分发 |
| QW `baseUrl/method-path` | 当前代码已承接 | 若 5 月 7 日要切真实 QW，需确认直连证书 / 商户号 / 签名物料 |
| 科技平台域名 | 文档仍未完全闭合 | 需提供测试域名和必要接口路径 |

### 3.3 联调前启动日志必须检查

后端启动后，至少确认下面 3 组日志：

1. `qw benefit client routing initialized mode=...`
2. `yunka gateway client initialized enabled=true mode=REST baseUrl=... gatewayPath=...`
3. `Started NexusfinEquityApplication`

联调前判定规则：

1. 如果仍看到 `baseUrl=http://127.0.0.1:18081`，说明当前还是本地 Yunka stub。
2. 如果仍看到 `baseUrl=https://yunka.local`，说明 Yunka 真实地址根本没配置。
3. 如果 `QW mode` 不是预期值，联调记录必须显式标注，不允许默认沿用旧口径。

## 4. Yunka / QW / 科技平台联调前提

### 4.1 Yunka

联调前必须确认：

1. Yunka gateway 真实测试地址
2. Yunka 统一入口 `gatewayPath`
3. `X-Trace-Id / X-Request-Id / X-Biz-Order-No / X-Timestamp / X-Channel-Code / X-Signature` 是否全部按当前协议受理
4. 借款试算路径真值是否为 `/loan/trial`
5. 当前 `KJ_NOT_READY / kj.private-key 配置无效` 的修复时间点
6. 电子签章接入方案是否已定稿

### 4.2 齐为

联调前必须确认：

1. 本轮是否继续使用 `QW=MOCK` 作为借款 / 权益主链联调基线
2. 如果要切真实齐为，测试环境 `baseUrl`、商户号、签名物料、证书路径是否齐备
3. `abs.sign.query / abs.sign.apply / abs.sign.confirm / abs.push.order` 的失败码和失败 `data` 结构是否已有对外样例
4. `cardNo` 与“续期 / 非首开”语义是否会影响当前会员同步联调

### 4.3 科技平台

联调前必须确认：

1. 测试环境域名
2. 登录 / 用户信息 / 页面跳转能力边界
3. 借款试算真实 path 和字段口径
4. 回调地址、header、签名、重试策略
5. 如需使用 `benefiturl`，对方是否接受当前“QW exercise redirect URL 来源”的实现语义

## 5. 日志与可观测性检查项

联调前每条代表性链路都必须能在同一 `traceId` 下串起关键日志。

### 5.1 必看字段

后端日志至少要有：

1. `traceId`
2. `path`
3. `requestId`
4. `bizOrderNo`
5. `requestBodyJson`
6. `responseBodyJson`
7. `elapsedMs`
8. `yunkaCode` 或等价上游返回码
9. `errorNo / errorMsg`

### 5.2 Yunka 出站日志检查

Yunka 代表性调用至少要看到：

1. `yunka gateway request begin`
2. `requestBodyJson=...`
3. `yunka gateway request success` 或 `yunka gateway request failed`
4. `responseBodyJson=...`

如日志里看不到完整报文，联调记录必须明确写：

1. 缺 `requestBodyJson`
2. 缺 `responseBodyJson`
3. 无法仅靠艾博生日志还原真实报文

### 5.3 主链关键观测点

联调时建议至少回查以下关键词：

1. `joint login entry resolved targetPage=`
2. `benefit redirect url resolved from qw exercise redirect url`
3. `benefits sync benefiturl generated`
4. `yunka gateway request begin`
5. `yunka gateway request success`
6. `yunka gateway request failed`

## 6. benefiturl 当前语义说明

当前必须按以下口径对外描述，不要扩写：

1. `benefiturl` 已经接入权益同步主链。
2. 当前值来源于 `BenefitRedirectUrlService.generate(...)`。
3. 当前运行态真实来源是 `QW exercise redirect URL`。
4. 当前已经满足主链同步字段要求和本地运行态闭环。
5. 当前不应自动泛化为“所有权益场景通用联登 URL”。

对外说法建议：

1. 可以说“字段已接线并真实出站”
2. 不要说“权益联登 URL 语义已经完全闭合”

## 7. joint-login 当前边界

当前 joint-login 对外只承诺下面这些：

1. `push` 场景可在无 `benefitOrderNo` 下建立登录态并跳转导流页
2. `exercise` 场景必须带 `benefitOrderNo`
3. `refund` 场景必须带 `benefitOrderNo`
4. 非 token 失效类上游 reject 已不再误报为 `JOINT_LOGIN_TOKEN_INVALID`

联调时若遇到 joint-login 问题，先区分：

1. token 无效 / 过期
2. 上游路径未就绪
3. 外部用户信息接口未就绪
4. 页面场景参数缺失
