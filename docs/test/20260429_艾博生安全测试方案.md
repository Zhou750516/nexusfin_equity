## 2. 安全维度划分

| 编号 | 维度 | 关键问题 |
|---|---|---|
| SEC-A | 认证（Authentication）| 谁在调用？JWT 是否有效、未篡改、未过期？ |
| SEC-B | 鉴权（Authorization）| 调用者能不能访问/操作这个资源？水平/垂直越权？ |
| SEC-C | 签名校验 | 第三方回调是否真的来自合作方？签名是否启用并生效？ |
| SEC-D | 输入安全 | 参数注入、类型混淆、超长输入、Content-Type 攻击 |
| SEC-E | 敏感数据保护 | 响应、日志、异常信息是否脱敏？AES 密钥保护？ |
| SEC-F | 速率限制 / 防刷 | 短时间内同 IP / 同 userId / 同卡号高频调用是否被拦截？ |
| SEC-G | 传输安全 | HTTPS 强制、Cookie 安全属性、TLS 版本 |

## 3. 当前安全机制盘点

| 机制 | 实现位置 | 现状 |
|---|---|---|
| JWT 认证 | `JwtAuthenticationFilter`、`AuthProperties.Jwt`、`JwtUtil` | ✅ Cookie `NEXUSFIN_AUTH`，TTL 7200s，httpOnly，sameSite=Lax，secure=true（默认）；按 `protectedPathPrefixes` 白名单生效 |
| 签名拦截 | `SignatureInterceptor`、`SignatureProperties` | ⚠️ 仅覆盖 `protectedPathPrefixes` 配置项；未配置即不校验。`X-App-Id / X-Timestamp / X-Nonce / X-Signature` 4 头必填，maxSkew 校验时间偏移 |
| AES 加解密 | `SensitiveDataCipher`、`CryptoProperties` | ✅ 手机号/身份证/银行卡等用 AES（参考 BankCardSign / AuthService / Repayment） |
| Nonce 防重放 | `SignatureInterceptor` | ❌ 仅校验时间窗，未持久化 nonce → **同窗口内同 nonce 可重放** |
| 鉴权（越权检查）| Service 层 | ⚠️ JWT Filter 解析 `userId` 入 `AuthContextUtil`；service 是否对所有资源访问做归属校验**需逐接口验证** |
| 速率限制 | — | ❌ 全局缺失（pom.xml 无 sentinel/resilience4j/RateLimit 依赖）|
| 审计日志 | — | ❌ ABS 侧无 `sys_audit` 表（云卡侧 `sys_audit_log` 不属于本服务）|
| HTTPS 强制 | 容器/网关层 | ⚠️ 应用本身不强制；依赖部署侧；本地开发 `cookie-secure=false` |

**关键缺口（codex 需重点验证）**：

1. `SignatureProperties.protectedPathPrefixes` 实际配置了哪些路径？（决定签名是否真的在保护回调）
2. 各业务接口是否都做了**资源归属校验**（A 用户不能访问 B 用户的订单）
3. Nonce 不持久化是否构成重放风险（在 maxSkew 时间窗内）
4. 错误响应是否泄露上游错误细节、SQL 错误、栈帧

## 5. P0 高风险用例集

### 5.1 SEC-A 认证（JWT）

#### SEC-A.1 缺失 cookie

```bash
curl -sS -m 10 -i \
  "http://127.0.0.1:8080/api/equity/orders/${ORDER_A}"
```

通过条件：

1. HTTP `200`（统一封装）或 `401`（按实现）
2. 业务码体现 `UNAUTHORIZED`，errorMsg `Missing auth cookie`
3. **不命中受保护资源**（不返回订单详情）
4. 日志含 `errorNo=UNAUTHORIZED`

#### SEC-A.2 cookie 篡改（payload 修改）

```bash
# 把 JWT_A 的 payload 部分改 userId（不重签），构造 JWT_BAD
curl -sS -m 10 -i -b "NEXUSFIN_AUTH=${JWT_BAD}" \
  "http://127.0.0.1:8080/api/equity/orders/${ORDER_A}"
```

通过条件：

1. HTTP 401 / 业务码 UNAUTHORIZED，errorMsg `Invalid auth cookie`
2. 不返回订单数据
3. 日志记录 `IllegalArgumentException` 被 filter 捕获

#### SEC-A.3 cookie 过期

```bash
curl -sS -m 10 -i -b "NEXUSFIN_AUTH=${JWT_EXPIRED}" \
  "http://127.0.0.1:8080/api/equity/orders/${ORDER_A}"
```

通过条件：同 SEC-A.2，errorMsg 体现"过期"语义；同时**不应触发自动续期**（除非业务约定）

#### SEC-A.4 跨保护路径白名单测试

```bash
# 健康检查应可未登录访问
curl -sS -m 10 -i "http://127.0.0.1:8080/api/equity/health"

# 受保护接口未登录应被拒
curl -sS -m 10 -i "http://127.0.0.1:8080/api/loan/calculator-config"
```

通过条件：

1. `/api/equity/health` 返回 `success=true`，无认证要求
2. `/api/loan/*` 未登录被拒
3. **关键风险标注**：核对 `AuthProperties.protectedPathPrefixes / excludedPathPrefixes` 的实际配置值，确保所有写操作都在 protected 列表内

### 5.2 SEC-B 鉴权（越权）

#### SEC-B.1 水平越权 — 查询他人订单

```bash
# A 用 自己的 cookie 查 B 的订单
curl -sS -m 10 -i -b "NEXUSFIN_AUTH=${JWT_A}" \
  "http://127.0.0.1:8080/api/equity/orders/${ORDER_B}"
```

通过条件：

1. HTTP 200 / 业务码 = `FORBIDDEN` 或 `RESOURCE_NOT_FOUND`（**不能返回 B 的订单详情**）
2. SQL 验证：A 拿不到 B 的 member_id 关联数据
3. **关键风险标注**：若返回详情，记录为 P0 越权缺陷

#### SEC-B.2 水平越权 — 操作他人订单

```bash
# A 尝试用 B 的订单号发起退款
curl -sS -m 10 -i -b "NEXUSFIN_AUTH=${JWT_A}" \
  -X POST "http://127.0.0.1:8080/api/refund/apply" \
  -H "Content-Type: application/json" \
  -d "{\"requestId\":\"REQ_SEC_B_2_001\",\"benefitOrderNo\":\"${ORDER_B}\",\"reason\":\"test\"}"
```

通过条件：

1. 业务码 = FORBIDDEN / NOT_FOUND
2. SQL 验证：B 的订单状态未被改动；refund 表无新增
3. 日志含越权拦截记录

#### SEC-B.3 水平越权 — 通过 path 参数

对每个 path 含 `{id} / {applicationId} / {repaymentId} / {benefitOrderNo} / {refundId}` 的接口至少跑一次（A 用 B 的 ID）：

| 接口 | path 参数 |
|---|---|
| GET `/api/equity/orders/{benefitOrderNo}` | benefitOrderNo |
| GET `/api/equity/exercise-url/{benefitOrderNo}` | benefitOrderNo |
| GET `/api/loan/approval-status/{applicationId}` | applicationId |
| GET `/api/loan/approval-result/{applicationId}` | applicationId |
| GET `/api/repayment/info/{loanId}` | loanId |
| GET `/api/repayment/result/{repaymentId}` | repaymentId |
| GET `/api/refund/info/{benefitOrderNo}` | benefitOrderNo |
| GET `/api/refund/result/{refundId}` | refundId |
| GET `/api/benefit-dispatch/context/{benefitOrderNo}` | benefitOrderNo |

通过条件：每个接口都拒绝；任一返回他人数据 → 立即记录该接口为越权 FAIL

#### SEC-B.4 垂直越权 — 普通用户访问 admin

```bash
# 假设有 admin endpoint
curl -sS -m 10 -i -b "NEXUSFIN_AUTH=${JWT_A}" \
  "http://127.0.0.1:8080/api/admin/benefit-orders?pageSize=10"
```

通过条件：

1. HTTP 403 / 业务码 = `FORBIDDEN`
2. **若 admin 不存在**：SKIP 并标注；当前 ABS 缺乏 admin 端鉴权机制是已知架构缺口

### 5.3 SEC-C 签名校验（回调入口）

#### SEC-C.1 签名保护路径配置确认（前置）

```bash
# 开发者运行时获取或贴出：
# SignatureProperties.protectedPathPrefixes = [...]
# 如果列表为空或不含 /api/callbacks/* → 全部回调接口未受签名保护
```

通过条件：列表非空且至少包含 `/api/callbacks/`、`/api/callbacks/payment-protocol`、`/api/callbacks/exercise-equity`、`/api/callbacks/refund` 等核心回调路径

**若配置为空**：直接判 P0 FAIL，整个 SEC-C 后续用例可继续跑作为缺陷验证

#### SEC-C.2 缺少签名 header

```bash
curl -sS -m 10 -i -X POST \
  "http://127.0.0.1:8080/api/callbacks/payment-protocol" \
  -H "Content-Type: application/json" \
  -d '{"requestId":"REQ_SEC_C_2_001","bizOrderNo":"BO_SEC_C_2_001","status":"SUCCESS"}'
```

通过条件：

1. 业务码 = `SIGNATURE_MISSING`
2. 回调副作用未执行（SQL 验证 protocol 未变更）

#### SEC-C.3 签名错误

```bash
curl -sS -m 10 -i -X POST "${URL}" \
  -H "X-App-Id: ${VALID_APP_ID}" \
  -H "X-Timestamp: $(date +%s)" \
  -H "X-Nonce: nonce-001" \
  -H "X-Signature: BAD_SIGNATURE" \
  -H "Content-Type: application/json" \
  -d '{...}'
```

通过条件：业务码 = `SIGNATURE_INVALID`；副作用未执行

#### SEC-C.4 时间戳超出 maxSkew

```bash
# 时间戳取 1 小时前
OLD_TS=$(($(date +%s) - 3600))
curl ... -H "X-Timestamp: ${OLD_TS}" ...
```

通过条件：业务码 = `SIGNATURE_EXPIRED`

#### SEC-C.5 nonce 重放（**关键**）

```bash
# 使用合法签名发一次
curl ... -H "X-Nonce: nonce-replay-001" -H "X-Signature: ${SIG_001}" ...
# 立即用同 nonce 同 signature 同 timestamp 重发
curl ... -H "X-Nonce: nonce-replay-001" -H "X-Signature: ${SIG_001}" ...
```

通过条件：

1. 第一次成功
2. **第二次应被拒为重放**
3. **预期 FAIL**：当前 SignatureInterceptor 不持久化 nonce，第二次会成功 → 记录为 P0 重放缺陷

#### SEC-C.6 appId 不匹配

```bash
curl ... -H "X-App-Id: WRONG_APP_ID" ...
```

通过条件：业务码 = `SIGNATURE_INVALID`，errorMsg `Invalid app id`

## 6. P1 用例集

### 6.1 SEC-D 输入安全

#### SEC-D.1 SQL 注入（productCode 字段）

```bash
curl -sS -m 10 -i -b "NEXUSFIN_AUTH=${JWT_A}" -X POST \
  "http://127.0.0.1:8080/api/equity/orders" \
  -H "Content-Type: application/json" \
  -d '{"requestId":"REQ_SEC_D_1_001","productCode":"P001'"'"' OR '"'"'1'"'"'='"'"'1","loanAmount":100000,"agreementSigned":true}'
```

通过条件：

1. HTTP 200，业务码 = `PARAM_ERROR` 或 `PRODUCT_NOT_FOUND`（参数未被作为 SQL 拼接）
2. 数据库未发生未授权读取
3. 错误响应不泄露 SQL 错误堆栈

#### SEC-D.2 路径参数注入（订单号）

```bash
curl ... GET "http://127.0.0.1:8080/api/equity/orders/$(printf 'BO\xff%s' 'INJECT')"
curl ... GET "http://127.0.0.1:8080/api/equity/orders/../admin"
```

通过条件：业务码 = `PARAM_ERROR` 或 `RESOURCE_NOT_FOUND`；不出 5xx

#### SEC-D.3 数值类型混淆

```bash
curl ... -d '{"loanAmount":"abc",...}'
curl ... -d '{"loanAmount":[1,2,3],...}'
curl ... -d '{"loanAmount":null,...}'
```

通过条件：业务码 = `PARAM_ERROR`（由 `@MethodArgumentTypeMismatchException` / `@MethodArgumentNotValidException` 处理）；不出 5xx

#### SEC-D.4 超长字符串

```bash
LONG=$(printf 'A%.0s' {1..10000})
curl ... -d "{\"productCode\":\"${LONG}\",...}"
```

通过条件：业务码 = `PARAM_ERROR`；DB 写入未发生（如有 `@Size` 校验）；不出 OOM / 5xx

#### SEC-D.5 Content-Type 攻击

```bash
# 用 form 提交 JSON
curl ... -H "Content-Type: application/x-www-form-urlencoded" \
  --data-urlencode 'productCode=P001&loanAmount=100000'

# 缺少 Content-Type
curl ... -d '{"productCode":"P001"}'
```

通过条件：业务码体现"不支持的 Content-Type"或"消息体解析失败"；不命中 controller 业务逻辑

#### SEC-D.6 大 body / 慢 body

```bash
# 100MB body
dd if=/dev/zero bs=1M count=100 2>/dev/null | curl ... --data-binary @-
```

通过条件：被 server 拒绝（多大边界以 `server.tomcat.max-http-form-post-size` 等配置为准）

### 6.2 SEC-E 敏感数据保护

#### SEC-E.1 响应中敏感字段脱敏

```bash
# 查询订单/签约/还款信息，检查响应中的 mobile/idNo/accountNo
curl ... GET "http://127.0.0.1:8080/api/equity/orders/${ORDER_A}"
curl ... GET "http://127.0.0.1:8080/api/repayment/info/${LOAN_A}"
curl ... GET "http://127.0.0.1:8080/api/users/me"
```

通过条件：响应中的手机号、身份证、银行卡号必须**脱敏返回**（如 `138****1234`、`6228****5678`、`110101********1234`）；**绝不能返回密文密钥**或明文值

#### SEC-E.2 日志中敏感字段脱敏

```bash
# 触发一次创建订单
curl ... POST "http://127.0.0.1:8080/api/equity/orders" -d '{...含 mobile / idNo}'
# 然后检查日志（需开发者协助 grep）
grep "13800001234\|110101199001011234" /path/to/app.log
```

通过条件：

1. 日志中**不含手机号 / 身份证号原文**
2. 若需记录，使用 `SensitiveDataUtil.sha256` 或脱敏中段
3. **预期 FAIL**：当前未发现统一日志脱敏配置；记录为 P1 缺陷

#### SEC-E.3 错误响应不泄露内部信息

```bash
# 触发各种异常
curl ... GET "http://127.0.0.1:8080/api/equity/orders/NONEXISTENT"
curl ... POST "http://127.0.0.1:8080/api/equity/orders" -d 'malformed json'
```

通过条件：

1. errorMsg 对外文案简洁（不含 SQL 片段、不含完整堆栈、不含表名）
2. 5xx 兜底响应是否泄露了 GlobalExceptionHandler 之外的堆栈（如未捕获异常）

#### SEC-E.4 加密密钥配置审查（必要的白盒）

需开发者在结果报告贴出：

- `nexusfin.crypto.aes-key` 是否从环境变量 / 密钥管理服务获取，**不能 hardcode 在 application.yml**
- 测试环境与生产环境密钥是否分离

通过条件：环境变量驱动 + 测试/生产分离

## 7. P2 用例集

### 7.1 SEC-F 速率限制 / 防刷

> **现状**：项目无 rate limit 框架。本组用例**预期全部 FAIL**，目的是量化"无防护"的风险面。

#### SEC-F.1 同 IP 高频调用

```bash
# 100 次/秒 调一个高代价接口
for i in $(seq 1 100); do
  curl -sS -m 5 -b "NEXUSFIN_AUTH=${JWT_A}" \
    "http://127.0.0.1:8080/api/loan/calculate" \
    -H "Content-Type: application/json" \
    -d '{"requestId":"REQ_SEC_F_1_'$i'","loanAmount":100000,"loanPeriod":12}' &
done
wait
```

通过条件：

1. 服务可承受不崩
2. **理想**：触发 429 Too Many Requests / 业务码 RATE_LIMIT
3. **实际预期**：100 次都返回成功；记录为缺陷

#### SEC-F.2 同 userId 频繁触发短信

```bash
# 1 分钟内连发 20 次签约申请（若每次发短信，会消耗短信通道额度）
for i in $(seq 1 20); do
  curl ... POST /api/bank-card/sign-apply -d '{"requestId":"REQ_SEC_F_2_'$i'", ...}'
  sleep 3
done
```

通过条件：

1. **理想**：从某次起被拒绝（同 userId 短信频率限制）
2. **实际预期**：20 次都向上游齐为发起申请 → 记录为缺陷与潜在资损（短信费用）

#### SEC-F.3 同 accountNo 注册攻击

```bash
# 用同卡号 + 不同 userId 反复签约（探测他人卡是否已注册）
for uid in 10001 10002 10003 ...; do
  curl ... POST /api/bank-card/sign-status?accountNo=6228...
done
```

通过条件：

1. 不允许通过 status 接口枚举他人卡的签约状态（响应应去除区分性，或要求归属校验）
2. 当前实现**预期失败**：sign-status 不做归属校验，可被滥用

### 7.2 SEC-G 传输安全

#### SEC-G.1 HTTPS 强制（生产环境用例）


## 8. 执行顺序

```
[前置] 4.4 准备 JWT_A / JWT_B / JWT_BAD / JWT_EXPIRED + ORDER_A / ORDER_B
   ↓
SEC-A 认证（4 条）
   ↓
SEC-B 鉴权（4 条，B-3 横切多接口）
   ↓
SEC-C 签名校验（6 条）       ← C-1 必须先确认配置
   ↓
[阻断决策] SEC-A/B/C 任一 P0 FAIL（认证绕过 / 越权 / 签名缺保护）→ 标记主缺陷，可继续 P1
   ↓
SEC-D 输入安全（6 条）
   ↓
SEC-E 敏感数据保护（4 条）
   ↓
SEC-F 速率限制（3 条，预期大概率 FAIL）
   ↓
SEC-G 传输安全（3 条，部署侧）
```

**异常分支**：

- 服务返回 5xx / Tomcat 错误页 → 当条 FAIL，记录原始响应
- admin endpoint 不存在 → SKIP 并标注
- 越权用例返回 200 含他人数据 → 立即升级为 P0 FAIL，单独章节列出
