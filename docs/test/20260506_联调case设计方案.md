## 2. 当前真值前提

当前可直接复用的本地稳定链路：

1. `POST /api/auth/joint-login`
2. `POST /api/auth/redrect_benefit_url`
3. `POST /api/loan/calculate`
4. `GET /api/benefits/card-detail`
5. `POST /api/benefits/activate`
6. `GET /api/repayment/info/{loanId}`
7. `POST /api/repayment/submit`

当前必须继续保留的语义边界：

1. `benefiturl` 已接线且已出站，但当前运行态来源是 `QW exercise redirect URL`。
2. `benefiturl` 当前满足主链同步字段要求，不自动等价于“所有权益场景通用联登 URL”。
3. `joint-login` 当前边界已拆清：
   - `push` 不要求 `benefitOrderNo`
   - `exercise` 要求 `benefitOrderNo`
   - `refund` 要求 `benefitOrderNo`

## 3. 当前 FE+BE 规范入口

### 3.1 结论

当前 FE+BE 入口不是单一入口，而是**两者并存但分场景**：

| 入口 | 当前角色 | 适用场景 |
|---|---|---|
| H5 `/joint-entry` -> `POST /api/auth/joint-login` | 当前 joint-login 规范入口 | `push / exercise / refund` 联登场景；`redrect_benefit_url` 子链 |
| `GET /api/auth/sso-callback` | 当前本地 / 非 joint SSO 登录入口 | 本地 tech-user stub 登录、普通 H5 基线用户态建立、非 joint 页面手工演练 |

### 3.2 执行口径

1. 涉及 `push / exercise / refund` 的 case，一律以 `joint-login` 为规范入口。
2. 涉及本地普通用户态、非 joint 页面 baseline、手工建立 cookie 的 case，可继续使用 `/api/auth/sso-callback`。
3. 不把 `/api/auth/sso-callback` 误写成 joint-login 主入口。
4. 不把 `joint-login` 误写成通用 SSO 唯一入口。

### 3.3 与科技平台边界的关系

1. 当前 `/api/auth/sso-callback` 在本地主要承接 tech-user stub。
2. 当前 `joint-login` 在本地主要承接 Yunka `/user/token`、`/user/query` 子链。
3. 这两条入口都**不应**被表述为“艾博生已直接对接科技平台真实服务”的证明。

### 4.3 当前 §4 完成判定

当前文档下，`§4` 不再要求“单条全链一遍过”，而采用下面的完成判定：

1. `joint-login / redrect_benefit_url / loan/calculate / benefits/card-detail / benefits/activate / repayment/info / repayment/submit` 至少各有 1 条可复用代表性证据。
2. 每条代表性证据至少包含：
   - 请求入口
   - 响应结果
   - backend 日志
   - 必要时的 SQL 或页面证据
3. 不把 callback、真实 Yunka、公网 QW、科技平台直连细节混入本节完成标准。

## 5. 当前关键异常 / 边界 case

### 5.1 当前优先级说明

`§5` 的目标不再是一次性把旧异常矩阵全部跑完，而是先按当前基线给出四类状态：

1. `已有证据覆盖`
2. `待执行`
3. `N/A（因边界已收紧）`
4. `需先澄清入口`

### 5.3 当前 §5 完成判定

当前文档下，`§5` 的可执行目标是：

1. 已有证据覆盖项不再重复报成 blocker。
2. `待执行` 项由测试会话按入口清晰、依赖可控的顺序补跑。
3. `N/A` 项不再混入 FE+BE 主联调完成标准。
4. `需先澄清入口` 项先由主决策会话或开发会话确认入口，再转测试会话执行。

## 6. 当前收口标准

当前版本不再按“D1 / D2 / D3”打卡，而按下面标准收口：

1. 文档里的“已有证据覆盖”项，必须能在现有文档中找到可复用证据来源。
2. 文档里的“待执行”项，必须入口清晰、责任清晰，不混入外部待确认边界。
3. 文档里的“N/A”项，必须写清楚为什么不再作为当前 FE+BE 主联调项。
4. 文档里的“需先澄清入口”项，必须写清楚到底卡在入口、协议还是场景边界。

## 7. 后续轨道关系

本文件只定义当前 FE+BE 可执行 case，不把后续轨道混入执行标准。

后续轨道仍然存在，但仅保留边界说明：

1. BE↔Yunka：继续由云卡统一收口外部 path / 签名 / callback / 重试 / 电子签章问题。
2. BE↔齐为：继续围绕签约 3 步和会员同步口径推进。
3. 科技平台：默认不作为直接问题接收方；除非云卡明确存在直连 token / 回调 / 跳转边界。

## 8. 当前风险

| 风险 | 当前处理 |
|---|---|
| 把本地 mock 证据误说成公网真实联调结论 | 严禁；必须显式标注“本地稳定基线” |
| 把 `benefiturl` 误说成泛化权益联登 URL | 严禁；只写当前 runtime source 为 QW exercise redirect URL |
| 把 `/api/auth/sso-callback` 和 `joint-login` 混写成一个入口 | 严禁；必须分场景描述 |
| 把 callback / 科技平台接口细节混进 FE+BE 主执行集 | 严禁；callback 线独立，科技平台默认由云卡统一收口 |

## 9. 执行约定

1. 缺陷、阻塞、待确认项必须落文档，不口头流转。
2. case 执行时优先复用现有 `traceId`、日志、SQL、截图证据结构。
3. 所有外部服务细节问题，默认先收敛到云卡，不直接拆给科技平台。

## 10. 当前真实协作模型

### 10.1 主决策会话

职责：

1. 定义当前边界和优先级
2. 决定某个 case 是继续执行、下沉开发、还是标记 N/A
3. 确认哪些问题属于外部 blocker，哪些属于本地缺陷

### 10.2 开发会话

职责：

1. 修本地代码或文档中的真实缺陷
2. 准备可执行环境、日志口径、stub / mock 能力
3. 对“需先澄清入口”的 case 给出当前实现真值

### 10.3 测试会话

职责：

1. 按文档执行 `已有入口且依赖明确` 的 case
2. 记录请求、响应、日志、SQL、截图等证据
3. 不擅自扩边界；遇到入口不清或环境阻塞时回抛给主决策会话

### 10.4 协作规则

1. 主决策会话负责边界。
2. 开发会话负责实现和文档真值。
3. 测试会话负责执行与留证。
4. 不再使用旧版 6 角色 codex 并行执行模型。
