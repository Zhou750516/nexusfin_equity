# 2026-04-30 Yunka 真实联调启动清单

> 目的：为后续 Yunka 真实联调提供一份不会再跑偏到 QW 签约链的最小执行清单。本文档只解决 3 件事：
> 1. 哪些接口属于 Yunka，哪些不属于
> 2. 启动真实 Yunka 测试前必须先确认什么
> 3. 第一轮应该先打哪几条请求、看到什么日志、何时停止

## 1. 适用范围

- 适用于 `ABS -> Yunka` 的真实联调、恢复确认、最小烟测
- 不适用于 `QW` 签约异常线
- 不适用于本地 `Yunka stub` happy-path 回归

## 3. 启动真实 Yunka 测试前必须满足的前置

### 3.1 运行时必须满足

启动后先看 backend log，只要不满足以下任一条，就不要继续跑用例：

1. `yunka gateway client initialized enabled=true mode=REST ...`
2. `baseUrl` 不能是 `http://127.0.0.1:18081`
3. `baseUrl` 不能是占位值 `https://yunka.local`
4. `/api/equity/health` 正常

硬规则：

- 如果日志里仍是 `baseUrl=http://127.0.0.1:18081`，说明当前仍在本地 `Yunka stub`
- 这种情况下应直接停止，并回报：
  - `前置失败：当前不是 Yunka 真实服务，而是本地 Yunka stub`

### 3.2 本轮必须记录的基础信息

每次真实 Yunka 测试开跑前，必须先记录：

1. `git rev-parse --short HEAD`
2. backend 启动时间
3. Yunka 初始化日志原文
4. 本轮使用的真实 Yunka `baseUrl`
5. 当前用户真值：
   - `memberId`
   - `externalUserId`
6. `/api/equity/health` 结果

### 3.3 本轮不允许混入的内容

- 不跑 `sign-status / sign-apply / sign-confirm`
- 不把 `QW timeout`、`QW reject` 的结果写进 Yunka 结论
- 不把 `127.0.0.1:18081` 的本地 stub 结果当成真实 Yunka 结果

## 4. 推荐的最小执行顺序

### 4.1 第一步：恢复确认

目标：

- 先验证“昨天的 Yunka 真实阻塞是否已经解除”

推荐动作：

1. 优先重放昨天失败的同一条真实 Yunka 请求
2. 若没有保留原始请求体，则先补齐：
   - 请求 path
   - Header
   - body
   - 当时的响应体
   - 当时的 `traceId / requestId`

通过标准：

- 不再返回昨天同类阻塞错误
- 本地 backend log 出现真实 Yunka 出站日志

失败即停止：

- 如果昨天同类错误仍复现，立即停止真联调
- 不继续扩更多 Yunka case

### 4.2 第二步：最小 Yunka 主链烟测

恢复确认通过后，再补一条最小烟测。推荐顺序：

1. `POST /api/loan/calculate`
2. `GET /api/loan/approval-status/{applicationId}` 或 `GET /api/loan/approval-result/{applicationId}`
3. 如需更深主链，再进入 `POST /api/loan/apply`

这样排的原因：

- `loan/calculate` 是纯 Yunka 请求，前置最少
- `loan/apply` 会先碰 QW 权益前置，再碰 Yunka，链路更长
- 如果一开始就打 `loan/apply`，容易再次把 QW 问题和 Yunka 问题混在一起

### 4.3 还款线的启动顺序

如果借款查询线通过，再进还款线，顺序建议为：

1. `GET /api/repayment/info/{loanId}`
2. `POST /api/repayment/submit`
3. `GET /api/repayment/result/{repaymentId}`

## 5. 每条 Yunka 请求必须留的证据

只要声称“今天请求了 Yunka”，至少要拿到下面这些证据：

1. 触发接口的 curl 或页面请求截图 / 命令
2. 本地接口返回：
   - HTTP 状态
   - 响应体
   - `X-Trace-Id`
3. backend log 中的 Yunka 出站日志，至少包含：
   - `yunka gateway request begin`
   - `yunka gateway request success` 或 `yunka gateway request failed`
   - `path=/loan/...` 或 `path=/repay/...`
4. 必要时补本地 SQL / 状态结果

如果缺少第 3 类日志，则不能下结论说“已经请求到 Yunka”。

## 6. 停止规则

出现以下任一情况，立即停止本轮 Yunka 测试：

1. `baseUrl` 仍指向 `127.0.0.1:18081`
2. 触发的接口实际属于 QW，不属于 Yunka
3. 没有 Yunka 出站日志
4. 复现昨天同类真实 Yunka 阻塞
5. 新出现更高优先级真实 blocker，且已经不再是本地可控问题

## 7. 推荐输出模板

测试会话跑完 Yunka 真实请求后，建议统一按下面结构回报：

1. 实际执行了哪些步骤
2. 本轮 Yunka 运行时是否已切换到真实服务
3. 本轮真正请求到 Yunka 的接口有哪些
4. 每条 Yunka 请求的返回结果
5. 关键 Yunka 出站日志
6. 关键本地状态 / SQL 结果
7. 是否确认“Yunka 真实链路可继续”
8. 最小通过/失败结论
