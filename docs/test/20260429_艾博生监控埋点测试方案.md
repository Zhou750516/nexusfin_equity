## 3. 当前监控埋点盘点

| 机制 | 实现位置 | 现状 |
|---|---|---|
| traceId 生成 / 传递 | `TraceIdUtil`（SLF4J MDC？） | ⚠️ 类存在；MDC 传递、跨上游透传需用例验证 |
| 结构化 key/value 日志 | 部分代码（如 `JwtAuthenticationFilter` writeUnauthorized log）| ⚠️ 用 SLF4J 占位符 `traceId={} bizOrderNo={} ...`；非全量、非 JSON |
| Logback 配置 | 仅 `application.yml logging.level.*` | ❌ 无 `logback-spring.xml`，无 JSON encoder、无 MDC pattern、无滚动策略可见 |
| Metrics 框架 | — | ❌ pom.xml 无 actuator / micrometer / prometheus 依赖 |
| 指标埋点 | — | ❌ 无 `Counter / Timer / Gauge` 调用 |
| Health endpoint | `HealthController` | ✅ `GET /api/equity/health` 存在；内容深度需验证 |
| 上游耗时 elapsedMs | `RestYunkaGatewayClient`（line 80-107）| ✅ Yunka 调用记录 elapsedMs；QW 是否记录待验 |
| 补偿任务可观测性 | `async_compensation_task` 表 | ✅ 表内字段完整（status / retry_count / next_retry_ts / last_error）；积压指标待计算 |
| 审计日志（ABS 侧）| — | ❌ 无 `abs_audit_log` 表 |
| 告警通路 | — | ❌ 未发现 alert webhook、邮件 sink、Prometheus alert manager |

**关键缺口**（codex 必须明确验证并标注）：

1. **无 metrics 框架**：所有指标用例只能基于"日志可计算性"，不能依赖 Prometheus / Grafana
2. **无 logback.xml**：日志结构、滚动、JSON 化都未配置；MDC 传播是否生效需要打实际日志验证
3. **无审计表**：ABS 侧关键操作不可追溯
4. **无告警通路**：5xx 即使产生，没有自动通知机制

## 5. P0 用例集

### 5.1 MON-A 链路追踪

#### MON-A.1 traceId 入口生成

```bash
# 不带 X-Trace-Id 调用受保护接口
curl -sS -m 10 -i -b "NEXUSFIN_AUTH=${JWT}" \
  "http://127.0.0.1:8080/api/equity/orders/${ORDER}"
```

通过条件：

1. 响应 header / body 包含 ABS 生成的 `X-Trace-Id`（或在响应内的 `traceId` 字段）
2. 应用日志同 traceId 与该次请求关联
3. **若响应不暴露 traceId**：标注为缺陷（用户无法用 traceId 找到自己的请求）

#### MON-A.2 traceId 透传（外部传入）

```bash
curl -sS -m 10 -i -b "NEXUSFIN_AUTH=${JWT}" \
  -H "X-Trace-Id: TRACE_MON_A_2_FROM_CALLER" \
  "http://127.0.0.1:8080/api/equity/orders/${ORDER}"
```

通过条件：

1. 响应 traceId == `TRACE_MON_A_2_FROM_CALLER`（透传，不被覆盖）
2. 日志同 traceId

#### MON-A.3 traceId 跨上游透传

```bash
# 触发一次需要调齐为/云卡的接口
curl -sS -m 30 -i -b "NEXUSFIN_AUTH=${JWT}" -X POST \
  "http://127.0.0.1:8080/api/equity/orders" \
  -H "X-Trace-Id: TRACE_MON_A_3" \
  -H "Content-Type: application/json" \
  -d '{"requestId":"REQ_MON_A_3","productCode":"P001","loanAmount":29900,"agreementSigned":true}'
```

通过条件（需查日志）：

1. ABS 入口日志含 traceId=TRACE_MON_A_3
2. **QW 出站日志** 含 traceId=TRACE_MON_A_3（透传到 QW 请求 header）
3. **Yunka 出站日志** 含 traceId=TRACE_MON_A_3（在 RestYunkaGatewayClient log line 80-107 验证）
4. **若 QW 未透传**：标注为 P0 缺陷（影响生产排查）

#### MON-A.4 异步线程 traceId 传递

```bash
# 触发会员同步超时 → 异步补偿任务
curl ... -d '{"requestId":"REQ_MON_A_4_FAULT_TIMEOUT","productCode":"P001",...}' \
  -H "X-Trace-Id: TRACE_MON_A_4"
# 等待 worker 重试一次（约 60s）
sleep 65
# 检查 worker 重试日志的 traceId
```

通过条件：

1. 主请求日志 traceId=TRACE_MON_A_4
2. **Worker 重试日志**应保留原 traceId（持久化在 task payload 中）
3. **若 worker 用全新 traceId**：标注为缺陷（无法关联用户原请求与后台补偿）

### 5.2 MON-B 关键日志字段

#### MON-B.1 入口日志结构

触发：任意一次创建订单。

通过条件（需查日志）：日志条目至少包含：

| 字段 | 必有 |
|---|---|
| `traceId` | ✅ |
| `bizOrderNo` 或 `requestId` | ✅ |
| `userId` | ✅ |
| `path` / `method` | ✅ |
| `elapsedMs`（耗时）| ✅ |
| `errorNo` / `errorMsg`（异常时）| ✅ |
| 时间戳（ISO 8601 + 时区）| ✅ |

#### MON-B.2 上游出站日志

触发：任意一次创建订单（涉及 QW / Yunka 调用）。

通过条件：

1. 每次上游调用产生 1 条出站日志，含 `provider=QW/Yunka`、`path`、`elapsedMs`、`upstreamCode`
2. 失败时含 `errorNo`、`providerMessage`
3. **不能含敏感字段明文**（手机号 / 身份证 / 银行卡 / 签名密钥）

#### MON-B.3 异常日志（GlobalExceptionHandler）

触发：参考 `20260429_艾博生异常case测试方案.md` P0-1.x（QW 超时）。

通过条件：

1. 日志级别：`code >= 500` → ERROR；其他 → WARN（per `GlobalExceptionHandler.java:25`）
2. 日志内容含完整堆栈（仅在 ERROR 级）
3. WARN 级日志内容简洁，不夹杂大段堆栈
4. **不输出敏感字段**

#### MON-B.4 补偿任务日志

触发：参考异常方案 P1-2（补偿 worker 退避）。

通过条件：每次 worker 轮询、claim、执行、成功/失败都产生 1 条日志，含 `taskId`、`taskType`、`bizKey`、`retryCount`、`status`

#### MON-B.5 鉴权日志（含被拒）

触发：参考安全方案 SEC-A.1 / SEC-A.2 / SEC-B.x。

通过条件：

1. 鉴权失败日志含 `errorNo=UNAUTHORIZED / FORBIDDEN`
2. 包含被拒资源 path
3. 包含调用者 IP（用于排查暴力枚举）
4. **预期可能 FAIL**：当前实现是否记录 IP 待验

#### MON-B.6 关键操作审计字段

触发：金额变更（支付/退款/还款）、状态推进（订单/借款/签约）。

通过条件：日志包含 `operation=PAY/REFUND/REPAY/STATE_CHANGE`、变更前后值（如 `from=PENDING to=SUCCESS`）

### 5.3 MON-C 异常告警通路

> **现状缺口**：未发现告警通路。本组用例验证缺口存在性。

#### MON-C.1 5xx 告警

触发：构造 unhandled 异常（如内部空指针）—— 需开发者协助；mock 模式下可通过特殊故障注入触发

通过条件：

1. 日志含 ERROR 级 + 完整堆栈
2. **理想**：触发告警通知（钉钉 / 邮件 / Slack）
3. **实际预期**：仅日志，无主动通知 → 标注为 P0 缺口

#### MON-C.2 补偿任务到达 max retries

触发：参考异常方案 P1-3。

通过条件：

1. SQL：`async_compensation_task.status=FAILED`，`last_error` 含完整摘要
2. **理想**：自动告警（人工介入）
3. **实际预期**：仅 DB 标记，无主动通知 → P0 缺口

#### MON-C.3 上游超时率突增

触发：连续 10 次 `_FAULT_TIMEOUT`。

通过条件：

1. **理想**：超过阈值（如 5 分钟内超时率 > 20%）触发告警
2. **实际预期**：无告警；甚至无相关指标 → P0 缺口

#### MON-C.4 鉴权失败激增（暴力探测）

触发：1 分钟内 100 次 SEC-A.2（cookie 篡改）。

通过条件：

1. **理想**：触发安全告警 / 临时封 IP
2. **实际预期**：所有请求都被处理但无告警 → P0 缺口

## 6. P1 用例集

### 6.1 MON-D 慢调用监控

#### MON-D.1 elapsedMs 字段一致性

触发：所有上游调用类型各跑一次（QW 各方法、Yunka 各 path）。

通过条件：

1. 每次调用日志均含 `elapsedMs` 字段
2. 字段单位为毫秒，可解析
3. **若部分调用未记录**：标注为缺陷

#### MON-D.2 慢请求识别

触发：使用 `_FAULT_DELAY_2000` 后缀触发慢调用。

```bash
curl ... -d '{"requestId":"REQ_MON_D_2_FAULT_DELAY_2000",...}'
```

通过条件：

1. 日志 elapsedMs ≈ 2000ms
2. **理想**：超过慢调用阈值（如 1500ms）触发 WARN 日志或独立告警
3. **实际预期**：仅记录耗时，不触发慢请求告警 → 标注为缺陷

#### MON-D.3 慢请求采样率（白盒）

通过条件：是否对慢请求做特殊采样（如全量打 trace）；当前预期 SKIP

### 6.2 MON-E 业务指标可计算性

#### MON-E.1 订单转化漏斗

触发：跑若干完整流程（创建/支付/失败混合）后，运行 §4.3 SQL。

通过条件：

1. SQL 能产出"创建 / 支付成功 / 放款成功 / 失败"的分布
2. 各状态计数自洽（创建 ≥ 支付 ≥ 放款）

#### MON-E.2 签约成功率

通过条件：SQL 能产出 24 小时内的签约成功率；可推算趋势

#### MON-E.3 上游超时率

通过条件：SQL 能从 `async_compensation_task.last_error` 推算超时率；**预期可计算但需要 LIKE 模糊查询**（不健壮）

#### MON-E.4 补偿任务积压

通过条件：SQL 产出按 `task_type` 的 INIT / RETRY_WAIT 数量；超过预设阈值（如 100 条）应有告警

#### MON-E.5 业务指标可视化（缺口）

通过条件：

1. **理想**：Grafana / 内部监控大盘有上述指标
2. **实际预期**：无 → 标注为缺口

| 无告警通路 | 故障无主动通知 | 引入 Sentry / 钉钉 / 邮件 sink |
| /health 浅 | 无法反映真实健康度 | 增加 DB / 上游探活 / 补偿 worker 心跳 |
| traceId 跨线程传递未明 | 异步补偿日志难以关联用户原请求 | 在 task payload 持久化原 traceId；worker 重试时 MDC.put 还原 |
| QW 出站 elapsedMs 缺失（待验）| 上游耗时不可观测 | 在 QwBenefitClientImpl HTTP 路径加 elapsedMs 日志 |
| 慢调用阈值缺失 | 无法识别"成功但慢" | 配置 slow-threshold-ms；超阈值打 WARN 日志 |
