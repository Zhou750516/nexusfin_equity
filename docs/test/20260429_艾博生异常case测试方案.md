## 2. 测试目标与判定原则

### 2.1 目标

回答以下 5 个问题：

1. **异常透传是否一致**：上游各类异常（超时 / 业务失败 / 空响应 / 非法响应）是否被 ABS 服务层正确捕获并以**统一错误结构**返回前端，不出现 5xx 默认错误页或异常吞没？
2. **本地状态是否一致**：异常发生时，ABS 本地业务状态（订单 / 签约 / 借款 / 还款 / 支付记录）是否处于**可恢复**态，不出现"上游已成功但本地无记录"的不可逆态？
3. **异步补偿是否生效**：超时类异常是否正确入队 `async_compensation_task`，worker 是否按退避重试，达到上限是否标记 FAILED？
4. **幂等保护是否生效**：相同 `requestId` 重发、回调重复投递、并发请求，是否产生重复业务效果？
5. **状态机是否健壮**：状态倒退（已成功又被失败覆盖）、跳跃（跳过中间态）是否被拒绝？

### 2.2 通过判定原则

每个用例必须明确以下 4 条中**所有适用项**，缺一即 FAIL：

| 维度 | 判定要点 |
|---|---|
| HTTP 层 | 不出现 5xx；不出现 Tomcat 默认错误页；响应结构遵循 `Result<T>` 统一封装 |
| 业务码 | 错误码 / 错误消息可识别（不是 `null` / `Internal server error`）|
| 持久化 | 业务表状态符合预期；不出现孤儿记录 / 倒退状态 |
| 副作用 | 异步补偿任务正确入队 / 不入队；不重复触发上游 |

## 4. Mock 增强需求（执行先决条件）

### 4.1 故障触发协议（推荐 requestId 后缀方案）

ABS 调用 mock 时，按 `requestId` 后缀触发对应故障；同时支持 `traceId` 后缀作为兜底（应对部分接口未透传 requestId 的情况）。**suffix 大小写敏感**。

| 后缀 | mock 行为 | 模拟的 HTTP 路径异常 |
|---|---|---|
| `_FAULT_TIMEOUT` | 抛 `UpstreamTimeoutException` | 读超时 / 连接超时 |
| `_FAULT_REJECT_<N>` | 抛 `BizException(N, "<UPSTREAM>_UPSTREAM_REJECTED", msg)`，N 为业务码 | 上游返回 code != 0/200 |
| `_FAULT_EMPTY` | 抛 `BizException("<UPSTREAM>_RESPONSE_EMPTY", msg)` | 上游返回空 / null data |
| `_FAULT_MALFORMED` | 抛 `BizException("<UPSTREAM>_RESPONSE_INVALID", msg)` | 上游返回非法 JSON |
| `_FAULT_NETWORK` | 抛 `BizException("<UPSTREAM>_NETWORK_ERROR", msg)` | 连接拒绝 / DNS 失败 |
| `_FAULT_DELAY_<MS>` | `Thread.sleep(MS)` 后正常返回 | 慢响应（不超时） |

`<UPSTREAM>` 在 QW mock 中为 `QW`，Yunka mock 中为 `YUNKA`。

**示例 requestId**：

- `REQ_EX_P0_1_1_FAULT_TIMEOUT` → 触发 QW/Yunka 模拟读超时
- `REQ_EX_P0_3_1_FAULT_REJECT_500` → 触发上游业务失败 code=500
- `REQ_EX_P3_5_FAULT_DELAY_2500` → 上游延迟 2.5 秒返回

### 4.2 QW mock 增强点

文件：`src/main/java/com/nexusfin/equity/thirdparty/qw/QwBenefitClientImpl.java`

在 `mockSyncMemberOrder / mockGetExerciseUrl / mockNotifyLending / mockQuerySignStatus / mockApplySign / mockConfirmSign` 各方法**入口处第一行**调用：

```java
private void applyMockFault(String requestId) {
    if (requestId == null) return;
    if (requestId.contains("_FAULT_TIMEOUT")) {
        throw new UpstreamTimeoutException("Mock QW timeout: " + requestId);
    }
    Matcher m = REJECT_PATTERN.matcher(requestId);
    if (m.matches()) {
        int code = Integer.parseInt(m.group(1));
        throw new BizException(code, "QW_UPSTREAM_REJECTED",
                "Mock QW rejection code=" + code);
    }
    if (requestId.contains("_FAULT_EMPTY")) {
        throw new BizException("QW_RESPONSE_EMPTY", "Mock QW empty response");
    }
    if (requestId.contains("_FAULT_MALFORMED")) {
        throw new BizException("QW_RESPONSE_INVALID", "Mock QW malformed response");
    }
    if (requestId.contains("_FAULT_NETWORK")) {
        throw new BizException("QW_NETWORK_ERROR", "Mock QW network error");
    }
    Matcher delay = DELAY_PATTERN.matcher(requestId);
    if (delay.matches()) {
        try { Thread.sleep(Long.parseLong(delay.group(1))); }
        catch (InterruptedException ie) { Thread.currentThread().interrupt(); }
    }
}
private static final Pattern REJECT_PATTERN = Pattern.compile(".*_FAULT_REJECT_(\\d+)$");
private static final Pattern DELAY_PATTERN  = Pattern.compile(".*_FAULT_DELAY_(\\d+)$");
```

### 4.3 Yunka mock 增强点

文件：`src/main/java/com/nexusfin/equity/thirdparty/yunka/RestYunkaGatewayClient.java`

`proxy()` 方法的 mock 分支（line 56-63 周围），在返回固定空壳之前调用 `applyMockFault(request.getRequestId())`，错误类型与 QW 对齐但前缀改为 `YUNKA_`。

## 6. P0 高风险用例集（必跑）

> 这一层针对架构调研发现的"现有兜底缺失"区域。任一 FAIL 即视为本轮联调存在阻塞性缺陷，需先修后再跑 P1+。

### 6.1 签约链路无补偿（`BankCardSignServiceImpl`）

**已知缺陷**：`querySignStatus / applySign / confirmSign` 三个方法的超时异常**不入队补偿任务**，直接抛给前端。本组用例验证当前实际行为，并暴露缺陷影响面。

#### P0-1.1 querySignStatus 超时

```bash
curl -sS -m 30 -i -X GET \
  "http://127.0.0.1:8080/api/bank-card/sign-status?accountNo=6228480012345678&requestId=REQ_EX_P0_1_1_FAULT_TIMEOUT"
```

通过条件：

1. HTTP `200`，业务码非 0；`errorNo` 含 `TIMEOUT` 语义
2. SQL 检查：`async_compensation_task` 表无新增（验证当前未做补偿）
3. SQL 检查：`member_payment_protocol` 无脏写
4. **结论标注**：当前架构未做补偿；记录为已知缺陷，建议修复

#### P0-1.2 applySign 超时（最高风险：齐为可能已发短信）

```bash
curl -sS -m 30 -i -X POST http://127.0.0.1:8080/api/bank-card/sign-apply \
  -H "Content-Type: application/json" \
  -d '{"requestId":"REQ_EX_P0_1_2_FAULT_TIMEOUT","phone":"13800001234","name":"测试","accountNo":"6228480012345678","idNo":"110101199001011234"}'
```

通过条件：

1. HTTP `200`，业务码非 0；errorNo 含 TIMEOUT
2. SQL：本地无 `userSignId` 落库（确认 ABS 在超时时不会伪造 userSignId）
3. SQL：`async_compensation_task` 无新增（确认无补偿）
4. **关键风险标注**：齐为侧实际可能已下发短信；用户重试时 ABS 无去重 → 可能二次下发

#### P0-1.3 confirmSign 超时（最严重：齐为已签约但 ABS 无 userSignId/agreementNo）


1. HTTP `200`，业务码非 0
2. SQL：`member_payment_protocol` 中该 userSignId 的 `agreementNo` 为空，`status` 不是"已签约"
3. SQL：`async_compensation_task` 无新增

## 7. P1 用例集 — 权益同步异步补偿

> 验证 `BenefitOrderServiceImpl.createOrder()` 在 QW 超时时的补偿链路。

### P1-1 syncMemberOrder 超时 → 补偿任务入队

```bash
curl -sS -m 30 -i -X POST http://127.0.0.1:8080/api/equity/orders \
  -H "Content-Type: application/json" \
  -d '{"requestId":"REQ_EX_P1_1_FAULT_TIMEOUT","productCode":"P001","loanAmount":100000,"agreementSigned":true}'
```

通过条件：

1. HTTP `200`，响应内含 `benefitOrderNo`（订单已落库）
2. SQL：`benefit_order.sync_status = SYNC_FAIL`
3. SQL：`async_compensation_task` 中存在 1 条 `task_type=QW_BENEFIT_PURCHASE_RETRY`、`status=INIT`、`biz_key=<orderId>`
4. 抛 `BenefitPurchaseSyncTimeoutCompensationException`，但订单仍持久化（`@Transactional(noRollbackFor=...)` 生效）

### P1-2 补偿 worker 退避

前置：P1-1 已入队补偿任务；保持 mock 增强为 `_FAULT_TIMEOUT` 持续超时。

观察：每隔 ~3s（worker 轮询间隔）观察 `async_compensation_task`：

```sql
SELECT id, retry_count, next_retry_ts, status FROM async_compensation_task
  WHERE task_type='QW_BENEFIT_PURCHASE_RETRY' AND biz_key=<orderId>;
```

通过条件：

1. `retry_count` 增长（0→1→2→3...）
2. `next_retry_ts` 间隔符合指数退避（60s, 120s, 240s, 480s, 900s 上限）
3. `status` 在 `INIT`/`PROCESSING`/`RETRY_WAIT` 之间正常切换

### P1-3 补偿到达 max retries → FAILED

前置：让 P1-2 的任务一路 fail 至 `retry_count=5`（视配置 max-retries）。

通过条件：

1. SQL：任务最终 `status=FAILED`
2. `last_error` 字段含完整异常摘要
3. `benefit_order.sync_status` 仍为 `SYNC_FAIL`，无误标
4. **运维提示**：是否有告警钩子或人工介入入口？记录为待确认项

### P1-4 补偿 worker crash（lease 过期）

模拟方式：在补偿任务被 worker `claim` 后（`status=PROCESSING`），手工 SQL 篡改 `lease_owner` 与 `lease_expire_ts` 模拟 lease 过期。

通过条件：

1. 下一轮 worker 轮询能重新 claim 该任务
2. 不出现"永久 PROCESSING"挂死

### P1-5 入队幂等（同 bizKey 重复入队）

模拟方式：同一 `benefitOrderNo` 连续两次触发 P1-1。

通过条件：

1. SQL：`async_compensation_task` 中相同 `task_type + biz_key` 的记录 = 1
2. 第二次入队应被去重，不产生第二条任务
3. 若实现未做去重，记录为 P1 待修复

### P1-6 补偿恢复后 sync_status 推进

前置：让 P1-1 入队，等几秒后改 `requestId` 为非故障型，触发 mock 切回 happy-path（实际操作：让 worker 在重试时拿到的 payload 不再触发故障——可通过修改 task payload 或改 mock 配置实现）。

通过条件：

1. SQL：任务 `status=SUCCESS`
2. `benefit_order.sync_status=SYNC_SUCCESS`
3. 上游齐为业务侧确认有最终成功记录（mock 模式下省略）

## 12. 已知风险与不覆盖项

| UpstreamTimeoutDetector 字符串匹配 | Spring 升级可能失效 | 不直接覆盖，标为长期风险 |

### 12.2 本轮不覆盖

- **性能/压测**：本方案不验证 QPS、并发安全（除 P0-3.1 的最小并发幂等场景）
- **真实上游**：齐为 / 云卡均处于 mock 模式；真实联调缺陷（签名、加密、字段语义）由 `docs/third-part/` 下的联调方案承接
- **H5 前端**：H5 的异常展示、loading 兜底、重试 UI 由 H5 测试文档承接
- **数据库故障**：DB 连接断、主从延迟、唯一键冲突等场景
- **JVM/容器层**：OOM、GC 长停、容器重启
- **认证/鉴权**：登录态过期、JWT 篡改
