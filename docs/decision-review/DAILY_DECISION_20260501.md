### DECISION-078: 后续联调与测试默认只看艾博生直接边界，不再默认下钻科技平台内部服务

**决策日期**: 2026-05-01  
**决策人**: 技术负责人 + 协调 / 审查会话  
**所属分类**: 测试边界 / 外部协同  
**优先级**: ⭐⭐⭐⭐ 高  
**影响范围**: 大

#### 问题背景

前期联调文档和测试方向里，仍残留一些“默认继续下钻科技平台内部 path / 域名 / 签名 / 服务可用性”的惯性。这和当前已经确认的边界不一致：

- 艾博生不直接感知科技平台服务
- 科技平台相关细节应由云卡统一收口
- 只有在明确存在直接 `token / 回调 / 跳转` 边界时，艾博生才需要面对科技平台

如果不及时收紧，后续测试会继续发散，问题归属也会越来越混。

#### 最终决策

**选择**: 后续默认只验证艾博生直接边界：

- FE <-> BE
- ABS <-> Yunka
- ABS <-> QW
- 被明确确认存在的直连边界

不再默认扩展到科技平台内部服务。

#### 理由

1. 这能让测试结果更干净，责任边界更清楚。
2. 外部问题接收方也会更明确，默认只追云卡、齐为。
3. 可以避免把“云卡未统一收口”误报成“艾博生还要自己接科技平台”。

#### 实施结果

- [x] `docs/test/20260501_对外联调前技术检查清单.md` 已按该原则收紧
- [x] `docs/third-part/20260501_外部联调待确认问题清单.md` 已改成默认只追云卡、齐为
- [x] 测试会话后续范围已明确收紧

2. 可以防止覆盖矩阵表面变大，但主链仍然不稳。

### DECISION-081: 还款重复提交先用 `120` 秒短窗口防重收口，作为当前联调阶段的最小可用方案

**决策日期**: 2026-05-01  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 风险控制 / 业务保护  
**优先级**: ⭐⭐⭐ 中高  
**影响范围**: 中

#### 问题背景

重复还款的真实问题已经暴露：同一笔还款可以连续两次打到 `/repay/apply`。  
这在当前联调阶段属于必须先拦住的风险点。

但要做“长期严格幂等”，需要更重的设计，例如：

- 显式客户端请求号
- 更完整的本地还款单状态
- 更稳定的去重主键

当天不适合再扩大实现范围。

#### 最终决策

**选择**: 先采用 `uid + loanId + amount` 维度的 `120` 秒短窗口防重，作为当前最小可用方案。

#### 理由

1. 能先拦住最危险的“重复打 `/repay/apply`”。
2. 改动范围小，适合当前联调节奏。
3. 不会把当天任务从“修 bug”升级成“重做还款幂等体系”。

#### 实施结果

- [x] 同窗口内第二次同金额还款会返回 `REPAYMENT_SUBMIT_DUPLICATED`
- [x] 第二次请求不会再真实打 `/repay/apply`
- [x] 当前方案的边界和残余风险已在开发输出里明确记录
