### DECISION-050: 异步补偿日志链路按“入队 + worker 执行”补齐闭环

**决策日期**: 2026-04-23  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 可观测性 / 排障闭环  
**优先级**: ⭐⭐⭐ 高  
**影响范围**: 中

#### 问题背景

前一阶段已完成借款、还款、云卡客户端、统一异常等主链路日志补齐，但当 `loan apply` 因云卡超时进入异步补偿后，链路会从同步调用切换到后台任务。

如果异步补偿入队和 worker 执行日志不完整，排障时只能看到“进入补偿”，但无法继续判断：

1. 任务是否入队成功
2. 任务被哪个 worker 认领
3. 任务执行成功还是失败
4. 失败是否还能重试
5. 是否已经进入死任务

#### 最终决策

**选择**: 将异步补偿日志拆成“入队链路”和“worker 执行链路”两段补齐。

#### 理由

1. 与异步补偿实际生命周期一致
2. 便于从 `bizOrderNo` 下钻到 `taskId`
3. 便于从 `taskId` 下钻到 `workerId / partitionNo`
4. 可以和现有任务表、尝试表形成互相校验

#### 实施结果

- [x] 入队成功日志补齐
- [x] 重复入队日志补齐
- [x] worker 认领日志补齐
- [x] worker 成功日志补齐
- [x] worker 失败 / 死任务日志补齐
- [x] 日志专项文档已回填

### DECISION-051: worker 失败日志统一使用 `errorNo / errorMsg`

**决策日期**: 2026-04-23  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 日志规范 / 错误口径  
**优先级**: ⭐⭐⭐ 高  
**影响范围**: 中

#### 问题背景

worker 失败日志此前存在 `error=` 这样的非标准字段，和当前主链路已经收口的 `errorNo / errorMsg` 不一致。

如果继续保留混用口径，会导致联调排障时出现：

1. 同类失败日志检索字段不统一
2. 异步补偿失败无法和统一异常处理对齐
3. 后续结构化采集字段不稳定

#### 最终决策

**选择**: worker 失败日志统一输出 `errorNo / errorMsg`。

#### 理由

1. 与云卡、统一异常、业务 Service 失败日志保持一致
2. 方便按错误码聚合检索
3. 方便后续对接日志平台时做统一字段采集

#### 实施结果

- [x] `BizException` 失败取 `getErrorNo()` / `getErrorMsg()`
- [x] 非业务异常取异常类名作为 `errorNo`
- [x] 非业务异常取异常消息作为 `errorMsg`
- [x] 单元测试已覆盖

### DECISION-052: worker 可重试失败打 `WARN`，死任务打 `ERROR`

**决策日期**: 2026-04-23  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 日志分级 / 运维排障  
**优先级**: ⭐⭐⭐ 高  
**影响范围**: 中

#### 问题背景

异步补偿失败并不一定意味着最终失败。有些失败还会进入 `RETRY_WAIT`，等待下一次重试；而达到最大重试次数后进入 `DEAD`，才表示需要重点人工排查。

如果两类场景都打成同一日志等级，会造成：

1. 可重试失败误报为严重故障
2. 死任务无法从普通失败中凸显出来
3. 运维排障优先级不清晰

#### 最终决策

**选择**:

- 可重试失败：`WARN`
- 死任务：`ERROR`

#### 理由

1. `WARN` 表示业务仍有自动恢复机会
2. `ERROR` 表示已达到死任务，需要人工重点关注
3. 与前期日志等级设计中 `WARN / ERROR` 的语义边界一致

#### 实施结果

- [x] `dead=false` 输出 `WARN`
- [x] `dead=true` 输出 `ERROR`
- [x] 测试覆盖可重试失败和死任务日志字段
