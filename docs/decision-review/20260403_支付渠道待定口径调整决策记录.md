# 2026-04-03 支付渠道待定口径调整决策记录

- 决策标题：支付渠道待定，通联口径降级为阶段性假设
- 决策日期：`2026-04-03`
- 决策人：用户 / 项目推进负责人
- 决策分类：技术管理 / 外部依赖 / 风险控制
- 优先级：`高`
- 当前状态：`已决策`

## 2. 问题背景

截至 `2026-04-02`，齐为联调口径明确指向通联协议号，因此文档和设计方案均按“通联真实协议号”进行第一轮收敛。

但截至 `2026-04-03`，新增信息表明：

1. 支付渠道后续可能调整
2. 因而“通联就是最终正式渠道”这一前提当前不再稳定

如果继续把通联写成最终唯一正式方案，会带来两类问题：

1. 文档失真，后续容易误导开发和上线准备
2. 设计被错误绑定到单一渠道，影响后续渠道切换

## 3. 可选方案

| 方案 | 优势 | 劣势 | 成本 | 风险 | 评分 |
|------|------|------|------|------|------|
| 继续按通联作为最终方案推进 | 口径简单 | 一旦支付渠道变化，文档立即失真 | 低 | 高 | ⭐⭐ |
| 暂停所有协议号相关文档整理 | 风险最低 | 会让已完成的结构沉淀失去承接价值 | 中 | 中 | ⭐⭐⭐ |
| **将通联口径降级为阶段性假设，保留通用结构设计（推荐）** | 既保留已完成成果，又为后续渠道切换留出空间 | 需要同步改文档口径 | 低 | 低 | ⭐⭐⭐⭐⭐ |

## 4. 最终决策

**选择**：将通联口径降级为阶段性假设，保留“用户级协议号资产 + 回写入口 + 订单快照 + override 边界”的通用结构设计

## 5. 决策理由

1. 当前变化的是“最终支付渠道是否仍为通联”，不是“支付协议号必须被用户级管理”这件事本身
2. 当前已经完成的主流程结构改造具有渠道无关性，仍然有效
3. 当前最稳妥的做法不是推翻已有文档，而是把它们降级为“阶段性基线”
4. 等最终支付渠道确定后，再将 `provider_code`、来源值和对接说明切换为正式版本即可

## 6. 当前有效与失效的部分

### 6.1 继续有效的部分

1. 用户级协议号资产单独落库
2. 订单保留协议号快照与来源快照
3. 回写入口 `POST /api/callbacks/payment-protocol`
4. 测试 override 与正式环境隔离
5. 主流程只使用有效协议号

### 6.2 需要降级的部分

1. “最终正式渠道一定为通联”的表述
2. “生产环境最终只会出现 `ALLINPAY`”的绝对化表述
3. “通联服务对接说明”为最终定版文档的表述
