### DECISION-026: H5 页面维度联调统一切到真实 MySQL + 本地 Yunka REST stub 口径

**决策日期**: 2026-04-15  
**决策人**: 技术负责人 + 项目推进负责人  
**所属分类**: 联调 / 环境  
**优先级**: ⭐⭐⭐ 高  
**影响范围**: 大

#### 问题背景

今天需要从页面维度完整走查 H5 主流程，并要求关键业务数据写入真实 MySQL。

但联调链路涉及多个前置条件：

1. H5 本地页面
2. Java 后端
3. 真实 MySQL
4. 本地科技平台 stub
5. 本地 Yunka stub

如果环境口径不统一，即使页面可打开，也会出现“假异常”。

#### 可选方案

| 方案 | 优势 | 劣势 | 成本 | 风险 | 评分 |
|------|------|------|------|------|------|
| 继续沿用默认本地启动方式 | 操作简单 | 很容易混入 MOCK 口径 | 低 | 高 | ⭐⭐ |
| 只切 MySQL，不切 Yunka REST | 部分接近真实环境 | 会产生误导性异常 | 低 | 高 | ⭐⭐ |
| **统一切到真实 MySQL + Yunka REST stub（推荐）** | 页面、接口、数据库链路一致 | 启动参数更严格 | 低 | 低 | ⭐⭐⭐⭐⭐ |

#### 最终决策

**选择**: 本地页面联调统一以“真实 MySQL + Yunka REST stub”作为标准环境口径

**理由**:

- 页面主流程测试的重点是验证真实接口链路和落库行为。
- 只切 MySQL 不切 Yunka REST，会制造错误的结论。
- 统一口径后，问题定位更稳定、证据更可信。

#### 实施计划

```text
第一阶段: 启动 H5 / Java / MySQL / 科技平台 stub / Yunka stub
第二阶段: 完成登录态、支付协议、产品种子准备
第三阶段: 从首页走到还款成功页，并核对 MySQL 落库
```

#### 风险与缓解

| 风险 | 严重度 | 缓解方案 |
|------|--------|---------|
| 漏配 Yunka REST 参数导致误判 | 高 | 固化启动命令与检查清单 |
| 联调前置数据缺失导致链路失败 | 中 | 固化 `HUXUAN_CARD` / 支付协议检查步骤 |

#### 下步行动

- [x] 固化正确启动命令
- [x] 固化本地联调检查清单
- [x] 固化人工测试执行步骤

### DECISION-027: 本轮审批拒绝与还款金额为 0 的问题，先按环境根因排查，不先改代码

**决策日期**: 2026-04-15  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 排障 / 风险控制  
**优先级**: ⭐⭐⭐ 高  
**影响范围**: 大

#### 问题背景

首轮页面联调中出现两个明显异常：

1. `approval-result` 返回拒绝
2. `repaymentAmount` 返回 `0.00`

此时需要决定，是直接改代码，还是先完成根因排查。

#### 可选方案

| 方案 | 优势 | 劣势 | 成本 | 风险 | 评分 |
|------|------|------|------|------|------|
| 直接修改业务代码 | 反馈快 | 极易误修 | 低 | 高 | ⭐ |
| 先看接口现象再猜测修复 | 操作直观 | 容易停留在症状层 | 中 | 中 | ⭐⭐⭐ |
| **先排查运行环境与数据流，再决定是否改代码（推荐）** | 能定位根因 | 需要多一步验证 | 低 | 低 | ⭐⭐⭐⭐⭐ |

#### 最终决策

**选择**: 严格先做环境与数据流排查，确认根因后再决定是否改代码

**理由**:

- 两个异常同时出现，且都与下游返回值为空高度相关。
- 多组件联调场景中，环境问题概率很高。
- 事实证明，根因是后端仍跑在 `YUNKA_MODE=MOCK`，而不是业务代码错误。

#### 实施计划

```text
第一阶段: 核对 Yunka 运行模式与请求链路
第二阶段: 以正确参数重启后端
第三阶段: 重新验证 approval-result 与 repayment-info
```

#### 风险与缓解

| 风险 | 严重度 | 缓解方案 |
|------|--------|---------|
| 将环境问题误判为代码 bug | 高 | 所有修复前先核对运行参数 |
| 反复改代码造成新问题 | 中 | 先证据后结论 |

#### 下步行动

- [x] 完成根因复核
- [x] 在测试报告中补充修正结论
- [x] 在联调清单中明确该风险点
