### DECISION-070: 云卡公网联调在 `B-01` 停住，不与本地异常线混跑

**决策日期**: 2026-04-29  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 联调边界 / 风险隔离  
**优先级**: ⭐⭐⭐⭐ 高  
**影响范围**: 大

#### 问题背景

按公网联调方案执行到 `B-01` 时，请求没有进入预期的“KJ 不可达”语义，而是在云卡侧出站编码阶段提前失败，报错为：

- `providerCode=KJ_REQUEST_ENCODE_ERROR`
- `providerMessage=InvalidKeyException ... DerInputStream.getLength()...`

这已经不是 ABS 侧入口、参数、日志或本地 mock 的问题。

#### 最终决策

**选择**: 将该问题明确归类为云卡侧到 KJ 的外部阻塞，在对方回复前停止公网链路继续压测，不把它和本地 mock 异常线混在一起推进。

#### 理由

1. 继续跑公网用例不会增加 ABS 侧有效信息，只会反复命中同一外部阻塞。
2. 本地 mock 异常线仍可独立推进，不应被公网阻塞拖住。
3. 将外部问题单独收敛，方便晚间由云卡同学直接接手排查。

#### 实施结果

- [x] 生成云卡侧问题描述与复现命令
- [x] 停止公网链路后续 case 扩测
- [x] 后续重心转回本地联调与异常线

2. QW 继续留在应用内 MOCK，能避免在权益前置上引入新的外部不确定性。

### DECISION-072: 异常线按 Layer A 先闭合、Layer B 再逐条推进，不并行铺开矩阵

**决策日期**: 2026-04-29  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 测试策略 / 执行节奏  
**优先级**: ⭐⭐⭐⭐ 高  
**影响范围**: 大

#### 问题背景

Claude 给出的异常方案覆盖面广，但其中混有：

1. 当前可执行基线
2. 未来 mock 增强后才可执行的场景
3. 一些与现有表结构不一致的旧假设

如果直接全量执行，测试会在半途不断撞到环境和基线问题。

#### 最终决策

**选择**: 先把异常线拆成 Layer 0 / A / B：

- Layer A：先闭合鉴权、参数、前置签约、fake key
- Layer B：待 fault hook 可用后，再一条条推进 timeout / reject 场景

每次只推进一个最小 case，未打通就停住，不并行扩矩阵。

#### 理由

1. 先闭合 Layer A，能优先消除最容易混淆业务判断的错误成功态和前置缺口。
2. Layer B 涉及 mock 能力与补偿链路，先单点验证 fault hook 是否真的可用更稳。
3. 这种推进方式更适合“主实现会话 + 审查 / 测试会话”协作。

#### 实施结果

- [x] `A1` / `A2` / `A3` / `A4` 已全部闭合
- [x] `58d9438` 已补齐 Layer B 最小 fault hook
- [x] 首轮 `B-P0-1` 已暴露触发协议问题，并在当天停在最小失败点
