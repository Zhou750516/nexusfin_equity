### DECISION-068: Loan 纵向拆分与异步补偿 typed payload 改造必须保持当前真实业务语义不变

**决策日期**: 2026-04-28  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 语义稳定 / 风险控制  
**优先级**: ⭐⭐⭐⭐ 高  
**影响范围**: 大

#### 问题背景

当天推进 `Phase 2A ~ 2E` 时，多次遇到“计划期望”和“当前真实实现”可能不完全一致的情况，主要集中在：

1. `LoanServiceImpl.apply(...)` 的 rejected / timeout 语义
2. `buildLoanFailedResponse(...)` 的返回口径
3. 异步补偿 producer 改 typed payload 后，executor 仍需消费兼容 JSON shape

如果借重构顺手改动这些语义，会把“结构收敛”升级成“业务规则变更”，风险明显升高。

#### 最终决策

**选择**: 所有 `Phase 2` 拆分与 typed payload 改造均以“保持当前真实业务语义不变”为最高约束，只做结构收敛，不借机重设计返回口径或补偿策略。

#### 理由

1. 当前阶段目标是降摩擦，不是重写业务规则
2. 先守住语义，才能让重构后的测试结果可对比、可追责
3. executor 生产代码本轮不动，是验证 `request_payload` 兼容性的最直接方式

#### 实施结果

- [x] `LoanApplicationService` 拆分后保持 rejected / timeout 语义不变
- [x] `LoanApplicationGateway` 抽取后保持 mapping 查询口径不变
- [x] `AsyncCompensationEnqueueService` 改 typed payload 后，落库仍为 JSON string
- [x] `YunkaLoanApplyCompensationExecutorTest` 与 `QwBenefitPurchaseCompensationExecutorTest` 通过，证明消费兼容性保留

### DECISION-069: 协调会话固定只做计划、审查、验证，主实现会话按完整任务串行推进

**决策日期**: 2026-04-28  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 协作方式 / 交付效率  
**优先级**: ⭐⭐⭐ 中  
**影响范围**: 中

#### 问题背景

当天工作大量采用“双会话协作”：

1. 一个会话负责主实现
2. 当前会话负责计划拆分、阶段放行、独立验证和最终收口

如果每个 task 都来回复制 prompt，节奏会越来越慢；但如果协调会话直接接管主实现，又会破坏角色分工。

#### 最终决策

**选择**: 协调会话继续固定为“计划 + 审查 + 验证”角色，不接管主实现；同时在当天后半段改为向主实现会话下发完整阶段总指令，减少频繁往返。

#### 理由

1. 保留审查独立性，可以减少“实现者自己判定自己完成”的盲区
2. 统一总指令后，能显著降低重复复制 prompt 的沟通成本
3. 这种模式适合后续 `Phase 0 / 5 / 3 / 4` 继续推进

#### 实施结果

- [x] 当前会话始终未接管 `Phase 1 / 2` 主实现
- [x] 每个关键阶段都做了独立验证与边界检查
- [x] 已形成可直接发给主实现会话的后续总任务指令
