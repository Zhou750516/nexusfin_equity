## 决策详情

### DECISION-044: 自动调度心跳语义统一为“认领任务后 RUNNING，tick 结束回 IDLE”

**决策日期**: 2026-04-20  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 调度可观测 / 运行口径  
**优先级**: ⭐⭐⭐ 高  
**影响范围**: 中

#### 问题背景

自动调度第一阶段骨架已经落地，但如果心跳语义不清晰，联调 / 运维同学会难以判断：

1. 当前分片是否真的在处理任务
2. `RUNNING` 是表示“本轮 tick 启动”还是“已经认领具体任务”
3. `currentTaskId` 应该在什么时候出现、什么时候清空

#### 最终决策

**选择**: 统一为“任务认领成功后上报 `RUNNING + currentTaskId`，tick 结束后回到 `IDLE + currentTaskId = null`”

#### 理由

1. 避免空转 tick 被误解释成正在处理任务
2. 让 `currentTaskId` 可以直接对应本轮认领到的任务
3. 保持当前调度层只做轮询、心跳和编排，不把业务逻辑塞进 scheduler 层

#### 实施结果

- [x] `worker service` 增加任务认领回调
- [x] `coordinator` 在任务认领成功时上报 `RUNNING + currentTaskId`
- [x] tick 结束后统一回到 `IDLE + currentTaskId = null`
- [x] 没有认领到任务时不误打 `RUNNING`
- [x] 输出 `plan/20260420_自动调度运行口径与启用说明.md`

### DECISION-045: 自动调度继续坚持“默认关闭、显式开启、显式分片”

**决策日期**: 2026-04-20  
**决策人**: 技术负责人 + 项目推进负责人  
**所属分类**: 上线控制 / 风险控制  
**优先级**: ⭐⭐⭐ 高  
**影响范围**: 大

#### 问题背景

自动调度已经具备可启用能力后，容易出现“是否默认开启”的争议。如果默认开启，可能会：

1. 污染现有联调环境
2. 让未准备好的环境开始消费补偿任务
3. 在分片归属不清晰时造成排查困难

#### 最终决策

**选择**: 继续坚持“默认关闭、显式开启、显式分片”

#### 理由

1. 当前目标是让能力可启用、可观察，不是让所有环境自动运行
2. 显式 `workerId` 和 `ownedPartitions` 更利于联调排查
3. 当前尚未进入多实例自动抢占、rebalance、生产级常驻编排阶段

#### 实施结果

- [x] 保持默认关闭策略
- [x] 补单实例启用配置示例
- [x] 补双实例显式分片配置示例
- [x] 明确启用后观察项

### DECISION-046: 自动调度交付材料按“启用说明 + 观察 SOP”承接

**决策日期**: 2026-04-20  
**决策人**: 技术负责人 + 项目推进负责人  
**所属分类**: 运维交接 / 联调准备  
**优先级**: ⭐⭐ 中  
**影响范围**: 中

#### 问题背景

自动调度如果只完成代码和测试，后续联调 / 运维仍会缺少“怎么看是否启用成功、异常时先看哪里”的操作说明。

#### 最终决策

**选择**: 当前交付材料先按“运行口径与启用说明 + 启用后观察与人工介入 SOP”两份文档承接

#### 理由

1. 一份说明解决“怎么启用、怎么配置、语义是什么”
2. 一份 SOP 解决“启用后怎么看、异常时怎么排查”
3. 当前不需要先扩展成完整生产运维手册，避免文档重、落地慢

#### 实施结果

- [x] 输出 `plan/20260420_自动调度运行口径与启用说明.md`
- [x] 输出 `plan/20260420_自动调度启用后观察与人工介入SOP.md`
- [x] 明确启动日志、分片运行时表、任务表、尝试表四类观察点
