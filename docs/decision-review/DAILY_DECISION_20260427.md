### DECISION-064: H5/API 后续任务与联合登录允许并行，但必须脱离 `main` 在独立工作区推进

**决策日期**: 2026-04-27  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 开发策略 / 风险控制  
**优先级**: ⭐⭐⭐⭐ 高  
**影响范围**: 大

#### 问题背景

当日目标已经从“purpose 最小闭环”切换为两条并行开发流：

1. H5/API 后续任务
2. 联合登录真实实现

但盘点工作区后发现，`main` 上已经堆积了一批未提交改动，其中大部分明显属于 H5/API 线。  
如果继续在 `main` 上并行推进，两条线会很快从“可并行”滑到“难以 review 和回滚”。

#### 最终决策

**选择**: 两条线允许并行，但必须分别在 `feature/h5-api` 与 `feature/joint-login` 的独立工作区推进，`main` 不再承载业务实现。

#### 理由

1. 可以把 H5/API 与联合登录的未提交改动物理隔离
2. 能显著降低灰区文件的冲突概率
3. 后续验证、合并、回滚都更可控

#### 实施结果

- [x] 确认并保留两个独立工作区：
  - `feature/h5-api`
  - `feature/joint-login`
- [x] 将明确属于 H5/API 的代码改动迁回 `feature/h5-api`
- [x] 将联合登录新增实施方案文档放入 `feature/joint-login`
- [x] 将 `main` 收敛为协调与文档视角，不继续承载双线业务开发

### DECISION-065: 灰区文件必须明确 owner，未授权不得跨线改动

**决策日期**: 2026-04-27  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 边界管理 / 冲突控制  
**优先级**: ⭐⭐⭐⭐ 高  
**影响范围**: 大

#### 问题背景

用户已明确指出以下文件属于灰区，必须重点盯防：

- `H5/client/src/App.tsx`
- `H5/client/src/lib/api.ts`
- `H5/client/src/lib/route.ts`
- `H5/client/src/contexts/LoanContext.tsx`
- `H5/client/src/lib/auth-api.ts`
- 后端统一鉴权、session、jwt、filter、interceptor 相关文件

其中 `LoanContext.tsx` 已被 H5/API 线触达；联合登录若继续改同类文件，将直接引发冲突。

#### 最终决策

**选择**: 对灰区文件实行显式 owner 制。未经 owner 同意，另一条线不得修改。

#### 理由

1. 灰区文件冲突代价远高于普通页面或测试文件
2. 这些文件承载的是跨页面、跨链路的共享状态与认证逻辑
3. owner 机制比“写完再解决冲突”更便宜

#### Owner 结论

1. **H5/API owner**
   - `H5/client/src/contexts/LoanContext.tsx`
   - `purpose` 及借款 / 权益 / 还款闭环相关 FE / BE / DB / test
2. **联合登录 owner**
   - `H5/client/src/lib/auth-api.ts`
   - 后端统一 `jwt/session/filter/interceptor` 相关文件
   - `JointLogin*` / `Auth*` / `Xiaohua*` 主链
3. **默认冻结**
   - `H5/client/src/App.tsx`
   - `H5/client/src/lib/api.ts`
   - `H5/client/src/lib/route.ts`

#### 实施结果

- [x] 确认 `LoanContext.tsx` 明确归 H5/API 线
- [x] 明确联合登录若必须改公共鉴权链路，由联合登录 owner 承担
- [x] 在协调结论中固定冻结文件清单
- [x] 截至当日收口时，未发现两边同时改同一灰区文件的实际冲突

### DECISION-066: 合并顺序固定为 `feature/h5-api` → `feature/joint-login`，每条线完成后先验证再合并

**决策日期**: 2026-04-27  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 集成策略 / 收口节奏  
**优先级**: ⭐⭐⭐ 高  
**影响范围**: 中

#### 问题背景

两条线并行开发时，即使没有文件冲突，也可能在集成阶段出现：

1. i18n 词条交叉
2. H5 公共 API 行为耦合
3. 联合登录入口对后续业务页面的跳转影响

因此必须提前固定合并顺序，而不是等两边都完成后临时决定。

#### 最终决策

**选择**: 先合 `feature/h5-api`，再让 `feature/joint-login` 基于最新 `main` 做一次收口验证，最后再合。

#### 理由

1. H5/API 线包含更多借款链路实体与页面状态变更，先落主线更稳
2. 联合登录线更聚焦在入口承接与认证链路，后合并更容易对齐最新主线
3. 先验证、再 merge，能避免“两个都觉得完成，但主线一起过不了”

#### 实施结果

- [x] 固定合并顺序为 `feature/h5-api` → `feature/joint-login`
- [x] 明确每个子任务完成后先做收口验证
- [x] 明确协调角色只负责验证、审查、最小冲突修复，不接管主实现
