### DECISION-083: `EX-C.5` 重复 confirm 的真实业务 bug 已闭合，口径应更新为“复用本地已生效协议”

**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 签约幂等 / 状态一致性  
**优先级**: ⭐⭐⭐⭐ 高  
**影响范围**: 中

#### 问题背景

之前的真实问题是：

- 同一笔 `sign-confirm` 第二次提交会再次打齐为
- 还会返回新的 `agreementNo`
- 但本地只保留一条 `ACTIVE` 协议

这会导致接口返回值和本地实际状态不一致。

#### 最终决策

**选择**: 将重复 confirm 的行为收口为：

- 第二次直接复用本地已生效协议
- 不再再次出站 QW confirmSign
- 不再返回新的 `agreementNo`

#### 理由

1. 这能直接修掉真实的不一致问题。
2. 这是当前联调用得上的最小幂等收口。
3. 比继续争论并发强幂等更符合当前节奏。

#### 实施结果

- [x] `EX-C.5` 已回归通过
- [x] 本地 `ACTIVE QW_SIGN` 仍只有一条
- [x] 覆盖矩阵已更新成最新真值

### DECISION-084: `EX-F.3` 已从“吞坏数据”修成“先校验，再映射”，后续不再保留为活跃问题

**决策日期**: 2026-05-02  
**决策人**: 技术负责人 + 开发负责人  
**所属分类**: 查询链路 / 错误收口  
**优先级**: ⭐⭐⭐⭐ 高  
**影响范围**: 中

#### 问题背景

之前 `loan/query` 对坏上游数据的处理是错误的：

- `data=null` 还能被映射成 `reviewing`
- 缺 `status` 还能被映射成 `reviewing`
- 缺 `loanId` 甚至还能返回 `approved + repaymentPlan`

这属于把坏报文当正常状态吞掉了。

#### 最终决策

**选择**: 改成先校验、再映射：

- `data=null` -> `YUNKA_RESPONSE_EMPTY`
- 缺 `status` / 缺 `loanId` -> `YUNKA_RESPONSE_INVALID`
- 正常 happy path 保持不变

#### 理由

1. 坏数据必须走受控异常，不能继续翻译成正常业务态。
2. 这个收口能让审批结果链路的口径更稳定。
3. 正常路径不应被这类修复误伤。

#### 实施结果

- [x] `EX-F.3` 已回归通过
- [x] 正常 `happy path` 没有被带坏
- [x] 当前不再存在活跃的 `EX-F.3` 问题
