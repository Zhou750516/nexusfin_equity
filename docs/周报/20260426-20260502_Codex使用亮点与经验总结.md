## 二、本周 AI Coding 在项目开发中的核心思考

### 1. 当项目进入联调阶段后，AI 的价值会从“继续写功能”转向“持续区分真问题和假阻塞”

本周一个很明显的变化是，Codex 的高价值不再主要来自“补更多功能”，而是来自：

1. 先判断一个失败到底是业务 bug、stub 缺口、外部阻塞，还是边界定义不清；
2. 再决定是让开发会话修代码，还是让测试会话继续跑，还是先停；
3. 最后把结论同步回覆盖矩阵和日报口径。

这周反复出现的情况就是：

1. `EX-A.3` 一开始看起来像登录异常没闭环，最后发现是 stub 缺 tamper 语义；
2. `EX-D.4` 一开始是审批拒绝造不出来，最后发现是本地 `loan/query -> rejected` fault 不足；
3. `EX-F.3` 一开始也是 `BLOCKED`，但在补完 stub 能力后，被确认成后端真实吞坏数据的问题；
4. `EX-C.5` 则不是 stub 问题，而是签约确认的真实业务幂等缺陷。

这说明在联调期，AI 最重要的价值之一，不是“直接给出更多实现”，而是帮助团队迅速分辨：**现在到底该修代码，还是该补测试能力，还是该把责任推给外部。**

### 2. AI Coding 很适合做“边界收紧器”，防止测试范围越跑越散

本周项目最大的管理价值之一，是把边界真正收紧了：

1. 艾博生不直接感知科技平台服务；
2. 科技平台细节默认由云卡统一收口；
3. 除非明确存在 `token / 回调 / 跳转` 直连边界，否则不向科技平台索要接口级细节。

这个决策看起来是文档口径，其实直接影响了技术推进方式：

1. 测试会话不再默认去测科技平台内部 path / 域名 / 签名；
2. 覆盖矩阵里一批旧入口和旧 callback 线被明确打成 `N/A`；
3. 主联调范围从“可能测很多东西”收口成“只测艾博生直接边界”。

Codex 在这里的价值，不是替团队拍板业务规则，而是把已经隐含存在的边界 **压成明确、可执行、可写进测试任务的规则**。这能明显减少后面一周的无效扩测。

### 3. AI Coding 在多会话协作里，最有价值的是“节奏控制”

本周很典型的一点是，开发会话和测试会话不是并行乱跑，而是在 Codex 协调下形成了稳定节奏：

1. 测试会话先把 `PENDING` 和 `BLOCKED` 打出最小证据；
2. 主决策会话判断：这是业务 bug、stub 缺口还是外部阻塞；
3. 只有遇到真实 bug，才让开发会话按最小修复原则介入；
4. 修完后测试会话只做差量回归，不回头刷整张矩阵。

这周被这样收口的典型问题有：

1. `repayment submit timeout -> 500`
2. 超额还款
3. 重复还款
4. 重复 `sign-confirm`
5. `loan/query` 坏数据吞没

这说明当 AI 不只是“参与一个代码任务”，而是持续管理 **测试 -> 判断 -> 修复 -> 回归 -> 更新文档** 的节奏时，项目推进的确定性会更高。

### 4. AI Coding 的高价值，正在从“写代码”扩展到“让项目可被证明”

本周另一个明显变化是，Codex 不只是帮忙修问题，而是在持续逼近一个更高标准：

> 不是“我觉得已经好了”，而是“我能拿出请求、响应、日志、出站 path 和次数、覆盖矩阵状态来证明它已经好了”。

具体体现为：

1. `requestBodyJson / responseBodyJson / errorNo / errorMsg` 被当成联调排障基础设施，而不是顺手日志；
2. 所有关键 case 都要求请求、响应、关键日志、是否真实出站、实际 path 与次数；
3. 覆盖矩阵从“写个状态”升级成“状态 + 证据索引”；
4. 联调完成度总结从“感觉差不多”升级成“明确知道还有没有 `PENDING / BLOCKED`”。

这类工作很能体现 AI Coding 的另一个价值：它不仅能推动实现，还能推动团队把结果变成 **可复用的证据资产**。

## 三、本周最有价值的使用场景

### 1. `benefiturl` 主链接线与语义风险拆开管理

本周 `benefiturl` 这条线很能体现 Codex 的价值。

项目里一开始其实有两件容易混在一起的事：

1. `benefiturl` 有没有真的接到“权益订单信息同步”主链；
2. 当前这个 `benefiturl` 的真实语义，能不能被讲成通用“权益联登 URL”。

Codex 本周做的不是简单说“已完成”或“未完成”，而是把这两件事拆开：

1. 主链接线：已完成，已经能真实出站；
2. 字段语义：仍有风险，因为当前运行态来源仍是 `QW exercise redirect URL`。

这个拆分很重要，因为它避免了团队在对外联调前把“接线完成”误说成“语义已被三方完全确认”。

### 2. 联调 case 从“散点验证”升级成“矩阵化收口”

这周 Codex 协助把 `20260506` 联调方案真正升级成了三层结构：

1. `联调结果`：记录每轮真实请求、响应、日志和 bug；
2. `覆盖矩阵`：记录每个 case 当前状态和证据索引；
3. `联调完成度总结`：给后续会话一个一眼能看懂的总体真值入口。

这使得项目不再只是“今天跑了什么”，而是变成：

1. 哪些 case 已有运行态证据；
2. 哪些 case 是 `N/A`；
3. 哪些问题已经修复；
4. 当前到底还有没有活跃 `PENDING / BLOCKED`。

这对多天、多会话协作非常关键。

### 3. 还款线与签约线的真问题闭环

本周 Codex 参与的最有代表性的“问题闭环”场景，是还款线和签约线。

还款线先后暴露出：

1. timeout 落通用 `500`
2. 超额还款未拦截
3. 重复还款未防重

签约线则暴露出：

1. 重复 `sign-confirm` 返回两个不同 `agreementNo`

这些问题如果没有统一节奏，很容易演变成：

1. 测试不断补更多失败样本；
2. 开发顺手改更多链路；
3. 文档状态越来越混乱。

Codex 本周的高价值，在于把这些问题都收口成：

1. 先测试确认是真问题；
2. 再最小修复；
3. 再只做差量回归；
4. 最后更新矩阵与总结。

这样项目不会在一个问题上越陷越深。

### 4. 文档体系第一次真正形成“真值入口”

本周持续补齐的 `日报 / decision-review / plan / checklist / 联调结果 / 覆盖矩阵 / 完成度总结`，不是简单堆文档，而是第一次让这套项目材料形成了真正的“真值入口”。

体现在：

1. 每天的判断和第二天的动作能接上；
2. 历史问题能知道是 bug、stub 问题还是外部阻塞；
3. 今天收口后的矩阵，不会再被旧日报误导；
4. 后续外部联调能直接拿完成度总结做对外口径基础。

这类治理型工作，非常适合交给 AI 持续参与，因为它比单纯写一段代码更需要连续上下文。

## 五、本周总结出的经验

### 经验一：联调期 AI 最重要的能力不是“继续写”，而是“先分清楚这是不是该修的真问题”

本周证明了一点：当项目进入联调期，最贵的成本往往不是写代码慢，而是 **把假的阻塞当成真的，把环境问题当成业务问题，把边界问题当成代码问题**。

AI 在这时候最有价值的事情，是帮助团队先分清：

1. 这是代码 bug；
2. 这是 stub 缺口；
3. 这是外部阻塞；
4. 这是旧入口不该再测；
5. 这是需要写成 `N/A` 的边界项。

如果这一步做错，后面的开发、测试和外部沟通都会越来越乱。

### 经验二：覆盖矩阵不是为了好看，而是为了防止同一个问题被不同人重复讲成不同状态

本周 `覆盖矩阵 + 联调结果 + 完成度总结` 的组合很有代表性。  
如果没有这套组合，很容易出现：

1. 测试会话说这个还没过；
2. 开发会话说这个其实已经修了；
3. 主决策会话又拿着前一天的旧状态做判断。

有了矩阵后，至少能做到：

1. 当前是不是 `COVERED / PENDING / BLOCKED / N/A` 一眼可见；
2. 证据在哪一轮日志里也能追；
3. 修完以后能把旧 FAIL 描述及时修正。

这类文档治理本身，就是 AI 很适合长期承担的工作。

### 经验三：AI 最适合推动“最小修复”，不适合在联调期顺手扩成更大的架构升级

这周几个问题的修复都说明，联调期更需要的是：

1. 最小修复
2. 差量回归
3. 清楚写明残余风险

例如：

1. 还款防重先做短窗口防重，而不是立刻重做完整幂等体系；
2. 重复 `sign-confirm` 先复用本地已生效协议，而不是直接冲向强并发级幂等；
3. `loan/query` 先把坏数据改成受控错误，而不是顺手做全仓同类逻辑普查。

AI 在这种“以当前问题为中心”的收口节奏里，表现会比“大而全改造”更稳。

### 经验四：一旦边界收紧，AI 的推进效率会明显上升

本周从“默认还会去问科技平台”切到“默认只看艾博生直接边界”之后，推进速度明显更顺。

因为一旦边界清楚了：

1. 测什么不再反复讨论；
2. `N/A` 项能更果断地打掉；
3. 外部问题接收方也更明确；
4. 开发和测试会话不会在错误范围里空转。

这说明 AI Coding 在复杂项目里的一个关键前提，就是先把边界讲清楚，否则再强的执行力也会被浪费在错误范围内。

### 经验五：AI 的价值已经扩展到“让项目结果可复述”

本周结束时，团队已经不只是“做完了一轮本地联调”，而是能够清楚地说出：

1. 本轮测了什么；
2. 哪些是 `N/A`；
3. 哪些问题真修了；
4. 当前还有没有活跃 `PENDING / BLOCKED`；
5. 后续应该转向什么。

这种“结果可复述”的能力，对于对外联调、群里同步、领导汇报都非常重要。  
AI 在这里的作用，已经不只是研发提效，而是直接帮助团队提高项目透明度和汇报质量。

## 六、当前使用 Codex 仍需注意的问题

| 问题 | 说明 |
| --- | --- |
| 外部联调材料仍不能由 AI 代替拍板 | `benefiturl` 字段语义、云卡正式口径、齐为联调环境等仍要靠外部确认，AI 只能先整理本地真值和问题归属。 |
| 当前本地全绿不等于外部联调无风险 | 本周收口的是本地主联调范围，不代表云卡、齐为、商务环境等外部条件已经全部就绪。 |
| 最小修复不等于长期终态 | 重复还款、重复签约等当前已达到联调可用，但还不是并发级终态方案。 |
| 文档真值需要持续维护 | 如果后续不继续同步日报、decision-review、覆盖矩阵和完成度总结，旧 FAIL 口径仍可能重新污染判断。 |
| `N/A` 项需要持续守住边界 | 已经收紧为 `N/A` 的旧入口和旧 callback 线，后续不能又被当成本轮待办拉回来。 |

## 七、下周建议

1. 继续让 Codex 承担“对外联调输入整理助手”角色，优先整理：
   - 当前主范围已闭合的本地真值
   - 云卡 / 齐为方向的最小问题归属
   - `benefiturl` 当前接线状态与语义风险说明
2. 继续让 Codex 承担“差量验证协调助手”角色，后续只有在外部反馈出现时，再按单点差量方式启用开发 / 测试会话。
3. 对当前已闭合的问题继续维持：
   - 不回头重刷整张矩阵
   - 不把 `N/A` 拉回主联调范围
   - 不把最小修复误说成终态架构
4. 对日报、decision-review、计划、checklist 和联调总结继续保持连续沉淀，确保对外联调时任何结论都能追到来源。

## 八、一句话总结

本周 Codex 在惠聚项目中的最大价值，已经从“帮助团队细化方案和重构代码”进一步升级为 **帮助团队在联调阶段收紧边界、识别真问题、驱动最小修复、沉淀可证明的运行态证据，并最终把整轮联调收口成一套可直接对外复述的真值体系**。
