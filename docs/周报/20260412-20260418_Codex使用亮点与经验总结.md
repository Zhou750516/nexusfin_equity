## 二、本周 AI Coding 在项目开发中的核心思考

### 1. AI Coding 已经可以承担“从计划到验证”的连续推进工作

本周最明显的变化是，Codex 不再只是单点参与某一类工作，而是贯穿了完整交付链路：

1. 先根据已确认决策整理技术方案和接口文档。
2. 再分析现有代码与目标接口的差异。
3. 再拆分任务组并确定实施顺序。
4. 再进入后端、H5、多语言、联调、补偿机制等具体实现。
5. 最后通过测试、构建、MySQL 回归和文档沉淀完成收口。

这说明在项目上下文足够完整、任务边界持续维护的情况下，AI Coding 可以从“辅助执行工具”升级为“连续交付助手”。它的价值不只在每一步更快，而在于每一步之间能保持一致的事实链条。

### 2. AI Coding 特别适合处理“多系统串联 + 多角色协作”的复杂项目

本周项目涉及多个系统和角色：

1. 艾博生 H5 与后端。
2. 云卡 gateway。
3. 小花科技接口与联合登录。
4. 齐为权益供应商。
5. 本地 MySQL。
6. 科技平台跳转、状态回推与消息提醒。

这类项目的难点不是某一个接口本身，而是每个系统的边界和职责都在变化。Codex 在本周体现出的优势是：

1. 能把“谁提供页面、谁提供接口、谁只做 gateway、谁负责状态回推”整理成稳定文档。
2. 能把外部未确认事项从开发任务中剥离出来，避免抢跑。
3. 能把本侧可做的内容拆成骨架、占位层、联调检查清单。
4. 能在真实协议未到位时，先把替换点和验证路径准备好。

因此，AI Coding 在多方协作项目中的关键价值，是帮助团队把“可推进项”和“待确认项”持续分开。

### 3. AI Coding 的高价值不只是生成代码，而是减少返工

本周大量工作都体现了“先判断、再执行”的节奏：

1. `Phase 9` 不是直接开写 10 个接口，而是先做现有实现差异分析。
2. H5 多语言不是零散替换文案，而是先确定统一文案资源中心方案。
3. 小花联合登录不是在文档缺失时伪造真实协议，而是先落本地骨架和真实协议适配占位清单。
4. 齐为绑卡不是抢跑接口开发，而是维护 TODO 和设计入口。
5. 退款能力不是伪造真实齐为协议，而是先补 `RefundClient` 替换点。
6. 异步补偿不是一开始扩到所有链路，而是先覆盖权益购买和用信提交两条主链路。

这类处理方式的本质，是让 AI 帮团队控制范围、明确边界、降低返工概率。代码产出只是结果之一，更重要的是执行顺序变得更稳。

### 4. AI Coding 在联调阶段的价值，主要体现在“快速定位假异常”

本周 H5 页面联调中出现过多个容易误判的问题：

1. 后端连接了 MySQL，但云卡仍处于默认 `MOCK` 模式。
2. H2 重启导致本地协议态和映射态丢失。
3. 多语言切换后静态文案刷新了，但动态接口字段没有重拉。
4. `loan_failed` 场景后端仍返回 `applicationId`，导致前端误查结果接口。
5. `approval-result.tip` 被上游通用中文 `remark` 覆盖，导致英文 / 越南文下仍展示中文。

这些问题如果只从页面现象判断，很容易误以为是业务链路失败。Codex 在联调阶段的价值，是能快速把现象拆成：

1. 环境配置问题。
2. 本地联调数据问题。
3. 前后端契约问题。
4. 后端国际化字段问题。
5. 外部依赖暂不可用问题。

这使问题定位从“猜原因”变成“按证据链排查”。

## 三、本周最有价值的使用场景

### 1. Phase 9 后端接口从准备态到完成态

本周 Codex 先完成 `Phase 9` 决策分析、接口对齐、现有实现差异分析，再将任务拆为 A/B/C/D/E 五组推进：

1. 任务组 A：本地配置与权益相关接口。
2. 任务组 B：云卡 gateway 基础设施。
3. 任务组 C：借款试算与审批查询。
4. 任务组 D：还款链路。
5. 任务组 E：借款申请编排。

最终形成了 H5 所需 10 个后端接口的主体实现，并完成阶段性集成测试和 checkstyle 验证。

### 2. H5 多语言从方案设计到端到端修复

本周多语言支持不是简单页面替换，而是覆盖：

1. 浏览器语言自动匹配。
2. 手动切换与持久化。
3. H5 静态文案。
4. 后端接口展示字段。
5. 协议标题、提示语、长文案和富文本类内容。
6. `Accept-Language` / `Content-Language` 请求响应口径。
7. 语言切换后的动态接口重拉。

Codex 不仅参与了方案设计，也参与了代码改造、类型检查、构建验证和浏览器级复核记录。

### 3. H5 页面维度联调测试与资料沉淀

本周 Codex 支撑完成了 H5 页面维度全流程联调资料建设：

1. 本地联调资料索引。
2. 本地联调启动与检查清单。
3. 页面维度 MySQL 全流程测试报告。
4. 人工执行步骤与关键检查项。
5. `test/README.md` 和资料索引入口补齐。

这些材料让联调不再依赖个人记忆，而是具备可复用、可交接的执行路径。

### 4. 小花联合登录、分发页、退款页、状态回推的方案骨架

在小花正式文档尚未完全到位的情况下，Codex 协助完成：

1. 小花联合登录方案骨架。
2. 联合登录核心入口实施计划。
3. 联合登录真实协议适配占位清单。
4. `/joint-entry`、`/joint-dispatch`、`/joint-refund-entry` 第一轮本地骨架。
5. 分发页 / 退款页 / 状态回推方案骨架。
6. 分发页 / 退款页 / 状态回推实施计划。
7. 三块能力联调检查清单。

这类工作最大的价值，是在外部文档未完全到位前，仍然让本侧有清晰的承接入口和后续替换点。

### 5. 异步消息补偿机制从业务决策到 MySQL 回归

本周围绕“同步调用超时或报错后如何处理”的业务问题，Codex 协助形成并落地方案 A：

1. 同步接口先给前端稳定承接口径。
2. 后端持久化异步补偿任务。
3. worker 按分片处理补偿任务。
4. supervisor 监控心跳、租约、积压和死任务。
5. 补偿执行器覆盖权益购买和用信提交。
6. MySQL 集成测试验证任务入队、执行、尝试日志、心跳、租约回收和死任务扫描。

这说明 AI Coding 可以参与的不只是普通 CRUD，也可以参与具备运行保障性质的可靠性机制建设。

## 五、本周总结出的经验

### 经验一：复杂项目中，AI 最好先做“拆解器”，再做“执行器”

本周 `Phase 9` 的推进证明，直接让 AI 写完整功能并不是最佳方式。更稳妥的顺序是：

1. 先让 AI 梳理现有实现。
2. 再让 AI 对齐目标接口。
3. 再拆任务组和执行顺序。
4. 最后进入实现和验证。

这种方式能显著降低“代码写出来了，但方向不对”的风险。

### 经验二：AI Coding 要和测试、构建、联调证据绑定

本周多次验证了一个原则：AI 输出不能只看“看起来合理”，必须绑定验证证据。例如：

1. 后端接口要有集成测试。
2. H5 逻辑要有类型检查和构建。
3. 页面问题要有浏览器复核。
4. 数据落库要有 MySQL 核验。
5. 补偿机制要有 MySQL 集成测试。

当 AI 生成的代码和文档都能回到验证证据上，可信度会明显提高。

### 经验三：外部文档未到位时，最好的 AI 输出不是“完整实现”，而是“替换点”

本周小花、齐为、科技平台都有文档未完全到位的情况。实践表明，此时 Codex 最适合输出：

1. 本地骨架。
2. 占位客户端。
3. 配置边界。
4. 字段映射待确认项。
5. 联调检查清单。
6. 后续替换点说明。

这比直接伪造真实协议更安全，也更利于后续接正式文档。

### 经验四：AI 很适合维护“项目事实链条”

本周每天都有大量决策、实现、测试和风险变化。如果只靠口头同步，很容易丢失上下文。Codex 持续维护：

1. 当日计划。
2. checklist。
3. 日报。
4. decision-review。
5. 明日计划。
6. 周报。

这让项目事实链条持续可追踪，也让“为什么这么做”有据可查。

### 经验五：AI 的高价值场景正在从“写功能”扩展到“设计可运营机制”

异步消息补偿机制是本周很重要的信号。它不是简单页面或接口开发，而是涉及：

1. 任务状态流转。
2. 分片与 worker。
3. supervisor 监控。
4. 重试退避。
5. 幂等保护。
6. 死任务识别。
7. MySQL 回归。

这说明当需求边界清楚、验证目标明确时，AI Coding 可以协助建设更偏运行保障和可靠性的工程能力。

## 六、当前使用 Codex 仍需注意的问题

| 问题 | 说明 |
| --- | --- |
| 外部协议不能由 AI 猜测 | 小花联合登录、齐为退款、齐为绑卡、科技平台状态回推等真实协议仍必须以对方文档为准，Codex 只能先做占位和待确认清单。 |

| AI 需要持续校验上下文一致性 | 多天连续推进时，必须让 Codex 回看计划、日报、checklist 和最新决策，避免沿用旧口径。 |
| 对外材料仍需人工最终把关 | 周报、对外确认清单、接口文档等正式材料仍需人工确认措辞和责任边界。 |

## 七、下周建议

1. 继续让 Codex 承担“真实协议到位后的快速适配助手”角色，优先用于：
   - 小花联合登录正式协议字段映射
   - 齐为绑卡接口设计
   - 齐为退款协议替换点实现
   - 科技平台权益状态回推协议接入
2. 对异步消息补偿机制，继续让 Codex 补齐：
   - 运维承接说明
   - 人工介入手册
   - 控制器级返回口径核对
   - 当前运行边界说明
3. 对真实联调继续采用“资料索引 + 启动清单 + 测试报告 + decision-review”的方式，保证每轮问题都有证据链。
4. 对所有外部待确认项继续维持三类标签：
   - 已确认
   - 待确认
   - 暂不进入
   防止范围继续膨胀。
5. 在后续真实环境联调中，继续让 Codex 辅助区分：
   - 环境问题
   - 数据问题
   - 契约问题
   - 本侧代码问题
   - 外部依赖问题

## 八、一句话结论

本周实践表明，Codex 在惠聚项目中的价值，已经从“边界收口工具”进一步升级为 **端到端交付闭环助手**：既能辅助方案和文档，也能推进代码实现、联调排障、测试验证和决策沉淀，并在多方外部依赖场景下持续降低返工风险。
