## 二、本周 AI Coding 在项目开发中的核心思考

### 1. AI Coding 在“项目进入复杂调整期”时，价值会从写代码转向重排方向

本周最明显的变化是，Codex 的价值不再主要体现为“继续补功能”，而更多体现为：

1. 把本周已经做完的技术事实重新整理出来；
2. 判断哪些结论已经稳定、哪些只是阶段性假设；
3. 判断旧任务哪些还能继续做、哪些必须停下来改写；
4. 把新的推进顺序重新写清楚；
5. 把这些判断同步到日报、decision-review、周报和明日计划中。

这说明在项目进入复杂调整期后，AI Coding 的高价值并不只是继续加快实现，而是帮助团队及时从“埋头推进”切换到“先把方向固定”。如果没有这一步，后面写得越快，返工反而可能越大。

### 2. AI Coding 很适合做“阶段性技术真相提炼”

本周项目里有一个非常典型的场景：

1. 本地 H5、借款、权益、还款、回调骨架其实已经有了；
2. 日志专项和异步补偿也已经形成一轮收口；
3. 但项目并不能因此直接判断为“下一步就是继续按旧任务写完”。

Codex 在本周真正帮团队做的，是把“当前到底已经具备什么”“到底还缺什么”“到底为什么现在不能直接开改”解释清楚。比如：

1. 当前仓库不是从零开始缺能力，而是第一版主链已跑通；
2. 真正缺的是 Shared Foundation、typed DTO、统一 facade、动态协议 / 绑卡 / 回调模型等对齐项；
3. 真正的主要阻塞也不是本地完全不会做，而是云卡状态机、小花 `4.22` 细节、齐为文档、法大大资料和商务审核。

这种“提炼当前真实状态”的能力，在复杂项目里比单纯写一批代码更重要。

### 3. AI Coding 的高价值，正在从“补实现”扩展到“减少错误推进”

本周差异分析和任务重排证明了一点：AI 的价值不只是更快写出代码，而是更快发现 **哪些代码现在不该写**。

具体体现为：

1. 不是看到 `004` 还有任务就直接做，而是先判断任务前提是否已经过期；
2. 不是看到新主链明确了就直接大改 `/api/loan/apply`，而是先确认云卡状态机和状态查询接口还没定版；
3. 不是看到 H5 自动埋点要做就直接开改页面，而是先要求把页面、事件、公参、隐私边界先写清楚；
4. 不是看到补偿机制做出来就默认开，而是先把“默认关闭、显式开启、显式分片、人工介入”这些运行边界讲明白。

这类动作的本质，是让 AI 帮团队减少“看似在推进、实则在跑偏”的风险。

### 4. AI Coding 在多方依赖项目里的价值，越来越像“边界管理员”

本周项目继续涉及：

1. 艾博生后端与 H5；
2. 云卡 gateway 和状态机；
3. 小花 `4.22` 文档和联合登录；
4. 齐为绑卡、短信签约；
5. 法大大电子签章；
6. 商务合同与审核流程。

这些问题的难点已经不只是接口本身，而是谁先做、谁后做、谁还没拍板、谁只是本侧预留。

Codex 在这里的优势，不是替代这些外部确认，而是持续把它们分层：

1. 已确认的结论；
2. 本侧可以先做的事项；
3. 必须等待外部接口定稿的事项；
4. 当前绝对不该抢跑的事项。

在多方协作项目里，这种“边界管理员”角色非常重要。因为很多返工不是技术不会做，而是把不该先做的事情做早了。

## 三、本周最有价值的使用场景

### 1. 异步补偿自动调度从技术能力走向运维可承接

本周 Codex 参与的不只是补偿代码本身，而是进一步把补偿机制补成“可用体系”：

1. 明确 `workerId` 默认生成与显式配置建议；
2. 明确 `IDLE / RUNNING / currentTaskId` 的观察语义；
3. 补齐启用说明、显式分片策略和示例配置；
4. 补齐启用后观察点；
5. 补齐 backlog 不下降、失败增长、`DEAD` 增长时的人工排查顺序；
6. 补齐人工重试、补单、关闭建议。

这说明 Codex 的价值已经从“帮忙把 worker 写出来”继续扩展到“帮忙把这个能力变成团队可接手的运行方案”。

### 2. 阶段性快照冻结与日志闭环补齐

本周 Codex 先协助完成：

1. 整体快照提交；
2. 完整后端回归验证；
3. 后续增量开发的安全基线建立。

然后在这个基线上继续做：

1. 异步补偿入队日志补齐；
2. worker 执行链路日志补齐；
3. `errorNo / errorMsg` 统一；
4. `WARN / ERROR` 分级；
5. 日志专项文档和排障清单回填。

这类工作很能体现 AI Coding 的价值：不是“要么全是文档，要么全是代码”，而是在同一轮推进里把基线、验证、代码、日志、文档串成闭环。

### 3. 小花 `4.22` 差异分析与旧任务体系纠偏

本周一个很有代表性的使用场景，是 Codex 协助回答了一个非常关键的问题：

> 现在项目到底应该继续做 `004-benefit-loan-flow` 的剩余任务，还是应该先整体重排？

Codex 通过差异分析明确指出：

1. 当前调用方向、回调方向、接口骨架并不是空白；
2. 但路径配置、typed DTO、动态协议、userCards、状态映射、还款短信确认、标准回调模型等与小花 `4.22` 仍有系统性缺口；
3. `004` 中 `T013-T033` 的任务前提已经和 `2026-04-23` 新链路不一致。

最终团队不再把旧任务直接当主清单，而是先重排任务，再明确 Shared Foundation 作为新的阻塞前置。这类“纠偏式使用”很有价值，因为它直接避免了错误方向上的继续投入。

### 4. 周报、日报、决策审查形成连续治理材料

本周 Codex 还承担了大量“看起来不 flashy、但非常重要”的工作：

1. 补齐缺失的 `2026-04-24` 日报和决策审查；
2. 完成 `2026-04-25` 日报、决策审查与技术周报；
3. 把本周的技术结论、外部阻塞、推进顺序统一成可对内对外复用的口径；
4. 继续生成下一日计划。

这类工作的价值在于：项目不再只靠“今天记得发生了什么”，而是把“为什么这么判断、后续怎么做、哪些还不能做”都留在文档里。

## 五、本周总结出的经验

### 经验一：项目进入调整期后，AI 最该先做的是“重排方向”，不是“继续冲实现”

本周很典型的一点是：如果只看表面，还有很多事情可以继续做；但如果先把事实拉平，就会发现很多事情必须先重排。

所以在复杂项目进入中后期后，正确顺序往往是：

1. 先回看当前真实状态；
2. 再判断哪些结论已经过期；
3. 再重排任务顺序；
4. 最后才进入实现。

这一步如果做对，AI 可以显著减少返工。

### 经验二：AI Coding 很适合输出“阶段真相”，而不是只输出“阶段成果”

很多时候，团队需要的不只是“本周做了什么”，更需要“本周其实说明了什么”。

例如本周真正重要的结论是：

1. 本地能力并不是空白；
2. 但项目主阻塞已经转到外部接口和商务审核；
3. 旧任务体系已经和新链路方向不一致；
4. 下阶段最重要的是 Shared Foundation 和三类方案。

这种“阶段真相”的提炼，是 AI 非常适合持续做的事。

### 经验三：AI Coding 在外部依赖不稳定时，最好的输出是“替换点 + 边界说明”

本周再次证明，外部接口没定时，最好的 AI 输出不是伪造一个完整实现，而是：

1. 把待确认项列清楚；
2. 把本地可控项先拆清楚；
3. 把替换点、状态位、映射层、配置位准备好；
4. 把“现在不要做什么”写清楚。

这比直接把代码写死要安全得多。

### 经验四：AI 要持续绑定“管理文档”，否则多天推进很容易口径漂移

本周从 `2026-04-20` 到 `2026-04-25`，结论变化很多。如果不持续让 Codex 回看：

1. 日报；
2. decision-review；
3. plan / checklist；
4. 周报；

那很容易在不同天继续沿用旧判断。

因此，AI 在连续推进型项目中，必须始终和这些管理材料绑定，才能保持真正的上下文一致性。

### 经验五：AI 的高价值场景已经扩展到“项目治理”

本周 Codex 体现出的价值，已经不只是实现能力本身，而是包括：

1. 哪些事项需要被记成决策；
2. 哪些事项只适合先记成待确认；
3. 哪些事项需要进入周报对外同步；
4. 哪些事项需要落成明日计划；
5. 哪些旧任务应该被判定为过期。

这说明 AI Coding 在成熟用法下，已经不只是“研发提效工具”，而是在一定程度上成为“项目治理助手”。

## 六、当前使用 Codex 仍需注意的问题

| 问题 | 说明 |
| --- | --- |
| 外部接口不能由 AI 代替拍板 | 云卡状态机、小花 `4.22` 细节、齐为正式文档、法大大资料仍必须等外部确认，Codex 只能先做本侧边界和替换点。 |
| 总结材料不能替代真实落地 | 本周大量结论已经清晰，但 Shared Foundation、状态映射、埋点方案仍需继续往可执行层推进。 |
| 阶段结论要及时回填 | 如果日报、decision-review、plan 不连续，AI 很容易继续引用旧判断。 |
| 不能把“本地已具备基础能力”误解成“真实联调已完成” | 当前仍有不少能力停留在本地骨架、测试通过或方案预留阶段。 |
| 周报口径仍需人工把关 | 对外同步时，责任边界、阻塞判断和措辞仍需人工最终确认。 |
