### 2.2 本周关键实现点

#### 实现点一：把订单幂等从“异常补偿”前移到“请求语义层”

本周最重要的基础修正之一，是让订单创建在进入写库前就完成重复请求识别。这样做的直接收益是：

1. 同一请求不会产生重复业务副作用。
2. 可以通过 `requestId` 更准确地关联请求和订单。
3. 后续排障不再依赖人工猜测重试链路。

#### 实现点二：把回调审计状态从“结果态”调整为“阶段态”

本周对通知链路的调整重点不是改一个枚举值，而是让审计状态真正可用：

1. 收到消息不等于业务处理成功。
2. 缺单和异常场景必须落到失败态。
3. 只有业务真正完成后，通知日志才应标记为 `PROCESSED`。

#### 实现点三：把业务会员身份从“外部参数”切换为“认证上下文”

跨域认证改造后，`/api/equity/**` 不再依赖外部显式传入 `memberId`，而是统一从本地登录态识别当前会员。这是本周最关键的安全边界调整之一。

#### 实现点四：把浏览器态和回调态鉴权边界拆清楚

当前系统边界已经清晰分层：

1. `/api/equity/**` 与 `/api/users/me` 走 JWT + Cookie。
2. `/api/callbacks/**` 保持签名认证。

这使后续扩展前端能力或继续对接合作方接口时，不会混用同一套鉴权语义。

#### 实现点五：把齐为接入从“分析文档”推进到“协议适配层 + 业务接线”

本周确认的关键事实是，齐为接入已经不只是接口分析：

1. 已存在齐为 AES 报文加解密和外层协议封装。
2. 已实现 3 个主动调用接口。
3. 已经接入订单、联登和放款通知相关主链路。

这意味着系统重点已经从“本地闭环正确”进一步进入“本地下游协同正确”阶段。

### 2.3 本周遇到的问题与解决方式

#### 问题一：审查建议很多，但不能全部无差别落地

问题本质：

- 审查内容同时包含“当前必须修复的问题”和“可后续增强的建议”。

解决方式：

1. 先判断是否违反当前规格。
2. 再判断是否影响当前系统正确性和边界清晰度。
3. 本周只落地必须进入主链的问题，其余内容在文档中说明原因。

#### 问题二：工具默认流程与项目实际推进方式不完全一致

问题本质：

- 部分流程默认要求切分支或走脚本，但当前项目明确要求直接在 `main` 上推进。

解决方式：

1. 不把工具流程当成阻塞条件。
2. 直接以设计文档、任务文档和当前工程约束为准推进实现。
3. 用测试、打包、checkstyle、实库回归和文档同步补齐过程控制。

#### 问题三：真实 MySQL 回归首次暴露 schema 漂移

问题本质：

- H2 和编译通过，并不能说明复用本地数据库的场景一定通过。
- 本地存量数据库与当前代码 schema 存在偏差，`member_info` 缺少 `tech_platform_user_id`。

解决方式：

1. 根据失败 SQL 锁定根因。
2. 在集成测试中增加测试前置 schema 对齐逻辑。
3. 只做增量补齐，不重建本地数据库。
4. 修复后重新执行 MySQL 集成测试，确认主链可回归通过。

#### 问题四：任务清单状态和代码真实进度不同步

问题本质：

- `tasks.md` 的勾选状态滞后，不能直接代表当前工作区真实完成度。

解决方式：

1. 同时对照代码、测试、文档和 Git 状态判断事实。
2. 先核清真实进度，再做验收记录和后续收口。

#### 问题五：如果只改代码，不补文档，Review 会失真

问题本质：

- 本周多项改动属于语义型和边界型调整，单靠 diff 很难完整表达设计意图。

解决方式：

1. 同步更新修改说明和 quickstart。
2. 新增验收记录，明确验证项和验证结果。
3. 保留“负责人可读摘要 + 研发团队复盘”的双层表达。
