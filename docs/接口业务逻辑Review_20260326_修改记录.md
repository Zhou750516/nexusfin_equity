# 接口业务逻辑 Review 20260326 修改记录

## 1. 评估结论

针对 `docs/接口业务逻辑Review_20260326.md` 中的建议，本次处理结论如下：

1. `POST /api/users/register` 相关建议不直接采纳到接口层。
原因：当前业务主链已经切换为 `GET /api/auth/sso-callback` 驱动的 SSO 即时建档，`/api/users/register` 已下线，不应恢复旧接口。

2. 敏感信息安全处理建议部分采纳，并迁移到现行 SSO 建档链路。
采纳内容：手机号、身份证、姓名在登录建档时进行入站解码、格式校验、AES-GCM 加密存储和 hash 索引写入。

3. “根据 `channelCode` 提供解密方法”部分采纳为可配置扩展点。
采纳内容：上游用户资料支持可选 `channelCode`，并按渠道配置不同的入站解码模式；当前仅实现 `PLAIN` / `BASE64` 两种模式，避免在缺少上游密钥协商协议时伪造 AES 解密实现。

4. `GET /api/equity/products/{productCode}` 关于“需要考虑登录问题”的建议不新增代码。
原因：当前接口已经受 JWT 认证过滤器保护，Controller 也只从认证上下文读取 `memberId`。

5. `POST /api/equity/orders` 关于“状态字段合并为一个 `order_status`”的建议不采纳。
原因：当前 `benefit_order` 需要同时表达订单主状态、首扣、兜底代扣、行权、退款、放款等多个维度；强行合并会损害状态表达能力，并扩大回调、对账和测试修改面。

## 2. 本次代码修改

### 2.1 SSO 建档敏感数据处理

- 新增配置类 `nexusfin.crypto`，用于维护敏感数据 DEK 和入站解码模式。
- 新增 `SensitiveDataCipher` 组件，使用 `AES/GCM/NoPadding` 进行加密，密文格式为 `keyId:keyVersion:payload`。
- `AuthServiceImpl` 不再使用 Base64 伪加密，而是在建档/同步会员信息时：
  - 按 `channelCode` 对上游字段做入站解码
  - 校验手机号、身份证、姓名格式
  - 将手机号、身份证、姓名以 AES-GCM 加密后落库
  - 手机号和身份证仍保留 SHA-256 hash 作为检索索引

### 2.2 渠道能力补充

- `TechPlatformUserProfileResponse` 新增 `channelCode` 字段。
- `TechPlatformUserClientImpl` 从上游用户资料中读取可选 `channelCode`。
- `AuthServiceImpl` 绑定会员渠道时优先使用上游返回的 `channelCode`，缺省时回退到系统默认渠道。

### 2.3 保持不变的部分

- `/api/equity/products/**`、`/api/equity/orders/**` 继续沿用本地 JWT 认证，不接受显式 `memberId`。
- `benefit_order` 保持主状态 + 子状态并存的数据结构，不做压扁式改造。
