## User Scenarios & Testing *(mandatory)*

### User Story 1 - 在艾博生完成权益开通前确认 (Priority: P1)

作为用户，我希望在艾博生权益页清楚看到权益内容、支付卡信息和协议确认入口，以便我决定是否继续开通权益并进入后续借款流程。

**Why this priority**: 这是整个流程的起点。用户在此页面无法完成信息确认，就不会进入借款试算和后续链路。

**Independent Test**: 只验证艾博生权益页即可独立完成测试，确认页面在不依赖后续借款页的情况下，能够完整展示权益信息、规则、支付卡和协议勾选控制。

**Acceptance Scenarios**:

1. **Given** 用户已进入艾博生权益页，**When** 页面加载完成，**Then** 用户能够看到固定借款金额、权益信息、规则入口、支付卡信息和协议勾选区域。
2. **Given** 用户未勾选协议，**When** 页面展示操作按钮，**Then** “立即开通权益”按钮必须保持不可点击状态。
3. **Given** 用户已勾选协议，**When** 页面刷新按钮状态，**Then** “立即开通权益”按钮必须变为可点击并可进入下一步。

---

### User Story 2 - 在云卡确认借款并查看审核结果 (Priority: P2)

作为用户，我希望在云卡借款页确认固定借款金额、收款银行卡和还款计划，并在提交借款后持续看到审核状态，直到得到成功或失败结果。

**Why this priority**: 这是用户完成借款申请的核心闭环，也是三方协作最密集的页面流程。

**Independent Test**: 只验证云卡借款确认页和审核页，即可独立测试借款试算展示、确认借款操作和状态更新流程。

**Acceptance Scenarios**:

1. **Given** 用户进入云卡借款确认页，**When** 页面初始化完成，**Then** 页面必须展示固定借款金额、收款银行卡、协议内容和还款计划结果。
2. **Given** 用户点击“确认借款”，**When** 借款申请提交成功，**Then** 系统必须进入借款审核页面并显示处理中状态。
3. **Given** 借款审核页持续获取状态，**When** 返回成功状态，**Then** 用户必须被自动引导到权益生效页面。
4. **Given** 借款审核页持续获取状态，**When** 返回失败状态，**Then** 页面必须明确告知用户审核失败而不是停留在处理中。

---

### User Story 3 - 在艾博生完成还款确认 (Priority: P3)

作为用户，我希望从科技平台还款入口进入艾博生还款页，查看本期应还金额和支付方式，并在身份验证通过后完成还款。

**Why this priority**: 还款是借后关键场景，虽然优先级低于开通和借款，但必须在当前理解版本中保留完整路径。

**Independent Test**: 只验证艾博生还款页即可独立测试金额展示、支付方式展示、身份验证触发和确认还款动作。

**Acceptance Scenarios**:

1. **Given** 用户从科技平台入口进入艾博生还款页，**When** 页面加载完成，**Then** 用户必须能看到本期应还金额和默认还款银行卡信息。
2. **Given** 用户点击“确认还款”，**When** 身份验证尚未通过，**Then** 系统必须先触发身份验证而不能直接执行最终扣款。
3. **Given** 用户已完成身份验证，**When** 用户继续确认还款，**Then** 系统必须进入还款执行流程并向用户反馈处理结果。

---

### Edge Cases

- 当权益信息、规则说明或银行卡信息未加载成功时，页面必须给出清晰提示，不能静默空白。
- 当用户在艾博生权益页未勾选协议时，系统不得允许其进入后续开通和借款流程。
- 当云卡借款审核状态长时间未返回最终结果时，页面必须持续维持明确的处理中状态，而不是展示为成功或失败。
- 当还款身份验证失败、取消或超时时，系统必须阻止最终扣款并给出可理解的提示。
- 当结清证明责任方尚未确定时，该能力必须被标记为待补充定义，而不能默认归属某一方。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须支持一条从艾博生权益页开始、经云卡借款页、最终覆盖还款页的完整页面交互主链。
- **FR-002**: 艾博生权益页必须展示固定借款金额、权益信息、规则入口、支付卡信息和协议确认区。
- **FR-003**: 艾博生权益页必须要求用户勾选协议后，才允许继续进入后续流程。
- **FR-004**: 用户在艾博生权益页继续操作后，系统必须进入云卡借款确认流程，并向用户明确展示页面跳转中的处理中反馈。
- **FR-005**: 云卡借款确认页必须展示固定借款金额、还款计划、收款银行卡和协议内容，供用户在提交前完成确认。
- **FR-006**: 用户在云卡确认借款后，系统必须进入借款审核页，并持续向用户反馈审核状态直到产生最终结果。
- **FR-007**: 当借款审核成功时，系统必须将用户引导到权益生效页面；当借款审核失败时，系统必须向用户明确反馈失败结果。
- **FR-008**: 用户从科技平台还款入口进入艾博生还款页时，系统必须展示本期应还金额和默认还款银行卡信息。
- **FR-009**: 用户确认还款时，系统必须先完成身份验证，再进入最终还款执行。
- **FR-010**: 系统必须具备存储和展示电子签章合同的能力，以支持借款和权益相关合同查看。
- **FR-011**: 系统必须将结清证明能力标记为待明确范围，并在正式定稿前确认由哪一方提供。
- **FR-012**: 本规格必须把当前内容视为第一版页面交互理解，允许后续根据业务评审结果调整页面顺序、数据来源和交互细节。

### Key Entities *(include if feature involves data)*

- **权益首页视图**: 用户在艾博生看到的开通入口页面，包含固定借款金额、权益列表、规则说明、支付卡摘要和协议确认状态。
- **借款确认视图**: 用户在云卡确认借款的页面，包含固定借款金额、收款银行卡、协议内容和还款计划摘要。
- **借款审核状态**: 借款提交后用户持续看到的处理状态，至少需要覆盖处理中、成功和失败三类结果。
- **还款确认视图**: 用户在艾博生完成还款前确认的页面，包含本期应还金额、默认还款银行卡和身份验证触发点。
- **合同与证明资料**: 与权益和借款相关的电子签章合同及后续结清证明能力，其中结清证明当前仍处于待明确归属状态。
