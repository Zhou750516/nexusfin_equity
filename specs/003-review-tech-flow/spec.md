# Feature Specification: 科技平台接口-业务流程分析合理性评审

**Feature Branch**: `003-review-tech-flow`  
**Created**: 2026-03-30  
**Status**: Draft  
**Input**: User description: "分析科技平台接口-业务流程分析.pdf，形成科技平台标准接口与业务流程一致性评审规格"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 判断现有分析稿是否可直接用于联调 (Priority: P1)

作为业务负责人或技术负责人，我希望判断 `科技平台接口-业务流程分析.pdf` 是否可以直接作为联调和设计依据，以便我决定是否可以据此推进开发、联调和评审。

**Why this priority**: 这是最直接的决策问题。如果不能先判断现有分析稿是否可靠，后续时序设计、联调职责划分和接口实现都会建立在错误前提上。

**Independent Test**: 只阅读目标分析稿和两份源文档，独立输出“合理 / 部分合理 / 不合理”的明确结论，并给出支持结论的依据，即可验证本故事是否成立。

**Acceptance Scenarios**:

1. **Given** 存在目标分析稿和两份源文档，**When** 评审人员完成逐项对照，**Then** 文档必须给出目标分析稿是否可直接用于联调的明确结论。
2. **Given** 目标分析稿存在合理与不合理并存的情况，**When** 输出评审结论，**Then** 文档必须明确说明哪些判断可以保留、哪些判断需要修正。

---

### User Story 2 - 识别接口方向和流程闭环偏差 (Priority: P2)

作为联调负责人，我希望识别目标分析稿中关于接口调用方向、同步查询、异步通知和流程闭环的偏差，以便及时修正对接责任矩阵。

**Why this priority**: 接口方向看反或漏掉关键环节，会直接导致联调双方职责错位、接口部署位置错误和验收口径混乱。

**Independent Test**: 只检查接口清单和章节归属，即可独立验证目标分析稿是否正确区分了“平台调机构”和“机构调平台”，并确认关键查询与通知接口是否齐全。

**Acceptance Scenarios**:

1. **Given** 标准接口文档区分了“平台调机构”和“机构调平台”， **When** 评审目标分析稿，**Then** 文档必须明确指出每个关键接口的正确调用方向。
2. **Given** 借款和还款流程涉及状态查询、通知和跳转入口，**When** 评审目标分析稿，**Then** 文档必须指出缺失的关键接口和闭环缺口。

---

### User Story 3 - 输出可落地的修订建议 (Priority: P3)

作为产品、研发和联调三方的协同人员，我希望获得一份可直接用于修订现有分析稿的建议清单，以便快速形成统一版本。

**Why this priority**: 仅指出问题不够，必须给出更合理的接口映射和待确认问题，才能推动后续文档收敛。

**Independent Test**: 在不查看原始分析稿正文的情况下，只阅读输出规格，也能知道应该如何重写标题、如何修正接口映射、还需要哪些待确认问题。

**Acceptance Scenarios**:

1. **Given** 目标分析稿存在方向性错误，**When** 评审文档输出建议，**Then** 必须提供一版更合理的借款和还款接口映射。
2. **Given** 源文档之间存在视角差异， **When** 评审文档收尾，**Then** 必须列出需要业务、产品、技术三方补充确认的问题。

---

### Edge Cases

- 当目标分析稿中的业务表述与标准接口章节归属冲突时，评审必须明确区分“页面/产品视角”和“外部标准接口视角”，不能直接混用。
- 当某一业务能力在业务流程文档中存在，但标准接口文档没有现成定义时，评审必须标记为“待补充约定”，而不能强行归到已有接口。
- 当标准接口文档提供了查询接口和通知接口两种机制时，评审必须分别说明二者作用，不能只保留其中一种。
- 当目标分析稿遗漏了关键入口、协议、合同或结果回查环节时，评审必须指出该遗漏会导致的流程断点。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 规格必须以 `docs/科技平台接口-业务流程分析.pdf` 为被评审对象，并以 `docs/惠选卡权益与借款流程-20260330-V1.0.pdf` 和 `docs/科技平台导流对接标准接口文档.docx` 作为主要对照依据。
- **FR-002**: 规格必须给出目标分析稿的总体结论，并明确其是否可直接作为联调、设计或实现依据。
- **FR-003**: 规格必须逐项评估目标分析稿中对借款阶段、还款阶段和异步通知阶段的接口映射是否合理。
- **FR-004**: 规格必须明确区分“页面/产品流程视角”和“标准外部接口视角”，并说明两者不能直接等同。
- **FR-005**: 规格必须指出目标分析稿中所有影响联调职责划分的关键偏差，包括但不限于接口调用方向、同步查询与异步通知关系、流程闭环缺口。
- **FR-006**: 规格必须识别目标分析稿遗漏的关键能力，并明确这些能力在业务流程中的作用，例如状态查询、还款入口、协议展示、合同展示和结果回查。
- **FR-007**: 规格必须输出一版更合理的接口映射，至少覆盖借款主链和还款主链两个部分。
- **FR-008**: 规格必须保留目标分析稿中合理的判断，并说明其成立依据，避免把正确结论一并否定。
- **FR-009**: 规格必须形成待确认问题清单，聚焦会影响范围、职责边界或联调口径的事项。
- **FR-010**: 规格必须生成一份适合业务、产品、研发共同阅读的 Markdown 文档，结论、依据、偏差和建议都应清晰可追溯。

### Key Entities *(include if feature involves data)*

- **Source Document**: 参与评审的原始材料，包括业务流程文档、标准接口文档和被评审分析稿。关键属性包括文档名称、文档角色和在评审中的证据用途。
- **Interface Mapping Claim**: 被评审分析稿中对某个接口用途、调用方、被调方、流程环节的判断。关键属性包括接口路径、业务环节、当前结论和对照结果。
- **Rationality Finding**: 对单个判断或一组判断形成的评审结果。关键属性包括结论类型、证据来源、影响范围和修订建议。
- **Open Question**: 当前文档无法独立消除的业务或职责边界问题。关键属性包括问题主题、冲突来源、潜在影响和需确认角色。

### Assumptions

- 本次规格目标是“形成结构化评审与修订依据”，不是直接修改原 PDF 或输出新的成品 PDF。
- 本次规格默认输出介质为 Markdown 分析文档，供后续继续修订成评审稿、联调稿或 PPT。
- 本次规格以现有三份文档为唯一正式依据，不引入外部系统口头约定作为定论。
- 由于用户明确要求不切换分支，本次仅生成 `specs/003-review-tech-flow/` 目录，不执行 git 分支切换。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 评审输出能够在一页内给出目标分析稿“是否可直接用于联调”的明确判断，且判断不是模糊描述。
- **SC-002**: 评审输出至少识别并解释 5 处以上会影响联调职责划分或流程闭环的关键偏差。
- **SC-003**: 评审输出覆盖借款主链和还款主链两类流程，并为每类流程提供一版修订后的接口映射。
- **SC-004**: 评审输出能够列出至少 3 个需要业务、产品、研发三方统一确认的问题，用于推动后续文档收敛。
- **SC-005**: 业务、产品或研发人员在只阅读评审输出的情况下，能够明确知道现有分析稿哪些内容可以保留、哪些必须重写。
